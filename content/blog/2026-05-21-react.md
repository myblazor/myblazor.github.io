## The Moment It All Falls Apart

You have been building the new customer portal in React for three months. It is going well. Components are rendering. Data is loading. The dashboard page looks beautiful. Then somebody asks you to add a notification badge to the top navigation bar. The badge should show the number of unread messages. The count comes from the messages page, which is an entirely separate part of the component tree.

You open your code and realize: the navigation bar component is at the top of the component tree. The messages count lives in a service that the messages page uses. How do you get the count from deep inside the messages page up to the navigation bar?

This question — "how do I share state between components that are not in a parent-child relationship?" — is the state management problem. It is one of the oldest and most contentious problems in front-end development. The React ecosystem has solved it approximately seven times, with each solution eventually being found wanting, leading to the next one.

Blazor, meanwhile, has had a simple and complete answer to this question since it was first released: dependency injection. Register a service as a singleton, inject it into any component that needs it, and they all share the same instance. The navigation bar injects the message service. The messages page injects the message service. They see the same data. Done.

But that does not make React's journey through state management uninteresting. It is a fascinating archaeological dig through a decade of front-end engineering thinking, and understanding it helps you understand React's component model at a much deeper level.

## Part 1: The History — From Prop Drilling to Atomic State

### Era 1: Prop Drilling (2013–2015)

When React was first released, there was no state management library, no context, no hooks. There were class components. State lived in class instances. If a child component needed data from a parent, the parent passed it as a prop. If a grandchild needed data from a grandparent, the grandparent passed it to the parent, who passed it to the grandchild.

This is called "prop drilling" — passing data through intermediate components that don't themselves need the data, just to get it to a component further down the tree.

```jsx
// The grandparent has the data
class App extends React.Component {
    state = { user: { name: 'Alice', unreadMessages: 5 } };
    
    render() {
        return <Layout user={this.state.user} />;
    }
}

// The intermediate component doesn't need 'user' — it just passes it along
function Layout({ user }) {
    return (
        <div>
            <NavBar user={user} />  {/* ← drilling through Layout */}
            <MainContent />
        </div>
    );
}

// Finally, the component that actually needs it
function NavBar({ user }) {
    return <div>Messages: {user.unreadMessages}</div>;
}
```

Prop drilling is simple and explicit — you can always trace exactly where data comes from by following the props. But it becomes unmanageable for deeply nested trees or data that is needed in many different parts of the tree. Adding a new piece of data requires updating every intermediate component in the chain, even if those components don't use the data.

### Era 2: Flux and Redux (2014–2019)

In 2014, Facebook released Flux — an architectural pattern (not a library) that introduced the concept of a unidirectional data flow:

```
Action → Dispatcher → Store → View
         ↑                      |
         └──────────────────────┘ (user interactions create actions)
```

The basic idea: all application state lives in a central "store." Components read state from the store. When state needs to change, a component dispatches an "action" (a plain JavaScript object describing what happened). The store receives the action and produces a new state.

Redux, created by Dan Abramov in 2015, is the most successful implementation of this pattern. It became enormously popular because it solved the prop drilling problem — any component could connect to the Redux store and read whatever state it needed, without prop drilling. It also made state changes traceable — every state change was a dispatched action that was logged and could be replayed.

Here is what Redux looks like for our messages counter:

```javascript
// Redux: Define the reducer (state + action → new state)
function messagesReducer(state = { unread: 0 }, action) {
    switch (action.type) {
        case 'messages/loaded':
            return { ...state, unread: action.payload.unreadCount };
        case 'messages/markRead':
            return { ...state, unread: Math.max(0, state.unread - 1) };
        default:
            return state;
    }
}

// Create the store
const store = Redux.createStore(messagesReducer);

// A component reads from the store (using react-redux hooks)
function NavBar() {
    const unread = useSelector(state => state.unread);
    return <div>Messages: {unread}</div>;
}

// Another component dispatches an action
function MessageList() {
    const dispatch = useDispatch();
    
    const markAsRead = (messageId) => {
        dispatch({ type: 'messages/markRead', payload: { messageId } });
    };
    
    return <div>...</div>;
}
```

Redux solved prop drilling but introduced significant boilerplate: actions, reducers, action creators, and later "sagas" or "thunks" for async operations. A simple feature could require five files and hundreds of lines of boilerplate code. This boilerplate was one of Redux's biggest criticisms.

Redux Toolkit (RTK), released in 2019, drastically reduced this boilerplate with `createSlice`, `createAsyncThunk`, and other utilities. RTK is now the recommended way to use Redux:

```javascript
// Redux Toolkit: much less boilerplate
import { createSlice, createAsyncThunk } from '@reduxjs/toolkit';

const fetchMessages = createAsyncThunk('messages/fetch', async (userId) => {
    const response = await fetch(`/api/users/${userId}/messages`);
    return response.json();
});

const messagesSlice = createSlice({
    name: 'messages',
    initialState: { unread: 0, loading: false },
    reducers: {
        markRead(state) {
            state.unread = Math.max(0, state.unread - 1);
            // ← Note: RTK uses Immer, which allows this "mutating" syntax
            // even though the underlying state is immutable
        }
    },
    extraReducers: builder => {
        builder
            .addCase(fetchMessages.pending, state => { state.loading = true; })
            .addCase(fetchMessages.fulfilled, (state, action) => {
                state.unread = action.payload.unreadCount;
                state.loading = false;
            });
    }
});

export const { markRead } = messagesSlice.actions;
export default messagesSlice.reducer;
```

RTK is genuinely better than vanilla Redux. But for many applications, even RTK is overkill.

### Era 3: Context API (React 16.3, 2018)

React 16.3 introduced the Context API as a first-party alternative to Redux for simpler state sharing scenarios. Context allows you to create a "context" — a value that is accessible to any component in a subtree without passing it through props.

```jsx
// Create a context with a default value
const MessageContext = React.createContext({ unread: 0 });

// A provider wraps the component tree and provides the value
function App() {
    const [messageState, setMessageState] = useState({ unread: 0 });
    
    return (
        <MessageContext.Provider value={messageState}>
            <Layout />
        </MessageContext.Provider>
    );
}

// Any component in the tree can consume the context
function NavBar() {
    const { unread } = useContext(MessageContext);
    return <div>Messages: {unread}</div>;
}
```

Context is simpler than Redux and works well for low-frequency updates: themes, language settings, authentication state, feature flags. However, Context has a significant performance problem: when the context value changes, **every component that calls `useContext(SomeContext)` re-renders**, regardless of whether the specific part of the context they care about actually changed.

If you put all your application state in a single context, and you update any part of it frequently, you will trigger unnecessary re-renders across the entire component tree. For this reason, Context is often split into multiple smaller contexts, or combined with `useReducer` to create a Redux-like pattern without Redux.

```jsx
// Split contexts to minimize unnecessary re-renders
const MessageCountContext = React.createContext(0);
const MessageActionsContext = React.createContext({ markRead: () => {} });
// Components that only need the count don't re-render when actions change
// and vice versa
```

### Era 4: Atomic State — Zustand, Jotai, Recoil (2019–present)

The atomic state libraries — Zustand (created by pmndrs in 2019), Jotai (created by Daishi Kato in 2020), and Recoil (created by Facebook in 2020, now largely unmaintained) — offer a different model for global state.

**Zustand** provides a simple store with selector-based subscriptions. Components subscribe to specific parts of the state, and only re-render when those specific parts change:

```javascript
import { create } from 'zustand';

const useMessageStore = create((set) => ({
    unread: 0,
    loading: false,
    
    fetchMessages: async (userId) => {
        set({ loading: true });
        const data = await fetch(`/api/users/${userId}/messages`).then(r => r.json());
        set({ unread: data.unreadCount, loading: false });
    },
    
    markRead: () => set(state => ({ unread: Math.max(0, state.unread - 1) })),
}));

// NavBar only re-renders when 'unread' changes, not when 'loading' changes
function NavBar() {
    const unread = useMessageStore(state => state.unread);
    return <div>Messages: {unread}</div>;
}

// MessageList re-renders when any store state changes that it selects
function MessageList() {
    const { fetchMessages, markRead } = useMessageStore();
    useEffect(() => { fetchMessages(currentUserId); }, []);
    
    return (
        <button onClick={markRead}>Mark as Read</button>
    );
}
```

Zustand's selector-based subscription model solves the Context performance problem. Components subscribe to exactly the state slices they need, and re-render only when those slices change.

**Jotai** goes further with an atomic model — state is broken into individual "atoms," and components can compose atoms into derived state:

```javascript
import { atom, useAtom, useAtomValue, useSetAtom } from 'jotai';

// Individual atoms
const unreadCountAtom = atom(0);
const loadingAtom = atom(false);

// Derived atom (computed from other atoms)
const hasUnreadAtom = atom(get => get(unreadCountAtom) > 0);

// NavBar subscribes only to unreadCountAtom
function NavBar() {
    const unread = useAtomValue(unreadCountAtom);
    return <div>Messages: {unread}</div>;
}

// Badge icon subscribes only to hasUnreadAtom
function BadgeIcon() {
    const hasUnread = useAtomValue(hasUnreadAtom);
    return <span className={hasUnread ? 'badge-active' : 'badge-empty'} />;
}
```

Jotai's model is particularly elegant and its per-atom subscription granularity means minimal unnecessary re-renders.

### Era 5: Signals (2022–present)

Signals are a reactive primitive that originated in the fine-grained reactivity systems of frameworks like Preact, SolidJS, and Angular (Angular 18 added signals as a stable feature in 2024). In React 19.2, the `useEffectEvent` hook is a step toward signal-like semantics, and there has been significant discussion in the React community about integrating signals more deeply into React's model.

A signal is an observable value — any computation that reads a signal's value is automatically subscribed to changes in that value. When the signal changes, only the computations that read it (and their dependent computations) are re-evaluated.

In SolidJS (which popularized signals in the front-end context):

```javascript
import { createSignal, createMemo } from 'solid-js';

const [unreadCount, setUnreadCount] = createSignal(0);

// This memo is automatically re-evaluated when unreadCount changes
const hasUnread = createMemo(() => unreadCount() > 0);

// NavBar re-renders only when unreadCount changes
function NavBar() {
    return <div>Messages: {unreadCount()}</div>;
}
```

Signals solve the re-render problem at a fundamental level — rather than relying on memoization to prevent unnecessary work, signals ensure that only the code that actually reads a changed value is ever re-evaluated. This is more efficient and requires less manual optimization.

## Part 2: The Immutability Problem — Why Mutation Silently Fails in React

One of the most common bugs for developers new to React is mutating state directly:

```jsx
// BROKEN — This will NOT cause the component to re-render
function BuggyComponent() {
    const [items, setItems] = useState([{ id: 1, name: 'First' }]);
    
    const addItem = () => {
        items.push({ id: 2, name: 'Second' }); // ← Direct mutation!
        // items now has 2 elements, but React doesn't know
        // because the reference to 'items' hasn't changed
        setItems(items); // ← This sets state to the same array reference
                         // React compares old and new with ===
                         // Same reference → assumes nothing changed → no re-render
    };
    
    return (
        <div>
            <button onClick={addItem}>Add Item</button>
            <ul>{items.map(i => <li key={i.id}>{i.name}</li>)}</ul>
        </div>
    );
}
```

When you call `setItems(items)` after mutating the array in place, React compares the new value to the old value using JavaScript's `===` operator. For objects and arrays, `===` compares **references**, not contents. Both the old `items` and the new `items` point to the same array object — so they are `===` equal — so React concludes nothing changed and skips the re-render.

The fix is to create a new array instead of mutating the old one:

```jsx
const addItem = () => {
    // Create a NEW array — spread the old one and add the new item
    setItems([...items, { id: items.length + 1, name: 'New Item' }]);
    //         ↑ This creates a new array reference
    //         React sees old !== new → triggers re-render
};
```

This immutability requirement pervades React state management. You cannot mutate nested objects either:

```jsx
// BROKEN
const updateItemName = (id, newName) => {
    const item = items.find(i => i.id === id);
    item.name = newName; // ← Mutates the item object in place
    setItems(items);     // ← Same array reference, no re-render
};

// CORRECT — immutable update
const updateItemName = (id, newName) => {
    setItems(items.map(i => 
        i.id === id 
            ? { ...i, name: newName }  // New object for the changed item
            : i                         // Same reference for unchanged items
    ));
};
```

The `map` approach creates a new array containing new objects for changed items and the same references for unchanged items. React sees the new array reference → triggers re-render → the diffing algorithm sees that most items are the same reference → skips updating their DOM nodes. Only the changed item is actually updated in the DOM.

### Why Object Identity Is React's Source of Truth

Understanding the immutability requirement requires understanding that React uses JavaScript reference equality (`===`) as the fundamental way to detect changes. This is a design choice baked into React's architecture: comparing two JavaScript objects by value (checking every property of every nested object) would be too expensive to do on every render. Reference comparison is O(1). Value comparison is O(n) where n is the size of the object graph.

So React's bet is: "If the reference is the same, the value is the same." This is only valid if you never mutate objects in place — hence the immutability requirement.

Let us draw this out as a mental model, because for C# developers it maps to a concept you already know: `readonly` records and value semantics.

```
State before:
items = [ref:0xA001] → Array
                        [0]: [ref:0xB001] → { id: 1, name: "Alice" }
                        [1]: [ref:0xB002] → { id: 2, name: "Bob" }

After MUTATION (wrong):
items = [ref:0xA001] → Same Array (same reference!)
                        [0]: [ref:0xB001] → { id: 1, name: "Alice" }
                        [1]: [ref:0xB002] → { id: 2, name: "Carol" }  ← mutated
// setItems(items) → React compares 0xA001 === 0xA001 → true → no re-render

After IMMUTABLE UPDATE (correct):
items = [ref:0xA003] → NEW Array (new reference!)
                        [0]: [ref:0xB001] → { id: 1, name: "Alice" }  ← same ref, reused
                        [1]: [ref:0xB004] → { id: 2, name: "Carol" }  ← new object
// setItems(items) → React compares 0xA001 === 0xA003 → false → RE-RENDER
```

The new array (`0xA003`) contains the same reference for `Alice` (`0xB001`) and a new reference for the updated `Carol` entry (`0xB004`). When React diffs the virtual DOM, it compares the `key` prop values and can skip re-rendering the Alice item (same data) while updating the Carol item (new data). This is efficient — but only if you follow the immutability rule.

## Part 3: Dependency Injection in Blazor — The .NET Answer

Now let us switch to Blazor and look at how .NET handles global state. The answer is dependency injection, and it is so natural for .NET developers that it requires almost no explanation — but let us be explicit about why it solves the same problems React has been wrestling with.

### Registering a Service

In a Blazor WebAssembly application, services are registered in `Program.cs`:

```csharp
// Program.cs
var builder = WebAssemblyHostBuilder.CreateDefault(args);

// Singleton: one instance for the lifetime of the application
// — like Redux store, global state
builder.Services.AddSingleton<IMessageService, MessageService>();

// Scoped: one instance per "scope" (in WASM, same as singleton — there is only one scope)
builder.Services.AddScoped<IUserService, UserService>();

// Transient: new instance every time it is requested
// — for stateless services
builder.Services.AddTransient<IEmailValidator, EmailValidator>();

builder.RootComponents.Add<App>("#app");

await builder.Build().RunAsync();
```

### The MessageService

The service holds the shared state and exposes it to components:

```csharp
public interface IMessageService
{
    int UnreadCount { get; }
    Task LoadAsync(string userId);
    void MarkRead();
    event Action? OnChange; // Components subscribe to this to know when to re-render
}

public class MessageService : IMessageService
{
    private readonly HttpClient _http;
    
    public MessageService(HttpClient http) { _http = http; }
    
    public int UnreadCount { get; private set; } = 0;
    
    public event Action? OnChange;
    
    public async Task LoadAsync(string userId)
    {
        var response = await _http.GetFromJsonAsync<MessagesResponse>(
            $"/api/users/{userId}/messages");
        UnreadCount = response?.UnreadCount ?? 0;
        OnChange?.Invoke(); // ← Notify subscribers
    }
    
    public void MarkRead()
    {
        if (UnreadCount > 0)
        {
            UnreadCount--;
            OnChange?.Invoke();
        }
    }
}
```

### Consuming the Service in Components

Any component in the application tree can inject and use this service:

```razor
@* NavBar.razor — at the top of the component tree *@
@inject IMessageService MessageService
@implements IDisposable

<div class="nav-badge">
    Messages: @MessageService.UnreadCount
</div>

@code {
    protected override void OnInitialized()
    {
        // Subscribe to state changes
        MessageService.OnChange += HandleStateChange;
    }
    
    private void HandleStateChange()
    {
        // Called when MessageService.OnChange fires
        InvokeAsync(StateHasChanged);
    }
    
    public void Dispose()
    {
        // Always unsubscribe to prevent memory leaks
        MessageService.OnChange -= HandleStateChange;
    }
}
```

```razor
@* MessageList.razor — deep in the component tree *@
@inject IMessageService MessageService

<button @onclick="MarkRead">Mark as Read</button>

@code {
    private async Task MarkRead()
    {
        MessageService.MarkRead();
        // The NavBar will automatically update because they share the same
        // MessageService instance (singleton) and NavBar subscribed to OnChange
    }
}
```

Both `NavBar` and `MessageList` inject `IMessageService`. Because it is registered as a singleton, they both get the same instance. When `MessageList` calls `MessageService.MarkRead()`, the `UnreadCount` changes on the shared instance, `OnChange` fires, `NavBar` receives the event, and calls `StateHasChanged()` to update its display.

This is the same pattern as Redux's store — a central place for shared state, with components subscribing to changes — but expressed using .NET idioms that every C# developer already knows.

### Making It More Ergonomic With a Base Class

Subscribing to `OnChange`, implementing `IDisposable`, and calling `InvokeAsync(StateHasChanged)` in every component that consumes a service is boilerplate. You can eliminate this with a base class:

```csharp
// A base class for components that need to react to service state changes
public abstract class ServiceConsumerBase : ComponentBase, IDisposable
{
    private readonly List<Action> _subscriptions = new();
    
    protected void Subscribe(ref Action? changeEvent)
    {
        void Handler() => InvokeAsync(StateHasChanged);
        changeEvent += Handler;
        
        // Store cleanup action
        _subscriptions.Add(() => changeEvent -= Handler);
    }
    
    public void Dispose()
    {
        foreach (var cleanup in _subscriptions)
            cleanup();
    }
}
```

Now your components become much cleaner:

```razor
@inherits ServiceConsumerBase
@inject IMessageService MessageService

<div class="nav-badge">Messages: @MessageService.UnreadCount</div>

@code {
    protected override void OnInitialized()
    {
        var changeEvent = MessageService.OnChange;
        Subscribe(ref changeEvent!);
        // ← handles subscription, re-render, and disposal automatically
    }
}
```

Or even simpler, use a library like FluxR or Fluxor that provides a Redux-like pattern on top of Blazor's DI system.

## Part 4: Signals in React — Closing the Gap

As mentioned in Part 1, signals are making their way into React's ecosystem. While native signal support in React core is still under discussion, several popular libraries provide signal-like reactivity in React applications.

The most mature is the Signals library from Preact (which works in React too via a compatibility layer):

```javascript
import { signal, computed, effect } from '@preact/signals-react';

// Signals exist outside of components — they are global reactive values
const unreadCount = signal(0);
const hasUnread = computed(() => unreadCount.value > 0);

// These functions can be used anywhere
function markRead() {
    if (unreadCount.value > 0) {
        unreadCount.value--;
    }
}

async function loadMessages(userId) {
    const data = await fetch(`/api/users/${userId}/messages`).then(r => r.json());
    unreadCount.value = data.unreadCount;
}

// Components automatically subscribe to signals they read
// and only re-render when those specific signals change
function NavBar() {
    return <div>Messages: {unreadCount}</div>;
    // ← No useSelector, no useContext, no useMemo
    // This component re-renders ONLY when unreadCount changes
}

function MessageList() {
    return (
        <div>
            <button onClick={markRead}>Mark as Read</button>
        </div>
    );
}
```

Signals are conceptually closer to Blazor's singleton services with change events than any previous React state management solution. The signal is a reactive value that lives outside components. Components subscribe by reading it. When it changes, only components that read it re-render. No boilerplate, no explicit subscriptions, no selectors.

The difference from Blazor's DI approach is primarily ergonomic: in Blazor, the state change notification mechanism (the `OnChange` event) is explicit. With signals, the subscription is implicit — the framework tracks which signals were read during each render and sets up subscriptions automatically.

## Part 5: The Key Insight — Why DI Is Not Just a Technical Choice

The difference between React's state management landscape and Blazor's DI-based approach is not just technical — it reflects a deeper philosophical difference about where complexity should live.

React's approach puts complexity at the component level. Individual components (or hooks) must explicitly opt into the state they need, manage subscriptions, and ensure they clean up. This maximizes flexibility (any component can subscribe to any state) at the cost of requiring developers to manage these concerns explicitly.

.NET's DI approach puts complexity at the service registration level. When you decide that `IMessageService` should be a singleton, that decision propagates automatically to every component that injects it. Components do not need to know that it is a singleton — they just inject the interface and use it. The infrastructure manages the lifetime.

For business application development — where the same patterns repeat across dozens of features — .NET's DI approach dramatically reduces boilerplate and the opportunity for lifecycle management bugs. Every .NET developer already knows how to register a service and inject it. Nobody has to learn a new state management library.

For highly interactive consumer applications — where state needs to be extremely fine-grained, where different parts of the UI need different update frequencies, and where the component tree might be enormous — React's explicit subscription model (especially with signals) gives you more precise control.

## Conclusion: Choosing Your State Model

Let us be concrete about the guidance here.

**Choose React + Redux Toolkit** when:
- Your application has complex state that involves async data fetching, loading states, error states, and optimistic updates across many entities.
- Your team is already familiar with the Redux mental model.
- You need advanced dev tools (Redux DevTools is exceptional for debugging state changes).

**Choose React + Zustand** when:
- You want centralized global state without Redux's complexity.
- Your application's state is relatively straightforward.
- You want selector-based subscriptions with minimal boilerplate.

**Choose React + Signals** when:
- You are building something that updates very frequently (real-time dashboards, games, collaborative editing).
- You want the most fine-grained reactivity with the least unnecessary re-rendering.

**Choose Blazor + DI Services** when:
- Your team is composed primarily of .NET developers who already know `IServiceCollection`.
- Your application follows standard CRUD and line-of-business patterns.
- You want state management that integrates naturally with the same patterns you use on the server.
- You value the ability to easily unit-test your state logic using standard .NET test tools.
