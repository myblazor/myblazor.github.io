## Introduction: The Redux Dev Tool That Opened To Six Panes

A few years ago we sat in on a code review for a React app that was struggling with performance. A new feature — an autocomplete dropdown in a navbar — had been built by a junior developer who was excited to use Redux. He set it up properly. Reducer. Action creators. Thunks for async. Selectors. `connect()`. Memoisation. He had done everything right according to the official Redux tutorial, which he had just completed.

The state tree for the dropdown looked like this:

```javascript
{
  navbar: {
    autocomplete: {
      input: { value: "", focused: false, cursor: 0 },
      results: { loading: false, items: [], error: null, page: 1 },
      selection: { activeIndex: -1, hovered: null, dragging: false },
      history: { recent: [], favorites: [], cleared: false },
      ui: { isOpen: false, position: null, anchor: null },
      abort: { controller: null, signal: null },
      cache: { byQuery: {}, byId: {}, byTag: {} },
    }
  }
}
```

Every change — every keystroke — went through a reducer, through the middleware chain, through the selector layer, produced a new immutable state snapshot, triggered a React re-render, propagated through useSelector hooks, and updated the UI. When the user pressed a key, the Redux DevTool timeline recorded an action. The action was in the "performance" tab with flame graphs. The developer was proud of the observability.

We asked him to measure the time to paint between keystroke and dropdown update. It was 180 milliseconds on a mid-range laptop. The autocomplete felt laggy. On mobile it was nearer 400ms.

The original jQuery version of this autocomplete — which the new React version was replacing — took 15 milliseconds per keystroke. It was one file, 160 lines, and it did not use a state-management library because it did not need one. A text input emitted a debounced event; the event handler did a fetch; the results went into a `<ul>`. That was the whole thing.

This is not a cautionary tale about React. It is a cautionary tale about ceremony. State management is not an inherent problem. It is a problem only when we make it one. The majority of the JavaScript in "state management" libraries — Redux, Zustand, Jotai, MobX, Recoil, Valtio, Effector — is solving problems that only exist *because* the library exists. Pure cognitive load generated by the tool itself.

Today we are going to do state management without a library. Not poorly, not minimally — properly. We are going to build from first principles: the Observer pattern, reactive stores, signals, Proxy-based reactivity, and a complete global store with undo. The final code is about 200 lines of plain JavaScript. You will understand every line. It will do 95% of what the frameworks do. It will be faster than every framework on your actual hardware. And it will not create a dependency you have to maintain.

This is Part 9 of 15 in our no-build web series. After [Day 7's module system](/blog/2026-05-30-no-build-web-es-modules) and [Day 8's DOM primitives](/blog/2026-05-31-no-build-web-dom-and-events), we finally have all the tools we need to build real applications. Today we cover how data flows through those applications.

## Part 1: The Question We Are Actually Asking

Before we write any code, let us be clear about the problem. "State management" is a vague term that bundles several distinct concerns:

1. **Storage.** Where does the data live? In a variable? In `localStorage`? In the URL? In the DOM?
2. **Change notification.** When the data changes, who needs to know?
3. **Derived values.** Some state is computed from other state. How do we keep derivations in sync with their inputs?
4. **Async orchestration.** Some state comes from network requests. How do we track loading, error, and success states?
5. **Time travel / undo.** Can we step backward through state history?
6. **Persistence.** Does state survive a page reload?
7. **Cross-tab sync.** If the same app is open in two tabs, should they share state?
8. **Debugging.** How do we introspect and debug state changes?

Every state library tries to address all of these, with varying weight. Redux emphasises 2, 5, and 8. Zustand emphasises 1 and 2. MobX emphasises 2 and 3. Jotai emphasises 3. None of them is exactly wrong; they have different priorities.

Here is the thing we have learned after fifteen years of doing this: **for most applications, the hard problems are 2, 3, and 4. The rest are either solved by the browser (7 is `BroadcastChannel`; 6 is `localStorage`; 8 is DevTools) or are not problems you have.**

So we will focus on those three. Change notification, derived values, and async orchestration. Do those well, and you have state management.

## Part 2: The Observer Pattern — 40 Lines Of Mental Model

The foundation of every reactive system is the **Observer pattern**: objects that hold state, and other objects that listen for changes. This is as old as software engineering — it is in the Gang of Four book from 1994 — and it predates JavaScript by decades.

The simplest possible observable:

```javascript
// @ts-check
/**
 * @template T
 */
export class Observable {
  /** @param {T} initial */
  constructor(initial) {
    this.value = initial;
    this.listeners = new Set();
  }

  subscribe(listener) {
    this.listeners.add(listener);
    listener(this.value);
    return () => this.listeners.delete(listener);
  }

  set(newValue) {
    if (Object.is(this.value, newValue)) return;
    this.value = newValue;
    for (const listener of this.listeners) listener(newValue);
  }

  update(fn) {
    this.set(fn(this.value));
  }
}
```

That is it. Fourteen lines including whitespace. Let us walk through what it does.

- `value` holds the current state.
- `listeners` is a `Set` of functions to call on change. Using a `Set` means you cannot double-subscribe the same listener accidentally, and deletion is O(1).
- `subscribe(listener)` adds a listener, *immediately invokes it* with the current value (the common pattern called "fire on subscribe" or "eager subscription"), and returns an `unsubscribe` function.
- `set(newValue)` updates the value. If nothing changed (`Object.is` check — handles `NaN` and `-0` correctly), do nothing; otherwise notify all listeners.
- `update(fn)` is syntactic sugar: takes a function that produces a new value from the old one.

Usage:

```javascript
const count = new Observable(0);

const unsub = count.subscribe(value => {
  document.getElementById("count").textContent = value;
});

document.getElementById("increment").addEventListener("click", () => {
  count.update(n => n + 1);
});

// Later, cleanup:
unsub();
```

That is a working counter. The DOM stays in sync with the state. `unsub()` removes the listener when no longer needed.

This is the heart of every reactive framework. Redux wraps this in reducers and actions. Zustand wraps it in hooks. MobX wraps it in Proxies. Jotai wraps it in atoms and derivations. The core is a box-of-value that notifies listeners. Everything else is ergonomics.

### Why return an unsubscribe function

The subscribe-returns-unsubscribe convention is important. It lets the caller manage listener lifetime without having to preserve a reference to the listener itself:

```javascript
const unsub = count.subscribe(v => render(v));
// ... later, from anywhere that has `unsub`:
unsub();
```

This is essentially the same idiom as `AbortController` from Day 8 — an API where cleanup is *returned* rather than tracked by the caller. Pairs naturally with `signal`:

```javascript
subscribe(listener, signal) {
  if (signal?.aborted) return () => {};
  this.listeners.add(listener);
  listener(this.value);
  const unsub = () => this.listeners.delete(listener);
  signal?.addEventListener("abort", unsub, { once: true });
  return unsub;
}
```

With a signal, `controller.abort()` unsubscribes automatically. Your component's cleanup lifecycle unifies with the rest of the platform.

## Part 3: Signals — A More Natural API

`Observable` is fine, but the method-call syntax is clunky. You want to write `count.value + 1`, not `count.update(n => n + 1)`. Enter **signals**.

A signal is a primitive that is:

- **Readable** via a getter (`count()` or `count.value`).
- **Writable** via a setter (`count.set(n)` or `count.value = n`).
- **Observable** — subscribers are notified on change.
- **Optionally derivable** — you can define computed signals that depend on others.

Signals are the core reactive primitive of Solid, Angular (since 17), Vue (as `ref`), Preact, Qwik, and most 2020s JavaScript frameworks. They also happen to be on track to become [a TC39 proposal](https://github.com/tc39/proposal-signals) for inclusion in JavaScript itself. For now, we build our own — it is about 60 lines.

### A signal, from first principles

Here is the complete implementation:

```javascript
// @ts-check

/** Current "reactive context" — the computation currently running. */
let currentComputation = null;

/**
 * Create a writable signal.
 * @template T
 * @param {T} initialValue
 * @returns {{ (): T, set: (v: T) => void, subscribers: Set<() => void> }}
 */
export function signal(initialValue) {
  let value = initialValue;
  /** @type {Set<() => void>} */
  const subscribers = new Set();

  function read() {
    if (currentComputation) {
      subscribers.add(currentComputation);
    }
    return value;
  }

  read.set = (newValue) => {
    if (Object.is(value, newValue)) return;
    value = newValue;
    for (const sub of [...subscribers]) sub();
  };

  read.subscribers = subscribers;

  return read;
}

/**
 * Create a computed signal derived from other signals.
 * @template T
 * @param {() => T} fn
 */
export function computed(fn) {
  const result = signal(/** @type {T} */ (undefined));

  function recompute() {
    const prev = currentComputation;
    currentComputation = recompute;
    try {
      result.set(fn());
    } finally {
      currentComputation = prev;
    }
  }

  recompute();
  return result;
}

/**
 * Run a side effect whenever its dependencies change.
 * @param {() => void | (() => void)} fn
 * @returns {() => void}
 */
export function effect(fn) {
  let cleanup;

  function run() {
    if (typeof cleanup === "function") cleanup();
    const prev = currentComputation;
    currentComputation = run;
    try {
      cleanup = fn();
    } finally {
      currentComputation = prev;
    }
  }

  run();

  return () => {
    if (typeof cleanup === "function") cleanup();
  };
}
```

Sixty lines, fully typed. Let us trace what happens.

A `signal(x)` returns a callable function. Calling it without arguments reads the value. Calling `set(v)` on it writes the value. Under the hood, it maintains a set of subscribers — other computations that depend on this signal.

The magic is `currentComputation`. When you call `computed(() => a() + b())`, it runs the function `fn`. Before running, it sets `currentComputation = recompute`. Inside `fn`, when you call `a()`, the signal's `read` function sees that `currentComputation` is not null, so it adds `recompute` to its subscribers. Same for `b()`. After `fn` returns, `currentComputation` is restored to its previous value.

Now both `a` and `b` know that `recompute` depends on them. When either changes, it calls `recompute`, which re-runs `fn`, which re-reads `a` and `b` (picking up the new value), which writes to `result`. If anything is subscribed to `result`, it is notified.

**This is automatic dependency tracking.** You never declare dependencies manually. The system figures them out by observing which signals are read during the computation. This is how Solid, Svelte's `$:` syntax, and Vue's `computed()` all work under the hood.

### Usage

```javascript
const count = signal(0);
const doubled = computed(() => count() * 2);
const tripled = computed(() => count() * 3);
const sum = computed(() => doubled() + tripled());

effect(() => {
  console.log(`count=${count()}, doubled=${doubled()}, sum=${sum()}`);
});
// logs: count=0, doubled=0, sum=0

count.set(5);
// logs: count=5, doubled=10, sum=25

count.set(10);
// logs: count=10, doubled=20, sum=50
```

The `effect` runs exactly when its dependencies change. `sum` recomputes only because `doubled` and `tripled` recomputed. No manual wiring. No subscription management. No memoisation annotations.

Compare to React, where the same thing needs `useMemo` for `doubled`, `useMemo` for `tripled`, `useMemo` for `sum`, and a `useEffect` with a dependency array for the logging side effect — and you have to get the dependency array right or you get a subtle bug. Here, none of that ceremony exists, because the dependency is established by the act of reading.

### Cleanup in effects

`effect` lets the function return a cleanup callback:

```javascript
const disposeEffect = effect(() => {
  const timer = setInterval(() => console.log(count()), 1000);
  return () => clearInterval(timer);
});

// Later:
disposeEffect();   // calls the cleanup, kills the timer
```

The pattern matches React's `useEffect` return semantics precisely. The difference: the dependency tracking is automatic (no array), and the cleanup is called automatically on dependency change.

### Where this falls short

Our signal implementation has one rough edge worth being honest about: **glitches**. If `count` changes, and both `doubled` and `tripled` are subscribed to `count`, the `effect` that depends on both might run *twice* — once after `doubled` updates, once after `tripled` updates — each time with an inconsistent view of the world.

Solving this properly requires a "scheduler" that batches recomputations into a single flush at the end of the current task. Solid and the TC39 Signals proposal have sophisticated schedulers. For most application code, the glitches are invisible (the DOM reconciler runs on the final state, so intermediate updates are invisible to the user). For serious reactive work, you want a batched update mechanism:

```javascript
let pending = false;
const queue = new Set();

export function batch(fn) {
  pending = true;
  try { fn(); } finally {
    pending = false;
    const tasks = [...queue];
    queue.clear();
    for (const t of tasks) t();
  }
}
```

And inside your signal's `set`:

```javascript
read.set = (newValue) => {
  if (Object.is(value, newValue)) return;
  value = newValue;
  if (pending) {
    for (const sub of subscribers) queue.add(sub);
  } else {
    for (const sub of [...subscribers]) sub();
  }
};
```

Now `batch(() => { a.set(1); b.set(2); })` causes each subscriber to run at most once after both updates. This is the version you want for production.

Even with this simplification, the entire reactive library is still under 100 lines. It is runtime-complete — you could build a real application on top of it. Several of our personal projects do.

## Part 4: Proxy-Based Reactivity — When You Want Deep Objects

Signals work beautifully for discrete pieces of state (a counter, a boolean, a list). They are awkward for deep objects where you want to track mutations to nested properties.

Consider an editable form:

```javascript
const form = signal({
  name: "Alice",
  email: "alice@example.com",
  address: { city: "Paris", country: "France" },
});

// To change the email:
form.set({ ...form(), email: "new@example.com" });

// To change the city:
form.set({
  ...form(),
  address: { ...form().address, city: "Lyon" },
});
```

Tedious. Immutability is correct, but the syntax for deep updates is painful. Libraries like Immer solve this by giving you a mutable-feeling API that produces an immutable result. We can do it natively with Proxy.

### The Proxy-backed store

```javascript
// @ts-check
/**
 * Create a deeply reactive proxy for an object.
 * @template {object} T
 * @param {T} target
 */
export function reactive(target) {
  /** @type {Set<() => void>} */
  const subscribers = new Set();

  function notify() {
    for (const sub of [...subscribers]) sub();
  }

  const handler = {
    get(obj, key) {
      if (currentComputation) subscribers.add(currentComputation);
      const value = Reflect.get(obj, key);
      if (value && typeof value === "object") {
        // Recursively wrap nested objects — they share the same subscribers
        return new Proxy(value, handler);
      }
      return value;
    },
    set(obj, key, value) {
      const prev = obj[key];
      if (!Object.is(prev, value)) {
        Reflect.set(obj, key, value);
        notify();
      }
      return true;
    },
    deleteProperty(obj, key) {
      if (key in obj) {
        Reflect.deleteProperty(obj, key);
        notify();
      }
      return true;
    },
  };

  return new Proxy(target, handler);
}
```

Now deep mutation is a one-liner:

```javascript
const form = reactive({
  name: "Alice",
  email: "alice@example.com",
  address: { city: "Paris", country: "France" },
});

form.email = "new@example.com";            // triggers subscribers
form.address.city = "Lyon";                // also triggers subscribers
form.tags = ["vip", "active"];             // also triggers

effect(() => {
  console.log(`${form.name} lives in ${form.address.city}`);
});
```

The `effect` runs once initially, and once per mutation that affects read properties. Because the nested proxy shares the same subscriber set, mutations at any depth trigger the same effect.

This is the MobX / Vue Reactivity model, reduced to its essence. The full Vue Reactivity implementation is ~600 lines and handles edge cases we ignore here (Map, Set, array methods like `push`, ref-in-reactive unwrapping), but the core idea is exactly this.

### Trade-offs of the Proxy approach

**Pros:**
- Mutable syntax; no spread-everywhere.
- Works with any object shape.
- Nested mutations are tracked automatically.

**Cons:**
- Every property access creates a new Proxy (unless you cache — left as an exercise).
- Debugging can be surprising: `console.log(form)` shows a Proxy, not the raw object.
- Does not naturally support immutable history (no snapshots).
- Harder to reason about when things change (every read registers a dependency).

For most application code, reactive objects are the most pleasant ergonomic choice. For strict immutable state with time travel, signals are cleaner. Use both — they compose.

## Part 5: Async State — Loading, Error, Success

Most real applications are not "a counter." They are "a list of posts fetched from the server, which might be loading, which might have failed, which might need to be refetched, which needs to be cancellable, which has pagination..."

A common pattern crystallises this into one reusable primitive, which we will call `asyncSignal`:

```javascript
// @ts-check
/**
 * @template T
 * @param {(signal: AbortSignal) => Promise<T>} loader
 */
export function asyncSignal(loader) {
  const state = signal(/** @type {
    | { status: "idle" }
    | { status: "loading" }
    | { status: "success", data: T }
    | { status: "error", error: Error }
  } */ ({ status: "idle" }));

  let controller;

  async function refresh() {
    if (controller) controller.abort();
    controller = new AbortController();
    state.set({ status: "loading" });
    try {
      const data = await loader(controller.signal);
      state.set({ status: "success", data });
    } catch (error) {
      if (error.name !== "AbortError") {
        state.set({ status: "error", error });
      }
    }
  }

  function cancel() {
    controller?.abort();
    controller = null;
  }

  return { state, refresh, cancel };
}
```

Thirty lines, handles every async state. Usage:

```javascript
const posts = asyncSignal(async (signal) => {
  const response = await fetch("/api/posts", { signal });
  if (!response.ok) throw new Error(`HTTP ${response.status}`);
  return response.json();
});

// Trigger initial load
posts.refresh();

// Render whatever the current state is
effect(() => {
  const s = posts.state();
  if (s.status === "loading") renderLoading();
  else if (s.status === "error") renderError(s.error);
  else if (s.status === "success") renderList(s.data);
});

// Refresh button:
document.getElementById("refresh").addEventListener("click", () => posts.refresh());

// Cancel button:
document.getElementById("cancel").addEventListener("click", () => posts.cancel());
```

Calling `refresh` again while a previous fetch is in flight aborts the old one — no race conditions, no stale data overwriting fresh data. The `AbortController` we built on Day 8 does the work.

This pattern replaces 90% of what [TanStack Query](https://tanstack.com/query) (formerly React Query) does. It does not do HTTP caching, request deduplication across components, or stale-while-revalidate — add those as needed. But for the common case of "fetch, show loading, show error, show success, maybe refresh," this is enough.

## Part 6: A Complete Global Store

Putting the pieces together, here is a production-shape global store:

```javascript
// @ts-check
import { reactive } from "./reactive.js";
import { effect } from "./signals.js";

/**
 * @template {object} T
 * @param {T} initial
 * @param {{ persist?: string }} [options]
 */
export function createStore(initial, options = {}) {
  const { persist } = options;

  // Load persisted state if available
  let startingValue = initial;
  if (persist) {
    try {
      const saved = localStorage.getItem(persist);
      if (saved) startingValue = { ...initial, ...JSON.parse(saved) };
    } catch { /* ignore malformed */ }
  }

  const state = reactive(startingValue);

  // Persist on change
  if (persist) {
    effect(() => {
      // Read all top-level keys to establish dependencies
      const snapshot = Object.fromEntries(
        Object.keys(state).map(k => [k, state[k]])
      );
      localStorage.setItem(persist, JSON.stringify(snapshot));
    });
  }

  return state;
}
```

And usage:

```javascript
// stores/app-store.js
import { createStore } from "../utils/store.js";

export const appStore = createStore({
  theme: "auto",
  user: null,
  sidebarOpen: false,
  selectedPostId: null,
  search: { query: "", filters: {} },
}, { persist: "app-state" });
```

Elsewhere:

```javascript
import { appStore } from "@app/stores/app-store.js";
import { effect } from "@app/utils/signals.js";

// React to theme changes
effect(() => {
  document.documentElement.dataset.theme = appStore.theme;
});

// Change it from a click handler
themeToggle.addEventListener("click", () => {
  appStore.theme = appStore.theme === "dark" ? "light" : "dark";
});
```

That is the complete pattern. The store persists automatically. Theme changes trigger the effect, which updates the DOM. Any other part of the app can subscribe to the same store. There is no "provider." There is no "context." There is a module-level object that everyone imports.

### Cross-tab sync

If the user has the app open in two tabs and changes the theme in one, the other should update. Add this to the store:

```javascript
if (persist) {
  const channel = new BroadcastChannel(persist);
  channel.addEventListener("message", (event) => {
    Object.assign(state, event.data);
  });
  effect(() => {
    const snapshot = structuredClone(state);
    channel.postMessage(snapshot);
  });
}
```

`BroadcastChannel` is a native API for cross-tab messaging, [Baseline Widely Available since 2022](https://developer.mozilla.org/en-US/docs/Web/API/BroadcastChannel). Messages sent from one tab arrive in every other tab of the same origin. Ten lines, full cross-tab sync. No library.

There is a subtle concern: the receiving tab's `Object.assign` triggers effects, which can re-broadcast. You want either a debounce or a simple flag to avoid an echo loop. We omit it for brevity; add it in production.

## Part 7: Undo, Redo, And Time Travel

History is a matter of storing snapshots. For immutable state (signals), this is trivial. For reactive objects, you need `structuredClone`.

A simple history wrapper:

```javascript
// @ts-check
/**
 * @template {object} T
 * @param {T} state
 */
export function createHistory(state) {
  /** @type {T[]} */
  const past = [];
  /** @type {T[]} */
  const future = [];

  function snapshot() {
    past.push(structuredClone(state));
    future.length = 0;   // any new action discards redo history
  }

  function undo() {
    if (past.length === 0) return;
    future.push(structuredClone(state));
    const previous = past.pop();
    Object.assign(state, previous);
  }

  function redo() {
    if (future.length === 0) return;
    past.push(structuredClone(state));
    const next = future.pop();
    Object.assign(state, next);
  }

  function clear() {
    past.length = 0;
    future.length = 0;
  }

  return { snapshot, undo, redo, clear };
}
```

Wrap it around any reactive state:

```javascript
const drawing = reactive({ strokes: [] });
const history = createHistory(drawing);

function addStroke(stroke) {
  history.snapshot();
  drawing.strokes = [...drawing.strokes, stroke];
}

// Ctrl-Z
document.addEventListener("keydown", (event) => {
  if ((event.ctrlKey || event.metaKey) && event.key === "z") {
    event.preventDefault();
    if (event.shiftKey) history.redo();
    else history.undo();
  }
});
```

Full undo/redo in fifty lines. No library. Performance scales with snapshot size — `structuredClone` is fast but not free — so for very large states you want a patch-based history (Immer has a built-in patches feature). For most applications, cloning is fine.

## Part 8: The URL As State

A surprisingly often-forgotten form of state: the URL. Search queries, filters, selected tab, pagination page, selected item ID — all of these belong in the URL, not in JavaScript memory.

Why URL-as-state matters:

- **Shareable.** User copies the URL, sends it to a colleague, colleague sees the same view.
- **Bookmarkable.** The "interesting" page is a real URL, not a stateful detour from the home page.
- **Refresh-safe.** Reloading does not lose the user's position.
- **Back-button-safe.** The browser's back button does the expected thing.

A URL-state helper:

```javascript
// @ts-check
import { signal, effect } from "./signals.js";

/**
 * @param {string} key
 * @param {string} defaultValue
 */
export function urlParam(key, defaultValue) {
  function read() {
    return new URLSearchParams(location.search).get(key) ?? defaultValue;
  }

  const value = signal(read());

  // Update on navigation
  window.addEventListener("popstate", () => value.set(read()));

  // Write to URL on change
  effect(() => {
    const url = new URL(location.href);
    const v = value();
    if (v === defaultValue) url.searchParams.delete(key);
    else url.searchParams.set(key, v);
    const next = url.pathname + url.search;
    if (location.pathname + location.search !== next) {
      history.replaceState({}, "", next);
    }
  });

  return value;
}
```

Usage:

```javascript
const query = urlParam("q", "");
const filter = urlParam("filter", "all");
const page = urlParam("page", "1");

effect(() => {
  updateResults(query(), filter(), page());
});

input.addEventListener("input", () => query.set(input.value));
```

Now the URL is the source of truth. Typing in the search box updates the URL. Using the back button restores the previous query. Bookmarking preserves everything. Pasting a URL into another tab shows the same view. This is the state management most "state management libraries" overlook entirely, because they grew up in SPA culture where the URL was often treated as incidental.

## Part 9: Binding State To The DOM — The Render Step

We have state that notifies us on change. We have a DOM. The remaining question: how do we keep them in sync?

Three patterns, in order of increasing sophistication.

### Pattern 1: Direct effect

```javascript
effect(() => {
  document.getElementById("count").textContent = counter();
});
```

The effect reads `counter()`, updates the DOM, and re-runs on change. Direct, imperative, no framework. This is the jQuery-era pattern with a signal behind it. For small numbers of bindings, perfectly fine.

### Pattern 2: A render function

```javascript
function render(container, state) {
  container.innerHTML = "";
  const list = h("ul",
    ...state.posts.map(post =>
      h("li", {},
        h("h3", {}, post.title),
        h("p", {}, post.summary)
      )
    )
  );
  container.append(list);
}

const app = document.getElementById("app");
effect(() => render(app, appStore));
```

The render function wipes and rebuilds. Works fine for small regions. Inefficient for large dynamic lists because it recreates DOM on every change, which is bad for keyboard focus, scroll position, and animation.

### Pattern 3: Targeted updates

For anything with many items, you want to update only what changed. Without a virtual DOM, the idiom is "attribute your DOM with keys and reconcile":

```javascript
function renderList(container, items, createItem) {
  const existing = new Map();
  for (const el of container.children) {
    existing.set(el.dataset.key, el);
  }

  let lastChild = null;
  for (const item of items) {
    let el = existing.get(item.id);
    if (!el) {
      el = createItem(item);
      el.dataset.key = item.id;
    } else {
      existing.delete(item.id);
      updateItem(el, item);
    }
    if (lastChild) lastChild.after(el);
    else container.prepend(el);
    lastChild = el;
  }

  for (const el of existing.values()) el.remove();
}
```

This is a simplified version of what React and Preact do internally — key-based reconciliation. Thirty lines. Handles list updates efficiently without recreating DOM.

For complex apps you may want a proper reconciler — htm + Preact from esm.sh is 4KB and gives you JSX-like syntax with keyed reconciliation. For simple apps, the patterns above suffice.

### The full picture

A table of rendering strategies and when to use them:

| Pattern | When to use |
|---|---|
| Direct effect (`effect(() => dom.textContent = x)`) | Single value, few bindings |
| Attribute binding (`effect(() => el.toggleAttribute("disabled", x))`) | Boolean UI states |
| Render function (replace `innerHTML`) | Small, infrequently-changing regions |
| Template-clone-and-update | Static structure, dynamic content |
| Keyed reconciliation | Long lists, interactive items |
| Preact + htm (from esm.sh) | When the above becomes painful |

The honest answer: for most UIs, the direct-effect pattern covers 80% of cases, a render function covers 15%, and the last 5% want a reconciler. We use Preact for the capstone's blog post list (where the reconciler is genuinely helpful) and direct effects for everything else.

## Part 10: A Comparison With The Libraries You Probably Know

Let us map our primitives to the libraries the reader probably uses.

### Redux

Redux is a single store with reducers and actions. Every state change is an object `{ type: "...", payload: ... }` dispatched to a reducer, which produces a new immutable state.

Our equivalent:

```javascript
const store = reactive({
  count: 0,
  user: null,
});

// "Action creator"
function increment() {
  store.count++;
}

// "Reducer" is just... code
function login(user) {
  store.user = user;
}
```

No action objects. No reducer indirection. No middleware. Redux's strengths — time-travel debugging, action logging — we rebuild as the undo/redo history in Part 7, and as an effect that logs every change:

```javascript
if (import.meta.env?.DEV) {
  effect(() => console.log("State:", structuredClone(store)));
}
```

### Zustand

Zustand is a hooks-first store with a cleaner API than Redux. Ours:

```javascript
import { reactive } from "./reactive.js";

const bearStore = reactive({
  bears: 0,
  increasePopulation: () => bearStore.bears++,
  removeAllBears: () => bearStore.bears = 0,
});
```

Same shape. Same ergonomics. The difference is the React hook integration — Zustand gives you `useStore(state => state.bears)` with automatic subscription/unsubscription. We get the same with effects.

### MobX

MobX is the Proxy-reactive model taken seriously, with fine-grained reactivity and automatic dependency tracking. Our `reactive()` function is a simplified MobX. The main thing MobX adds that we skip: "computed" values as first-class observable properties on the reactive object (not separate signals). This is doable with another ~50 lines of code (a getter on the proxy that memoises), but we have not needed it.

### Jotai

Jotai is atoms-based: each piece of state is an independent atom, and derivations are atoms defined in terms of others. Our `signal` + `computed` is the same mental model:

```javascript
// Jotai
const countAtom = atom(0);
const doubledAtom = atom((get) => get(countAtom) * 2);

// Ours
const count = signal(0);
const doubled = computed(() => count() * 2);
```

Literally the same concept, different spelling.

### TanStack Query

TanStack Query (React Query) handles async server state — caching, deduplication, refetching, stale-while-revalidate. Our `asyncSignal` from Part 5 is the kernel; TanStack Query adds sophisticated cache invalidation, query keys, and optimistic updates. For most apps, our version is enough. When it is not, reach for the library.

### Signals proposal

As mentioned above, [TC39 is considering a Signals proposal](https://github.com/tc39/proposal-signals) to add signals natively to JavaScript. The proposed API is essentially what we built in Part 3, with a more robust scheduler. Our code will transfer directly to the native primitive when it ships.

## Part 11: The Practical Rules Of State

Fifteen years of writing stateful apps, distilled.

**Rule 1: Not everything is state.** The current scroll position is not state. The hover state of a link is not state. The focused element is not state. These are *ephemeral UI* and belong to the DOM. Do not store them.

**Rule 2: Derive when possible.** If `fullName` can be computed from `firstName` and `lastName`, it is not state. It is a derivation. Use `computed()` or a selector function. Storing derivations separately creates the risk of them getting out of sync with their inputs.

**Rule 3: Normalise collections.** A list of posts fetched from an API should be stored as `{ [id]: post }` and `order: [id, id, id]`, not as `[post, post, post]`. This makes individual-post updates O(1) and avoids the "find by id, splice, replace" dance.

**Rule 4: The URL is part of your state.** Search query, active tab, pagination, selected item — if it affects the view and should survive a reload, it goes in the URL.

**Rule 5: Persist judiciously.** `localStorage` is small (usually 5-10MB, quota enforced by the browser). User preferences belong there; cached API responses usually do not. Use IndexedDB (Day 13) for anything larger than a few kilobytes.

**Rule 6: Optimistic updates need rollback.** When you update the UI immediately and the server call fails, you need to revert. This is why the undo/redo pattern from Part 7 is useful even for non-undo features — it gives you "checkpoint before this action" for free.

**Rule 7: Assume the network can fail.** Every async call can timeout, get a 500, or have the user abandon it. Your state model needs to represent all four: idle, loading, success, error. The `asyncSignal` pattern in Part 5 formalises this.

**Rule 8: Effects should be idempotent.** If the effect runs twice with the same input, it should produce the same observable state. This is how React's Strict Mode works and why it catches subtle bugs — it intentionally double-runs effects to surface non-idempotency.

**Rule 9: Debugging is part of the design.** Expose `window.appStore = store` in dev mode. Log every state transition. Name your effects. The payoff — being able to paste a state snapshot into a bug report — is enormous.

**Rule 10: Start small. Add complexity only when forced.** A `let` variable is state. A `Map` is state. A single `reactive()` object is state. Ninety percent of apps never outgrow the last step. The other ten percent will be obvious when they need more.

## Part 12: A Complete Example — The Tag Filter

Let us finish with something you will actually write. A tag filter on a blog archive, with live URL sync, persisted active tags, and undo:

```javascript
// @ts-check
import { reactive } from "@app/utils/reactive.js";
import { effect, computed } from "@app/utils/signals.js";
import { createHistory } from "@app/utils/history.js";

/** @typedef {{ id: string, title: string, tags: string[] }} Post */

/** @param {Post[]} allPosts */
export function setupTagFilter(allPosts) {
  // 1. Parse initial active tags from URL
  const initial = new URLSearchParams(location.search).getAll("tag");

  // 2. State
  const state = reactive({
    activeTags: new Set(initial),
  });

  // 3. History
  const history = createHistory(state);

  // 4. Derived: posts matching the filter
  const filteredPosts = computed(() => {
    const active = state.activeTags;
    if (active.size === 0) return allPosts;
    return allPosts.filter(post =>
      post.tags.some(tag => active.has(tag))
    );
  });

  // 5. Mutation helpers
  function toggleTag(tag) {
    history.snapshot();
    if (state.activeTags.has(tag)) state.activeTags.delete(tag);
    else state.activeTags.add(tag);
    // Trigger: reactive Set doesn't fire without reassignment
    state.activeTags = new Set(state.activeTags);
  }

  // 6. Sync to URL
  effect(() => {
    const url = new URL(location.href);
    url.searchParams.delete("tag");
    for (const tag of state.activeTags) url.searchParams.append("tag", tag);
    history.replaceState?.({}, "", url);
  });

  // 7. Render tag chips
  const chipContainer = document.getElementById("tag-chips");
  const allTags = [...new Set(allPosts.flatMap(p => p.tags))].sort();

  effect(() => {
    chipContainer.innerHTML = "";
    for (const tag of allTags) {
      const chip = document.createElement("button");
      chip.type = "button";
      chip.textContent = tag;
      chip.dataset.tag = tag;
      chip.toggleAttribute("aria-pressed", state.activeTags.has(tag));
      chip.addEventListener("click", () => toggleTag(tag));
      chipContainer.append(chip);
    }
  });

  // 8. Render filtered list
  const listContainer = document.getElementById("posts");
  effect(() => {
    listContainer.innerHTML = "";
    for (const post of filteredPosts()) {
      const li = document.createElement("li");
      li.textContent = post.title;
      listContainer.append(li);
    }
  });

  // 9. Keyboard: Ctrl-Z / Cmd-Z for undo
  document.addEventListener("keydown", (event) => {
    if ((event.ctrlKey || event.metaKey) && event.key === "z") {
      event.preventDefault();
      event.shiftKey ? history.redo() : history.undo();
    }
  });

  return { state, history };
}
```

Sixty lines for a fully working tag filter with:

- URL as source of truth.
- Reactive state, deriving filtered posts automatically.
- Tag chips with `aria-pressed` state.
- History with Ctrl-Z / Ctrl-Shift-Z.
- All keyboard-accessible, all screen-reader-friendly.

No Redux. No Zustand. No React. No framework at all. Sixty lines on top of ~200 lines of reusable primitives. The whole thing runs in any evergreen browser with zero build step.

This is what "state management without a library" looks like in practice. It is not minimal; it is full-featured. It just does not have a 70KB import.

## Part 13: When To Reach For A Library

We are not categorical. Here is when a real state library earns its keep:

1. **Very large apps (100+ routes, 1000+ components).** The extra abstraction pays back in team coordination. Redux Toolkit's patterns are well-documented and widely understood.

2. **Server-state heavy apps.** TanStack Query's caching, deduplication, and stale-while-revalidate are genuinely hard to reimplement. If you have dozens of API endpoints used from multiple screens, use it.

3. **Complex form state.** For one-off forms, the Constraint Validation API (Day 12) is fine. For hundreds of forms with shared validation rules, a library like TanStack Form or React Hook Form is justified.

4. **Optimistic concurrency with CRDTs.** Collaborative editing (Google Docs-style) needs Yjs or Automerge; do not write this yourself.

5. **Strict immutability requirements.** If your team has decided (for whatever reason) that all state must be immutable and time-travel-debuggable, Redux's discipline pays off.

For everything else — which is most apps — the primitives we built today are enough.
