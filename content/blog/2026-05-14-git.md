## Part 25 — The Three-Way Merge Algorithm: How Git Decides What Conflicts

In Part 1 of this guide we showed you that merging two branches that changed the same line produces a conflict. But we did not explain exactly how Git decides what is a conflict and what is not. This section explains the three-way merge algorithm line by line.

### 25.1 — Why "Three-Way"?

When Git merges two branches, it does not just compare the two branch tips. It compares three versions:

1. **The merge base** — the most recent common ancestor commit of the two branches. This is the last point where the two branches shared the same code.
2. **Ours** — the current branch (the one you are on when you run `git merge`). In Visual Studio, this is the "left side" of the merge editor.
3. **Theirs** — the branch being merged in. In Visual Studio, this is the "right side."

Git finds the merge base using an algorithm that traverses the commit graph backward from both branch tips until it finds the earliest commit that is an ancestor of both. In most cases, this is the commit where the branch was created.

### 25.2 — The Line-by-Line Decision

For every line in every file, Git makes one of four decisions:

**Case 1: No one changed the line.** The line is the same in the base, ours, and theirs. Git keeps it. No conflict.

**Case 2: Only "ours" changed the line.** The line in "theirs" matches the base, but "ours" is different. Git takes our version. No conflict.

**Case 3: Only "theirs" changed the line.** The line in "ours" matches the base, but "theirs" is different. Git takes their version. No conflict.

**Case 4: Both changed the line to the same thing.** The base is different from both, but "ours" and "theirs" are identical. Git takes the shared version. No conflict.

**Case 5: Both changed the line to different things.** The base is different from both, and "ours" and "theirs" disagree. Git cannot decide. **Conflict.**

Let us work through a concrete example. Suppose the base version of a file has these lines:

```
line 1: apple
line 2: banana
line 3: cherry
line 4: date
line 5: elderberry
```

"Ours" changed the file to:

```
line 1: apple
line 2: BLUEBERRY
line 3: cherry
line 4: date
line 5: FIG
```

"Theirs" changed the file to:

```
line 1: AVOCADO
line 2: banana
line 3: cherry
line 4: DRAGONFRUIT
line 5: GRAPE
```

Now Git goes line by line:

- **Line 1:** Base is `apple`. Ours is `apple` (unchanged). Theirs is `AVOCADO` (changed). Only theirs changed → take theirs → `AVOCADO`. No conflict.
- **Line 2:** Base is `banana`. Ours is `BLUEBERRY` (changed). Theirs is `banana` (unchanged). Only ours changed → take ours → `BLUEBERRY`. No conflict.
- **Line 3:** Base is `cherry`. Ours is `cherry`. Theirs is `cherry`. Nobody changed it → keep it → `cherry`. No conflict.
- **Line 4:** Base is `date`. Ours is `date` (unchanged). Theirs is `DRAGONFRUIT` (changed). Only theirs changed → take theirs → `DRAGONFRUIT`. No conflict.
- **Line 5:** Base is `elderberry`. Ours is `FIG` (changed). Theirs is `GRAPE` (changed). Both changed it to different things → **CONFLICT**.

The merged result would be:

```
line 1: AVOCADO
line 2: BLUEBERRY
line 3: cherry
line 4: DRAGONFRUIT
<<<<<<< HEAD
line 5: FIG
=======
line 5: GRAPE
>>>>>>> theirs
```

Four out of five lines were merged automatically. Only line 5, where both sides made incompatible changes, requires human intervention.

### 25.3 — What Counts as "the Same Line"?

Git does not actually compare files line by line in a rigid positional way. It uses a diff algorithm (by default, the "Myers" diff algorithm, named after Eugene W. Myers who published it in 1986) to determine which lines in each version correspond to which lines in the base. The algorithm finds the longest common subsequence and treats everything else as insertions or deletions.

This means that if "ours" added three new lines at the top of the file, Git does not naively think every line shifted and everything conflicts. It recognizes that lines were inserted and correctly matches the remaining lines to their counterparts in the base and "theirs."

### 25.4 — The ORT Merge Strategy

Git 2.33 (August 2021) introduced a new default merge strategy called **ORT** (Ostensibly Recursive's Twin). Before ORT, Git used the "recursive" strategy. ORT is a ground-up rewrite that is faster and handles edge cases better — particularly when files are renamed on one or both branches.

ORT can detect that a file was renamed on one branch and modified on the other. It correctly applies the modification to the renamed file instead of treating it as a delete-and-create. It also handles "directory/file conflicts" (where one branch creates a file and the other creates a directory with the same name) more gracefully.

You almost never need to specify the merge strategy manually, but if you do:

```bash
# Use the ORT strategy explicitly
git merge -s ort feature-branch

# Use the recursive strategy (old default)
git merge -s recursive feature-branch
```

### 25.5 — Merge Strategy Options

Within a merge strategy, you can pass options that change behavior:

```bash
# In case of conflict, automatically pick our version
git merge -X ours feature-branch

# In case of conflict, automatically pick their version
git merge -X theirs feature-branch
```

These are called strategy OPTIONS (note the `-X`, not `-s`). They do not prevent automatic merges — Git still merges lines that can be merged automatically. They only resolve the ambiguous cases (Case 5 above) by automatically choosing one side.

There is also a strategy (not an option) called `ours` that is completely different:

```bash
# Merge but KEEP EVERYTHING from our branch, discard all their changes
git merge -s ours feature-branch
```

This creates a merge commit that records the merge but throws away all changes from the other branch. The result is identical to the current branch. This is rarely used, but can be helpful when you want to record "we considered this branch and decided not to take any of its changes."

### 25.6 — The learningbydoing Scenario Revisited, Line by Line

Let us revisit the scenario from Part 1 with this three-way merge framework.

The base (commit A, `main` before any changes):
```
line 5: main
```

Branch-1 (commit B):
```
line 5: branch-1
```

Branch-2 (commit C):
```
line 5: branch-2
```

**Merge 1: branch-1 into main (fast-forward)**

Main is at A, branch-1 is at B. B is a direct descendant of A. Git fast-forwards — no three-way merge needed. Main now points to B.

**Merge 2: branch-2 into main (three-way merge)**

Main is now at B. Branch-2 is at C. Their common ancestor is A.

- Base (A): `main`
- Ours (B, main): `branch-1`
- Theirs (C, branch-2): `branch-2`

Line 5: Base says `main`. Ours says `branch-1` (changed from base). Theirs says `branch-2` (changed from base). Both changed, to different values. **Conflict.**

This is exactly what we observed. The math is unambiguous.


## Part 26 — Every Type of Merge Conflict You Will Encounter, with Resolution

### 26.1 — Same Line, Different Content (The Classic)

This is what we just discussed. Both branches changed the same line to different text. You see the `<<<<<<<`, `=======`, `>>>>>>>` markers. Open the file, decide which version to keep (or write something new), remove the markers, stage, and commit.

### 26.2 — One Branch Modified, Other Branch Deleted

One branch edits a file. The other branch deletes the same file.

```
CONFLICT (modify/delete): README.md deleted in feature-branch and modified in HEAD.
Version HEAD of README.md left in tree.
```

Git leaves the file in your working directory (the modified version) and tells you about the conflict. You decide:

- To keep the file: `git add README.md`
- To accept the deletion: `git rm README.md`

Then commit.

### 26.3 — Both Branches Added a File with the Same Name

Two branches independently create a file with the same path but different content.

```
CONFLICT (add/add): Merge conflict in config.json
```

Git merges the file and inserts conflict markers at every point where the content differs. Resolve like a regular content conflict.

### 26.4 — File Renamed on One Branch, Modified on the Other

Branch A renames `Utils.cs` to `Utilities.cs`. Branch B adds new methods to `Utils.cs`.

Git (with the ORT strategy) detects the rename and tries to apply Branch B's changes to the renamed file. If the changes are compatible, it succeeds automatically. If Branch B also modified lines that Branch A modified, you get a conflict inside the renamed file.

### 26.5 — File Renamed on Both Branches to Different Names

Branch A renames `Utils.cs` to `Utilities.cs`. Branch B renames `Utils.cs` to `Helpers.cs`.

```
CONFLICT (rename/rename): Utils.cs renamed to Utilities.cs in HEAD 
and to Helpers.cs in feature-branch.
```

Git cannot decide which name to use. You must choose:

```bash
# Keep the name from your branch
git add Utilities.cs
git rm Helpers.cs

# Or keep the name from their branch
git add Helpers.cs
git rm Utilities.cs
```

### 26.6 — Directory vs. File Conflict

Branch A creates a file called `docs`. Branch B creates a directory called `docs/` with files inside it. A file and a directory cannot have the same name.

Git renames one of them to avoid the collision and reports the conflict. You must restructure to resolve.

### 26.7 — Binary File Conflicts

Git cannot merge binary files (images, compiled DLLs, PDFs). If both branches modified the same binary file:

```
warning: Cannot merge binary files: logo.png
CONFLICT (content): Merge conflict in logo.png
```

Git keeps one version in your working directory. You must manually choose which version to keep:

```bash
# Keep our version
git checkout --ours logo.png
git add logo.png

# Keep their version
git checkout --theirs logo.png
git add logo.png
```

In Visual Studio 2026, right-click the conflicted binary file in Git Changes → choose **Take Current** or **Take Incoming**.

### 26.8 — The .csproj Merge Conflict (Every .NET Developer's Nemesis)

`.csproj` files are XML. When two branches add different NuGet packages or files, they often add XML elements at the same position in the file. This produces conflicts that look like:

```xml
<ItemGroup>
<<<<<<< HEAD
    <PackageReference Include="Newtonsoft.Json" Version="13.0.3" />
=======
    <PackageReference Include="Serilog" Version="3.1.1" />
>>>>>>> feature-logging
</ItemGroup>
```

The resolution is almost always "keep both lines":

```xml
<ItemGroup>
    <PackageReference Include="Newtonsoft.Json" Version="13.0.3" />
    <PackageReference Include="Serilog" Version="3.1.1" />
</ItemGroup>
```

**Prevention tip:** The SDK-style `.csproj` format used by modern .NET (including .NET 10) produces far fewer merge conflicts than the old verbose `.csproj` format. If you are still on the old format, migrating to SDK-style is one of the best things you can do for your Git workflow.

You can also configure a custom merge driver for `.csproj` files in `.gitattributes`, but for most teams, manually resolving the occasional conflict is faster than setting up custom tooling.

### 26.9 — The .sln and .slnx Merge Conflict

`.sln` files use a proprietary text format that is fragile and prone to conflicts. Two branches adding different projects will conflict. The resolution involves keeping both project entries and both build configuration entries.

The newer `.slnx` format (introduced with .NET 10 and Visual Studio 2026) is simpler XML and produces fewer conflicts because the file is shorter and more regular.

### 26.10 — Conflicts in Auto-Generated Files

Files like `package-lock.json`, `yarn.lock`, or `.designer.cs` are auto-generated and difficult to merge manually. The best strategy is:

1. Accept one side entirely (usually theirs, since you will regenerate anyway)
2. Regenerate the file by running the appropriate tool (`npm install`, `dotnet build`, etc.)
3. Stage and commit

```bash
# Accept theirs for the lock file
git checkout --theirs package-lock.json
# Regenerate
npm install
git add package-lock.json
git commit
```

### 26.11 — Conflict Resolution in Visual Studio 2026

When Visual Studio detects conflicts after a merge or rebase, it shows a yellow warning banner at the top of the Git Changes window. Conflicted files show a warning icon.

**Option 1: Visual Studio's built-in merge editor.** Double-click a conflicted file. Visual Studio opens a three-pane merge editor:
- Left pane: **Incoming Changes** (theirs)
- Right pane: **Current Changes** (ours)
- Bottom pane: **Result** (the merged output)

You can click checkboxes next to each conflict region to accept the left side, right side, or both. You can also edit the result pane directly. When done, click **Accept Merge** at the top.

**Option 2: Use an external merge tool.** Configure an external tool like Beyond Compare, KDiff3, or Visual Studio Code. More on this in a later section.

**Option 3: Accept one side entirely.** Right-click the conflicted file → **Accept Current** (keep ours) or **Accept Incoming** (keep theirs).


## Part 27 — The Complete tf.exe to Git Command Mapping

If you have used TFS from the Developer Command Prompt, you know the `tf` command. This section maps every commonly used `tf` command to its Git equivalent.

### 27.1 — Workspace and Repository Operations

| TFS Command | What It Does | Git Equivalent |
|---|---|---|
| `tf workspace /new` | Create a new workspace | `git clone <url>` |
| `tf workspace /delete` | Delete a workspace | Delete the local folder |
| `tf workspaces` | List workspaces | `git remote -v` (list remotes) |
| `tf get` | Download latest from server | `git pull` |
| `tf get /version:C12345` | Get a specific changeset version | `git checkout <hash>` |
| `tf get /recursive` | Get latest recursively | `git pull` (always recursive) |

### 27.2 — Checking In and Committing

| TFS Command | What It Does | Git Equivalent |
|---|---|---|
| `tf checkin` | Check in pending changes | `git add . && git commit -m "msg" && git push` |
| `tf checkin /comment:"msg"` | Check in with a comment | `git commit -m "msg" && git push` |
| `tf checkin /noprompt` | Skip the check-in dialog | Same commands (no dialog in CLI) |
| `tf status` | Show pending changes | `git status` |
| `tf checkout file.cs` | Check out (make writable) | No equivalent needed (files are always writable) |
| `tf undo` | Undo pending changes | `git restore .` |
| `tf undo /recursive` | Undo all pending changes | `git restore . && git clean -fd` |

### 27.3 — Shelving and Stashing

| TFS Command | What It Does | Git Equivalent |
|---|---|---|
| `tf shelve "name"` | Shelve pending changes | `git stash push -m "name"` |
| `tf shelve /delete "name"` | Delete a shelveset | `git stash drop stash@{N}` |
| `tf unshelve "name"` | Restore shelved changes | `git stash pop` or `git stash apply` |
| `tf shelvesets` | List shelvesets | `git stash list` |

### 27.4 — Branching and Merging

| TFS Command | What It Does | Git Equivalent |
|---|---|---|
| `tf branch $/src $/dest` | Create a branch (server folder copy) | `git switch -c new-branch` |
| `tf merge $/src $/dest` | Merge changes | `git merge source-branch` |
| `tf merge $/src $/dest /recursive` | Merge recursively | `git merge source-branch` (always recursive) |
| `tf merge /candidate` | See what changesets can be merged | `git log main..feature --oneline` |
| `tf merge /discard` | Merge and discard (mark as merged without taking changes) | `git merge -s ours feature` |
| `tf branches` | Show branch hierarchy | `git branch -a` + `git log --graph` |

### 27.5 — History and Information

| TFS Command | What It Does | Git Equivalent |
|---|---|---|
| `tf history` | Show changeset history | `git log` |
| `tf history /itemmode` | History for a specific item | `git log -- file.cs` |
| `tf history /noprompt` | Output to console | `git log` (always console) |
| `tf changeset 12345` | Show a specific changeset | `git show <hash>` |
| `tf annotate file.cs` | Show who changed each line | `git blame file.cs` |
| `tf diff` | Show differences | `git diff` |
| `tf diff /version:C100~C200` | Diff between changesets | `git diff <hash1>..<hash2>` |

### 27.6 — Labels and Tags

| TFS Command | What It Does | Git Equivalent |
|---|---|---|
| `tf label "v1.0"` | Create a label | `git tag -a v1.0 -m "Version 1.0"` |
| `tf labels` | List labels | `git tag` |
| `tf label /delete "v1.0"` | Delete a label | `git tag -d v1.0` |

### 27.7 — Locking

| TFS Command | What It Does | Git Equivalent |
|---|---|---|
| `tf lock file.cs` | Lock a file | No core equivalent. Use Git LFS locking. |
| `tf lock /unlock file.cs` | Unlock a file | `git lfs unlock file.cs` |

### 27.8 — Properties and Info

| TFS Command | What It Does | Git Equivalent |
|---|---|---|
| `tf info file.cs` | Show file info (server path, version, etc.) | `git log -1 file.cs` + `git ls-files file.cs` |
| `tf properties file.cs` | Show file properties | No direct equivalent |
| `tf localversions` | Show local file versions | `git status` + `git log -1` |


## Part 28 — Team Explorer (Old) vs. the New Git Experience in Visual Studio

### 28.1 — The Old Team Explorer

If you have used Visual Studio 2017 or 2019 with TFS or Git, you know Team Explorer. It was a side panel that handled both TFS and Git through the same interface. It had tabs for Home, Connect, Changes, Branches, Sync, Settings, and more. It worked, but it was clunky. The Git support felt bolted-on because it was — Team Explorer was originally designed for TFS.

### 28.2 — The New Git Experience

Starting with Visual Studio 2019 version 16.8 (November 2020), Microsoft introduced a new Git experience that replaced Team Explorer for Git operations. By Visual Studio 2022, the new experience was the default. In Visual Studio 2026, Team Explorer is still available for TFVC connections, but Git uses the dedicated Git menu, Git Changes window, and Manage Branches window exclusively.

The new experience is faster, more intuitive, and designed specifically for Git workflows. Key improvements:

- **Top-level Git menu** instead of buried Team Explorer options
- **Git Changes window** that shows status at a glance
- **Branch picker** in the status bar for instant switching
- **Inline diff viewer** when hovering over changed lines
- **Integrated merge conflict resolution** with a proper three-pane merge editor
- **Repository picker** when working with multiple repos
- **Commit graph** visualization in the branch history view

### 28.3 — If You Still Need Team Explorer

Team Explorer is still there in Visual Studio 2026 for connecting to TFS/TFVC repositories. You can open it from **View** → **Team Explorer**. But for Git repositories, use the Git menu and Git Changes window. Do not mix the two — it will only confuse you.


## Part 29 — Configuring External Merge and Diff Tools

### 29.1 — Why Use an External Tool?

Visual Studio's built-in merge editor is good enough for most conflicts. But some developers prefer specialized tools that offer features like:

- Three-way merge with a fourth "ancestor" pane
- Syntax highlighting during merge
- Folder-level comparison
- Better handling of long lines or large files

### 29.2 — Configuring Beyond Compare

Beyond Compare (by Scooter Software) is a popular diff and merge tool. It is not free, but it has a trial period and many enterprises have licenses.

```bash
# Set Beyond Compare as the diff tool
git config --global diff.tool bc
git config --global difftool.bc.path "C:/Program Files/Beyond Compare 5/BComp.exe"

# Set Beyond Compare as the merge tool
git config --global merge.tool bc
git config --global mergetool.bc.path "C:/Program Files/Beyond Compare 5/BComp.exe"
```

Then to use it:

```bash
# Open diff in Beyond Compare
git difftool

# Open merge conflicts in Beyond Compare
git mergetool
```

### 29.3 — Configuring Visual Studio Code

VS Code works well as both a diff and merge tool and is free:

```bash
git config --global diff.tool vscode
git config --global difftool.vscode.cmd "code --wait --diff \$LOCAL \$REMOTE"

git config --global merge.tool vscode
git config --global mergetool.vscode.cmd "code --wait --merge \$REMOTE \$LOCAL \$BASE \$MERGED"
```

### 29.4 — Configuring KDiff3

KDiff3 is a free, open-source three-way merge tool:

```bash
git config --global merge.tool kdiff3
git config --global mergetool.kdiff3.path "C:/Program Files/KDiff3/kdiff3.exe"
```

### 29.5 — Preventing .orig Backup Files

By default, after resolving conflicts with `git mergetool`, Git leaves `.orig` backup files in your working directory. To prevent this:

```bash
git config --global mergetool.keepBackup false
```


## Part 30 — Azure DevOps Branch Policies in Exhaustive Detail

Branch policies in Azure DevOps prevent unreviewed or untested code from reaching protected branches. This section covers every policy option.

### 30.1 — Accessing Branch Policies

In Azure DevOps: **Repos** → **Branches** → click the three dots (`...`) next to the branch you want to protect (usually `main`) → **Branch policies**.

### 30.2 — Require a Minimum Number of Reviewers

Set the minimum number of people who must approve a pull request before it can be completed. Common settings:

- **Minimum reviewers: 1** — At least one other person must approve.
- **Allow requestors to approve their own changes: Off** — The author cannot approve their own PR.
- **When new changes are pushed, reset approval votes** — If the author pushes new commits after approval, the approval is revoked and reviewers must re-approve.
- **Allow completion even if some reviewers vote "Wait" or "Reject"** — Usually left off for strict teams.

### 30.3 — Check for Linked Work Items

Require that every PR has at least one linked work item (user story, bug, task). This ensures traceability — you can always trace a code change back to the work item that required it.

### 30.4 — Check for Comment Resolution

Require that all PR comments are resolved before the PR can be completed. This prevents code from being merged while there are still outstanding review questions.

### 30.5 — Limit Merge Types

You can restrict which merge strategies are allowed:

- **Basic merge (no fast-forward):** The standard merge commit.
- **Squash merge:** All commits from the branch are squashed into one.
- **Rebase and fast-forward:** Commits are rebased onto target, then fast-forwarded.
- **Rebase with merge commit:** Rebase followed by a merge commit.

Many teams restrict to only one or two types for consistency.

### 30.6 — Build Validation

Add one or more build pipelines that must pass before the PR can be completed. This is the Azure DevOps equivalent of CI checks.

Configuration:
- **Build pipeline:** Select the pipeline to run.
- **Trigger:** Choose between "Automatic" (runs on every push to the PR) or "Manual."
- **Policy requirement:** "Required" means the build must pass. "Optional" means it runs but does not block.
- **Build expiration:** How long a build result is valid. If the PR branch or target branch changes, the build may be re-triggered.

### 30.7 — Status Checks

External services can post status updates to a PR. You can require specific status checks to pass. This is used for things like:

- SonarQube code quality analysis
- Security scanners
- License compliance checks
- Custom validation scripts

### 30.8 — Automatically Included Reviewers

Specify people or groups who are automatically added as reviewers when a PR is created. You can scope this by path — for example, require the DBA team as reviewers when any file in `src/Database/` changes.

### 30.9 — Path-Based Policies

You can set different policies for different paths in the repository. For example:
- Changes to `src/` require 2 reviewers and build validation.
- Changes to `docs/` require only 1 reviewer and no build.
- Changes to `.github/workflows/` require a specific security team reviewer.


## Part 31 — GitHub Branch Protection Rules in Exhaustive Detail

GitHub's equivalent of Azure DevOps branch policies is called "Branch protection rules."

### 31.1 — Accessing Branch Protection

**Settings** → **Branches** → **Branch protection rules** → **Add rule**. Enter the branch name pattern (e.g., `main` or `release/*` for wildcard matching).

### 31.2 — Available Rules

**Require a pull request before merging:** No direct pushes to the branch. All changes must go through a PR.

**Require approvals:** Set the number of required approving reviews (1–6).

**Dismiss stale pull request approvals when new commits are pushed:** If the author pushes new commits, previous approvals are dismissed.

**Require review from Code Owners:** If a `CODEOWNERS` file exists, require approval from the designated owner for each changed file.

**Require status checks to pass before merging:** Specify GitHub Actions workflows or external checks that must succeed.

**Require branches to be up to date before merging:** The PR branch must be up-to-date with the target branch. This prevents "merge skew" where a PR was approved but the target branch moved.

**Require conversation resolution before merging:** All PR comments must be resolved.

**Require signed commits:** All commits must have a verified GPG or SSH signature.

**Require linear history:** Only allow squash or rebase merges (no merge commits). This produces a clean, linear history on the protected branch.

**Lock branch:** Make the branch read-only. No pushes, no PR merges. Used for archived branches.

**Require deployments to succeed before merging:** If using GitHub Environments, require a successful deployment.

**Do not allow bypassing the above settings:** Even administrators must follow the rules.

### 31.3 — CODEOWNERS File

The `CODEOWNERS` file (placed in the repository root, `.github/`, or `docs/`) specifies who owns which parts of the codebase:

```
# Default owners for everything
* @tech-lead

# Database changes need DBA review
src/Database/ @dba-team

# Infrastructure changes need DevOps review
.github/workflows/ @devops-team
infra/ @devops-team

# Frontend changes
src/Frontend/ @frontend-team
```

When a PR modifies files matching a pattern, the corresponding team or person is automatically requested as a reviewer.


## Part 32 — CI/CD Pipeline Integration with Git

### 32.1 — Azure Pipelines Triggers

Azure Pipelines can be triggered by Git events. In a YAML pipeline (`azure-pipelines.yml`):

```yaml
# Trigger on push to main
trigger:
  branches:
    include:
      - main
    exclude:
      - 'feature/*'
  paths:
    include:
      - src/
    exclude:
      - docs/

# Trigger on PR to main
pr:
  branches:
    include:
      - main
  paths:
    include:
      - src/

pool:
  vmImage: 'ubuntu-latest'

steps:
  - task: UseDotNet@2
    inputs:
      packageType: 'sdk'
      version: '10.0.x'

  - script: dotnet restore
    displayName: 'Restore dependencies'

  - script: dotnet build --configuration Release
    displayName: 'Build'

  - script: dotnet test --no-build --verbosity normal
    displayName: 'Run tests'
```

**Trigger types:**
- `trigger:` — Runs when commits are pushed to specified branches.
- `pr:` — Runs when a PR is created or updated against specified branches.
- `schedules:` — Runs on a cron schedule.
- `resources:` — Triggers when another pipeline completes, a container image is updated, or a package is published.

### 32.2 — GitHub Actions Triggers

GitHub Actions uses workflow files in `.github/workflows/`:

```yaml
name: Build and Test

on:
  push:
    branches: [main]
    paths:
      - 'src/**'
      - 'tests/**'
  pull_request:
    branches: [main]
  schedule:
    - cron: '0 6 * * *'  # Daily at 6 AM UTC
  workflow_dispatch:       # Manual trigger

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
      - uses: actions/setup-dotnet@v5
        with:
          dotnet-version: '10.0.x'
      - run: dotnet restore
      - run: dotnet build --configuration Release
      - run: dotnet test --no-build --verbosity normal
```

**Key trigger events:**
- `push:` — Code pushed to a branch.
- `pull_request:` — PR opened, updated, or reopened.
- `pull_request_target:` — Like `pull_request` but runs in the context of the target branch (useful for forks).
- `schedule:` — Cron-based scheduling.
- `workflow_dispatch:` — Manual trigger with optional inputs.
- `release:` — When a GitHub Release is created.
- `tag:` — Filtered by tag patterns.

### 32.3 — Required Status Checks

Both Azure DevOps and GitHub can require that CI pipelines pass before a PR can be merged. This means:

1. Developer pushes a branch and creates a PR.
2. CI runs automatically (build, test, lint, security scan).
3. If CI fails, the merge button is disabled.
4. Developer fixes the issue, pushes again.
5. CI runs again. If it passes, the merge button is enabled.

This is the single most valuable Git workflow practice. It catches bugs before they reach the main branch. In TFS, the closest equivalent was "gated check-ins" — but gated check-ins ran AFTER you clicked Check In and held your code in a queue. Git PR checks run in parallel while you continue working.


## Part 33 — The Fork-and-Pull Model

### 33.1 — What Is a Fork?

A fork is a server-side copy of a repository under a different account. It is a concept provided by GitHub, Azure DevOps, GitLab, and Bitbucket — not a core Git feature.

When you fork a repository on GitHub:
- GitHub creates a complete copy of the repository under your account.
- You can clone your fork, make changes, and push to it freely.
- When you want to contribute changes back to the original repository, you create a PR from your fork to the original.

### 33.2 — Why Forks Exist

Forks solve a permissions problem. In the direct contribution model, every developer needs push access to the repository. In the fork model, only maintainers have push access to the main repository. Everyone else works in their own fork.

This is the standard model for open-source collaboration. If you want to contribute to an open-source project on GitHub, you fork it, make your changes, and submit a PR.

### 33.3 — Setting Up a Fork Workflow

```bash
# Step 1: Fork the repository on GitHub (click the "Fork" button in the UI)

# Step 2: Clone YOUR fork (not the original)
git clone https://github.com/YOUR-USERNAME/repo.git
cd repo

# Step 3: Add the original repository as an "upstream" remote
git remote add upstream https://github.com/ORIGINAL-OWNER/repo.git

# Verify remotes
git remote -v
# origin    https://github.com/YOUR-USERNAME/repo.git (fetch)
# origin    https://github.com/YOUR-USERNAME/repo.git (push)
# upstream  https://github.com/ORIGINAL-OWNER/repo.git (fetch)
# upstream  https://github.com/ORIGINAL-OWNER/repo.git (push)
```

### 33.4 — Keeping Your Fork Up to Date

The original repository continues to evolve after you fork it. To keep your fork current:

```bash
# Fetch the latest from the original repo
git fetch upstream

# Merge upstream's main into your main
git switch main
git merge upstream/main

# Push to your fork
git push origin main
```

Or with rebase for a cleaner history:

```bash
git switch main
git rebase upstream/main
git push origin main
```

### 33.5 — Contributing a Change

```bash
# Create a branch on your fork
git switch -c fix-typo-in-readme

# Make changes, commit
git add .
git commit -m "Fix typo in README"

# Push to your fork
git push origin fix-typo-in-readme

# Go to GitHub and create a PR from your fork's branch to the original repo's main
```


## Part 34 — Working with Multiple Remotes

### 34.1 — What Is a Remote?

A remote is a named URL that points to another copy of the repository. When you clone, Git automatically creates a remote called `origin` pointing to the URL you cloned from. You can add more remotes.

```bash
# List all remotes
git remote -v

# Add a remote
git remote add upstream https://github.com/original/repo.git

# Rename a remote
git remote rename origin backup

# Remove a remote
git remote remove upstream

# Change a remote's URL
git remote set-url origin https://github.com/new/url.git
```

### 34.2 — Common Multi-Remote Scenarios

**Scenario 1: Fork workflow** (described above). You have `origin` (your fork) and `upstream` (the original repo).

**Scenario 2: Migrating from one host to another.** You are moving from Azure DevOps to GitHub. You add the new GitHub repository as a second remote, push all branches and tags to it, then remove the old remote.

```bash
git remote add github https://github.com/org/repo.git
git push github --all
git push github --tags
git remote remove origin
git remote rename github origin
```

**Scenario 3: Mirror backup.** You push to both GitHub and GitLab for redundancy.

```bash
git remote add gitlab https://gitlab.com/org/repo.git
git push origin main
git push gitlab main
```

Or configure multiple push URLs on a single remote:

```bash
git remote set-url --add --push origin https://github.com/org/repo.git
git remote set-url --add --push origin https://gitlab.com/org/repo.git
# Now `git push origin main` pushes to both
```


## Part 35 — Git Rerere: Reuse Recorded Resolution

### 35.1 — The Problem

Imagine you are rebasing a long-lived branch. The branch has 20 commits, and rebase replays each one on top of the latest main. If you get a conflict on commit 5, you resolve it. But then you abort the rebase and try again later. You get the same conflict on commit 5 and have to resolve it again. And again. And again.

### 35.2 — The Solution: Rerere

`rerere` stands for "REuse REcorded REsolution." When enabled, Git remembers how you resolved a specific conflict and automatically applies the same resolution if it sees the same conflict again.

```bash
# Enable rerere globally
git config --global rerere.enabled true
```

With rerere enabled:
1. You hit a conflict and resolve it manually.
2. Git records the pre-conflict state and the resolution.
3. Next time Git sees the same conflict (same three versions of the same hunk), it automatically applies your previous resolution.

You still need to verify and stage the result, but you do not have to re-resolve the same conflict from scratch.

### 35.3 — Managing Rerere

```bash
# See recorded resolutions
git rerere status

# Forget a resolution (if you resolved it wrong)
git rerere forget path/to/file.cs

# Clear all recorded resolutions
git rerere gc
```


## Part 36 — Patch Files: Sharing Changes Without a Remote

### 36.1 — What Is a Patch?

A patch is a text file that describes changes (diffs) in a standard format. You can create a patch from commits, email it to someone, and they can apply it to their repository. This was the original way the Linux kernel developers shared changes before Git hosting platforms existed.

### 36.2 — Creating Patches

```bash
# Create a patch file for the last commit
git format-patch -1

# Create patch files for the last 3 commits (one file per commit)
git format-patch -3

# Create a patch for a range of commits
git format-patch main..feature-branch

# Create a single combined patch
git format-patch main..feature-branch --stdout > changes.patch
```

Each patch file contains the commit message, author, date, and the diff. It has everything needed to recreate the commit.

### 36.3 — Applying Patches

```bash
# Apply a patch file, preserving commit metadata
git am 0001-Fix-login-bug.patch

# Apply a series of patches
git am *.patch

# Apply a raw diff (without commit metadata)
git apply changes.diff
```

### 36.4 — When to Use Patches

- Sharing changes across repositories that do not have a shared remote
- Contributing to projects that use mailing-list-based workflows (the Linux kernel still works this way)
- Archiving specific changes
- Applying a fix from an unrelated repository


## Part 37 — Git Bundle: Carrying a Repository in Your Pocket

### 37.1 — What Is a Bundle?

A Git bundle is a single file that contains part or all of a Git repository. Unlike a patch (which contains diffs), a bundle contains actual Git objects — commits, trees, blobs. You can clone from a bundle, fetch from a bundle, and use it to transport repository data without a network connection.

### 37.2 — Creating and Using Bundles

```bash
# Create a bundle of the entire repository
git bundle create repo.bundle --all

# Create a bundle of just the main branch
git bundle create main.bundle main

# Create a bundle of commits since a specific point
git bundle create update.bundle main ^v1.0

# Verify a bundle
git bundle verify repo.bundle

# Clone from a bundle
git clone repo.bundle my-project

# Fetch from a bundle into an existing repo
git fetch repo.bundle main:refs/remotes/origin/main
```

### 37.3 — When to Use Bundles

- Transferring a repository to a machine with no internet (air-gapped environments)
- Initial clone when network bandwidth is limited (copy the bundle on a USB drive)
- Incremental updates in restricted environments


## Part 38 — Shallow Clones and Partial Clones

### 38.1 — The Problem with Large Repositories

Some repositories are enormous. The Linux kernel repository has over a million commits. Cloning it downloads hundreds of megabytes of history that you may never need. If you just want the latest code to build, waiting 10 minutes for the full clone is wasteful.

### 38.2 — Shallow Clone

A shallow clone downloads only the most recent N commits:

```bash
# Clone with only the last 1 commit (just the current state)
git clone --depth 1 https://github.com/torvalds/linux.git

# Clone with the last 100 commits
git clone --depth 100 https://github.com/torvalds/linux.git
```

A shallow clone is faster and uses less disk space, but you cannot do certain operations that require full history (like `git log` going back further than the depth, or `git blame` on old code).

You can "unshallow" later:

```bash
# Download the full history
git fetch --unshallow
```

### 38.3 — Partial Clone (Blobless Clone)

Git 2.22+ introduced partial clones, which download the commit and tree objects but skip blob (file content) objects until they are needed:

```bash
# Clone without downloading file content
git clone --filter=blob:none https://github.com/org/repo.git
```

When you check out a branch or view a file, Git downloads the blobs on demand. This is much faster for initial clone and uses less disk space, while still giving you access to the full commit history.

### 38.4 — Treeless Clone

Even more aggressive — skip both blob and tree objects:

```bash
git clone --filter=tree:0 https://github.com/org/repo.git
```

This downloads only commit objects initially. Trees and blobs are fetched on demand. Useful for CI/CD where you only need to build one specific commit.

### 38.5 — Combining with Sparse Checkout

For maximum speed in a monorepo, combine partial clone with sparse checkout:

```bash
git clone --filter=blob:none --sparse https://github.com/org/monorepo.git
cd monorepo
git sparse-checkout set src/my-service tests/my-service
```

This gives you a working tree with only the files you need, and only downloads those files' content from the server.


## Part 39 — Monorepo Strategies

### 39.1 — What Is a Monorepo?

A monorepo is a single Git repository that contains multiple projects, services, or applications. Google, Meta, Microsoft, and many other large companies use monorepos. The entire MyBlazor Magazine project is itself a monorepo — it contains the Blazor web app, the content processor tool, tests, blog content, and GitHub Actions workflows all in one repository.

### 39.2 — Advantages of a Monorepo

- **Atomic changes across projects.** If you change a shared library, you can also update every consumer in the same commit.
- **Simplified dependency management.** No versioning hell between internal packages.
- **Unified CI/CD.** One pipeline that builds and tests everything.
- **Discoverability.** All code is in one place.

### 39.3 — Challenges of a Monorepo

- **Clone time.** Large repos take longer to clone (mitigated by partial/sparse clone).
- **Build time.** Building everything takes longer (mitigated by incremental builds and caching).
- **Permission management.** Everyone can see everything (mitigated by CODEOWNERS and path-based policies).
- **Git performance.** Very large repos (millions of files) can make Git slow.

### 39.4 — Tools for Large Monorepos

**Scalar** (by Microsoft, now part of Git): Optimizes Git for large repositories. It was originally created for the Windows repository (which has over 300,000 source files). Scalar enables partial clone, sparse checkout, background maintenance, and filesystem monitor integration.

```bash
# Initialize Scalar optimizations on an existing clone
scalar register

# Clone with Scalar optimizations
scalar clone https://github.com/org/huge-repo.git
```

**Git maintenance:** Git 2.29+ includes built-in maintenance tasks that keep the repository fast:

```bash
# Enable scheduled maintenance
git maintenance start

# This enables:
# - commit-graph: Speeds up log and merge-base operations
# - prefetch: Downloads objects from remotes in the background
# - loose-objects: Packs loose objects
# - incremental-repack: Optimizes pack files
# - pack-refs: Packs reference files
```


## Part 40 — Git Worktrees: Multiple Working Copies

### 40.1 — The Problem

You are deep in a feature branch with 30 modified files. Your manager says "drop everything, we need a hotfix on main." In TFS, you might have had a second workspace mapped to the main branch. In Git, you could stash your changes and switch branches, but stashing 30 files is risky and slow.

### 40.2 — Worktrees to the Rescue

Git worktrees let you check out multiple branches simultaneously in different directories, all sharing the same `.git` database:

```bash
# Add a new worktree for the hotfix
git worktree add ../hotfix-build main
cd ../hotfix-build

# You are now on main in a separate directory
# Make your fix, commit, push

# Go back to your original work
cd ../my-project
# Your feature branch is exactly as you left it

# Clean up the worktree when done
git worktree remove ../hotfix-build
```

### 40.3 — Key Worktree Rules

- Each worktree must be on a different branch. You cannot have two worktrees on `main`.
- All worktrees share the same `.git` objects, so they do not duplicate disk space.
- Commits made in any worktree are visible to all other worktrees (after fetch/pull).
- Use `git worktree list` to see all active worktrees.
- Use `git worktree prune` to clean up stale worktree references.

### 40.4 — Worktrees in Visual Studio 2026

Visual Studio 2026 supports opening worktrees as separate folders. After creating a worktree with the command line, open the worktree folder in a new Visual Studio instance. Each instance shows its own branch and changes independently.


## Part 41 — Authentication: SSH vs. HTTPS

### 41.1 — HTTPS Authentication

When you clone via HTTPS (`https://github.com/user/repo.git`), Git prompts for your username and password. But GitHub, Azure DevOps, and most platforms no longer accept plain passwords. You need either:

- **Personal Access Token (PAT):** A generated token that acts as your password. Create one in your account settings.
- **Credential Manager:** Git Credential Manager (GCM) handles authentication automatically, including multi-factor authentication and OAuth.

On Windows, GCM is included with Git for Windows and integrates with Windows Credential Manager:

```bash
git config --global credential.helper manager
```

Visual Studio 2026 handles HTTPS authentication automatically when you sign in with your Microsoft or GitHub account.

### 41.2 — SSH Authentication

When you clone via SSH (`git@github.com:user/repo.git`), Git uses SSH key pairs instead of passwords.

**Setting up SSH:**

```bash
# Generate an SSH key pair
ssh-keygen -t ed25519 -C "your.email@company.com"
# Press Enter for default file location (~/.ssh/id_ed25519)
# Enter a passphrase (or leave empty)

# Start the SSH agent
eval "$(ssh-agent -s)"

# Add your key to the agent
ssh-add ~/.ssh/id_ed25519

# Copy the public key to your clipboard
cat ~/.ssh/id_ed25519.pub
# (On Windows: clip < ~/.ssh/id_ed25519.pub)
```

Then add the public key to your GitHub or Azure DevOps account settings (SSH keys section).

**Test the connection:**

```bash
ssh -T git@github.com
# Output: Hi username! You've successfully authenticated...
```

### 41.3 — Which Should You Use?

**HTTPS** is simpler to set up and works through corporate firewalls and proxies. Recommended for most developers, especially on Windows.

**SSH** avoids the need for credential managers and tokens. Preferred by developers on macOS and Linux, and for CI/CD systems where you want key-based authentication.


## Part 42 — Dealing with Large Repositories: Performance Tuning

### 42.1 — Symptoms of a Slow Repository

- `git status` takes several seconds
- `git log` is sluggish
- `git merge` or `git rebase` takes a long time
- `git add .` takes several seconds

### 42.2 — Solutions

**Enable the filesystem monitor:**

Git 2.36+ can watch for file changes instead of scanning the entire working tree:

```bash
git config core.fsmonitor true
git config core.untrackedcache true
```

On Windows, this uses the built-in File System Monitor daemon. On macOS, it uses FSEvents. This dramatically speeds up `git status` and related commands.

**Enable the commit graph:**

The commit graph is a precomputed data structure that speeds up operations that traverse the commit history:

```bash
git commit-graph write --reachable
# Or enable automatic updates:
git config gc.writeCommitGraph true
```

**Pack references:**

If you have thousands of branches or tags, packing them into a single file speeds up reference lookups:

```bash
git pack-refs --all
```

**Repack the repository:**

Over time, a repository accumulates loose objects. Repacking compresses them:

```bash
git repack -a -d --depth=250 --window=250
```

**Run Git maintenance:**

```bash
git maintenance start
```

This schedules all the above optimizations to run automatically.

### 42.3 — The Nuclear Option: Scalar

For truly enormous repositories (hundreds of thousands of files), use Scalar:

```bash
scalar register
```

Scalar configures all of the above optimizations plus partial clone, background prefetch, and sparse checkout support.


## Part 43 — History Cleanup: git filter-repo and BFG Repo-Cleaner

### 43.1 — When You Need to Rewrite History

Sometimes you need to remove something from the entire history of a repository:
- An accidentally committed secret (API key, password)
- A large file that should never have been tracked (database dump, video file)
- Sensitive data that must be purged for compliance

### 43.2 — Using git filter-repo

`git-filter-repo` is the officially recommended tool for history rewriting (it replaced the older `git filter-branch`):

```bash
# Install
pip install git-filter-repo

# Remove a file from all history
git filter-repo --invert-paths --path secrets.json

# Remove a folder from all history
git filter-repo --invert-paths --path database-dumps/

# Remove files larger than 10 MB from all history
git filter-repo --strip-blobs-bigger-than 10M

# Replace text in all files across all history
git filter-repo --replace-text expressions.txt
# expressions.txt contains: literal:API_KEY_VALUE==>***REDACTED***
```

### 43.3 — Using BFG Repo-Cleaner

BFG is a simpler alternative that focuses on the most common use cases:

```bash
# Remove all files larger than 100 MB
java -jar bfg.jar --strip-blobs-bigger-than 100M repo.git

# Remove a specific file by name
java -jar bfg.jar --delete-files password.txt repo.git

# Replace text (passwords, API keys)
java -jar bfg.jar --replace-text passwords.txt repo.git
```

### 43.4 — After Rewriting History

History rewriting changes every commit hash from the rewritten point forward. You must force-push the rewritten history:

```bash
git push --force --all
git push --force --tags
```

All collaborators must re-clone the repository. Their existing clones will have the old (unrewritten) history and will cause conflicts if they try to push.

This is why history rewriting is a drastic operation. Rotate any exposed secrets immediately — do not rely on history rewriting alone to protect them.


## Part 44 — Disaster Recovery

### 44.1 — "I Force-Pushed and Overwrote Someone Else's Work"

If someone else pushed commits to a branch and you force-pushed over them:

1. Ask the other person to push their commits again (they still have them locally).
2. If they do not have them, check the server: GitHub and Azure DevOps keep a reflog-like record. GitHub's Events API or Azure DevOps audit logs may show the lost commit hashes.
3. If you can find the commit hash, cherry-pick or reset to recover.

**Prevention:** Use `--force-with-lease` instead of `--force`:

```bash
git push --force-with-lease
```

This only force-pushes if the remote branch is at the state you expect. If someone else pushed in the meantime, the push is rejected.

### 44.2 — "I Accidentally Deleted a Branch"

**Locally:**
```bash
# Find the commit it pointed to
git reflog
# Look for "checkout: moving from branch-name to ..."

# Recreate the branch
git branch branch-name abc1234
```

**Remotely (on GitHub/Azure DevOps):**

GitHub keeps recently deleted branches for a short period. In the Branches view, look for "Restore" buttons. Azure DevOps also has a "Deleted branches" view where you can restore.

### 44.3 — "My Repository Is Corrupted"

This is extremely rare but can happen due to disk errors or interrupted operations.

```bash
# Check repository integrity
git fsck --full

# If objects are missing, try fetching from the remote
git fetch origin

# If the remote has the same corruption, re-clone
git clone https://github.com/org/repo.git fresh-clone
```

### 44.4 — "I Am in a Detached HEAD State and Made Commits"

Detached HEAD means HEAD is pointing at a specific commit, not a branch. If you make commits in this state, they are "orphaned" — no branch references them.

```bash
# You realize you are in detached HEAD
git status
# HEAD detached at abc1234

# Option 1: Create a branch to save your commits
git switch -c save-my-work

# Option 2: If you already switched away and lost the commits
git reflog
# Find the commit hash
git branch save-my-work abc1234
```

### 44.5 — "I Rebased and Now Everything Is a Mess"

```bash
# Abort the rebase if it is still in progress
git rebase --abort

# If you completed the rebase and want to undo it
git reflog
# Find the commit hash of the branch BEFORE the rebase
git reset --hard abc1234
```

### 44.6 — "I Merged the Wrong Branch"

If you have not pushed yet:

```bash
# Undo the merge commit
git reset --hard HEAD~1
```

If you have already pushed:

```bash
# Revert the merge commit (specifying which parent to keep)
git revert -m 1 HEAD
git push
```

The `-m 1` tells Git to keep the first parent (the branch you were on before the merge) as the mainline.


## Part 45 — Git Attributes: Beyond .gitignore

### 45.1 — What Is .gitattributes?

While `.gitignore` tells Git which files to ignore, `.gitattributes` tells Git how to handle files that are tracked. It controls:

- Line ending normalization
- Diff rendering
- Merge behavior
- Binary vs. text detection
- Git LFS tracking

### 45.2 — Line Ending Normalization (In Depth)

This is the most important use of `.gitattributes` for cross-platform teams.

```gitattributes
# Normalize all text files on commit (convert CRLF → LF in the repo)
# and convert to the OS-native line ending on checkout
* text=auto

# Force these file types to always use LF, even on Windows
*.cs text eol=lf
*.csproj text eol=lf
*.json text eol=lf
*.xml text eol=lf
*.yml text eol=lf
*.md text eol=lf
*.razor text eol=lf
*.css text eol=lf
*.js text eol=lf
*.html text eol=lf
*.sh text eol=lf

# Force batch files to use CRLF (Windows-native)
*.bat text eol=crlf
*.cmd text eol=crlf

# Binary files (do not normalize or diff)
*.png binary
*.jpg binary
*.gif binary
*.ico binary
*.woff binary
*.woff2 binary
*.ttf binary
*.eot binary
*.pdf binary
*.zip binary
*.dll binary
*.exe binary
```

**How it works:** When you commit a file, Git normalizes line endings to LF (Unix-style) in the repository. When you check out a file, Git converts to the line ending specified by `eol=` (or your OS default if using `text=auto`). This means the repository always has consistent LF endings, regardless of which OS the developer is on.

**Fixing existing line endings after adding .gitattributes:**

```bash
# Renormalize all files
git add --renormalize .
git commit -m "Normalize line endings"
```

### 45.3 — Custom Diff Drivers

You can tell Git to use a different tool for diffing specific file types:

```gitattributes
# Use a custom diff driver for images
*.png diff=exif
```

Configure the driver in `.gitconfig`:

```bash
git config diff.exif.textconv exiftool
```

Now `git diff` on a PNG file will show EXIF metadata changes instead of a useless binary diff.

### 45.4 — Custom Merge Drivers

You can define custom merge strategies for specific file types:

```gitattributes
# Always use "ours" version for package-lock.json
package-lock.json merge=ours
```

Configure the driver:

```bash
git config merge.ours.driver true
```

This automatically resolves conflicts in `package-lock.json` by keeping "ours." You would then regenerate the lock file with `npm install`.

### 45.5 — Export Ignore

Mark files that should not be included in `git archive` exports:

```gitattributes
.gitignore export-ignore
.gitattributes export-ignore
tests/ export-ignore
.github/ export-ignore
```

### 45.6 — Linguist Overrides (GitHub)

GitHub uses the Linguist library to detect languages and calculate repository statistics. You can override it:

```gitattributes
# Mark vendor files so they are not counted in language stats
vendor/* linguist-vendored
docs/generated/* linguist-generated

# Override language detection
*.razor linguist-language=C#
```


## Part 46 — Author vs. Committer, and Date Manipulation

### 46.1 — Author and Committer Are Separate

Every Git commit has TWO identities:

- **Author:** The person who originally wrote the change.
- **Committer:** The person who committed the change to the repository.

Usually these are the same person. But they can differ:

- When you cherry-pick someone else's commit, the author is the original person and the committer is you.
- When a maintainer applies a patch from an email, the author is the patch sender and the committer is the maintainer.
- When you rebase, the author stays the same but the committer changes to you (because you are creating new commits).

```bash
# See both author and committer
git log --format="%H %an <%ae> %cn <%ce>"
```

### 46.2 — Author Date vs. Committer Date

Similarly, every commit has two dates:

- **Author date:** When the change was originally written.
- **Committer date:** When the commit was created in its current form.

After a rebase, the author date stays the same (preserving when the work was originally done), but the committer date updates to the current time.

```bash
# See both dates
git log --format="%H %ai %ci"
```

### 46.3 — Overriding Dates

You can set dates manually (useful for migration or special scenarios):

```bash
# Set the author date
GIT_AUTHOR_DATE="2026-01-01T12:00:00" git commit -m "Message"

# Set the committer date
GIT_COMMITTER_DATE="2026-01-01T12:00:00" git commit -m "Message"
```


## Part 47 — Signed Commits and Tags (In Depth)

### 47.1 — Why Sign Commits?

When you see a commit by "John Developer <john@example.com>," how do you know John actually made that commit? Anyone can set their `user.name` and `user.email` to anything. Commit signing uses cryptographic signatures to prove that a commit came from the person it claims to come from.

### 47.2 — GPG Signing

```bash
# Generate a GPG key
gpg --full-generate-key
# Choose RSA and RSA, 4096 bits, key does not expire

# List your keys
gpg --list-secret-keys --keyid-format=long
# Copy the key ID (the hex string after "sec rsa4096/")

# Configure Git to use the key
git config --global user.signingkey YOUR_KEY_ID
git config --global commit.gpgsign true

# Export your public key (to add to GitHub/Azure DevOps)
gpg --armor --export YOUR_KEY_ID
```

Add the public key to your GitHub account: **Settings** → **SSH and GPG keys** → **New GPG key**.

### 47.3 — SSH Signing (Git 2.34+)

Git 2.34 and later support signing commits with SSH keys (no GPG needed):

```bash
git config --global gpg.format ssh
git config --global user.signingkey ~/.ssh/id_ed25519.pub
git config --global commit.gpgsign true
```

This is simpler than GPG and uses the SSH key you may already have.

### 47.4 — Verification

On GitHub, signed commits show a green "Verified" badge. On Azure DevOps, signed commits are indicated in the commit details.

```bash
# Verify a commit signature
git verify-commit abc1234

# Verify a tag signature
git verify-tag v1.0

# Show signature info in log
git log --show-signature
```


## Part 48 — Orphan Branches

### 48.1 — What Is an Orphan Branch?

An orphan branch is a branch with no parent commits. It starts with a fresh, empty history. The first commit on an orphan branch has no parent — it is a new root.

```bash
git checkout --orphan gh-pages
git rm -rf .
echo "<h1>GitHub Pages</h1>" > index.html
git add index.html
git commit -m "Initial GitHub Pages commit"
```

### 48.2 — When to Use Orphan Branches

- **GitHub Pages:** A `gh-pages` branch with only built HTML, separate from the source code history.
- **Documentation:** A separate history for generated documentation.
- **Unrelated content in the same repo:** Configuration files that have no relationship to the source code history.

The MyBlazor Magazine repository could, for example, use an orphan branch for deployment artifacts if it separated build output from source.


## Part 49 — Git Notes

### 49.1 — What Are Git Notes?

Git notes let you attach metadata to commits without modifying the commits. Since commits are immutable (changing anything changes the hash), notes are stored separately and linked to commits by hash.

```bash
# Add a note to a commit
git notes add -m "Reviewed by QA" abc1234

# Show notes for a commit
git log --show-notes

# Edit a note
git notes edit abc1234

# Remove a note
git notes remove abc1234
```

### 49.2 — Use Cases

- QA signoff without modifying commit history
- Linking commits to external ticket systems
- Build metadata (build number, artifact URL)
- Performance benchmark results

Notes are stored in `refs/notes/` and can be pushed and pulled like branches:

```bash
git push origin refs/notes/*
git fetch origin refs/notes/*:refs/notes/*
```


## Part 50 — More Real-World Scenarios

### 50.1 — Scenario: "I Need to Squash My 15 WIP Commits Before Creating a PR"

You committed 15 times over three days with messages like "WIP," "more WIP," "fix typo," "oops," "actually fix it this time." Before creating a PR, you want to clean this up.

```bash
# Count how many commits you have ahead of main
git log main..HEAD --oneline
# 15 commits

# Interactive rebase
git rebase -i main
```

The editor opens. Change all but the first `pick` to `squash` (or `fixup` if you want to discard the messages):

```
pick abc1234 Start implementing user registration
fixup def5678 WIP
fixup 111aaaa more WIP
fixup 222bbbb fix typo
fixup 333cccc oops
fixup 444dddd actually fix it this time
...
```

Save and close the editor. Git combines all 15 commits into one. You get a chance to write a new commit message:

```
Implement user registration with email validation

- Add registration form with client-side validation
- Add server-side email uniqueness check
- Add confirmation email flow
- Add unit tests for RegistrationService

Refs: JIRA-5678
```

Push (force-push since you rewrote history):

```bash
git push --force-with-lease
```

### 50.2 — Scenario: "I Need to Split One Commit Into Two"

You made a commit that does two unrelated things. Reviewers asked you to split it.

```bash
# Interactive rebase
git rebase -i HEAD~1

# Change "pick" to "edit" for the commit you want to split
# Save and close

# Git stops at that commit. Undo it but keep changes:
git reset HEAD~1

# Now stage and commit the first part
git add src/Login.cs tests/LoginTests.cs
git commit -m "Fix login validation"

# Stage and commit the second part
git add src/Registration.cs tests/RegistrationTests.cs
git commit -m "Add registration email check"

# Continue the rebase
git rebase --continue
```

### 50.3 — Scenario: "I Need to Move a File and Preserve Its History"

Git does not explicitly track renames. Instead, it detects renames heuristically after the fact (by comparing file content similarity). If you move a file, Git will show it as a delete-and-add, but `git log --follow` will follow the rename:

```bash
git mv src/OldName.cs src/NewName.cs
git commit -m "Rename OldName to NewName"

# View history across the rename
git log --follow src/NewName.cs
```

In Visual Studio 2026, if you rename a file through Solution Explorer, it automatically stages both the delete and the add. Git will detect the rename.

### 50.4 — Scenario: "I Need to Undo a Merge That Was Already Pushed"

```bash
# Find the merge commit
git log --oneline --merges
# a1b2c3d Merge branch 'feature-bad' into main

# Revert the merge, keeping the first parent (main's history)
git revert -m 1 a1b2c3d
git push
```

**Important caveat:** After reverting a merge, if you later try to merge the same branch again, Git thinks those changes are already in main (because they were, then they were reverted). To re-merge, you must first revert the revert:

```bash
# Revert the revert
git revert <hash-of-the-revert-commit>
# Now you can merge the branch again
git merge feature-bad
```

### 50.5 — Scenario: "Two Developers Are Working on the Same Feature Branch"

If two people are pushing to the same feature branch, they will run into the same push-rejection problem as pushing to main. The solution:

```bash
# Before pushing, pull with rebase
git pull --rebase origin feature-shared
git push
```

This rebases your local commits on top of whatever your colleague pushed, then pushes the result. It keeps the branch history linear.

Alternatively, create sub-branches:

```bash
# Developer A works on feature-login-frontend
# Developer B works on feature-login-backend
# Both merge into feature-login through PRs
```

### 50.6 — Scenario: "I Need to Copy a File from Another Branch Without Merging"

```bash
# Check out a single file from another branch
git checkout feature-branch -- src/Utils.cs

# The file is now in your working directory and staged
git commit -m "Copy Utils.cs from feature-branch"
```

This is different from cherry-pick (which copies a commit's changes) and merge (which combines branch histories). This literally copies the file's current state from another branch.

### 50.7 — Scenario: "I Want to See What Changed Between Two Tags"

```bash
# List files that changed
git diff --name-only v1.0..v2.0

# Show the full diff
git diff v1.0..v2.0

# Show the commit log between two tags
git log v1.0..v2.0 --oneline

# Count commits between tags
git rev-list v1.0..v2.0 --count
```

### 50.8 — Scenario: "I Need to Bisect to Find When a Bug Was Introduced"

A detailed walkthrough:

```bash
# Start bisecting
git bisect start

# The current commit has the bug
git bisect bad

# Tag v3.0 from last month did not have the bug
git bisect good v3.0

# Git checks out a commit halfway between. There are 100 commits
# between v3.0 and HEAD, so Git picks commit ~50.

# Test: does this commit have the bug?
# If YES:
git bisect bad
# Git now picks a commit halfway between v3.0 and commit 50 (around commit 25)

# If NO:
git bisect good
# Git picks a commit halfway between commit 50 and HEAD (around commit 75)

# After about 7 steps (log2(100) ≈ 7), Git identifies the exact commit:
# abc1234 is the first bad commit

# You can automate this with a test script:
git bisect run dotnet test --filter "TestName=LoginTest"
# Git runs the test at each bisect step automatically

# When done
git bisect reset
```

This is dramatically faster than manually checking out changesets in TFS. For 1,000 commits, bisect finds the culprit in about 10 steps.

### 50.9 — Scenario: "My .gitignore Is Not Working"

The most common reason: the files were tracked before you added them to `.gitignore`. Ignore rules only apply to untracked files.

```bash
# Remove the file from tracking (but keep it on disk)
git rm --cached offending-file.txt

# For a directory
git rm -r --cached bin/

# Commit the removal
git commit -m "Stop tracking files that should be ignored"
```

Now the ignore rule takes effect.

### 50.10 — Scenario: "I Want to Create a Release"

```bash
# Make sure main is up to date
git switch main
git pull

# Create an annotated tag
git tag -a v2.1.0 -m "Release 2.1.0 - Add user registration and email verification"

# Push the tag
git push origin v2.1.0

# If using GitHub, create a Release from the tag in the GitHub UI
# This generates release notes automatically from commit messages and PR titles
```

On GitHub, tags trigger `release:` events in GitHub Actions, so you can build and publish release artifacts automatically.

### 50.11 — Scenario: "I Accidentally Committed a Large File and Cannot Push"

GitHub rejects pushes containing files larger than 100 MB. If you committed one:

```bash
# If it was the last commit, amend it
git rm --cached huge-database.bak
echo "*.bak" >> .gitignore
git add .gitignore
git commit --amend

# If it was several commits ago, use filter-repo
git filter-repo --invert-paths --path huge-database.bak
git push --force-with-lease
```

### 50.12 — Scenario: "I Need to Work on a Feature That Depends on Another Unmerged Feature"

Feature B depends on Feature A, but Feature A's PR has not been merged yet.

```bash
# Create Feature B's branch from Feature A's branch (not from main)
git switch feature-a
git switch -c feature-b

# Work on Feature B, which includes Feature A's changes

# When Feature A is merged into main:
git switch feature-b
git rebase main
# This replays Feature B's unique commits on top of main (which now includes A)
```

This is called "stacking branches" or "branch stacking." It is common in teams with rapid development cycles.


## Part 51 — Git's Relationship to Other Version Control Systems

### 51.1 — Git vs. Subversion (SVN)

SVN is another centralized version control system, similar to TFS/TFVC. Many older open-source projects used SVN before migrating to Git. Git can interoperate with SVN using `git-svn`:

```bash
git svn clone http://svn.example.com/repo -s
```

This creates a Git repository from an SVN repository, preserving history.

### 51.2 — Git vs. Mercurial (Hg)

Mercurial was a distributed VCS created around the same time as Git (2005). It was simpler to learn but ultimately lost the adoption battle. Many former Mercurial projects (including Python itself and Facebook's monorepo) migrated to Git. Facebook created a tool called `hg-git` for interop and later contributed to extensions that made Mercurial work with Git servers.

### 51.3 — Git vs. Perforce (Helix Core)

Perforce is a centralized VCS popular in the game development industry because of its strong binary file handling and file locking. Epic Games (Unreal Engine) and many game studios use Perforce. Google also uses a Perforce-based system (Piper) internally. Git-p4 provides interop:

```bash
git p4 clone //depot/project/main
```

### 51.4 — The Convergence

As of 2026, Git has won the version control wars for general-purpose software development. TFVC, SVN, and Mercurial are all in maintenance mode or declining. Perforce retains a niche in game development. The only serious alternative to Git for new projects is not another VCS — it is the question of where to host it (GitHub, GitLab, Azure DevOps, Bitbucket, or self-hosted).


## Part 52 — Semantic Versioning and Git Tags

### 52.1 — Semantic Versioning (SemVer)

Semantic Versioning uses a `MAJOR.MINOR.PATCH` format:

- **MAJOR** — Breaking changes (v1.x → v2.0)
- **MINOR** — New features, backward-compatible (v1.1 → v1.2)
- **PATCH** — Bug fixes, backward-compatible (v1.1.0 → v1.1.1)

Pre-release versions: `v2.0.0-beta.1`, `v2.0.0-rc.1`

### 52.2 — Using Git Tags for Versions

```bash
# Tag a release
git tag -a v1.2.3 -m "Release 1.2.3"
git push origin v1.2.3

# List tags sorted by version
git tag --sort=-version:refname

# Find the latest tag
git describe --tags --abbrev=0

# Find how many commits since the last tag
git describe --tags
# Output: v1.2.3-14-gabc1234
# Meaning: 14 commits after v1.2.3, at commit abc1234
```

### 52.3 — Automating Version Numbers

In .NET projects, you can use the Git tag as the assembly version:

```xml
<!-- In your .csproj -->
<PropertyGroup>
  <Version>1.2.3</Version>
</PropertyGroup>
```

Or use tools like `GitVersion` or `MinVer` that compute the version from Git tags automatically:

```xml
<PackageReference Include="MinVer" Version="6.0.0" PrivateAssets="All" />
```

MinVer reads Git tags and sets the version number without any manual configuration.


## Part 53 — The Security Model of Git

### 53.1 — Content-Addressable Storage

Every object in Git is addressed by the SHA-1 hash of its content. If you change one bit of a file, the hash changes. If you change one bit of a commit message, the hash changes. If you change one commit, every descendant commit's hash also changes (because each commit includes its parent's hash).

This means tampering with history is detectable. You cannot surreptitiously alter an old commit — the hash chain breaks.

### 53.2 — SHA-1 vs. SHA-256

Git originally used SHA-1 (160-bit hashes). In 2017, researchers demonstrated a SHA-1 collision (the "SHAttered" attack). While this does not directly threaten Git's integrity model (the attack requires enormous computational resources and produces files with specific content), it motivated a migration to SHA-256.

Git 2.29 (October 2020) introduced experimental SHA-256 support. As of Git 2.53.0 (February 2026), SHA-256 repositories work but are not the default. Git 3.0, expected by late 2026, is planned to make SHA-256 the default for new repositories.

### 53.3 — Protecting the Remote

Git itself does not have access controls. Protection comes from the hosting platform:

- **GitHub:** Repository visibility (public/private), branch protection rules, team permissions.
- **Azure DevOps:** Project-level permissions, repository-level permissions, branch policies, security groups.
- **Self-hosted:** File system permissions, SSH key access, hook-based enforcement.


## Part 54 — A Day in the Life: A Complete TFS-to-Git Walkthrough

Let us walk through an entire day of work, side by side, showing exactly what you did in TFS and what you will do in Git.

### Morning: Arrive, Get Latest, Start Work

**TFS:**
1. Open Visual Studio 2019. Connect to TFS.
2. Right-click solution in Solution Explorer → Source Control → Get Latest Version.
3. Open the bug tracking system. Find bug #4521 — "Login page crashes when email is null."
4. Start editing `Login.aspx.cs` directly on the main branch.

**Git:**
1. Open Visual Studio 2026.
2. Git menu → Pull (or click the down-arrow in the status bar). This fetches and merges.
3. Open your work item tracker. Find bug #4521.
4. Click the branch name in the bottom-left → New Branch → type `bugfix/4521-login-null-email` → Create.
5. Edit `Login.aspx.cs` on your new branch.

### Mid-Morning: Colleague Needs Your Help

**TFS:**
1. You have pending changes. You do not want to check in yet.
2. Shelve your changes: Team Explorer → Pending Changes → Shelve → name it "WIP 4521."
3. Help your colleague. They show you their code on their screen.

**Git:**
1. You have uncommitted changes. Commit them (they are private on your branch):
   ```
   git add .
   git commit -m "WIP: Investigating null email crash"
   ```
2. Help your colleague. If you need to switch branches:
   ```
   git switch main
   ```
   Your branch is safe — the commit is saved locally.
3. When done:
   ```
   git switch bugfix/4521-login-null-email
   ```

### Afternoon: Finish the Fix, Submit for Review

**TFS:**
1. Unshelve your changes.
2. Finish the fix.
3. Check In: Team Explorer → Pending Changes → Write comment → Check In.
4. Your change is immediately on the server, in the main branch, affecting everyone.
5. Separately, send a code review request through TFS.

**Git:**
1. Your WIP commit is already there. Make more changes.
2. Commit:
   ```
   git add .
   git commit -m "Fix null reference when user has no email address

   Added null check before accessing User.Email in LoginHandler.
   Added unit test covering the null email scenario.

   Fixes: #4521"
   ```
3. Push: `git push -u origin bugfix/4521-login-null-email`
4. Create a Pull Request in Azure DevOps or GitHub.
5. Your change is on the server but NOT on main. It will be merged after review.

### Late Afternoon: Review Feedback, Final Merge

**TFS:**
1. Reviewer comments are separate from the check-in process. The code is already on main.

**Git:**
1. Reviewer comments appear on the PR. You make a small fix:
   ```
   git add .
   git commit -m "Address review: Extract null check to helper method"
   git push
   ```
2. The PR updates automatically. Reviewer approves.
3. Click "Complete" on the PR. The branch is merged into main.
4. The feature branch is deleted automatically.
5. Switch back to main and pull:
   ```
   git switch main
   git pull
   ```

### End of Day: Clean Up

**TFS:** Nothing to do.

**Git:**
```bash
# Delete merged local branches
git branch --merged main | grep -v main | xargs git branch -d

# Prune remote tracking branches
git fetch --prune
```

## Part 55 — Final Thoughts: It Gets Easier

You have now read two comprehensive articles covering every aspect of the TFS-to-Git transition. That is a lot of material. Here is the truth: you will not need most of it on any given day. Your daily Git workflow is five commands: pull, branch, add, commit, push. Everything else is for when something unusual happens — and now you have a reference for those situations.

The biggest adjustment is not the commands. It is the mental model. In TFS, you thought "check out from the server, edit, check in to the server." In Git, you think "everything is local until I push." Once that shift clicks, the rest follows naturally.

Your team will make mistakes during the transition. Someone will push to main directly. Someone will forget to pull before pushing. Someone will create a merge conflict and panic. These are not failures — they are learning moments. Every Git expert has an embarrassing story about force-pushing to main or deleting a branch they should not have deleted.

The Git community has a saying: "it is almost impossible to permanently lose work in Git." Between the reflog, the object database, and the remote server, your work is backed up in multiple places. Even when you think you have destroyed something, there is almost always a way to recover it.

Be patient with yourself. Be patient with your team. Bookmark this article. And remember: if all else fails, you can always delete the folder and clone again. Nobody will judge you. We have all been there.

Good luck. You are going to do great.
