# The Complete Guide to Relational Databases, SQL, and Normalization: From First Principles to Third Normal Form and Beyond

Let us begin with a confession.

You have almost certainly written bad database code. Not because you are a bad person. Not because you lack talent or ambition. You have written bad database code because nobody sat you down and explained, slowly and patiently, why the way you are storing data is quietly building a trap that future-you (and future-colleagues-you) will step into at the worst possible moment.

This article is that explanation. It is long. It does not apologize for being long. Understanding data storage is arguably the most consequential skill a software developer can develop, because every line of code you will ever write in your career either reads data, writes data, or transforms data in transit between those two operations. If the foundation on which that data rests is cracked, everything built on top of it eventually sinks.

We are going to start from the very beginning. We are going to talk about what data even is, why we need special tools to store it, and what the relational model is and why it represents one of the most significant intellectual achievements in the history of computing. Then we are going to work through SQL — the language you use to talk to relational databases — systematically and completely. Then we are going to cover normalization: the discipline of designing tables in ways that prevent a whole class of silent, insidious data corruption bugs that you probably already have in your database and do not know about yet.

There will be code. A lot of code. There will be bad code, and we will explain why it is bad. There will be good code, and we will explain why it is better. There will be SQL. There will be C#. There will be some ASP.NET. Every single example is real.

By the end of this article you will understand why every database book written in the last fifty years keeps talking about "the key, the whole key, and nothing but the key" — and more importantly, you will understand how to apply that understanding to the real software you write every day.

Let us begin.

## Part 1: Why Databases Exist at All — The Problem With Just Storing Things

### 1.1 The Pre-Database World

Before there were databases, there were files. Programs that needed to store data wrote that data to files on disk. A payroll program would write employee records to a file. A billing program would write invoice records to a different file. An inventory program would write product records to a third file.

This worked, after a fashion, for small problems. But it immediately created a cascade of problems that software developers spent decades fighting.

**Problem One: Duplication.** The payroll program stored employee names and addresses. The billing program also needed employee names and addresses, so it stored its own copy. The HR system stored them too. When an employee moved, the person updating the address had to know about every single place that address was stored. They often did not. So databases drifted out of sync with reality, and nobody knew.

**Problem Two: No shared access.** Files are owned by programs. If the payroll program had an employee file open, the billing program could not safely read it at the same time, let alone write to it. Multi-user access was essentially impossible without extremely fragile and custom-built locking schemes.

**Problem Three: No structure enforcement.** A file is bytes. Nothing about the file itself enforces that record number 1,500 has all its fields filled in correctly, that the employee ID is unique, or that the department code refers to a department that actually exists. You could store nonsense. Nothing stopped you. Programs spent enormous amounts of code checking for nonsense.

**Problem Four: Physical coupling.** Programs knew exactly where on disk their data lived, how it was laid out in bytes, and in what order records were stored. If you decided to add a new field to an employee record, you had to rewrite every program that read that file — not because the data itself required changing, but because those programs were encoded with specific knowledge of the physical layout of bytes on disk.

These are not minor inconveniences. They are architectural catastrophes that caused real software projects to fail. The payroll department ends up paying people at their old address. The inventory system shows products in a warehouse that was decommissioned two years ago. The billing system charges customers at a rate that was updated in one place but not another.

### 1.2 The Hierarchical and Network Models

Before relational databases, two models dominated: hierarchical and network.

**Hierarchical databases** (exemplified by IBM's IMS, Information Management System, which launched around 1968 and was famously used for the Apollo program) organized data as trees. A company had departments; departments had employees; employees had dependents. Each record existed in a strict parent-child hierarchy. Querying was fast if you knew exactly where you were in the tree. It was nearly impossible if you wanted to traverse the data in a direction the hierarchy did not anticipate. What if you wanted all employees whose salary exceeded a threshold, regardless of department? You had to traverse every branch of every tree. What if an employee could belong to multiple departments? You had to duplicate the employee record under each department, reintroducing exactly the duplication problem you were trying to solve.

**Network databases** (following the CODASYL standard from 1969) were more flexible. Instead of strict parent-child trees, records could participate in multiple relationships simultaneously. An employee could belong to multiple departments without duplication. But the query model required programmers to navigate the network manually, following pointers from record to record, in procedural code. The programmer was still responsible for knowing the physical layout. You wrote traversal code, not declarations about what you wanted. And when the data structure changed, all that traversal code broke.

Both models suffered from what we now call physical data dependence: your application code was coupled to the physical organization of the data on disk. Change the storage, change the code. This is a disaster for maintainability.

### 1.3 Enter Edgar F. Codd

In June 1970, a British computer scientist named Edgar Frank Codd, working at IBM's San Jose Research Laboratory, published a paper titled "A Relational Model of Data for Large Shared Data Banks" in the journal *Communications of the ACM*. It is, without exaggeration, one of the most important papers in the history of computer science.

Codd's insight was both mathematically rigorous and practically revolutionary. He proposed organizing data not as trees or networks, but as **relations** — two-dimensional tables of rows and columns where:

1. Each column had a name and represented a specific attribute.
2. Each row represented a single instance of the entity being described.
3. The order of rows did not matter.
4. The order of columns did not matter.
5. Every row in a table was unique (no duplicate rows allowed).
6. Each cell contained exactly one atomic value — a single, indivisible piece of data, not a list, not a sub-structure.

This sounds almost trivially simple. It is anything but. The profound insight buried in this structure is the principle of **physical data independence**: the application does not need to know where the data is physically stored, how it is indexed, or in what order it is laid out on disk. You express what data you want, and the database figures out how to retrieve it. You describe the query; the database engine chooses the execution plan. This separation of logical structure from physical storage is what makes relational databases so enduringly powerful.

Codd grounded his model in set theory and first-order predicate logic — real mathematics, not engineering intuition. This meant that operations on relational data could be reasoned about formally. Queries could be optimized by applying mathematical equivalences. Correctness could be proven, not merely hoped for.

He also introduced the concept of a **relational algebra**: a set of operations (selection, projection, join, union, difference, and a few others) that could be composed to express arbitrarily complex queries. Any query expressible in the relational algebra was said to be relationally complete. He used this to define a minimum bar for what a database query language needed to be capable of.

Codd died on April 18, 2003, at age 79. By that point, the industry he created was worth hundreds of billions of dollars. In 1981, he received the Turing Award — computing's highest honour — for his foundational contributions.

### 1.4 How IBM Fumbled Its Own Invention

Here is a small piece of history that is worth knowing, partly because it is fascinating and partly because it explains why Oracle exists.

Codd published his paper in 1970. IBM was his employer. You would expect IBM to immediately run with this revolutionary idea. They did not. IBM had already invested heavily in IMS (the hierarchical system mentioned earlier), and the IMS product group had no interest in being made obsolete by a researcher's paper. Internal politics delayed IBM's implementation of the relational model for years.

Meanwhile, a small California company called Oracle Corporation — founded by Larry Ellison, Bob Miner, and Ed Oates — read Codd's papers and built a relational database from scratch. Oracle Version 1 shipped internally in 1977. Oracle Version 2 (the first marketed version) shipped in 1979. This was actually before IBM released its own relational product, DB2, which came to market in 1983.

This is a lesson that repeats throughout the history of technology: the person who invents a thing is not always the person who commercializes it. It is also a lesson about organizational antibodies: large organizations often kill the innovations that would otherwise eat their existing products.

## Part 2: The Relational Model — What It Actually Is

### 2.1 Relations, Tables, Tuples, and Attributes

A **relation** is a mathematical structure. In practice, it corresponds to a table. Let us be precise about the vocabulary, because sloppy vocabulary leads to sloppy thinking.

- A **relation** is the abstract, mathematical object.
- A **table** is how a relation is typically represented in a database management system (DBMS) and in SQL.
- A **tuple** is a single row in a relation — a complete, ordered set of values for one entity.
- An **attribute** is a column — a named property that all tuples in a relation share.
- A **domain** is the set of all permissible values for an attribute (e.g., the domain of an `age` attribute might be "positive integers less than 150").
- A **schema** is the named relation together with its list of attributes and their domains.

When database people say "relation," they mean something specific. Not a relationship between things (that is a different concept, confusingly), but the table itself. Formally, a relation of degree *n* (meaning *n* attributes) contains a set of tuples where each tuple has exactly *n* values, one from each attribute's domain. Since a relation is a **set**, it cannot contain duplicate tuples, and it has no inherent ordering.

This mathematical foundation matters because it is why SQL behaves the way it does. The SQL `SELECT` statement is not retrieving "records" in a file; it is computing a new relation from existing relations by applying operations from the relational algebra.

### 2.2 Keys: The Cornerstone of Everything

The concept of a **key** is probably the single most important concept in all of database design. If you understand keys deeply — not just what they are, but why they exist and what they guarantee — you understand 80% of what you need to know about designing good databases.

Let us build up from first principles.

We said that a relation cannot contain duplicate tuples. This means that in any valid relation, there must exist some combination of attributes whose values are unique across all tuples. We call this a **superkey**.

A **candidate key** is a minimal superkey — a superkey such that if you removed any one attribute from it, it would no longer be a superkey. In other words, a candidate key is the smallest possible set of attributes that still uniquely identifies every row.

From among all candidate keys, we designate exactly one as the **primary key**. This is a convention, not a mathematical requirement, but it is an extremely important convention. The primary key is how we talk about and refer to individual rows. It is what foreign keys in other tables point to.

A **foreign key** is an attribute (or combination of attributes) in one table that must match the value of the primary key (or some candidate key) in another table. It establishes a relationship between tables. The database can enforce this — if you try to insert a row in the "orders" table with a `customer_id` that does not exist in the "customers" table, the database will refuse. This enforcement is called **referential integrity**, and it is one of the most valuable things a relational database provides.

Let us make this concrete.

```sql
-- The customers table. customer_id is the primary key.
CREATE TABLE customers (
    customer_id   INT           NOT NULL,
    first_name    VARCHAR(100)  NOT NULL,
    last_name     VARCHAR(100)  NOT NULL,
    email         VARCHAR(255)  NOT NULL,
    CONSTRAINT pk_customers PRIMARY KEY (customer_id),
    CONSTRAINT uq_customers_email UNIQUE (email)
);

-- The orders table. order_id is the primary key.
-- customer_id is a foreign key referencing customers.customer_id.
CREATE TABLE orders (
    order_id      INT           NOT NULL,
    customer_id   INT           NOT NULL,
    order_date    DATE          NOT NULL,
    total_amount  DECIMAL(10,2) NOT NULL,
    CONSTRAINT pk_orders PRIMARY KEY (order_id),
    CONSTRAINT fk_orders_customer 
        FOREIGN KEY (customer_id) 
        REFERENCES customers (customer_id)
);
```

Notice a few things in this DDL (Data Definition Language):

- Every constraint is named explicitly (`pk_customers`, `uq_customers_email`, `fk_orders_customer`). You should always name your constraints. Anonymous constraints are extremely difficult to reference when you need to drop or modify them, and database error messages become much more readable.
- `NOT NULL` is specified everywhere it should be. The absence of `NOT NULL` means the column allows `NULL`, which is a source of endless confusion we will discuss at length later.
- The foreign key constraint means the database will enforce that every `customer_id` in `orders` must exist in `customers`. You literally cannot insert an order for a nonexistent customer.

This structural enforcement — the database refusing to let you create nonsense data — is worth a hundred lines of validation code in your application. It works even when multiple applications are talking to the same database simultaneously. It works when a developer decides to bypass the application and manipulate data directly with a database client. It works when that "one-time script" your colleague wrote six months ago is still quietly running in a cron job.

### 2.3 NULL: The Thing That Causes More Bugs Than Anything Else

We need to have a serious conversation about `NULL`.

`NULL` in SQL does not mean zero. It does not mean empty string. It does not mean false. It means **unknown** or **inapplicable** — the absence of a value. Codd included it in his model, and Chris Date (one of the great database theorists) has spent decades arguing it was a mistake. Whether it was a mistake or not, you need to understand how it actually behaves, because it causes more bugs than almost anything else in database programming.

The fundamental problem with `NULL` is that SQL uses **three-valued logic** (3VL) when `NULL` is involved. Normal boolean logic has two values: TRUE and FALSE. In SQL, expressions involving `NULL` can evaluate to UNKNOWN, which is neither TRUE nor FALSE.

Here is the horror:

```sql
-- Does NULL equal NULL? No. NULL = NULL evaluates to UNKNOWN, not TRUE.
SELECT NULL = NULL;      -- Result: NULL (which evaluates as UNKNOWN/false in WHERE clauses)

-- Is NULL not equal to NULL? Also UNKNOWN.
SELECT NULL <> NULL;     -- Result: NULL

-- This query will NOT return any rows, even if some products have NULL price.
SELECT * FROM products WHERE price = NULL;

-- This is how you actually check for NULL.
SELECT * FROM products WHERE price IS NULL;

-- And this is how you check that it is not NULL.
SELECT * FROM products WHERE price IS NOT NULL;
```

This means that when you write `WHERE column = value`, rows where `column` is `NULL` are excluded from the result, even if `value` is also `NULL`. This trips up everyone at some point.

```sql
-- Aggregate functions silently ignore NULLs.
-- If 3 out of 10 products have NULL price, AVG(price) divides by 7, not 10.
SELECT AVG(price) FROM products;

-- COUNT(*) counts all rows. COUNT(price) counts rows where price is NOT NULL.
SELECT COUNT(*), COUNT(price) FROM products;
-- If 3 rows have NULL price, these will return different numbers.
```

The practical advice here is:

1. Apply `NOT NULL` to every column that should never be null. This is most columns. Be thoughtful and deliberate about allowing nulls, not permissive by default.
2. Never use `= NULL` in a `WHERE` clause. Always use `IS NULL` or `IS NOT NULL`.
3. When computing aggregates, be aware that `NULL` values are excluded, and decide whether that is the behavior you want.
4. Use `COALESCE(value, default)` to substitute a default value when an expression might be `NULL`. `COALESCE(price, 0)` returns `0` if `price` is `NULL`.
5. Use `NULLIF(value, unwanted)` to convert a specific value to `NULL`. `NULLIF(price, 0)` returns `NULL` if `price` is `0` — useful to avoid division-by-zero errors in expressions like `numerator / NULLIF(denominator, 0)`.

### 2.4 The Twelve Rules (Codd's Rules)

In 1985, fifteen years after his original paper, Codd published "Is Your DBMS Really Relational?" and "Does Your DBMS Run By the Rules?" in *Computerworld*. In these articles, he articulated twelve rules (often cited as "Codd's 12 Rules" or "Codd's Rules") that a database management system would need to follow to be considered truly relational. We will not reproduce all twelve in full detail, but the key points are:

1. **Information Rule**: All information in the database is represented in tables and only in tables.
2. **Guaranteed Access Rule**: Every piece of data must be accessible by table name, primary key value, and column name.
3. **Systematic Treatment of NULL**: `NULL` values must be handled consistently and independently of data type.
4. **Dynamic Online Catalog**: The database schema itself must be stored in the database as tables and queryable with the same SQL.
5. **Comprehensive Data Sublanguage Rule**: There must be a language that supports DDL, DML, data integrity, authorization, and transaction boundaries.
6. **View Updating Rule**: All theoretically updateable views must be updatable.
7. **High-Level Insert, Update, and Delete**: The language must support set-level operations (affecting multiple rows at once), not just row-at-a-time operations.
8. **Physical Data Independence**: Application programs must be insulated from changes to physical storage structures.
9. **Logical Data Independence**: Application programs must be insulated from changes to logical table structures (within the limits of non-breaking changes).
10. **Integrity Independence**: Integrity constraints must be storable in the catalog and enforced by the DBMS, not the application.
11. **Distribution Independence**: The system must work the same whether data is distributed across multiple nodes or stored in a single location.
12. **Non-Subversion Rule**: If the database provides a low-level language for row-by-row access, that language cannot be used to bypass the integrity rules of the higher-level SQL interface.

No commercial database system fully satisfies all twelve rules. SQL itself violates several of them (notably, SQL allows duplicate rows in intermediate result sets, which violates the mathematical definition of a relation). But these rules remain a useful standard against which to evaluate a system's design.

## Part 3: SQL — The Language of Relational Databases

### 3.1 What SQL Is and What It Is Not

SQL (Structured Query Language, pronounced either as "sequel" or as "S-Q-L" — both are correct, and anyone who insists otherwise is wrong) is a domain-specific language for managing and querying data in relational databases. It is not a general-purpose programming language. It does not have loops in the conventional sense. It does not have conventional variable declarations. You cannot write an HTTP server in SQL.

What SQL is is a **declarative** language. When you write a SQL query, you describe what data you want, not how to retrieve it. You say "give me all customers from New York who placed an order in the last 30 days" — you do not say "scan the customers table, for each customer check whether their city equals 'New York', then for each matching customer scan the orders table..." The database engine's query optimizer decides how to actually retrieve the data. This is the key distinction from procedural languages.

This declarative nature is both SQL's greatest strength and the source of much of its trickiness. Because you are not specifying the execution plan, you can sometimes be surprised by how the database chooses to execute your query — usually this is fine (the optimizer is smart), but occasionally you need to understand execution plans to diagnose performance problems.

SQL was standardized by ANSI in 1986 (SQL-86) and by ISO in 1987. The standard has been revised multiple times since: SQL-89, SQL-92, SQL:1999, SQL:2003, SQL:2008, SQL:2011, SQL:2016, and most recently SQL:2023. Each version adds features and clarifies ambiguities, but the core of the language has been stable since SQL-92. The most recent edition is ISO/IEC 9075:2023.

In practice, every major database system implements SQL with extensions and deviations. When you use PostgreSQL, you are using PostgreSQL's dialect of SQL. When you use SQL Server, you are using T-SQL (Transact-SQL). When you use MySQL, you are using MySQL's dialect. The core `SELECT`, `INSERT`, `UPDATE`, `DELETE`, `CREATE TABLE`, `ALTER TABLE`, and `DROP TABLE` are largely compatible across systems. Window functions, CTEs, JSON support, and procedural extensions vary considerably.

We will write SQL in this article that is broadly standard and should work in PostgreSQL, SQL Server, and MySQL with minor adjustments. Where there are significant dialect differences, we will note them.

### 3.2 SQL Sublanguages

SQL is actually composed of several sublanguages, each for a different purpose. Understanding this categorization helps you understand the structure of the language.

**DDL — Data Definition Language**: Statements that define the structure of the database.
- `CREATE TABLE`, `ALTER TABLE`, `DROP TABLE`
- `CREATE INDEX`, `DROP INDEX`
- `CREATE VIEW`, `DROP VIEW`
- `CREATE SCHEMA`, `DROP SCHEMA`

**DML — Data Manipulation Language**: Statements that manipulate data.
- `SELECT` — read data
- `INSERT` — add new rows
- `UPDATE` — modify existing rows
- `DELETE` — remove rows

**DCL — Data Control Language**: Statements that control access.
- `GRANT` — give permissions to users
- `REVOKE` — remove permissions from users

**TCL — Transaction Control Language**: Statements that manage transactions.
- `BEGIN TRANSACTION` (or `START TRANSACTION`)
- `COMMIT` — permanently save changes
- `ROLLBACK` — undo changes since the last commit
- `SAVEPOINT` — create a named point within a transaction you can roll back to

### 3.3 The SELECT Statement in Complete Detail

`SELECT` is the most important and most complex SQL statement. Let us cover it completely.

The general structure of a SELECT is:

```sql
SELECT [DISTINCT] column_list
FROM   table_or_subquery
[JOIN  other_table ON condition]
[WHERE filter_condition]
[GROUP BY grouping_columns]
[HAVING aggregate_filter]
[ORDER BY sort_columns]
[LIMIT n] [OFFSET m]  -- Standard SQL uses FETCH FIRST n ROWS ONLY
```

The logical order in which SQL processes these clauses is different from the order you write them, which is a source of confusion. The logical processing order is:

1. `FROM` — identify the source tables
2. `JOIN` — combine tables
3. `WHERE` — filter individual rows
4. `GROUP BY` — group remaining rows
5. `HAVING` — filter groups
6. `SELECT` — compute output expressions
7. `DISTINCT` — eliminate duplicate output rows
8. `ORDER BY` — sort the result
9. `LIMIT`/`OFFSET` — return a subset of rows

This order explains why you cannot use a column alias (defined in `SELECT`) in a `WHERE` clause — the `WHERE` clause is processed before `SELECT` computes the aliases. It also explains why you can use a column alias in `ORDER BY` in many databases — `ORDER BY` is processed after `SELECT`.

#### 3.3.1 Column Selection and Expressions

```sql
-- Select specific columns
SELECT customer_id, first_name, last_name
FROM   customers;

-- Rename columns with aliases (both syntaxes work; AS is clearer)
SELECT 
    customer_id   AS id,
    first_name    AS "First Name",  -- Use quotes for aliases with spaces
    last_name     AS "Last Name"
FROM customers;

-- Compute expressions
SELECT
    product_id,
    unit_price,
    units_in_stock,
    unit_price * units_in_stock AS total_inventory_value
FROM products;

-- String operations (PostgreSQL/SQL Server)
SELECT
    first_name || ' ' || last_name AS full_name  -- PostgreSQL
    -- CONCAT(first_name, ' ', last_name) AS full_name  -- MySQL/SQL Server
FROM customers;

-- Conditional expressions
SELECT
    product_id,
    product_name,
    unit_price,
    CASE
        WHEN unit_price < 10 THEN 'Budget'
        WHEN unit_price < 50 THEN 'Standard'
        ELSE 'Premium'
    END AS price_category
FROM products;
```

**Never write `SELECT *` in production code.** There are three reasons for this. First, it retrieves every column, including ones you do not need — this wastes network bandwidth, memory, and can prevent the database from using a covering index. Second, if someone adds a column to the table later, your query will silently start returning that extra column, which might break code that expects a specific number of columns. Third, `SELECT *` in a view definition is actively dangerous — the `*` is resolved at view creation time in some databases, not at query time, leading to deeply confusing behavior.

#### 3.3.2 The WHERE Clause and Predicates

```sql
-- Simple equality
SELECT * FROM products WHERE category_id = 5;

-- Inequality
SELECT * FROM products WHERE unit_price > 20;

-- Range (BETWEEN is inclusive on both ends)
SELECT * FROM products 
WHERE unit_price BETWEEN 10 AND 50;
-- Equivalent to: WHERE unit_price >= 10 AND unit_price <= 50

-- Set membership
SELECT * FROM products 
WHERE category_id IN (1, 3, 5);

-- Pattern matching
-- % matches any sequence of characters; _ matches exactly one character
SELECT * FROM customers 
WHERE last_name LIKE 'Sm%';    -- All last names starting with "Sm"

SELECT * FROM customers 
WHERE last_name LIKE '_mith';  -- Smith, Emith, etc. — 5 letters, ends in "mith"

-- NULL checks (never use = NULL)
SELECT * FROM products 
WHERE discontinued IS NULL;

SELECT * FROM products 
WHERE discontinued IS NOT NULL;

-- Compound predicates
SELECT * FROM products 
WHERE category_id = 1 
  AND unit_price > 20 
  AND units_in_stock > 0;

SELECT * FROM products 
WHERE category_id = 1 
   OR category_id = 2;

-- NOT operator
SELECT * FROM products 
WHERE NOT (category_id = 5);

-- Existence check with subquery
SELECT * FROM customers c
WHERE EXISTS (
    SELECT 1 
    FROM   orders o 
    WHERE  o.customer_id = c.customer_id
      AND  o.order_date >= '2026-01-01'
);
```

A common mistake is to write `WHERE column = NULL` when you mean `WHERE column IS NULL`. This is so common that it is worth repeating: the expression `column = NULL` always evaluates to UNKNOWN (not TRUE, not FALSE), so it never matches any rows. Always use `IS NULL`.

Another common mistake is using `NOT IN` when the subquery might return NULLs:

```sql
-- DANGEROUS: If any product has NULL category_id, this returns NO rows.
-- Because: 5 NOT IN (1, 2, NULL) evaluates to UNKNOWN (not TRUE or FALSE)
SELECT * FROM orders
WHERE category_id NOT IN (
    SELECT category_id FROM products  -- might contain NULLs
);

-- SAFE alternative:
SELECT * FROM orders
WHERE category_id NOT IN (
    SELECT category_id FROM products WHERE category_id IS NOT NULL
);

-- Or better still, use NOT EXISTS:
SELECT * FROM orders o
WHERE NOT EXISTS (
    SELECT 1 FROM products p WHERE p.category_id = o.category_id
);
```

#### 3.3.3 JOINs: Combining Tables

The `JOIN` clause is how you retrieve data from multiple tables simultaneously. It is the practical implementation of Codd's join operation from the relational algebra. Understanding JOINs thoroughly is absolutely essential. Let us go through each type.

**INNER JOIN**: Returns only rows where the join condition is satisfied in both tables.

```sql
-- Basic INNER JOIN
SELECT 
    o.order_id,
    o.order_date,
    c.first_name,
    c.last_name,
    c.email
FROM   orders o
INNER JOIN customers c ON o.customer_id = c.customer_id;

-- Multiple JOINs
SELECT 
    o.order_id,
    o.order_date,
    c.first_name || ' ' || c.last_name AS customer_name,
    p.product_name,
    od.quantity,
    od.unit_price
FROM       orders        o
INNER JOIN customers     c  ON o.customer_id   = c.customer_id
INNER JOIN order_details od ON o.order_id      = od.order_id
INNER JOIN products      p  ON od.product_id   = p.product_id;
```

When you write `FROM orders o INNER JOIN customers c ON o.customer_id = c.customer_id`, you are asking the database to find every pair of rows (one from `orders`, one from `customers`) where `orders.customer_id` equals `customers.customer_id`. Rows in `orders` with no matching customer are excluded. Rows in `customers` with no matching orders are excluded.

**LEFT JOIN (LEFT OUTER JOIN)**: Returns all rows from the left table, and matching rows from the right table. Where there is no match, the right table columns are filled with `NULL`.

```sql
-- All customers, including those with no orders
SELECT 
    c.customer_id,
    c.first_name,
    c.last_name,
    COUNT(o.order_id) AS order_count
FROM      customers c
LEFT JOIN orders    o ON c.customer_id = o.customer_id
GROUP BY  c.customer_id, c.first_name, c.last_name;

-- Find customers who have never placed an order
SELECT c.*
FROM      customers c
LEFT JOIN orders    o ON c.customer_id = o.customer_id
WHERE o.order_id IS NULL;  -- No matching order row means no orders
```

This last query — left join followed by `WHERE right_table.key IS NULL` — is the standard SQL idiom for finding rows in one table that have no corresponding rows in another. Memorize it.

**RIGHT JOIN (RIGHT OUTER JOIN)**: The mirror image of LEFT JOIN. Returns all rows from the right table. You can always rewrite a RIGHT JOIN as a LEFT JOIN by swapping the table order; most style guides prefer LEFT JOIN for consistency.

**FULL OUTER JOIN**: Returns all rows from both tables, with NULLs where there is no match.

```sql
-- All customers and all orders, whether or not they match
SELECT 
    c.customer_id,
    c.first_name,
    o.order_id,
    o.order_date
FROM      customers c
FULL OUTER JOIN orders o ON c.customer_id = o.customer_id;
```

**CROSS JOIN**: The Cartesian product. Every row in the left table combined with every row in the right table. Rarely what you want, but occasionally useful for generating combinations.

```sql
-- Generate all possible colour/size combinations for a product
SELECT 
    c.colour_name,
    s.size_name
FROM   colours c
CROSS JOIN sizes s;
```

**Self JOIN**: A table joined to itself, typically used when rows in a table reference other rows in the same table (e.g., an employees table where each employee has a `manager_id` that references another `employee_id` in the same table).

```sql
-- Each employee and their manager's name
SELECT 
    e.employee_id,
    e.first_name || ' ' || e.last_name AS employee_name,
    m.first_name || ' ' || m.last_name AS manager_name
FROM      employees e
LEFT JOIN employees m ON e.manager_id = m.employee_id;
```

#### 3.3.4 GROUP BY and Aggregate Functions

Aggregate functions compute a single value from a set of rows. The key aggregate functions are:

- `COUNT(*)` — count of all rows in the group
- `COUNT(expression)` — count of rows where the expression is not NULL
- `COUNT(DISTINCT expression)` — count of distinct non-NULL values
- `SUM(expression)` — sum of non-NULL values
- `AVG(expression)` — average of non-NULL values
- `MIN(expression)` — minimum non-NULL value
- `MAX(expression)` — maximum non-NULL value

When you use an aggregate function without `GROUP BY`, you get a single row summarizing the entire table. When you add `GROUP BY`, you get one summary row per distinct combination of the grouping columns.

```sql
-- Total number of orders
SELECT COUNT(*) AS total_orders
FROM   orders;

-- Orders per customer
SELECT 
    customer_id,
    COUNT(*)              AS order_count,
    SUM(total_amount)     AS total_spent,
    MIN(order_date)       AS first_order,
    MAX(order_date)       AS last_order
FROM orders
GROUP BY customer_id;

-- Customers who have spent more than 500 total (HAVING filters groups)
SELECT 
    customer_id,
    SUM(total_amount) AS total_spent
FROM   orders
GROUP BY customer_id
HAVING SUM(total_amount) > 500;

-- JOIN with GROUP BY
SELECT 
    c.customer_id,
    c.first_name || ' ' || c.last_name AS customer_name,
    COUNT(o.order_id)    AS order_count,
    SUM(o.total_amount)  AS total_spent
FROM      customers c
LEFT JOIN orders    o ON c.customer_id = o.customer_id
GROUP BY c.customer_id, c.first_name, c.last_name
HAVING COUNT(o.order_id) > 5
ORDER BY total_spent DESC;
```

The `HAVING` clause is `WHERE` for groups. It filters the result of `GROUP BY`, after aggregation. This is why you cannot use aggregate functions in a `WHERE` clause (because `WHERE` runs before `GROUP BY`) but you can use them in `HAVING`.

A classic mistake:

```sql
-- WRONG: Cannot use aggregate in WHERE
SELECT customer_id, SUM(total_amount)
FROM   orders
WHERE  SUM(total_amount) > 500  -- ERROR: aggregate not allowed in WHERE
GROUP BY customer_id;

-- CORRECT: Aggregate filter goes in HAVING
SELECT customer_id, SUM(total_amount) AS total_spent
FROM   orders
GROUP BY customer_id
HAVING SUM(total_amount) > 500;
```

Another classic mistake: selecting a column that is not in the `GROUP BY` clause and is not an aggregate function. This is technically invalid in standard SQL and in most databases (MySQL in its default mode will pick an arbitrary value, which is wrong).

```sql
-- WRONG: first_name is not in GROUP BY and is not aggregated
SELECT customer_id, first_name, COUNT(*)
FROM   orders
GROUP BY customer_id;
-- PostgreSQL and SQL Server will reject this.

-- CORRECT: Either include first_name in GROUP BY or aggregate it
SELECT customer_id, MAX(first_name) AS first_name, COUNT(*)
FROM   orders
GROUP BY customer_id;
```

#### 3.3.5 Window Functions

Window functions were introduced in SQL:2003 and are one of the most powerful additions to the language since SQL-92. They allow you to perform calculations across a set of rows that are related to the current row, without collapsing them into a single aggregate row.

```sql
-- Running total of orders per customer
SELECT
    order_id,
    customer_id,
    order_date,
    total_amount,
    SUM(total_amount) OVER (
        PARTITION BY customer_id 
        ORDER BY order_date
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS running_total
FROM orders;

-- Rank orders by amount within each customer
SELECT
    order_id,
    customer_id,
    total_amount,
    RANK() OVER (PARTITION BY customer_id ORDER BY total_amount DESC) AS amount_rank
FROM orders;

-- Compare each order to the customer's average
SELECT
    order_id,
    customer_id,
    total_amount,
    AVG(total_amount) OVER (PARTITION BY customer_id) AS customer_avg,
    total_amount - AVG(total_amount) OVER (PARTITION BY customer_id) AS diff_from_avg
FROM orders;

-- Previous and next row values
SELECT
    order_id,
    customer_id,
    order_date,
    total_amount,
    LAG(total_amount,  1) OVER (PARTITION BY customer_id ORDER BY order_date) AS prev_amount,
    LEAD(total_amount, 1) OVER (PARTITION BY customer_id ORDER BY order_date) AS next_amount
FROM orders;
```

The `OVER` clause defines the window. `PARTITION BY` divides the rows into groups (like `GROUP BY`, but without collapsing). `ORDER BY` within `OVER` specifies the ordering within each partition. `ROWS BETWEEN` specifies the frame — the specific subset of rows within the partition to include in the calculation.

Window functions are indispensable for reporting and analytics. They let you compute things like "this row compared to the running average," "rank within group," "previous and next values," and "cumulative sums" without the contortions that would be required in raw procedural code.

#### 3.3.6 Common Table Expressions (CTEs)

CTEs were introduced in SQL:1999. They allow you to name a subquery and reference it by name within the main query, improving readability dramatically and sometimes also enabling recursive queries.

```sql
-- Without CTE: nested subquery nightmare
SELECT customer_id, total_spent
FROM (
    SELECT customer_id, SUM(total_amount) AS total_spent
    FROM   orders
    WHERE  order_date >= '2026-01-01'
    GROUP BY customer_id
) AS customer_totals
WHERE total_spent > 1000;

-- With CTE: much clearer
WITH customer_totals AS (
    SELECT 
        customer_id, 
        SUM(total_amount) AS total_spent
    FROM   orders
    WHERE  order_date >= '2026-01-01'
    GROUP BY customer_id
)
SELECT customer_id, total_spent
FROM   customer_totals
WHERE  total_spent > 1000;

-- Multiple CTEs
WITH 
high_value_customers AS (
    SELECT customer_id
    FROM   orders
    GROUP BY customer_id
    HAVING SUM(total_amount) > 1000
),
recent_orders AS (
    SELECT *
    FROM   orders
    WHERE  order_date >= CURRENT_DATE - INTERVAL '30 days'
)
SELECT 
    c.first_name,
    c.last_name,
    ro.order_id,
    ro.total_amount
FROM      customers c
INNER JOIN high_value_customers hvc ON c.customer_id = hvc.customer_id
INNER JOIN recent_orders        ro  ON c.customer_id = ro.customer_id;
```

Recursive CTEs allow you to traverse hierarchical data:

```sql
-- Traverse an employee hierarchy: find all reports of employee 1
WITH RECURSIVE org_chart AS (
    -- Base case: the root employee
    SELECT employee_id, first_name, last_name, manager_id, 0 AS depth
    FROM   employees
    WHERE  employee_id = 1

    UNION ALL

    -- Recursive case: employees who report to someone already in the result
    SELECT e.employee_id, e.first_name, e.last_name, e.manager_id, oc.depth + 1
    FROM   employees e
    INNER JOIN org_chart oc ON e.manager_id = oc.employee_id
)
SELECT * FROM org_chart ORDER BY depth, last_name;
```

### 3.4 INSERT, UPDATE, and DELETE

```sql
-- INSERT: single row
INSERT INTO customers (customer_id, first_name, last_name, email)
VALUES (1001, 'Alice', 'Johnson', 'alice@example.com');

-- INSERT: multiple rows
INSERT INTO customers (customer_id, first_name, last_name, email)
VALUES 
    (1002, 'Bob', 'Smith', 'bob@example.com'),
    (1003, 'Carol', 'Williams', 'carol@example.com'),
    (1004, 'David', 'Brown', 'david@example.com');

-- INSERT from SELECT (copy rows matching a condition)
INSERT INTO archived_orders (order_id, customer_id, order_date, total_amount)
SELECT order_id, customer_id, order_date, total_amount
FROM   orders
WHERE  order_date < '2025-01-01';

-- UPDATE: change values in existing rows
UPDATE products
SET    unit_price = unit_price * 1.1  -- 10% price increase
WHERE  category_id = 3;

-- UPDATE with JOIN (SQL Server syntax)
UPDATE p
SET    p.unit_price = p.unit_price * 1.1
FROM   products p
INNER JOIN categories c ON p.category_id = c.category_id
WHERE  c.category_name = 'Electronics';

-- DELETE: remove rows
DELETE FROM orders
WHERE  order_date < '2024-01-01'
  AND  shipped = TRUE;

-- Careful: DELETE without WHERE deletes ALL rows
DELETE FROM products;  -- This deletes every product. No undo without a transaction.

-- TRUNCATE is faster than DELETE with no WHERE, but cannot be rolled back in most DBs
TRUNCATE TABLE staging_import_data;
```

**Always test your WHERE clause with a SELECT before running an UPDATE or DELETE.** Wrap everything in a transaction so you can roll back if needed:

```sql
BEGIN TRANSACTION;

-- First, verify what you're about to change
SELECT COUNT(*) FROM products WHERE category_id = 3;

-- Then make the change
UPDATE products
SET    discontinued = TRUE
WHERE  category_id = 3;

-- Verify the result
SELECT COUNT(*) FROM products WHERE discontinued = TRUE AND category_id = 3;

-- If it looks right, commit. If not, rollback.
COMMIT;
-- Or: ROLLBACK;
```

### 3.5 Transactions, ACID, and Why They Matter

A **transaction** is a sequence of operations treated as a single unit of work. Either all of the operations succeed, or none of them do. This is the all-or-nothing guarantee.

Imagine transferring money between bank accounts. You need to subtract $500 from account A and add $500 to account B. These are two separate database operations. Without transactions, if the system crashes between the debit and the credit, $500 just vanishes. With transactions, a crash before commit means the database reverts to the state before the transaction started.

Relational databases guarantee ACID properties for transactions:

**A — Atomicity**: The transaction is atomic. All operations succeed, or none do. There is no partial execution visible to the outside world.

**C — Consistency**: The transaction takes the database from one valid state to another valid state. All integrity constraints (NOT NULL, UNIQUE, foreign keys, CHECKs) are satisfied before and after.

**I — Isolation**: Concurrent transactions do not see each other's intermediate states. The degree of isolation is configurable (more on isolation levels below).

**D — Durability**: Once a transaction is committed, it stays committed. A crash immediately after a commit does not lose the data. The database uses a write-ahead log (WAL) to ensure this.

```csharp
// In C# with System.Data.SqlClient or Npgsql (PostgreSQL):
// Always use transactions when you have multiple related operations.

await using var connection = new NpgsqlConnection(connectionString);
await connection.OpenAsync();

await using var transaction = await connection.BeginTransactionAsync();

try
{
    // Debit from source account
    await using var debitCmd = new NpgsqlCommand(
        "UPDATE accounts SET balance = balance - @amount WHERE account_id = @source AND balance >= @amount",
        connection, transaction);
    debitCmd.Parameters.AddWithValue("amount", 500m);
    debitCmd.Parameters.AddWithValue("source", sourceAccountId);
    
    int rowsAffected = await debitCmd.ExecuteNonQueryAsync();
    if (rowsAffected == 0)
        throw new InvalidOperationException("Insufficient funds or account not found.");

    // Credit to destination account
    await using var creditCmd = new NpgsqlCommand(
        "UPDATE accounts SET balance = balance + @amount WHERE account_id = @dest",
        connection, transaction);
    creditCmd.Parameters.AddWithValue("amount", 500m);
    creditCmd.Parameters.AddWithValue("dest", destAccountId);
    
    await creditCmd.ExecuteNonQueryAsync();

    await transaction.CommitAsync();
}
catch
{
    await transaction.RollbackAsync();
    throw;
}
```

**Isolation levels** control how transactions interact when they run concurrently. The SQL standard defines four levels, in order of increasing isolation (and decreasing concurrency):

1. **READ UNCOMMITTED**: One transaction can read another transaction's uncommitted changes. This allows "dirty reads" — you might read data that gets rolled back. Almost never appropriate.

2. **READ COMMITTED**: A transaction only sees data that has been committed. Prevents dirty reads. Allows "non-repeatable reads" — if you read the same row twice in a transaction, you might get different values if another transaction commits a change in between.

3. **REPEATABLE READ**: If you read a row within a transaction, any subsequent reads of the same row will return the same value, even if another transaction modifies it. Prevents dirty reads and non-repeatable reads. Still allows "phantom reads" — a query run twice might return different numbers of rows if another transaction inserts rows that match the query's criteria.

4. **SERIALIZABLE**: The highest isolation level. Transactions behave as if they executed serially, one after another. No concurrency anomalies. The database may need to block or abort transactions to maintain this guarantee, which can reduce throughput.

Most production applications use READ COMMITTED (the default in PostgreSQL and SQL Server). For financial applications requiring absolute consistency, SERIALIZABLE or REPEATABLE READ is appropriate.

## Part 4: Database Design — From Chaos to Structure

### 4.1 The Problem This Section Solves

Here is a scenario you should recognize.

You are building a small application. Maybe it is an e-commerce store. Maybe it is a blog. Maybe it is a task management system. You are working quickly, you want to see results, and you are making pragmatic decisions to get something working.

You create a table. It looks like this:

```sql
CREATE TABLE orders (
    order_id        INT,
    customer_name   VARCHAR(200),
    customer_email  VARCHAR(255),
    customer_phone  VARCHAR(20),
    customer_city   VARCHAR(100),
    product_1_name  VARCHAR(200),
    product_1_qty   INT,
    product_1_price DECIMAL(10,2),
    product_2_name  VARCHAR(200),
    product_2_qty   INT,
    product_2_price DECIMAL(10,2),
    product_3_name  VARCHAR(200),
    product_3_qty   INT,
    product_3_price DECIMAL(10,2),
    sales_rep_name  VARCHAR(200),
    sales_rep_email VARCHAR(255),
    region_name     VARCHAR(100)
);
```

This table is a disaster. Not a potential disaster. Not a risky design choice. An actual disaster. But it is also an extremely natural table to build if nobody has ever explained why it is wrong.

Let us list the specific ways it will hurt you:

**Problem 1 — Data duplication**: Alice Johnson's email address appears in every row of every order she has ever placed. If she changes her email, you must find and update every single one of those rows. Miss one, and now you have inconsistent data. Alice exists in your database as two different people with two different email addresses, and you have no way to know which one is correct.

**Problem 2 — Update anomaly**: Updating the sales rep's email requires finding every order they ever handled and updating each row. One failed UPDATE (due to network glitch, bug, anything) and different rows say different things about the same person.

**Problem 3 — Insertion anomaly**: You cannot add a new sales representative to the system until they have placed at least one order. The sales rep only exists in the `orders` table, so if they have no orders, there is nowhere to put them.

**Problem 4 — Deletion anomaly**: If you delete the last order handled by a particular sales rep, you lose all information about that sales rep. The data describing the sales rep is coupled to the data describing the order; you cannot delete one without destroying the other.

**Problem 5 — The fixed-column products problem**: You have `product_1`, `product_2`, `product_3`. What happens when an order has four products? You either alter the table (breaking all code that reads it), or you start storing comma-separated product lists in a single column (which violates the rules of the relational model and makes querying essentially impossible), or you give up.

**Problem 6 — Wasted space**: Most orders have fewer than three products. The `product_3_*` columns are NULL for every such order.

**Problem 7 — Impossible queries**: "What is the total revenue from Product X?" You would have to search through `product_1_name`, `product_2_name`, and `product_3_name` separately and add them up. And if you later add `product_4`, you have to update every such query.

Every single one of these problems is solved by normalization. And normalization is not a magic technique or a clever hack. It follows mathematically from first principles about what it means for data to be consistently organized.

### 4.2 Functional Dependency — The Concept Underlying All Normalization

Before we can talk about the normal forms, we need to understand **functional dependency**. It is the foundational concept that makes normalization work.

A functional dependency means: given a value for attribute A, you can always determine the value of attribute B. We write this as A → B and say "A functionally determines B" or "B is functionally dependent on A."

Some examples:

- `employee_id → employee_name` (given the employee ID, you can always look up the employee's name — each employee has exactly one name)
- `zip_code → city` (given a zip code, you know which city it belongs to — each zip code is in exactly one city)
- `{order_id, product_id} → quantity` (given a specific order and a specific product within that order, you know the quantity — the quantity depends on both)

A functional dependency means this is a many-to-one relationship from the left side to the right side. Multiple employees can have the same name (name does not functionally determine employee_id), but each employee_id corresponds to exactly one name (employee_id functionally determines name).

The concept of **full functional dependency** means that the dependency on the left side is not reducible — if you removed any attribute from the left side, the dependency would break. If `{A, B} → C` and `A → C` alone already holds, then C is only partially dependent on `{A, B}` (it depends on just A, not the full composite key).

**Transitive dependency** means: A → B and B → C, and B → A does not hold (B is not a key). In this case, C is transitively dependent on A through B.

These two concepts — partial dependency and transitive dependency — are precisely what the normal forms are designed to eliminate.

### 4.3 Entity-Relationship Modelling — Designing Before You Build

Before writing a single `CREATE TABLE` statement, you should model your data conceptually. The standard tool for this is the **Entity-Relationship (ER) diagram**, developed by Peter Chen in 1976.

An ER diagram has three main concepts:

1. **Entities**: Things about which you store data. Examples: Customer, Product, Order, Employee. Each entity becomes a table.

2. **Attributes**: Properties of entities. Examples: Customer has name, email, phone; Product has name, price, SKU. Each attribute becomes a column.

3. **Relationships**: How entities relate to each other. Examples: A Customer *places* Orders; An Order *contains* Products. Relationships are expressed as foreign keys, or (for many-to-many relationships) as junction tables.

Relationships have **cardinality**:
- **One-to-one (1:1)**: Each row in A corresponds to at most one row in B. Example: each employee has exactly one HR record.
- **One-to-many (1:N)**: Each row in A corresponds to multiple rows in B, but each row in B corresponds to at most one row in A. Example: one customer places many orders; each order belongs to one customer.
- **Many-to-many (M:N)**: Each row in A corresponds to multiple rows in B, and each row in B corresponds to multiple rows in A. Example: one order can contain many products; one product can appear in many orders.

Many-to-many relationships cannot be directly represented with just two tables and foreign keys. They require a **junction table** (also called a bridge table, link table, or associative table) — a separate table whose primary key is a composite of the two foreign keys.

```sql
-- One customer, many orders (1:N)
-- Foreign key in orders table points to customers table
CREATE TABLE orders (
    order_id    INT  NOT NULL,
    customer_id INT  NOT NULL,  -- Foreign key → customers
    ...
    CONSTRAINT pk_orders PRIMARY KEY (order_id),
    CONSTRAINT fk_orders_customer FOREIGN KEY (customer_id) REFERENCES customers (customer_id)
);

-- Many orders, many products (M:N)
-- Junction table: order_details
CREATE TABLE order_details (
    order_id    INT           NOT NULL,
    product_id  INT           NOT NULL,
    quantity    INT           NOT NULL,
    unit_price  DECIMAL(10,2) NOT NULL,
    CONSTRAINT pk_order_details PRIMARY KEY (order_id, product_id),
    CONSTRAINT fk_od_order   FOREIGN KEY (order_id)   REFERENCES orders   (order_id),
    CONSTRAINT fk_od_product FOREIGN KEY (product_id) REFERENCES products (product_id)
);
```

Notice that `order_details` has a composite primary key: `(order_id, product_id)`. The combination uniquely identifies a row. Neither alone is sufficient — an order can have many products, and a product can appear in many orders.

## Part 5: The Normal Forms in Complete Detail

### 5.1 First Normal Form (1NF)

**Definition**: A relation is in First Normal Form if and only if every attribute contains only atomic (indivisible) values, and there are no repeating groups.

Let us unpack those two conditions.

**Atomic values**: Every cell in the table contains a single, indivisible value — not a list, not a set, not a sub-table. A cell that contains "Alice, Bob, Carol" is not atomic. A cell that contains "Smith" is atomic.

**No repeating groups**: There are no sets of columns that represent the same kind of information for different instances of the same thing. The `product_1_name`, `product_2_name`, `product_3_name` pattern we saw earlier is a repeating group. So is storing multiple phone numbers in a single comma-separated string.

Here is a table that violates 1NF:

```sql
-- VIOLATION: phone_numbers stores multiple values in one cell
-- VIOLATION: has repeating product columns
CREATE TABLE bad_orders (
    order_id       INT,
    customer_name  VARCHAR(200),
    phone_numbers  VARCHAR(500),  -- "555-1234, 555-5678, 555-9999"
    product_1_name VARCHAR(200),
    product_1_qty  INT,
    product_2_name VARCHAR(200),
    product_2_qty  INT
);
```

This is problematic for all the reasons listed earlier. How do you query "all orders from customers with phone number 555-5678"? You have to do a string search inside `phone_numbers`, which is both slow and error-prone. How do you join to a `phones` table? You cannot.

Converting to 1NF means:

1. Remove multi-valued attributes — move them to their own table.
2. Eliminate repeating groups by converting them to separate rows.

```sql
-- Customers table: one row per customer
CREATE TABLE customers (
    customer_id   INT           NOT NULL,
    customer_name VARCHAR(200)  NOT NULL,
    CONSTRAINT pk_customers PRIMARY KEY (customer_id)
);

-- Phone numbers table: one row per phone number
-- A customer can have multiple phone numbers, each gets its own row
CREATE TABLE customer_phones (
    customer_id  INT          NOT NULL,
    phone_type   VARCHAR(20)  NOT NULL,  -- 'mobile', 'home', 'work'
    phone_number VARCHAR(20)  NOT NULL,
    CONSTRAINT pk_customer_phones PRIMARY KEY (customer_id, phone_type),
    CONSTRAINT fk_cph_customer FOREIGN KEY (customer_id) REFERENCES customers (customer_id)
);

-- Orders table: one row per order
CREATE TABLE orders (
    order_id    INT  NOT NULL,
    customer_id INT  NOT NULL,
    order_date  DATE NOT NULL,
    CONSTRAINT pk_orders PRIMARY KEY (order_id),
    CONSTRAINT fk_orders_customer FOREIGN KEY (customer_id) REFERENCES customers (customer_id)
);

-- Order details: one row per product per order (replaces repeating product columns)
CREATE TABLE order_details (
    order_id   INT           NOT NULL,
    product_id INT           NOT NULL,
    quantity   INT           NOT NULL,
    unit_price DECIMAL(10,2) NOT NULL,
    CONSTRAINT pk_order_details PRIMARY KEY (order_id, product_id),
    CONSTRAINT fk_od_order   FOREIGN KEY (order_id)   REFERENCES orders   (order_id),
    CONSTRAINT fk_od_product FOREIGN KEY (product_id) REFERENCES products (product_id)
);
```

Now `customer_phones` has one row per phone number. Querying "all customers with phone number X" is a simple equality comparison on an indexed column. Adding a fourth or fifth phone number requires no schema change — just insert another row.

**The rule about atomic values and what "atomic" means**: This is actually more subtle than it appears. Whether a value is "atomic" depends on the domain and what you need to query. A customer's full name stored as "Alice Johnson" is arguably not atomic — it contains first name and last name. But if you never need to query by first name or last name separately, storing the full name as one column is acceptable. If you do need to query by first name or last name, you should store them separately.

The practical test: if you ever need to operate on parts of a value (sort by last name, search for first name, extract the year from a date string stored as text), that value is not atomic enough for your purposes and should be stored as separate columns or in a proper data type.

**Do not store JSON in SQL columns as a way to avoid schema design.** This is a common anti-pattern in modern applications. JSON in a SQL column breaks 1NF. You cannot query fields inside the JSON with index support (mostly), you cannot enforce constraints on the JSON contents, and you lose all the benefits of a relational database for that data. SQL Server, PostgreSQL, and MySQL all support JSON columns with some query capabilities, but this is for specific use cases — storing truly schemaless data, external API responses, event logs — not for avoiding the work of proper database design.

### 5.2 Second Normal Form (2NF)

**Definition**: A relation is in Second Normal Form if it is in First Normal Form and every non-key attribute is fully functionally dependent on every candidate key.

In simpler terms: every column must depend on the whole primary key, not just part of it.

This only applies to tables with **composite primary keys** (primary keys made up of more than one column). If your primary key is a single column, and your table is in 1NF, it is automatically in 2NF — you cannot have a partial dependency on a single-column key (you either depend on it or you do not).

When does this become an issue? Precisely when you have a junction table or any other table with a composite key, and you store attributes that only depend on part of the key.

Here is a table that violates 2NF. This is an `order_details` table:

```sql
-- VIOLATION of 2NF
CREATE TABLE order_details_bad (
    order_id        INT           NOT NULL,
    product_id      INT           NOT NULL,
    quantity        INT           NOT NULL,
    unit_price      DECIMAL(10,2) NOT NULL,
    -- These columns violate 2NF:
    product_name    VARCHAR(200)  NOT NULL,  -- depends only on product_id, not the whole key
    product_weight  DECIMAL(8,3),            -- depends only on product_id, not the whole key
    customer_id     INT           NOT NULL,  -- depends only on order_id, not the whole key
    order_date      DATE          NOT NULL,  -- depends only on order_id, not the whole key
    CONSTRAINT pk_order_details PRIMARY KEY (order_id, product_id)
);
```

Here, `product_name` and `product_weight` depend only on `product_id`, not on the combination `(order_id, product_id)`. The name of the product does not change based on which order it is in. Similarly, `customer_id` and `order_date` depend only on `order_id`.

This creates the same kinds of problems as before:

- If a product is renamed, you must update every row in `order_details` that mentions that product. Miss any, and the same product has multiple names.
- If you want to know all the products in the catalog, you have to look through all historical orders.
- If you delete all orders containing a particular product, you lose the product's name and weight.

The fix for 2NF violations is to decompose: move the partially dependent attributes to the table whose key they actually depend on.

```sql
-- Products table: product_name and product_weight belong here
CREATE TABLE products (
    product_id     INT           NOT NULL,
    product_name   VARCHAR(200)  NOT NULL,
    product_weight DECIMAL(8,3),
    list_price     DECIMAL(10,2) NOT NULL,
    CONSTRAINT pk_products PRIMARY KEY (product_id)
);

-- Orders table: customer_id and order_date belong here
CREATE TABLE orders (
    order_id    INT  NOT NULL,
    customer_id INT  NOT NULL,
    order_date  DATE NOT NULL,
    CONSTRAINT pk_orders PRIMARY KEY (order_id)
);

-- Order details: only attributes that depend on (order_id, product_id)
CREATE TABLE order_details (
    order_id   INT           NOT NULL,  -- references an order
    product_id INT           NOT NULL,  -- references a product
    quantity   INT           NOT NULL,  -- depends on which order AND which product
    unit_price DECIMAL(10,2) NOT NULL,  -- also depends on both (price at time of order)
    CONSTRAINT pk_order_details PRIMARY KEY (order_id, product_id),
    CONSTRAINT fk_od_order   FOREIGN KEY (order_id)   REFERENCES orders   (order_id),
    CONSTRAINT fk_od_product FOREIGN KEY (product_id) REFERENCES products (product_id)
);
```

Notice something interesting: `unit_price` stays in `order_details`, not in `products`. The current price of a product belongs in `products`. But the price at which the product was sold in a specific order depends on both the product and the order — different orders might have applied a discount, and the product's price changes over time. Storing `unit_price` in `order_details` records the historical price at the time of the sale. This is correct and intentional. It is not a 2NF violation — the price-for-this-order-line genuinely depends on both `order_id` and `product_id`.

### 5.3 Third Normal Form (3NF)

**Definition**: A relation is in Third Normal Form if it is in Second Normal Form and every non-key attribute is non-transitively dependent on each candidate key.

Or, in William Kent's famous mnemonic: **"[every non-key attribute must describe] the key, the whole key, and nothing but the key."**

A transitive dependency is a chain of dependencies: A → B → C, where B is not a key. If A is the primary key, B is a non-key attribute, and B → C, then C is transitively dependent on A through B. The problem is that C is really a fact about B, not a fact about A. When B changes, C might need to change too — and if you have stored C in A's table, you now have to find every row in A's table where B has a certain value and update C.

Let us build a concrete example. Imagine a table that stores employee information including their department:

```sql
-- VIOLATION of 3NF
CREATE TABLE employees_bad (
    employee_id     INT          NOT NULL,
    first_name      VARCHAR(100) NOT NULL,
    last_name       VARCHAR(100) NOT NULL,
    department_id   INT          NOT NULL,
    department_name VARCHAR(200) NOT NULL,  -- VIOLATION: depends on department_id, not employee_id
    department_head VARCHAR(200) NOT NULL,  -- VIOLATION: depends on department_id, not employee_id
    manager_id      INT,
    hire_date       DATE         NOT NULL,
    salary          DECIMAL(10,2) NOT NULL,
    CONSTRAINT pk_employees PRIMARY KEY (employee_id)
);
```

The transitive dependency here is:
- `employee_id → department_id` (each employee belongs to a department)
- `department_id → department_name` (each department has a name)
- `department_id → department_head` (each department has a head)

Therefore: `employee_id → department_id → department_name`. The `department_name` depends on `department_id`, not directly on `employee_id`. It is a fact about the department, not a fact about the employee.

The consequences:

- If the "Engineering" department is renamed to "Software Engineering," you must update every single row in `employees` where `department_name = 'Engineering'`. Miss one, and you have employees in both "Engineering" and "Software Engineering" — which are apparently different departments to your database.
- If the department head changes, same problem: update every employee in that department, or have inconsistent records.
- You cannot add a new department until it has at least one employee. Where do you store the department name and head otherwise?
- If you reassign all employees out of a department, you lose the department's name and head.

The fix is decomposition: move the facts about departments to a departments table.

```sql
-- Departments table: facts about departments belong here
CREATE TABLE departments (
    department_id   INT          NOT NULL,
    department_name VARCHAR(200) NOT NULL,
    department_head VARCHAR(200) NOT NULL,
    CONSTRAINT pk_departments PRIMARY KEY (department_id)
);

-- Employees table: only facts about employees
CREATE TABLE employees (
    employee_id   INT           NOT NULL,
    first_name    VARCHAR(100)  NOT NULL,
    last_name     VARCHAR(100)  NOT NULL,
    department_id INT           NOT NULL,   -- foreign key — the link to departments
    manager_id    INT,
    hire_date     DATE          NOT NULL,
    salary        DECIMAL(10,2) NOT NULL,
    CONSTRAINT pk_employees   PRIMARY KEY (employee_id),
    CONSTRAINT fk_emp_dept    FOREIGN KEY (department_id) REFERENCES departments (department_id),
    CONSTRAINT fk_emp_manager FOREIGN KEY (manager_id)   REFERENCES employees   (employee_id)
);
```

Now:
- Renaming a department requires a single `UPDATE departments SET department_name = ... WHERE department_id = ...`. All employees implicitly get the updated name because they reference the department by ID.
- You can create a department with zero employees.
- Removing all employees from a department does not delete the department.

Let us look at another common 3NF violation — the zip code / city / state triad:

```sql
-- VIOLATION of 3NF
CREATE TABLE shipping_addresses_bad (
    address_id    INT          NOT NULL,
    customer_id   INT          NOT NULL,
    street        VARCHAR(200) NOT NULL,
    city          VARCHAR(100) NOT NULL,
    state_code    VARCHAR(2)   NOT NULL,
    zip_code      VARCHAR(10)  NOT NULL,
    country       VARCHAR(100) NOT NULL DEFAULT 'US',
    -- city and state_code are functionally determined by zip_code:
    -- zip_code → city, zip_code → state_code
    CONSTRAINT pk_shipping_addresses PRIMARY KEY (address_id)
);
```

The dependency: `zip_code → city`, `zip_code → state_code`. If a zip code changes its city assignment (rare but it happens), or if you discover a data entry error, you must update every row with that zip code. And if you have different spellings of the same city name for the same zip code (e.g., "New York" vs "New York City"), your data is inconsistent.

The strictly correct 3NF resolution is to create a `zip_codes` table:

```sql
CREATE TABLE zip_codes (
    zip_code   VARCHAR(10)  NOT NULL,
    city       VARCHAR(100) NOT NULL,
    state_code VARCHAR(2)   NOT NULL,
    CONSTRAINT pk_zip_codes PRIMARY KEY (zip_code)
);

CREATE TABLE shipping_addresses (
    address_id  INT          NOT NULL,
    customer_id INT          NOT NULL,
    street      VARCHAR(200) NOT NULL,
    zip_code    VARCHAR(10)  NOT NULL,
    country     VARCHAR(100) NOT NULL DEFAULT 'US',
    CONSTRAINT pk_shipping_addresses PRIMARY KEY (address_id),
    CONSTRAINT fk_sa_zip FOREIGN KEY (zip_code) REFERENCES zip_codes (zip_code)
);
```

However — and this is an important nuance — strict 3NF is not always the right choice for every application. The zip code table is a practical example where you might choose not to fully normalize. Zip codes do not change their city assignments often. The performance cost of an extra join on every address lookup might not be worth the theoretical benefit of eliminating the transitive dependency. **Normalization is a tool, not a law.** We will come back to the question of when to denormalize in Part 7.

### 5.4 An Extended Case Study: Building a Library System from Scratch

Theory is important. But theory without practice is just philosophy. Let us build a real database schema from scratch, making all the mistakes deliberately so we can fix them systematically.

**Requirements**: We are building a library management system. The library needs to track:
- Books (title, ISBN, year published, publisher, author(s), genre(s))
- Authors (name, nationality, birth year)
- Patrons (members who borrow books)
- Loans (which patron borrowed which book, when, when it was returned)

**Iteration 0: The naive developer's first attempt**

This is the table a developer who has never heard of normalization might build:

```sql
CREATE TABLE library_everything (
    loan_id          INT,
    patron_name      VARCHAR(200),
    patron_email     VARCHAR(255),
    patron_phone     VARCHAR(20),
    patron_address   VARCHAR(500),
    book_isbn        VARCHAR(13),
    book_title       VARCHAR(500),
    book_author_1    VARCHAR(200),
    book_author_2    VARCHAR(200),
    book_author_3    VARCHAR(200),
    book_genres      VARCHAR(500),    -- "Fiction, Thriller, Mystery"
    book_publisher   VARCHAR(200),
    publisher_city   VARCHAR(100),
    publisher_country VARCHAR(100),
    book_year        INT,
    loan_date        DATE,
    due_date         DATE,
    return_date      DATE
);
```

This table has every violation in the book:
- 1NF: `book_genres` is multi-valued ("Fiction, Thriller, Mystery"). `book_author_1`, `book_author_2`, `book_author_3` are a repeating group.
- Partial dependency: Book information and patron information both depend on only part of what would be the key.
- Transitive dependencies everywhere: `publisher_city` and `publisher_country` depend on `book_publisher`, not on `loan_id`. The book's attributes depend on `book_isbn`, not on `loan_id`.

**Iteration 1: Apply 1NF**

Eliminate multi-valued attributes and repeating groups:

```sql
-- Patrons
CREATE TABLE patrons (
    patron_id   INT          NOT NULL,
    patron_name VARCHAR(200) NOT NULL,
    email       VARCHAR(255) NOT NULL,
    phone       VARCHAR(20),
    address     VARCHAR(500),
    CONSTRAINT pk_patrons PRIMARY KEY (patron_id)
);

-- Books (no repeating author columns, no multi-valued genre)
CREATE TABLE books (
    isbn          VARCHAR(13)  NOT NULL,
    title         VARCHAR(500) NOT NULL,
    publisher     VARCHAR(200) NOT NULL,
    publisher_city    VARCHAR(100),
    publisher_country VARCHAR(100),
    year_published INT          NOT NULL,
    CONSTRAINT pk_books PRIMARY KEY (isbn)
);

-- Authors (separate entity)
CREATE TABLE authors (
    author_id   INT          NOT NULL,
    author_name VARCHAR(200) NOT NULL,
    CONSTRAINT pk_authors PRIMARY KEY (author_id)
);

-- Book-Author (M:N junction table)
CREATE TABLE book_authors (
    isbn      VARCHAR(13) NOT NULL,
    author_id INT         NOT NULL,
    CONSTRAINT pk_book_authors PRIMARY KEY (isbn, author_id),
    CONSTRAINT fk_ba_book   FOREIGN KEY (isbn)      REFERENCES books   (isbn),
    CONSTRAINT fk_ba_author FOREIGN KEY (author_id) REFERENCES authors (author_id)
);

-- Genres (separate entity)
CREATE TABLE genres (
    genre_id   INT          NOT NULL,
    genre_name VARCHAR(100) NOT NULL,
    CONSTRAINT pk_genres PRIMARY KEY (genre_id)
);

-- Book-Genre (M:N junction table)
CREATE TABLE book_genres (
    isbn     VARCHAR(13) NOT NULL,
    genre_id INT         NOT NULL,
    CONSTRAINT pk_book_genres PRIMARY KEY (isbn, genre_id),
    CONSTRAINT fk_bg_book  FOREIGN KEY (isbn)     REFERENCES books  (isbn),
    CONSTRAINT fk_bg_genre FOREIGN KEY (genre_id) REFERENCES genres (genre_id)
);

-- Loans (one row per individual loan event)
CREATE TABLE loans (
    loan_id     INT         NOT NULL,
    patron_id   INT         NOT NULL,
    isbn        VARCHAR(13) NOT NULL,
    loan_date   DATE        NOT NULL,
    due_date    DATE        NOT NULL,
    return_date DATE,                 -- NULL if not yet returned
    CONSTRAINT pk_loans       PRIMARY KEY (loan_id),
    CONSTRAINT fk_loan_patron FOREIGN KEY (patron_id) REFERENCES patrons (patron_id),
    CONSTRAINT fk_loan_book   FOREIGN KEY (isbn)      REFERENCES books   (isbn)
);
```

This is in 1NF. Now let us check for 2NF violations.

**Iteration 2: Check for 2NF violations**

The tables with single-column primary keys (`patrons`, `books`, `authors`, `genres`) cannot have partial dependencies by definition.

The junction tables (`book_authors`, `book_genres`) have no non-key attributes at all — their entire row is the composite key. No 2NF violation possible.

The `loans` table has a single-column primary key (`loan_id`). No partial dependencies possible.

All tables are already in 2NF. Good. (In this particular design, 1NF got us to 2NF automatically because no table has a multi-column key with non-key attributes that only depend on part of the key.)

**Iteration 3: Check for 3NF violations**

The `books` table has a transitive dependency:

```
isbn → publisher → publisher_city
isbn → publisher → publisher_country
```

`publisher_city` and `publisher_country` are facts about the publisher, not about the book. A publisher has a single headquarters city. This creates the same update anomalies: if a publisher moves, we update every book they published.

Fix: extract publishers to their own table.

```sql
CREATE TABLE publishers (
    publisher_id   INT          NOT NULL,
    publisher_name VARCHAR(200) NOT NULL,
    city           VARCHAR(100),
    country        VARCHAR(100),
    CONSTRAINT pk_publishers PRIMARY KEY (publisher_id)
);

-- Update books to use publisher_id (foreign key) instead of publisher name
-- and remove publisher_city/publisher_country
CREATE TABLE books (
    isbn         VARCHAR(13)  NOT NULL,
    title        VARCHAR(500) NOT NULL,
    publisher_id INT          NOT NULL,
    year_published INT        NOT NULL,
    CONSTRAINT pk_books         PRIMARY KEY (isbn),
    CONSTRAINT fk_books_pub     FOREIGN KEY (publisher_id) REFERENCES publishers (publisher_id)
);
```

Now `books` is in 3NF: the only non-key attributes are `title`, `publisher_id`, and `year_published`, and all three directly describe the book, not any other entity. `publisher_id` is a foreign key — it is a fact about the book (which publisher published it) that directly depends on the ISBN.

The `authors` table: `author_id → author_name`. No other attributes. Fine.

The `patrons` table: `patron_id → patron_name, email, phone, address`. Do any of these depend on each other? Email does not determine name (two people can have the same name but different emails). Phone does not determine address. We might argue that `address → city, state, zip` is a transitive dependency hidden inside the `address` column — but since we stored the entire address as one column, it is moot for our purposes. If we wanted to query by city, we should store city as a separate column and potentially create a `cities` table. For now, let us say our queries do not require that, so we leave it.

**The final 3NF schema**:

```sql
CREATE TABLE publishers (
    publisher_id   INT          NOT NULL,
    publisher_name VARCHAR(200) NOT NULL,
    city           VARCHAR(100),
    country        VARCHAR(100),
    CONSTRAINT pk_publishers PRIMARY KEY (publisher_id)
);

CREATE TABLE authors (
    author_id     INT          NOT NULL,
    author_name   VARCHAR(200) NOT NULL,
    nationality   VARCHAR(100),
    birth_year    INT,
    CONSTRAINT pk_authors PRIMARY KEY (author_id)
);

CREATE TABLE genres (
    genre_id   INT          NOT NULL,
    genre_name VARCHAR(100) NOT NULL,
    CONSTRAINT pk_genres PRIMARY KEY (genre_id)
);

CREATE TABLE books (
    isbn           VARCHAR(13)  NOT NULL,
    title          VARCHAR(500) NOT NULL,
    publisher_id   INT          NOT NULL,
    year_published INT          NOT NULL,
    CONSTRAINT pk_books     PRIMARY KEY (isbn),
    CONSTRAINT fk_books_pub FOREIGN KEY (publisher_id) REFERENCES publishers (publisher_id)
);

CREATE TABLE book_authors (
    isbn      VARCHAR(13) NOT NULL,
    author_id INT         NOT NULL,
    CONSTRAINT pk_book_authors PRIMARY KEY (isbn, author_id),
    CONSTRAINT fk_ba_book   FOREIGN KEY (isbn)      REFERENCES books   (isbn),
    CONSTRAINT fk_ba_author FOREIGN KEY (author_id) REFERENCES authors (author_id)
);

CREATE TABLE book_genres (
    isbn     VARCHAR(13) NOT NULL,
    genre_id INT         NOT NULL,
    CONSTRAINT pk_book_genres PRIMARY KEY (isbn, genre_id),
    CONSTRAINT fk_bg_book  FOREIGN KEY (isbn)     REFERENCES books  (isbn),
    CONSTRAINT fk_bg_genre FOREIGN KEY (genre_id) REFERENCES genres (genre_id)
);

CREATE TABLE patrons (
    patron_id INT          NOT NULL,
    full_name VARCHAR(200) NOT NULL,
    email     VARCHAR(255) NOT NULL,
    phone     VARCHAR(20),
    address   VARCHAR(500),
    CONSTRAINT pk_patrons        PRIMARY KEY (patron_id),
    CONSTRAINT uq_patron_email   UNIQUE (email)
);

CREATE TABLE loans (
    loan_id     INT          NOT NULL,
    patron_id   INT          NOT NULL,
    isbn        VARCHAR(13)  NOT NULL,
    loan_date   DATE         NOT NULL,
    due_date    DATE         NOT NULL,
    return_date DATE,
    CONSTRAINT pk_loans       PRIMARY KEY (loan_id),
    CONSTRAINT fk_loan_patron FOREIGN KEY (patron_id) REFERENCES patrons (patron_id),
    CONSTRAINT fk_loan_book   FOREIGN KEY (isbn)      REFERENCES books   (isbn)
);
```

Now we can answer questions like "all books by author X" (join `books` to `book_authors` to `authors`), "all books in genre Y" (join `books` to `book_genres` to `genres`), "all patrons who have overdue books" (join `loans` to `patrons` where `return_date IS NULL AND due_date < CURRENT_DATE`), and so on. The data is consistent: there is exactly one place where each fact is stored.

### 5.5 Boyce-Codd Normal Form (BCNF)

**Definition**: A relation is in Boyce-Codd Normal Form if and only if for every non-trivial functional dependency X → Y, X is a superkey.

BCNF was developed in 1974 by Raymond Boyce and Edgar Codd (hence the name) to address anomalies that can persist in 3NF relations when there are multiple overlapping candidate keys. It is slightly stricter than 3NF.

The difference between 3NF and BCNF is subtle. In 3NF, we allow a column to be functionally dependent on a non-superkey attribute, as long as that column is a prime attribute (part of a candidate key). BCNF removes even this exception: every determinant must be a superkey.

Most tables in 3NF are also in BCNF. The cases where 3NF is satisfied but BCNF is not are specific and relatively rare — they require a table with multiple overlapping composite candidate keys.

The classic textbook example:

Imagine a course-scheduling table. The rules are:
- Each student is enrolled in exactly one course per teacher.
- Each teacher teaches exactly one subject.
- A student can take a subject from different teachers in different years.

| StudentID | Teacher | Subject |
|-----------|---------|---------|
| S1 | Dr. Smith | Mathematics |
| S1 | Dr. Jones | Physics |
| S2 | Dr. Smith | Mathematics |
| S3 | Dr. Brown | Mathematics |

The candidate keys are `{StudentID, Teacher}` (a student can have many teachers, a teacher can have many students, but each student-teacher pair is unique) and `{StudentID, Subject}` (a student can take many subjects, but only one class per subject).

Now, `Teacher → Subject` (each teacher teaches exactly one subject). But `Teacher` is not a superkey. This violates BCNF, even though this table is technically in 3NF (because `Subject` is a prime attribute — it is part of the candidate key `{StudentID, Subject}`).

The consequence: if Dr. Smith changes to teach Chemistry instead of Mathematics, you must update every row where `Teacher = 'Dr. Smith'`. Update one, miss another, and your database says Dr. Smith teaches both Mathematics and Chemistry to different students — an inconsistency.

The BCNF decomposition:

```sql
-- Teacher-to-subject mapping (BCNF: teacher is the key, subject depends on it)
CREATE TABLE teacher_subjects (
    teacher_id  INT          NOT NULL,
    teacher_name VARCHAR(200) NOT NULL,
    subject     VARCHAR(200) NOT NULL,
    CONSTRAINT pk_teacher_subjects PRIMARY KEY (teacher_id)
);

-- Student-teacher assignments (BCNF: composite key, no non-key non-key dependencies)
CREATE TABLE student_teachers (
    student_id INT NOT NULL,
    teacher_id INT NOT NULL,
    CONSTRAINT pk_student_teachers PRIMARY KEY (student_id, teacher_id),
    CONSTRAINT fk_st_teacher FOREIGN KEY (teacher_id) REFERENCES teacher_subjects (teacher_id)
);
```

Now Dr. Smith's subject is stored in exactly one place. Changing it requires one update.

The catch: BCNF decomposition does not always preserve functional dependencies losslessly. In the example above, we can no longer directly enforce "a student can only take each subject once" with a simple unique constraint — we would need a query to check. This is why 3NF is sometimes preferred over BCNF: it always guarantees a lossless decomposition that preserves all original functional dependencies.

### 5.6 Fourth Normal Form (4NF) and Beyond

4NF, introduced by Ronald Fagin in 1977, addresses **multi-valued dependencies (MVDs)**. An MVD X →→ Y means that for a given value of X, there are multiple independent values of Y that are not constrained to relate to each other in any way.

Imagine a table that stores the programming skills and the human languages spoken by employees:

| EmployeeID | ProgrammingSkill | SpokenLanguage |
|------------|-----------------|----------------|
| E1 | Python | English |
| E1 | Python | Spanish |
| E1 | C# | English |
| E1 | C# | Spanish |

The `EmployeeID` →→ `ProgrammingSkill` (employee E1 knows Python and C# independently of any spoken language). `EmployeeID` →→ `SpokenLanguage` (employee E1 speaks English and Spanish independently of any programming skill). The two facts are completely independent of each other.

The problem: if E1 learns a new language, French, you must add two new rows (E1, Python, French) and (E1, C#, French). If you miss one, you have an inconsistency. The solution is to decompose into two tables:

```sql
-- Employee programming skills
CREATE TABLE employee_skills (
    employee_id INT          NOT NULL,
    skill       VARCHAR(100) NOT NULL,
    CONSTRAINT pk_employee_skills PRIMARY KEY (employee_id, skill)
);

-- Employee spoken languages
CREATE TABLE employee_languages (
    employee_id INT          NOT NULL,
    language    VARCHAR(100) NOT NULL,
    CONSTRAINT pk_employee_languages PRIMARY KEY (employee_id, language)
);
```

5NF (Fifth Normal Form, also called Project-Join Normal Form) handles **join dependencies** — cases where a table cannot be reconstructed by joining two sub-tables but can be reconstructed by joining three or more. This is very rare in practice.

6NF (Domain-Key Normal Form / Sixth Normal Form) breaks each table down to exactly one non-key attribute per table. This is theoretically pure but impractical for transactional systems — it proliferates tables dramatically. It is used in some columnar data warehouses where the extreme decomposition provides storage and query advantages.

**In practice**: 3NF is the target for transactional systems (OLTP). BCNF addresses the edge cases you are likely to encounter. 4NF is occasionally relevant for junction tables storing independent many-to-many relationships. 5NF and beyond are almost never encountered in typical application development.

## Part 6: Indexes — Making Queries Fast

You cannot talk about relational databases without talking about indexes. A database without indexes is like a book without a table of contents: all the information is there, but you have to read the whole thing to find what you want.

### 6.1 What an Index Is

An **index** is a data structure separate from the table data that provides fast lookup of rows based on the values of one or more columns. The most common type is a **B-tree index** (balanced tree). It stores the indexed column values in sorted order, along with pointers to the actual table rows. Looking up a value in a B-tree takes O(log n) time instead of the O(n) full table scan.

When you define a `PRIMARY KEY`, the database automatically creates a unique index on those columns. When you define a `UNIQUE` constraint, the database automatically creates a unique index.

You create additional indexes explicitly when you know your queries will filter or sort on certain columns:

```sql
-- Index to speed up queries that filter by customer_id in the orders table
CREATE INDEX idx_orders_customer_id ON orders (customer_id);

-- Index to speed up lookups by email in customers
CREATE INDEX idx_customers_email ON customers (email);

-- Composite index: useful when queries filter on both columns together
CREATE INDEX idx_orders_customer_date ON orders (customer_id, order_date);

-- Unique index: enforces uniqueness AND provides fast lookup
CREATE UNIQUE INDEX uq_products_sku ON products (sku);
```

### 6.2 How the Query Optimizer Uses Indexes

When you execute a query, the database's query optimizer considers all available indexes and chooses an execution plan. The key question is: can the database use an index to satisfy the `WHERE`, `JOIN`, or `ORDER BY` clause, or does it have to do a full table scan?

Index usage depends on how the query is written:

```sql
-- Good: exact match on indexed column → index seek (very fast)
SELECT * FROM orders WHERE customer_id = 1001;

-- Good: range on indexed column → index range scan
SELECT * FROM orders WHERE order_date BETWEEN '2026-01-01' AND '2026-03-31';

-- Bad: function applied to indexed column → cannot use index
SELECT * FROM orders WHERE YEAR(order_date) = 2026;
-- The function YEAR() prevents index use. The db must compute YEAR() for every row.
-- Fix:
SELECT * FROM orders WHERE order_date >= '2026-01-01' AND order_date < '2027-01-01';

-- Bad: leading wildcard in LIKE → full scan
SELECT * FROM customers WHERE last_name LIKE '%son';
-- The database cannot skip to any particular point in the index because the
-- match is at the end of the string. Every entry must be checked.

-- Good: trailing wildcard → index range scan on prefix
SELECT * FROM customers WHERE last_name LIKE 'Jo%';
-- The database can jump to the 'Jo' portion of the index and scan from there.
```

### 6.3 Covering Indexes

A **covering index** is an index that contains all the columns needed by a query, so the database can satisfy the query entirely from the index without reading the actual table rows. This can dramatically improve performance.

```sql
-- Query: what are the names and emails of high-spending customers?
SELECT c.first_name, c.last_name, c.email, SUM(o.total_amount) AS total
FROM   customers c
JOIN   orders    o ON c.customer_id = o.customer_id
GROUP BY c.customer_id, c.first_name, c.last_name, c.email
HAVING SUM(o.total_amount) > 1000;

-- A covering index on orders that includes total_amount
-- allows the orders part of the query to be satisfied without touching table rows
CREATE INDEX idx_orders_covering ON orders (customer_id, total_amount);
-- Now the JOIN and SUM can be resolved from the index alone.
```

### 6.4 The Cost of Indexes

Indexes are not free. Every index:

- Consumes storage (sometimes significant — a table with 10 indexes can use 2-3x the storage of a table with none)
- Slows down writes (INSERT, UPDATE, and DELETE must update every index on the table)
- Can mislead the optimizer if statistics are stale

**Index guidelines**:

1. Always index foreign key columns. The database does not do this automatically, and unindexed foreign keys cause very slow `JOIN` operations.
2. Index columns that appear frequently in `WHERE` clauses, especially in high-cardinality columns (many distinct values).
3. Index columns used in `ORDER BY` if you frequently sort on them.
4. Do not index columns with very low cardinality (e.g., a boolean flag — only two distinct values, half the table is in each). The optimizer may ignore such indexes anyway.
5. Do not over-index write-heavy tables. Tables that receive thousands of inserts per second cannot afford many indexes.
6. Regularly review your indexes. Use the database's index usage statistics to find unused indexes and drop them.

```sql
-- PostgreSQL: check index usage
SELECT 
    indexrelname AS index_name,
    relname      AS table_name,
    idx_scan     AS times_used,
    idx_tup_read AS rows_read_via_index
FROM   pg_stat_user_indexes
JOIN   pg_stat_user_tables USING (relid)
ORDER BY idx_scan ASC;  -- Low idx_scan means the index might be unnecessary
```

## Part 7: Advanced Topics and Real-World Considerations

### 7.1 Surrogate Keys vs. Natural Keys

A **natural key** is a key that has meaning in the real world — an ISBN, an email address, a Social Security Number, a product SKU. A **surrogate key** is an artificially generated key with no real-world meaning — typically an auto-incremented integer (`IDENTITY` in SQL Server, `SERIAL` or `GENERATED ALWAYS AS IDENTITY` in PostgreSQL) or a UUID/GUID.

**The case for surrogate keys**:

1. Natural keys change. Email addresses change. People get divorced and change their last name. Companies merge and change their names. Product SKUs get renumbered. When a natural key changes, you must update every foreign key reference throughout your database. With a surrogate key, you update one column in one row, and the foreign keys keep pointing to the same row.

2. Natural keys are often wide. An email address is 50-100 bytes. An integer is 4 bytes. A UUID is 16 bytes. Wide keys slow down index operations and increase the size of every table that holds a foreign key to that table.

3. Some entities do not have obvious natural keys. What is the natural key of a "transaction event"? You probably need to generate one.

**The case for natural keys**:

1. They are self-documenting. `SELECT * FROM orders WHERE product_id = 7` tells you nothing. `SELECT * FROM orders WHERE product_sku = 'WDG-X100'` tells you what product you are talking about.

2. They avoid an extra join. If you always use `orders.customer_email` instead of `orders.customer_id`, you can look up an order without joining to `customers`.

3. They encode meaningful data you might want to query on.

**The pragmatic answer**: Use surrogate keys (auto-increment integers or UUIDs) as primary keys and foreign keys. Also enforce the natural keys as UNIQUE constraints where they exist. Best of both worlds: stable, efficient keys for relationships; business rules enforced by unique constraints.

```sql
CREATE TABLE customers (
    customer_id SERIAL        NOT NULL,  -- surrogate key (PostgreSQL auto-increment)
    email       VARCHAR(255)  NOT NULL,  -- natural key, enforced as unique
    first_name  VARCHAR(100)  NOT NULL,
    last_name   VARCHAR(100)  NOT NULL,
    CONSTRAINT pk_customers      PRIMARY KEY (customer_id),
    CONSTRAINT uq_customer_email UNIQUE (email)
);
```

### 7.2 UUID vs. Auto-Increment Integer Keys

Auto-increment integers (`BIGINT GENERATED ALWAYS AS IDENTITY`) are simple, compact, and sequential. Their sequentiality is both an advantage (sequential inserts are fast) and a disadvantage (it reveals information about your data — customer ID 1001 was created before customer 1002; how many customers do you have?).

UUIDs (Universally Unique Identifiers, 128-bit values) are globally unique, which means:
- Records can be created in distributed systems without a central authority assigning IDs.
- IDs can be generated client-side before the database is even contacted.
- IDs do not reveal the volume of your data.

The traditional GUID (random UUID v4) is problematic as a clustered primary key in SQL Server or as a B-tree index key in general, because random inserts into a sorted B-tree cause page splits and fragmentation.

The modern solution is **UUID v7** (introduced in the UUID specification update of 2022), which is time-ordered: the first bits encode a timestamp, making the UUIDs roughly sequential while still being globally unique and opaque. PostgreSQL 17 added a native `gen_random_uuid()` function that generates v4 UUIDs, and v7 support is available via extensions or application-level generation.

```csharp
// Generating a UUID v7 in C# (.NET 9+)
// As of .NET 9, Guid.CreateVersion7() is available
Guid newId = Guid.CreateVersion7();

// Use as a primary key when inserting a new record
await using var cmd = new NpgsqlCommand(
    "INSERT INTO events (event_id, event_type, event_data) VALUES (@id, @type, @data)",
    connection);
cmd.Parameters.AddWithValue("id", newId);
cmd.Parameters.AddWithValue("type", "user.registered");
cmd.Parameters.AddWithValue("data", JsonSerializer.Serialize(eventData));
await cmd.ExecuteNonQueryAsync();
```

### 7.3 When to Denormalize (and When Not To)

All the normalization work we have done reduces redundancy and prevents anomalies. So why would anyone ever denormalize?

**Performance**. Normalized schemas require JOINs. JOINs have a cost. In an OLTP (Online Transaction Processing) system handling thousands of short transactions per second, the cost of joins on properly indexed tables is usually acceptable. But in an OLAP (Online Analytical Processing) system — a data warehouse, a reporting database, a business intelligence platform — queries frequently aggregate across millions of rows and dozens of tables. The cost of joins at that scale can be prohibitive.

Data warehouses therefore often use **denormalized schemas** intentionally:
- **Star schema**: A central "fact table" (containing measurements like sales amounts, quantities) surrounded by "dimension tables" (containing descriptive attributes like customer name, product name, date). The fact table is deliberately denormalized — it contains some redundant data to avoid expensive joins.
- **Snowflake schema**: A more normalized variation of the star schema where dimension tables themselves have further normalized sub-tables.

The key insight: **the right level of normalization depends on the workload**. OLTP systems should be normalized. OLAP systems are often intentionally denormalized.

Even in OLTP systems, there are valid reasons to denormalize:

1. **Computed columns for frequently-needed aggregates**: If you always need `total_amount` on an order (sum of all order line items), computing it via `SUM(quantity * unit_price)` on every read is expensive. Storing it in a `total_amount` column on `orders` and maintaining it via triggers or application logic is a pragmatic denormalization.

2. **Materialized views**: A database-managed denormalized view that is refreshed periodically. You define the query; the database computes and stores the result; queries hit the pre-computed result.

3. **Read replicas and caching**: For extreme read throughput, you can have denormalized, read-only replicas of your data that are updated asynchronously from the normalized source.

**The rule**: normalize first. Denormalize only when you have measured a specific performance problem that denormalization solves, you understand the trade-offs, and you have a plan for maintaining consistency.

### 7.4 Views — Virtual Tables

A **view** is a named query that is stored in the database and can be used like a table in subsequent queries. Views do not store data (unless they are materialized views); they are just saved query definitions.

```sql
-- Create a view that joins customers and orders for a clean interface
CREATE VIEW v_customer_orders AS
SELECT
    c.customer_id,
    c.first_name || ' ' || c.last_name AS customer_name,
    c.email,
    o.order_id,
    o.order_date,
    o.total_amount
FROM      customers c
LEFT JOIN orders    o ON c.customer_id = o.customer_id;

-- Query the view as if it were a table
SELECT *
FROM   v_customer_orders
WHERE  order_date >= '2026-01-01'
ORDER BY total_amount DESC;
```

Views are useful for:
- **Simplifying complex queries**: The join lives in the view; application code queries the view.
- **Security**: Grant users access to a view that exposes only certain columns (e.g., a view without salary information).
- **Providing a stable interface**: If the underlying table structure changes, update the view; application code that queries the view continues to work.

### 7.5 Stored Procedures and Functions

**Stored procedures** are named, compiled SQL programs stored in the database that can be called by applications. They can contain procedural logic (loops, conditionals, variables).

```sql
-- PostgreSQL stored function (using PL/pgSQL)
CREATE OR REPLACE FUNCTION transfer_funds(
    p_source_account INT,
    p_dest_account   INT,
    p_amount         DECIMAL(10,2)
)
RETURNS VOID
LANGUAGE plpgsql
AS $$
BEGIN
    -- Check sufficient funds
    IF (SELECT balance FROM accounts WHERE account_id = p_source_account) < p_amount THEN
        RAISE EXCEPTION 'Insufficient funds in account %', p_source_account;
    END IF;

    UPDATE accounts SET balance = balance - p_amount WHERE account_id = p_source_account;
    UPDATE accounts SET balance = balance + p_amount WHERE account_id = p_dest_account;
    
    INSERT INTO transaction_log (source, dest, amount, logged_at)
    VALUES (p_source_account, p_dest_account, p_amount, NOW());
END;
$$;

-- Call the stored procedure
SELECT transfer_funds(1001, 1002, 500.00);
```

**Opinions on stored procedures vary widely** in the development community. Arguments for them: they encapsulate business logic at the database layer; they reduce network round-trips; they are compiled and cached. Arguments against them: they split business logic between the application and the database; they are hard to version-control and test; they create tight coupling between the application and the specific database.

In modern .NET development, the dominant approach is to keep business logic in the application (C# code) and use the database for storage and constraint enforcement, not complex logic. However, stored procedures remain appropriate for performance-critical bulk operations and for legacy systems.

### 7.6 Triggers

A **trigger** is code that automatically executes when a specified event (INSERT, UPDATE, DELETE) occurs on a table.

```sql
-- Automatically set updated_at timestamp when a row changes
CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS TRIGGER
LANGUAGE plpgsql
AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$;

CREATE TRIGGER trg_products_updated_at
BEFORE UPDATE ON products
FOR EACH ROW
EXECUTE FUNCTION set_updated_at();
```

Triggers are useful for: auditing (logging every change to a history table), maintaining denormalized aggregate values, enforcing complex business rules that cannot be expressed as simple constraints. They are dangerous for complex business logic — triggers that call other triggers, or triggers that modify tables that have triggers, create debugging nightmares. Use them sparingly and document them clearly.

## Part 8: SQL in C# and ASP.NET — Practical Integration

### 8.1 The Landscape: ADO.NET, Dapper, Entity Framework Core

In the .NET ecosystem, there are three main approaches to talking to a relational database:

**ADO.NET** is the lowest-level API. You write SQL strings, create connection objects, create command objects, execute them, and read results row by row. Maximum control, maximum verbosity.

**Dapper** is a micro-ORM built on top of ADO.NET. You still write SQL, but Dapper handles mapping query results to strongly typed C# objects. Very thin layer, minimal overhead, SQL stays in control.

**Entity Framework Core (EF Core)** is a full ORM (Object-Relational Mapper). You define C# classes (entities) and let EF Core generate SQL for you. You can write LINQ queries and never see the actual SQL. Highly productive for CRUD operations, can generate complex SQL for simple scenarios, can generate extremely bad SQL for complex scenarios.

The right choice depends on your project:

| Scenario | Best Choice |
|---|---|
| Complex, performance-critical queries | ADO.NET or Dapper |
| Simple CRUD with rapid development needs | EF Core |
| Mixed: simple CRUD + complex reporting | EF Core + Dapper (side by side) |
| Full control, no surprises | Dapper |
| Rapid prototyping | EF Core |

### 8.2 ADO.NET: The Foundation

```csharp
// Program.cs or a service class in your ASP.NET application
// Always use parameterized queries — never build SQL strings by concatenation.

public class CustomerRepository
{
    private readonly string _connectionString;

    public CustomerRepository(string connectionString)
    {
        _connectionString = connectionString;
    }

    public async Task<IReadOnlyList<Customer>> GetCustomersByCity(string city)
    {
        const string sql = """
            SELECT customer_id, first_name, last_name, email, phone
            FROM   customers
            WHERE  city = @city
            ORDER BY last_name, first_name
            """;

        var customers = new List<Customer>();

        await using var conn = new NpgsqlConnection(_connectionString);
        await conn.OpenAsync();

        await using var cmd = new NpgsqlCommand(sql, conn);
        cmd.Parameters.AddWithValue("city", city);

        await using var reader = await cmd.ExecuteReaderAsync();
        while (await reader.ReadAsync())
        {
            customers.Add(new Customer(
                reader.GetInt32(0),
                reader.GetString(1),
                reader.GetString(2),
                reader.GetString(3),
                reader.IsDBNull(4) ? null : reader.GetString(4)
            ));
        }

        return customers;
    }

    public async Task<int> InsertCustomer(Customer customer)
    {
        const string sql = """
            INSERT INTO customers (first_name, last_name, email, phone)
            VALUES (@firstName, @lastName, @email, @phone)
            RETURNING customer_id
            """;

        await using var conn = new NpgsqlConnection(_connectionString);
        await conn.OpenAsync();

        await using var cmd = new NpgsqlCommand(sql, conn);
        cmd.Parameters.AddWithValue("firstName", customer.FirstName);
        cmd.Parameters.AddWithValue("lastName", customer.LastName);
        cmd.Parameters.AddWithValue("email", customer.Email);
        cmd.Parameters.AddWithValue("phone", (object?)customer.Phone ?? DBNull.Value);

        var result = await cmd.ExecuteScalarAsync();
        return Convert.ToInt32(result);
    }
}
```

**SQL Injection**: The reason you always use parameterized queries (`@parameterName` syntax with `Parameters.AddWithValue`) and never build SQL by string concatenation is SQL injection. If you write:

```csharp
// NEVER DO THIS
var sql = $"SELECT * FROM users WHERE email = '{userInput}'";
```

...and the user inputs `'; DROP TABLE users; --`, the resulting SQL is:

```sql
SELECT * FROM users WHERE email = ''; DROP TABLE users; --'
```

This executes two statements: the SELECT, then a DROP TABLE. Your users table is gone. Parameterized queries prevent this completely — the parameter value is sent separately from the SQL text and cannot be interpreted as SQL.

This is not a theoretical risk. SQL injection attacks are one of the most common and most destructive categories of web application vulnerability. The OWASP Top 10 has included injection attacks (of which SQL injection is the most common) in virtually every edition since 2003.

### 8.3 Dapper: Simple and Effective

```csharp
// Add NuGet package: Dapper
// Dapper extends IDbConnection with extension methods

public class CustomerRepository
{
    private readonly string _connectionString;

    public CustomerRepository(string connectionString)
    {
        _connectionString = connectionString;
    }

    private NpgsqlConnection CreateConnection() =>
        new NpgsqlConnection(_connectionString);

    public async Task<IReadOnlyList<Customer>> GetCustomersByCity(string city)
    {
        const string sql = """
            SELECT 
                customer_id AS CustomerId,
                first_name  AS FirstName,
                last_name   AS LastName,
                email,
                phone
            FROM   customers
            WHERE  city = @City
            ORDER BY last_name, first_name
            """;

        await using var conn = CreateConnection();
        // Dapper maps query results to Customer objects automatically
        var customers = await conn.QueryAsync<Customer>(sql, new { City = city });
        return customers.AsList();
    }

    public async Task<Customer?> GetById(int customerId)
    {
        const string sql = """
            SELECT customer_id AS CustomerId, first_name AS FirstName, 
                   last_name AS LastName, email, phone
            FROM   customers
            WHERE  customer_id = @CustomerId
            """;

        await using var conn = CreateConnection();
        return await conn.QuerySingleOrDefaultAsync<Customer>(sql, new { CustomerId = customerId });
    }

    public async Task<IReadOnlyList<CustomerOrderSummary>> GetOrderSummaries()
    {
        const string sql = """
            SELECT 
                c.customer_id AS CustomerId,
                c.first_name  AS FirstName,
                c.last_name   AS LastName,
                COUNT(o.order_id) AS OrderCount,
                COALESCE(SUM(o.total_amount), 0) AS TotalSpent
            FROM      customers c
            LEFT JOIN orders    o ON c.customer_id = o.customer_id
            GROUP BY c.customer_id, c.first_name, c.last_name
            ORDER BY TotalSpent DESC
            """;

        await using var conn = CreateConnection();
        var results = await conn.QueryAsync<CustomerOrderSummary>(sql);
        return results.AsList();
    }
}

public record Customer(int CustomerId, string FirstName, string LastName, string Email, string? Phone);
public record CustomerOrderSummary(int CustomerId, string FirstName, string LastName, int OrderCount, decimal TotalSpent);
```

### 8.4 Entity Framework Core: The Productive Option

EF Core is the right choice when you want rapid development and are willing to accept some opacity in how queries are generated.

```csharp
// Define your entities (C# classes that represent database tables)
public class Customer
{
    public int    CustomerId { get; set; }
    public string FirstName  { get; set; } = "";
    public string LastName   { get; set; } = "";
    public string Email      { get; set; } = "";
    public string? Phone     { get; set; }

    // Navigation property: collection of orders
    public ICollection<Order> Orders { get; set; } = [];
}

public class Order
{
    public int      OrderId     { get; set; }
    public int      CustomerId  { get; set; }
    public DateTime OrderDate   { get; set; }
    public decimal  TotalAmount { get; set; }

    // Navigation properties
    public Customer           Customer     { get; set; } = null!;
    public ICollection<OrderDetail> OrderDetails { get; set; } = [];
}

// DbContext: the bridge between your entities and the database
public class AppDbContext : DbContext
{
    public AppDbContext(DbContextOptions<AppDbContext> options) : base(options) { }

    public DbSet<Customer> Customers { get; set; } = null!;
    public DbSet<Order>    Orders    { get; set; } = null!;

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        modelBuilder.Entity<Customer>(e =>
        {
            e.HasKey(c => c.CustomerId);
            e.HasIndex(c => c.Email).IsUnique();
            e.Property(c => c.FirstName).HasMaxLength(100).IsRequired();
            e.Property(c => c.LastName).HasMaxLength(100).IsRequired();
            e.Property(c => c.Email).HasMaxLength(255).IsRequired();
        });

        modelBuilder.Entity<Order>(e =>
        {
            e.HasKey(o => o.OrderId);
            e.Property(o => o.TotalAmount).HasPrecision(10, 2);
            e.HasOne(o => o.Customer)
             .WithMany(c => c.Orders)
             .HasForeignKey(o => o.CustomerId)
             .OnDelete(DeleteBehavior.Restrict);
        });
    }
}

// Register in Program.cs / Startup.cs
builder.Services.AddDbContext<AppDbContext>(options =>
    options.UseNpgsql(builder.Configuration.GetConnectionString("DefaultConnection")));

// Use in a service or controller
public class CustomerService
{
    private readonly AppDbContext _db;

    public CustomerService(AppDbContext db)
    {
        _db = db;
    }

    public async Task<List<CustomerOrderSummary>> GetTopCustomers(int count)
    {
        return await _db.Customers
            .Select(c => new CustomerOrderSummary
            {
                CustomerId  = c.CustomerId,
                FullName    = c.FirstName + " " + c.LastName,
                OrderCount  = c.Orders.Count(),
                TotalSpent  = c.Orders.Sum(o => o.TotalAmount)
            })
            .OrderByDescending(s => s.TotalSpent)
            .Take(count)
            .ToListAsync();
    }

    public async Task<Customer?> GetWithOrders(int customerId)
    {
        return await _db.Customers
            .Include(c => c.Orders)
            .FirstOrDefaultAsync(c => c.CustomerId == customerId);
    }
}
```

**EF Core warnings**:

1. Always check the SQL EF Core generates for complex queries. Call `ToQueryString()` on an IQueryable to see the generated SQL, or use the logging configuration to log all SQL to the console during development.

2. Avoid N+1 query problems. If you load a list of customers and then access `customer.Orders` for each customer in a loop, EF Core will issue one query per customer. Use `.Include(c => c.Orders)` or `.Select()` projections to load all data in a single query.

3. Do not use lazy loading in production. EF Core supports lazy loading (automatically loading related data when you access a navigation property), but it makes N+1 problems nearly invisible and is extremely difficult to reason about.

4. For complex reports and analytics, drop out of LINQ and use Dapper or raw SQL (EF Core supports `FromSqlRaw` and `SqlQuery` for this).

```csharp
// Using raw SQL in EF Core when LINQ cannot express the query efficiently
var results = await _db.Database
    .SqlQueryRaw<CustomerOrderSummary>("""
        SELECT 
            c.customer_id AS CustomerId,
            c.first_name || ' ' || c.last_name AS FullName,
            COUNT(o.order_id) AS OrderCount,
            COALESCE(SUM(o.total_amount), 0) AS TotalSpent
        FROM      customers c
        LEFT JOIN orders    o ON c.customer_id = o.customer_id
        GROUP BY  c.customer_id, c.first_name, c.last_name
        HAVING    COUNT(o.order_id) > @minOrders
        ORDER BY  TotalSpent DESC
        """, new NpgsqlParameter("minOrders", 5))
    .ToListAsync();
```

### 8.5 Connection Pooling and IHttpClientFactory Patterns

Database connections are expensive to create. Opening a connection involves a TCP handshake, authentication, and session setup. In a high-traffic web application, creating a new connection for every request would be catastrophically slow.

**Connection pooling** keeps a set of open connections ready to reuse. You "borrow" a connection from the pool, use it, then "return" it to the pool when you are done. The actual TCP connection is not closed; it is held in the pool for the next caller. Npgsql (PostgreSQL driver) and Microsoft.Data.SqlClient (SQL Server driver) both include built-in connection pooling.

In ASP.NET Core with EF Core, connection pooling is managed automatically if you use the correct DI registration:

```csharp
// CORRECT: Scoped lifetime (one DbContext instance per HTTP request)
// EF Core manages connection pooling internally
builder.Services.AddDbContext<AppDbContext>(options =>
    options.UseNpgsql(connectionString));

// For Dapper, inject a factory or the connection string and create connections on demand
// Do NOT inject NpgsqlConnection directly with Singleton lifetime
builder.Services.AddScoped<ICustomerRepository, CustomerRepository>();
```

The golden rules:
- **Scoped** lifetime for DbContext (one per HTTP request).
- **Transient** for anything that creates database connections — do not hold connections open longer than necessary.
- **Never** store a database connection as a class-level field unless you are explicitly managing its lifetime.
- Always await all async database calls — do not fire-and-forget.
- Always `using` or `await using` connections and commands to ensure they are returned to the pool.

## Part 9: Database Migration Strategies

### 9.1 Schema Evolution: The Hardest Part of Database Development

Getting your schema right the first time is nearly impossible. Requirements change. Business rules evolve. You discover that the way you modeled orders does not accommodate subscription billing. You realize your `customers` table needs to support organizations as well as individuals.

Changing a live database schema while preserving data and maintaining application availability is one of the hardest problems in software engineering. This section covers the main approaches.

### 9.2 EF Core Migrations

EF Core includes a built-in migrations system that tracks changes to your entity model and generates SQL scripts to apply those changes to the database.

```bash
# Install EF Core tools
dotnet tool install --global dotnet-ef

# Add a migration (after changing your entity model)
dotnet ef migrations add AddCustomerCity

# Apply pending migrations to the database
dotnet ef database update

# Generate a SQL script instead of applying directly (for production)
dotnet ef migrations script 0 AddCustomerCity --output migration.sql
```

The migration tool generates a C# class:

```csharp
public partial class AddCustomerCity : Migration
{
    protected override void Up(MigrationBuilder migrationBuilder)
    {
        migrationBuilder.AddColumn<string>(
            name: "city",
            table: "customers",
            type: "character varying(100)",
            maxLength: 100,
            nullable: true);
    }

    protected override void Down(MigrationBuilder migrationBuilder)
    {
        migrationBuilder.DropColumn(
            name: "city",
            table: "customers");
    }
}
```

EF Core migrations work well for small to medium applications. They become unwieldy when:
- Multiple teams are working on the same schema simultaneously
- The database is very large and migrations take a long time
- You need fine-grained control over generated SQL

### 9.3 Flyway and Liquibase

For more control, **Flyway** and **Liquibase** are widely used migration tools. Both work with raw SQL migration files and track which migrations have been applied. They are language-agnostic and work with any application.

Flyway uses a simple naming convention:

```
V1__Create_customers_table.sql
V2__Add_orders_table.sql
V3__Add_customer_city_column.sql
V4__Create_products_table.sql
```

Each file is applied in order, exactly once. Flyway tracks applied migrations in a `flyway_schema_history` table.

### 9.4 Zero-Downtime Migrations

Changing a live database without taking the application offline requires careful technique. The key rules:

1. **Never rename a column directly**. Add the new column, write code that writes to both old and new, deploy, backfill data, update code to read from new column, deploy, then drop the old column.

2. **Never remove a column until the code that reads it is fully deployed** and you have verified no traffic accesses it.

3. **Add NOT NULL columns with a default value** or in two steps (add as nullable first, backfill, add NOT NULL constraint).

4. **Create indexes CONCURRENTLY** (PostgreSQL) or use an `ONLINE` operation (SQL Server) to avoid locking the table during index creation.

```sql
-- PostgreSQL: create index without locking the table
CREATE INDEX CONCURRENTLY idx_orders_customer_id ON orders (customer_id);

-- SQL Server: similar
CREATE INDEX idx_orders_customer_id ON orders (customer_id) WITH (ONLINE = ON);
```

## Part 10: Common Anti-Patterns and How to Fix Them

### 10.1 The God Table

A "god table" is a single table that stores multiple different types of entities using a `type` discriminator column and many nullable columns that only apply to certain types.

```sql
-- ANTI-PATTERN: God table
CREATE TABLE entities (
    entity_id      INT          NOT NULL,
    entity_type    VARCHAR(50)  NOT NULL,  -- 'person', 'company', 'product', 'invoice'
    name           VARCHAR(200) NOT NULL,
    first_name     VARCHAR(100),           -- only for persons
    last_name      VARCHAR(100),           -- only for persons
    company_reg_no VARCHAR(50),            -- only for companies
    unit_price     DECIMAL(10,2),          -- only for products
    total_due      DECIMAL(10,2),          -- only for invoices
    -- ... 50 more nullable columns ...
    CONSTRAINT pk_entities PRIMARY KEY (entity_id)
);
```

This table is terrible for the same reasons as all the other bad tables we have seen: you cannot enforce NOT NULL on fields that are required for one entity type (what column can you put NOT NULL on for person-specific fields?), foreign keys are meaningless (who does `invoice.customer_id` reference — another entity? how do you ensure it is a company, not a product?), and queries become a jungle of `WHERE entity_type = 'person' AND first_name IS NOT NULL`.

The fix: separate tables for separate entities. If they share common attributes, use table inheritance (a base table and type-specific tables linked by a 1:1 foreign key).

```sql
-- BASE TABLE: attributes common to all parties
CREATE TABLE parties (
    party_id   INT          NOT NULL,
    party_type VARCHAR(50)  NOT NULL,   -- 'person' or 'company', for routing
    name       VARCHAR(200) NOT NULL,   -- display name
    email      VARCHAR(255),
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    CONSTRAINT pk_parties PRIMARY KEY (party_id)
);

-- PERSON: person-specific attributes
CREATE TABLE persons (
    party_id   INT          NOT NULL,
    first_name VARCHAR(100) NOT NULL,
    last_name  VARCHAR(100) NOT NULL,
    birth_date DATE,
    CONSTRAINT pk_persons PRIMARY KEY (party_id),
    CONSTRAINT fk_persons_party FOREIGN KEY (party_id) REFERENCES parties (party_id)
);

-- COMPANY: company-specific attributes
CREATE TABLE companies (
    party_id       INT          NOT NULL,
    legal_name     VARCHAR(200) NOT NULL,
    registration_no VARCHAR(50),
    CONSTRAINT pk_companies PRIMARY KEY (party_id),
    CONSTRAINT fk_companies_party FOREIGN KEY (party_id) REFERENCES parties (party_id)
);
```

### 10.2 Storing Relationships as Comma-Separated Strings

```sql
-- ANTI-PATTERN: storing tags as a comma-separated string
UPDATE posts SET tags = 'dotnet,csharp,blazor,webassembly' WHERE post_id = 42;

-- How do you query "all posts tagged 'blazor'"?
SELECT * FROM posts WHERE tags LIKE '%blazor%';
-- This does a full table scan and also matches 'blazor-webassembly' accidentally.
```

The correct approach is a junction table:

```sql
CREATE TABLE tags (
    tag_id   INT          NOT NULL,
    tag_name VARCHAR(100) NOT NULL,
    CONSTRAINT pk_tags       PRIMARY KEY (tag_id),
    CONSTRAINT uq_tags_name  UNIQUE (tag_name)
);

CREATE TABLE post_tags (
    post_id INT NOT NULL,
    tag_id  INT NOT NULL,
    CONSTRAINT pk_post_tags   PRIMARY KEY (post_id, tag_id),
    CONSTRAINT fk_pt_post     FOREIGN KEY (post_id) REFERENCES posts (post_id),
    CONSTRAINT fk_pt_tag      FOREIGN KEY (tag_id)  REFERENCES tags  (tag_id)
);

-- Query: all posts tagged 'blazor'
SELECT p.*
FROM   posts     p
JOIN   post_tags pt ON p.post_id = pt.post_id
JOIN   tags      t  ON pt.tag_id = t.tag_id
WHERE  t.tag_name = 'blazor';
```

This allows exact matching, indexing, and all the referential integrity guarantees of a proper foreign key.

### 10.3 Using VARCHAR(MAX) / TEXT for Everything

```sql
-- ANTI-PATTERN: no thought given to column sizes
CREATE TABLE users_bad (
    user_id  INT,
    name     TEXT,
    email    TEXT,
    zip_code TEXT,
    age      TEXT,   -- storing numeric data as text
    balance  TEXT    -- storing money as text
);
```

Problems: no size enforcement (a typo could insert a 50,000-character "name"), no type enforcement (what is `'twenty dollars'` as a balance?), sort order for numbers stored as text is alphabetical not numeric ("9" > "10" as text), no arithmetic possible on balance, no range validation on age.

Use appropriate types: `INT` for integers, `DECIMAL(precision, scale)` for money, `DATE`/`TIMESTAMP WITH TIME ZONE` for dates, `BOOLEAN` for booleans, `VARCHAR(n)` with a sensible limit for strings, `CHAR(n)` for fixed-length strings like country codes and currency codes.

### 10.4 Not Using Transactions for Multi-Step Operations

```csharp
// ANTI-PATTERN: each operation is independent, no transaction
await _db.ExecuteAsync("UPDATE accounts SET balance = balance - @amount WHERE id = @source", ...);
// If the application crashes here, money has been debited but not credited
await _db.ExecuteAsync("UPDATE accounts SET balance = balance + @amount WHERE id = @dest", ...);
```

Always wrap related operations in a transaction. See section 3.5.

### 10.5 Not Indexing Foreign Keys

```sql
-- ANTI-PATTERN: foreign key without an index
CREATE TABLE orders (
    order_id    INT NOT NULL,
    customer_id INT NOT NULL,
    ...
    CONSTRAINT pk_orders PRIMARY KEY (order_id),
    CONSTRAINT fk_orders_customer FOREIGN KEY (customer_id) REFERENCES customers (customer_id)
    -- No index on customer_id!
);
```

When you join `customers` to `orders` on `customer_id`, the database must search through the entire `orders` table for each customer. With an index, it can jump directly to the matching rows:

```sql
-- Add this after every foreign key definition
CREATE INDEX idx_orders_customer_id ON orders (customer_id);
```

PostgreSQL does not automatically index foreign key columns. SQL Server does not either. MySQL does, using its InnoDB storage engine — but the other databases do not. Manually add indexes on all foreign key columns that you query on.

## Part 11: Choosing a Database — PostgreSQL, SQL Server, MySQL, SQLite

Since this is a .NET / ASP.NET audience, you are likely using or considering one of these four databases. Here is a quick comparison for context.

### PostgreSQL

PostgreSQL (often just called "Postgres") is a free, open-source, advanced relational database. It is the strongest implementation of the SQL standard among the major databases, with excellent support for JSON, full-text search, geographic data (PostGIS), window functions, CTEs, and foreign data wrappers. Its MVCC (Multi-Version Concurrency Control) implementation is excellent.

For new .NET projects, PostgreSQL with Npgsql and EF Core is an excellent choice and completely free of cost for any use. The `Npgsql.EntityFrameworkCore.PostgreSQL` package is maintained by the Npgsql team and is excellent.

PostgreSQL version 17 was released in September 2024.

### SQL Server

Microsoft SQL Server is the natural choice in a pure Microsoft ecosystem. It integrates tightly with Azure, Azure Active Directory, and other Microsoft services. It is well-supported in EF Core and has excellent tooling (SQL Server Management Studio, Azure Data Studio). The free tier, SQL Server Express, is limited to 10 GB database size per database.

T-SQL (SQL Server's dialect) has some differences from standard SQL: `TOP` instead of `LIMIT`, `GETDATE()` instead of `NOW()`, identity columns via `IDENTITY(1,1)`. SQL Server 2022 was released in November 2022.

### MySQL

MySQL is the most widely deployed database in the world (by install count), largely due to its use in the LAMP stack (Linux, Apache, MySQL, PHP) that powered the early web. It is reliable for many use cases but has some significant deviations from standard SQL, particularly in older versions' handling of `GROUP BY`. MariaDB is a community-maintained fork.

For .NET projects, `MySql.EntityFrameworkCore` or `Pomelo.EntityFrameworkCore.MySql` (Pomelo is generally recommended) provide EF Core integration.

### SQLite

SQLite is a completely different kind of database. Rather than a client-server system (separate database server process), SQLite is an embedded database — a single `.db` file that your application reads and writes directly using a library linked into the application. No separate server process, no network, no installation.

SQLite is excellent for:
- Mobile applications (it is the embedded database for iOS and Android)
- Desktop applications
- Development and testing
- Unit tests (use an in-memory SQLite database for fast, isolated tests)
- Applications with modest data volumes and single-user or read-heavy access patterns

SQLite is not appropriate for:
- High-concurrency writes (it uses file-level locking for writes)
- Very large databases
- Multiple applications accessing the same database simultaneously

In .NET, `Microsoft.EntityFrameworkCore.Sqlite` provides EF Core integration. SQLite is included in .NET's `System.Data.SQLite` or via the `Microsoft.Data.Sqlite` package.

For the kind of showcase and sample project that a Blazor WebAssembly application might be, SQLite is a perfect fit for the backend database (if you have a backend at all). For a Blazor WASM app deployed to GitHub Pages with no backend, you would be loading data as static JSON files — there is no running database server.

## Part 12: Practical Database Testing in .NET

### 12.1 Why Testing Database Code Is Important

Business logic that touches the database is among the hardest code to test correctly, and also among the most important. Bugs in data access code cause data corruption, lost transactions, incorrect reports, and security vulnerabilities. You need tests that exercise the actual SQL you are running against an actual database.

There are two main approaches:

**Unit tests with mocked data access**: You mock the database interface and test the business logic in isolation. Fast but does not catch SQL syntax errors, query logic errors, or ORM mapping problems.

**Integration tests with a real database**: You run your tests against an actual database, set up test data, run operations, and assert on results. Slower but catches the real problems.

For the integration test database, you have options:
- **In-memory SQLite**: Fast, requires no setup, but SQLite and your production database (e.g., PostgreSQL) have dialect differences that can hide real bugs.
- **Real PostgreSQL in a container**: Use Docker or Testcontainers to spin up a PostgreSQL container for each test run. Slow to start (first time) but perfectly accurate. Testcontainers for .NET makes this easy.
- **Local PostgreSQL with a dedicated test database**: Simple, fast, but requires developer setup.

### 12.2 Integration Tests with Testcontainers

```csharp
// NuGet: Testcontainers.PostgreSql, xunit

public class CustomerRepositoryTests : IAsyncLifetime
{
    private PostgreSqlContainer? _postgresContainer;
    private AppDbContext?        _db;

    public async Task InitializeAsync()
    {
        _postgresContainer = new PostgreSqlBuilder()
            .WithImage("postgres:17-alpine")
            .Build();

        await _postgresContainer.StartAsync();

        var options = new DbContextOptionsBuilder<AppDbContext>()
            .UseNpgsql(_postgresContainer.GetConnectionString())
            .Options;

        _db = new AppDbContext(options);
        await _db.Database.MigrateAsync();  // Apply all migrations
    }

    public async Task DisposeAsync()
    {
        if (_db != null) await _db.DisposeAsync();
        if (_postgresContainer != null) await _postgresContainer.DisposeAsync();
    }

    [Fact]
    public async Task GetCustomersByCity_ReturnsOnlyMatchingCustomers()
    {
        // Arrange
        _db!.Customers.AddRange(
            new Customer { FirstName = "Alice", LastName = "A", Email = "alice@test.com", City = "London" },
            new Customer { FirstName = "Bob",   LastName = "B", Email = "bob@test.com",   City = "Paris"  },
            new Customer { FirstName = "Carol",  LastName = "C", Email = "carol@test.com", City = "London" }
        );
        await _db.SaveChangesAsync();

        var repo = new CustomerRepository(_postgresContainer!.GetConnectionString());

        // Act
        var result = await repo.GetCustomersByCity("London");

        // Assert
        Assert.Equal(2, result.Count);
        Assert.All(result, c => Assert.Equal("London", c.City));
    }
}
```

## Part 13: A Complete End-to-End Example — A Blog Engine Schema

Let us put all of this together with a real, practical example. We will design the database for a blog engine, which is a domain every web developer will immediately recognise.

### 13.1 Requirements

A blog engine needs to support:
- Authors who write posts
- Posts with title, body, publication date, status (draft/published)
- Tags (posts can have multiple tags; each tag can apply to multiple posts)
- Comments on posts (from readers)
- Likes/reactions on posts and comments

### 13.2 ER Analysis

Entities:
- `Author` (author_id, display_name, email, bio, avatar_url)
- `Post` (post_id, author_id, title, slug, body, status, created_at, published_at, updated_at)
- `Tag` (tag_id, name, slug)
- `Comment` (comment_id, post_id, author_name, author_email, body, status, created_at)
- `Reaction` (reaction_id, post_id, reaction_type, created_at, ip_address)

Relationships:
- Author writes Posts (1:N)
- Post has Tags (M:N — junction table `post_tags`)
- Post has Comments (1:N)
- Post has Reactions (1:N)

### 13.3 The Final Schema

```sql
-- Authors
CREATE TABLE authors (
    author_id  SERIAL        NOT NULL,
    email      VARCHAR(255)  NOT NULL,
    display_name VARCHAR(200) NOT NULL,
    bio        TEXT,
    avatar_url VARCHAR(500),
    created_at TIMESTAMPTZ   NOT NULL DEFAULT NOW(),
    CONSTRAINT pk_authors        PRIMARY KEY (author_id),
    CONSTRAINT uq_authors_email  UNIQUE (email)
);

-- Posts
CREATE TABLE posts (
    post_id      SERIAL        NOT NULL,
    author_id    INT           NOT NULL,
    title        VARCHAR(500)  NOT NULL,
    slug         VARCHAR(500)  NOT NULL,  -- URL-friendly identifier: "my-first-post"
    body         TEXT          NOT NULL,
    status       VARCHAR(20)   NOT NULL DEFAULT 'draft',  -- 'draft', 'published'
    created_at   TIMESTAMPTZ   NOT NULL DEFAULT NOW(),
    published_at TIMESTAMPTZ,
    updated_at   TIMESTAMPTZ   NOT NULL DEFAULT NOW(),
    CONSTRAINT pk_posts         PRIMARY KEY (post_id),
    CONSTRAINT uq_posts_slug    UNIQUE (slug),
    CONSTRAINT chk_posts_status CHECK (status IN ('draft', 'published')),
    CONSTRAINT fk_posts_author  FOREIGN KEY (author_id) REFERENCES authors (author_id)
);

CREATE INDEX idx_posts_author_id    ON posts (author_id);
CREATE INDEX idx_posts_status_date  ON posts (status, published_at DESC)
    WHERE status = 'published';  -- Partial index: only indexes published posts

-- Tags
CREATE TABLE tags (
    tag_id   SERIAL        NOT NULL,
    name     VARCHAR(100)  NOT NULL,
    slug     VARCHAR(100)  NOT NULL,
    CONSTRAINT pk_tags        PRIMARY KEY (tag_id),
    CONSTRAINT uq_tags_name   UNIQUE (name),
    CONSTRAINT uq_tags_slug   UNIQUE (slug)
);

-- Post-Tag junction
CREATE TABLE post_tags (
    post_id INT NOT NULL,
    tag_id  INT NOT NULL,
    CONSTRAINT pk_post_tags   PRIMARY KEY (post_id, tag_id),
    CONSTRAINT fk_pt_post     FOREIGN KEY (post_id) REFERENCES posts (post_id) ON DELETE CASCADE,
    CONSTRAINT fk_pt_tag      FOREIGN KEY (tag_id)  REFERENCES tags  (tag_id)  ON DELETE CASCADE
);

CREATE INDEX idx_post_tags_tag_id ON post_tags (tag_id);

-- Comments
CREATE TABLE comments (
    comment_id   SERIAL        NOT NULL,
    post_id      INT           NOT NULL,
    author_name  VARCHAR(200)  NOT NULL,
    author_email VARCHAR(255)  NOT NULL,
    body         TEXT          NOT NULL,
    status       VARCHAR(20)   NOT NULL DEFAULT 'pending',  -- 'pending', 'approved', 'spam'
    created_at   TIMESTAMPTZ   NOT NULL DEFAULT NOW(),
    CONSTRAINT pk_comments        PRIMARY KEY (comment_id),
    CONSTRAINT chk_comment_status CHECK (status IN ('pending', 'approved', 'spam')),
    CONSTRAINT fk_comments_post   FOREIGN KEY (post_id) REFERENCES posts (post_id) ON DELETE CASCADE
);

CREATE INDEX idx_comments_post_id ON comments (post_id);
CREATE INDEX idx_comments_status  ON comments (status);

-- Reactions
CREATE TABLE reactions (
    reaction_id   SERIAL       NOT NULL,
    post_id       INT          NOT NULL,
    reaction_type VARCHAR(50)  NOT NULL,   -- 'like', 'love', 'insightful', etc.
    created_at    TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    ip_address    INET,
    CONSTRAINT pk_reactions        PRIMARY KEY (reaction_id),
    CONSTRAINT fk_reactions_post   FOREIGN KEY (post_id) REFERENCES posts (post_id) ON DELETE CASCADE
);

CREATE INDEX idx_reactions_post_id ON reactions (post_id);
```

### 13.4 Useful Queries for the Blog Engine

```sql
-- All published posts with their author name and tag count
SELECT 
    p.post_id,
    p.title,
    p.slug,
    p.published_at,
    a.display_name AS author_name,
    COUNT(DISTINCT pt.tag_id) AS tag_count,
    COUNT(DISTINCT c.comment_id) AS approved_comment_count
FROM       posts     p
INNER JOIN authors   a  ON p.author_id = a.author_id
LEFT  JOIN post_tags pt ON p.post_id   = pt.post_id
LEFT  JOIN comments  c  ON p.post_id   = c.post_id AND c.status = 'approved'
WHERE  p.status = 'published'
GROUP BY p.post_id, p.title, p.slug, p.published_at, a.display_name
ORDER BY p.published_at DESC;

-- All posts with a given tag
SELECT p.title, p.slug, p.published_at
FROM       posts     p
INNER JOIN post_tags pt ON p.post_id = pt.post_id
INNER JOIN tags      t  ON pt.tag_id = t.tag_id
WHERE  t.slug    = 'dotnet'
  AND  p.status  = 'published'
ORDER BY p.published_at DESC;

-- Reaction counts per type for a specific post
SELECT 
    reaction_type,
    COUNT(*) AS count
FROM   reactions
WHERE  post_id = 42
GROUP BY reaction_type
ORDER BY count DESC;

-- Posts with the most reactions in the last 30 days
SELECT 
    p.post_id,
    p.title,
    p.slug,
    COUNT(r.reaction_id) AS reaction_count
FROM       posts     p
INNER JOIN reactions r ON p.post_id = r.post_id
WHERE  r.created_at >= NOW() - INTERVAL '30 days'
  AND  p.status = 'published'
GROUP BY p.post_id, p.title, p.slug
ORDER BY reaction_count DESC
LIMIT 10;
```

This schema is in 3NF:
- Every table has a clear primary key.
- No partial dependencies (no composite keys with partially dependent non-key attributes).
- No transitive dependencies (post body does not describe the author; reaction type does not describe the post).
- M:N relationships are properly expressed via junction tables.
- Foreign key constraints enforce referential integrity.
- Indexes on all foreign key columns and frequently-queried columns.
- CHECK constraints on status columns enforce valid domain values.
- `ON DELETE CASCADE` on comments and reactions means deleting a post automatically removes its comments and reactions (appropriate for this domain).

## Conclusion: The Key, The Whole Key, and Nothing But the Key

William Kent's paraphrase of the 3NF rule — a non-key field must provide a fact about the key, the whole key, and nothing but the key — is not just a mnemonic. It is a design philosophy.

When you internalize this principle, database design stops feeling like following arbitrary rules and starts feeling like telling the truth about your data. The tables you design mirror the actual structure of the information you are storing. Each table is about one thing. Each column tells you one fact about that one thing. The relationships between tables reflect real relationships between real concepts.

The bugs you prevent by designing to 3NF are not subtle, edge-case bugs. They are fundamental, inevitable bugs: facts stored in multiple places that eventually disagree with each other, operations that require updating an unknown number of rows to change a single fact, data that gets permanently lost when related data is deleted.

You have now seen:
- Where relational databases came from (Codd's 1970 paper, the relational algebra, the model that beat hierarchical and network alternatives)
- What SQL is and how every major clause works (SELECT in complete detail, JOINs, GROUP BY, window functions, CTEs, INSERT/UPDATE/DELETE, transactions)
- What functional dependency means and why it is the engine of normalization
- First, Second, and Third Normal Form with complete examples built step by step
- Boyce-Codd Normal Form and when it differs from 3NF
- How to design an ER model before touching a keyboard
- How to use indexes intelligently
- How to integrate databases into ASP.NET with ADO.NET, Dapper, and EF Core
- How to avoid the most common anti-patterns
- How to test database code properly

This is not the end of the journey. There is much more to learn: query optimization and execution plans, partitioning, replication, high availability, the internals of the B-tree, lock contention and deadlock resolution, database-as-code pipelines, event sourcing, and the many valid use cases for NoSQL systems. But every one of those topics builds on the foundation you now have.

The relational model has endured for over fifty years because it is genuinely correct. It captures something true about data: that facts belong with the things they describe, and that consistency is not optional. Build your databases with this in mind, and they will serve your applications faithfully for years.
