## The Meeting Nobody Wants to Have

You have been in the job for six weeks. You have done your homework. You have read through the legacy application, you understand the data model, and you have made a compelling case to your tech lead that Blazor WebAssembly is the right direction for the new UI rewrite. The tech lead agrees. The architecture review board agrees. The project plan is approved.

And then, two days before development starts, a product manager drops a requirement into the backlog: the new application needs to include the same interactive financial data grid that the trading desk currently uses on their standalone React application. Specifically, it needs to use AG Grid — the enterprise data grid library with thousands of rows of virtual scrolling, cell renderers, complex sorting and filtering, column dragging, and export-to-Excel functionality. AG Grid has a React binding that the trading desk is happy with. It does not have an official Blazor binding.

You have three options:

1. Tell the product manager that AG Grid is not available in Blazor and that they need to accept a different grid component.
2. Rewrite the entire application in React to get native access to AG Grid.
3. Find a way to use AG Grid inside your Blazor application by integrating it through JavaScript interop.

Option 1 might be the right answer depending on your situation — and we will discuss when it is. Option 2 is a significant cost to impose for a single component. Option 3 is what this article is about.

The good news is that Blazor's JavaScript interop (JSInterop) system is specifically designed for this kind of integration. The less good news is that crossing the WebAssembly/JavaScript boundary correctly requires understanding some important rules about memory management, serialization, and component lifecycle that are not obvious from the documentation.

## Part 1: The Strangler Fig Pattern — Your Migration Strategy

Before we get into the technical details of JSInterop, let us talk about the strategic context. If you are modernizing a legacy application, you probably should not rewrite everything at once. The "big bang" rewrite — tear everything down, rebuild it from scratch, release the new version on a specific date — is famously risky. Deadlines slip. Edge cases emerge. The legacy system gets new requirements while the rewrite is in progress. The release date moves. Morale suffers. Sometimes the rewrite is canceled, leaving you with an unmaintained legacy system and a half-finished new system.

The Strangler Fig pattern, named by Martin Fowler after a tree species that grows around and eventually replaces its host tree, is an alternative migration strategy. You build the new system incrementally, routing individual features or pages to the new code while keeping the old system running for the rest. Over time, the new system grows to cover more and more of the old system's functionality, until eventually the old system is completely surrounded and can be removed.

### Applying the Strangler Fig to a React-to-Blazor Migration

Imagine you have a legacy React SPA. It has fifteen pages. You want to migrate to Blazor. Instead of rewriting all fifteen pages simultaneously, you:

1. Keep the existing React SPA running.
2. Build one page in Blazor WebAssembly, hosted at a specific route (e.g., `/new/dashboard`).
3. Add a link in the React app that navigates to the Blazor page (a full-page navigation, not a SPA navigation).
4. Validate that the Blazor page works correctly.
5. Point the React app's `/dashboard` route to the Blazor URL.
6. Repeat for the next page.

This approach requires that both applications can coexist in the same deployment. There are several ways to do this:

**Path-based routing**: Serve the Blazor app from `/app/...` and the React app from `/legacy/...`. A reverse proxy (nginx, Azure Front Door, Cloudflare) routes requests to the appropriate app.

**Page-based migration**: For a traditional MVC application, each "page" might already be a server-rendered page load. You can replace individual server-rendered pages with Blazor pages one at a time, since each page navigation is a full HTTP request that the server can route to either the old or new handler.

**Island embedding**: Embed individual Blazor components within existing React or HTML pages. The React page stays intact, but specific sections of it render Blazor components via JSInterop.

### Applying the Strangler Fig to a Blazor-with-React-Components Scenario

The inverse scenario — building primarily in Blazor but embedding specific React components — is simpler. Your Blazor application is the host. Individual pages or sections that need a specific React component mount that component via JavaScript interop. The Blazor navigation model, state management, and business logic remain pure .NET. The React component is isolated in a specific area of the page, communicating with Blazor through a defined interface.

This is called the "React island" pattern (borrowing the term from Astro's "Islands Architecture"), and it is exactly what we will implement in this article.

## Part 2: JavaScript Interop in Blazor — The Foundations

Blazor's JSInterop system provides two directions of interoperability:

1. **C# calling JavaScript**: Use `IJSRuntime.InvokeAsync<T>()` or `IJSRuntime.InvokeVoidAsync()` to call JavaScript functions from C# code.
2. **JavaScript calling C# (callbacks)**: Pass a `DotNetObjectReference` to JavaScript, which allows JavaScript code to call back into C# instance methods via `dotNetRef.invokeMethodAsync()`.

Both directions serialize their arguments to JSON for transfer across the boundary and deserialize on the other side. This serialization step has performance implications and correctness constraints that we will discuss in detail.

### Setting Up: The Module Pattern

When you need to call a JavaScript function from Blazor, the recommended pattern in .NET 10 is to use ES modules — standard JavaScript module syntax that allows importing specific functions rather than polluting the global namespace.

Create a JavaScript module file in `wwwroot/js/`:

```javascript
// wwwroot/js/agGrid.js

let gridApi = null;

export function initializeGrid(containerId, initialData, dotNetRef) {
    const container = document.getElementById(containerId);
    if (!container) {
        throw new Error(`Container element '${containerId}' not found`);
    }

    // AG Grid initialization would happen here in a real implementation.
    // For this example, we use a minimal vanilla implementation to illustrate the pattern.
    gridApi = {
        container,
        dotNetRef,
        rows: initialData
    };

    renderGrid(gridApi);
    return true;
}

export function updateGridData(newData) {
    if (!gridApi) return;
    gridApi.rows = newData;
    renderGrid(gridApi);
}

export function destroyGrid() {
    if (!gridApi) {
        gridApi = null;
    }
    // In AG Grid, you would call gridApi.destroy() here
}

function renderGrid(api) {
    // Simple rendering for illustration purposes
    api.container.innerHTML = api.rows
        .map(row => `<div class="grid-row">
            <span>${row.id}</span>
            <span>${row.name}</span>
            <span>${row.value}</span>
        </div>`)
        .join('');
}
```

In your Blazor component:

```razor
@page "/data-grid"
@inject IJSRuntime JS
@implements IAsyncDisposable

<div id="@_containerId" class="data-grid-container">
    Loading grid...
</div>

@code {
    private IJSObjectReference? _gridModule;
    private DotNetObjectReference<DataGridPage>? _dotNetRef;
    private readonly string _containerId = $"grid-{Guid.NewGuid():N}";
    
    [Parameter] public List<GridRow>? Data { get; set; }
    
    // OnAfterRenderAsync because we need the DOM element to exist
    protected override async Task OnAfterRenderAsync(bool firstRender)
    {
        if (firstRender)
        {
            // Load the JS module once on first render
            _gridModule = await JS.InvokeAsync<IJSObjectReference>(
                "import", "./js/agGrid.js");
            
            // Create a reference that allows JS to call back into this C# object
            _dotNetRef = DotNetObjectReference.Create(this);
            
            // Initialize the grid
            await _gridModule.InvokeVoidAsync(
                "initializeGrid",
                _containerId,
                Data ?? [],
                _dotNetRef);
        }
    }
    
    // Called from JavaScript when a row is selected
    [JSInvokable]
    public async Task OnRowSelected(int rowId)
    {
        // This method is called from JavaScript
        Console.WriteLine($"Row {rowId} selected");
        // You would raise an EventCallback here in a real component
    }
    
    protected override async Task OnParametersSetAsync()
    {
        // Update grid data when Parameters change
        if (_gridModule is not null && Data is not null)
        {
            await _gridModule.InvokeVoidAsync("updateGridData", Data);
        }
    }
    
    public async ValueTask DisposeAsync()
    {
        // 1. Destroy the grid (release any JS-side resources)
        if (_gridModule is not null)
        {
            try
            {
                await _gridModule.InvokeVoidAsync("destroyGrid");
            }
            catch (JSDisconnectedException)
            {
                // Browser connection was lost — ignore
            }
            
            // 2. Dispose the JS module reference
            await _gridModule.DisposeAsync();
        }
        
        // 3. Dispose the .NET reference (prevents memory leaks on the JS side)
        _dotNetRef?.Dispose();
    }
}
```

There are several important patterns in this code that prevent common bugs:

**Unique container ID**: Using `Guid.NewGuid()` for the container element's ID ensures that if this component appears multiple times on the same page, each instance has its own grid. Hardcoding an ID like `id="grid"` would cause every instance to target the same element.

**Module imports via `import()`**: Rather than referencing global JavaScript functions, we import a JavaScript ES module using `JS.InvokeAsync<IJSObjectReference>("import", "./js/agGrid.js")`. This loads the module lazily (only when first needed), scopes the module's variables to the module itself (not the global window object), and allows proper tree-shaking of unused module code.

**Cleanup in DisposeAsync**: The `DisposeAsync` method releases three things: the grid itself (JavaScript-side cleanup), the `IJSObjectReference` to the module (which releases the JS module reference), and the `DotNetObjectReference` (which allows the JavaScript garbage collector to collect the .NET proxy object it holds). Missing any of these can cause memory leaks that accumulate over the lifetime of the application.

**JSDisconnectedException handling**: If the user navigates away from the page before the DisposeAsync completes, the SignalR connection (for Blazor Server) or the WebAssembly context may be gone. `JSDisconnectedException` is thrown in this case and should generally be swallowed — the cleanup happened by the browser unloading the page, and there is nothing else to do.

## Part 3: Mounting a Real React Component in Blazor

Now let us go deeper and actually mount a React component inside a Blazor page. This is the true hybrid architecture — running React's runtime inside a Blazor WebAssembly application.

First, some important context: Blazor WebAssembly is a .NET runtime running inside a browser tab. React is a JavaScript library running in the same browser tab. They are both running in the same JavaScript environment. They share the same global scope (though we avoid using global scope intentionally). They can interact with the same DOM.

The mechanism for mounting a React component is standard React: you call `ReactDOM.createRoot(container).render(<YourComponent />)`. You just have to call this from within the Blazor context.

### The JavaScript Wrapper

The key architectural decision here is to write defensive JavaScript wrappers that expose clean interop points to C#. The React component implementation details (props, state, internal event handling) are completely hidden from C#. C# sees only the wrapper API.

```javascript
// wwwroot/js/reactGridWrapper.js
// This is a wrapper module that initializes React and mounts a React component.
// It hides all React implementation details from C#.

// Import React and ReactDOM from a CDN or bundled file
// In a real project, you would use a bundler like Vite or esbuild to bundle React
import React from 'https://esm.sh/react@19.2.1';
import ReactDOM from 'https://esm.sh/react-dom@19.2.1/client';

// Simple React component — in a real project this might be AG Grid, a charting
// library, or any other complex React component
function DataGrid({ columns, rows, onRowClick }) {
    const [selectedRow, setSelectedRow] = React.useState(null);
    
    const handleRowClick = (row) => {
        setSelectedRow(row.id);
        if (onRowClick) onRowClick(row.id);
    };
    
    return React.createElement('div', { className: 'react-data-grid' },
        React.createElement('table', null,
            React.createElement('thead', null,
                React.createElement('tr', null,
                    ...columns.map(col => 
                        React.createElement('th', { key: col.field }, col.header)
                    )
                )
            ),
            React.createElement('tbody', null,
                ...rows.map(row =>
                    React.createElement('tr', {
                        key: row.id,
                        onClick: () => handleRowClick(row),
                        className: selectedRow === row.id ? 'selected' : ''
                    },
                        ...columns.map(col =>
                            React.createElement('td', { key: col.field }, row[col.field])
                        )
                    )
                )
            )
        )
    );
}

// Module state — keyed by container ID to support multiple instances
const instances = new Map();

export function mountReactComponent(containerId, columns, rows, dotNetRef) {
    const container = document.getElementById(containerId);
    if (!container) throw new Error(`Container '${containerId}' not found`);
    
    // Destroy any existing React root for this container
    if (instances.has(containerId)) {
        instances.get(containerId).root.unmount();
    }
    
    // Create a callback that calls back to .NET
    const handleRowClick = (rowId) => {
        dotNetRef.invokeMethodAsync('OnRowSelected', rowId)
            .catch(err => console.error('DotNet callback failed:', err));
    };
    
    // Create React root and render the component
    const root = ReactDOM.createRoot(container);
    root.render(
        React.createElement(DataGrid, { 
            columns, 
            rows, 
            onRowClick: handleRowClick 
        })
    );
    
    instances.set(containerId, { root, dotNetRef });
    return true;
}

export function updateReactData(containerId, rows) {
    const instance = instances.get(containerId);
    if (!instance) return;
    
    // Re-render the React component with new data
    // We need to get the current columns — in a real implementation,
    // we would store them in the instance
    instance.root.render(
        React.createElement(DataGrid, {
            columns: instance.columns || [],
            rows,
            onRowClick: (rowId) => {
                instance.dotNetRef.invokeMethodAsync('OnRowSelected', rowId)
                    .catch(err => console.error('DotNet callback failed:', err));
            }
        })
    );
}

export function unmountReactComponent(containerId) {
    const instance = instances.get(containerId);
    if (instance) {
        instance.root.unmount();
        instances.delete(containerId);
    }
}
```

And the Blazor component that uses this wrapper:

```razor
@page "/hybrid-grid"
@inject IJSRuntime JS
@implements IAsyncDisposable

<h1>Hybrid Data Grid</h1>

@if (_loading)
{
    <p>Initializing grid...</p>
}

<div id="@_containerId" style="min-height: 400px;">
    @* React component mounts here *@
</div>

@if (_lastSelectedRowId.HasValue)
{
    <p>Selected row ID: @_lastSelectedRowId</p>
}

@code {
    private IJSObjectReference? _module;
    private DotNetObjectReference<HybridGridPage>? _dotNetRef;
    private readonly string _containerId = $"react-grid-{Guid.NewGuid():N}";
    private bool _loading = true;
    private int? _lastSelectedRowId;
    
    private static readonly ColumnDefinition[] Columns = 
    [
        new("id", "ID"),
        new("name", "Name"),
        new("value", "Value"),
    ];
    
    private List<GridRow> _rows = 
    [
        new(1, "Alpha", 1500.00m),
        new(2, "Beta", 2300.00m),
        new(3, "Gamma", 800.00m),
    ];
    
    protected override async Task OnAfterRenderAsync(bool firstRender)
    {
        if (firstRender)
        {
            _module = await JS.InvokeAsync<IJSObjectReference>(
                "import", "./js/reactGridWrapper.js");
            
            _dotNetRef = DotNetObjectReference.Create(this);
            
            await _module.InvokeAsync<bool>(
                "mountReactComponent",
                _containerId,
                Columns,
                _rows,
                _dotNetRef);
            
            _loading = false;
            StateHasChanged();
        }
    }
    
    [JSInvokable]
    public void OnRowSelected(int rowId)
    {
        _lastSelectedRowId = rowId;
        InvokeAsync(StateHasChanged);
    }
    
    public async ValueTask DisposeAsync()
    {
        if (_module is not null)
        {
            try 
            { 
                await _module.InvokeVoidAsync("unmountReactComponent", _containerId); 
            }
            catch (JSDisconnectedException) { }
            await _module.DisposeAsync();
        }
        _dotNetRef?.Dispose();
    }
    
    private record ColumnDefinition(string Field, string Header);
    private record GridRow(int Id, string Name, decimal Value);
}
```

This hybrid architecture works. React renders its component tree inside the `<div>` that Blazor manages. Blazor manages the surrounding page, navigation, state management, and all the business logic. The React component is isolated to its container div and communicates back to Blazor only through the defined callback interface.

## Part 4: Memory Management Across the WASM/JS Boundary

The WebAssembly/JavaScript boundary is not free. Every interop call has overhead, and poor management of object references across the boundary can cause memory leaks. Let us understand the mechanics.

### The .NET Heap, the JavaScript Heap, and the Bridge

When Blazor WebAssembly runs in a browser, you have three separate memory spaces:

1. **The WebAssembly linear memory**: A flat array of bytes managed by the .NET garbage collector. This is where your .NET objects live.
2. **The JavaScript heap**: The V8 (or SpiderMonkey, or JavaScriptCore) object heap. This is where your JavaScript objects and React component trees live.
3. **The interop bridge**: A mechanism that allows objects in each heap to hold references to objects in the other heap without those objects being garbage collected prematurely.

When you create a `DotNetObjectReference<T>`, Blazor registers the .NET object in a table that prevents the .NET GC from collecting it, and passes a numeric handle to JavaScript. When JavaScript code calls `dotNetRef.invokeMethodAsync(...)`, it uses that handle to look up the .NET object and invoke the method. The object stays alive (from the .NET GC's perspective) as long as the `DotNetObjectReference` is alive in JavaScript.

This is why `DotNetObjectReference.Dispose()` is essential: it releases the registration, allowing the .NET GC to collect the object when there are no other references. Failing to dispose it creates a memory leak — the .NET object stays alive forever, accumulating if the component mounts and unmounts repeatedly.

Similarly, `IJSObjectReference` objects hold a reference to a JavaScript object (or module) from the .NET side. Disposing the `IJSObjectReference` releases the JavaScript-side reference, allowing the JavaScript GC to collect the object if nothing else references it.

### Serialization and Its Costs

Every interop call serializes its arguments to JSON. This means:

- Calling `InvokeAsync<bool>("mountReactComponent", _containerId, Columns, _rows, _dotNetRef)` serializes `_containerId` (a string), `Columns` (an array of column definition records), and `_rows` (a list of GridRow records) to JSON.
- Large payloads (thousands of rows) can take measurable time to serialize on the .NET side and deserialize on the JavaScript side.
- Passing complex objects with circular references will fail.
- The `_dotNetRef` is passed as a special object reference, not serialized — Blazor's runtime handles this specially.

**Practical rule**: For large data payloads (thousands of items), avoid passing them all at once through interop. Instead, page the data, or store the full dataset on the JavaScript side and pass only filter/sort parameters through interop.

### The ByteArray Optimization

For bulk data transfer, Blazor's JSInterop supports efficient byte array transfer without JSON serialization. This is critical for scenarios like passing image data or large binary payloads:

```csharp
// Efficient bulk data transfer using byte arrays
var imageData = await imageService.GetImageBytesAsync(imageId);

// This transfers the byte array directly, without JSON serialization
await JS.InvokeVoidAsync("renderImage", containerId, imageData);
```

```javascript
// JavaScript receives a Uint8Array, not a JSON-encoded array
export function renderImage(containerId, imageBytes) {
    const blob = new Blob([imageBytes], { type: 'image/jpeg' });
    const url = URL.createObjectURL(blob);
    document.getElementById(containerId).src = url;
}
```

For numeric data (arrays of numbers), you can also use typed arrays (`float[]`, `int[]`) which are transferred more efficiently than arbitrary object arrays.

## Part 5: Defensive JavaScript Wrappers

One of the most important principles of JSInterop architecture is writing defensive JavaScript wrappers. The JavaScript wrapper is the interface between two worlds — C# and React — and it needs to protect both sides from the other's assumptions.

### Defensive Pattern 1: Existence Checks

Never assume a DOM element exists just because you asked for it. The Blazor render cycle and the browser's DOM updates can be slightly out of sync:

```javascript
export function initializeComponent(containerId, options) {
    const container = document.getElementById(containerId);
    if (!container) {
        // Return a meaningful error rather than crashing with a null reference
        console.error(`[ReactWrapper] Container not found: ${containerId}`);
        return false;
    }
    // ... proceed with initialization
    return true;
}
```

### Defensive Pattern 2: Idempotent Initialization

Your init function might be called more than once (due to Blazor's rendering behavior). Make it safe to call multiple times:

```javascript
const instances = new Map();

export function initializeComponent(containerId, options) {
    // If already initialized for this container, update it instead
    if (instances.has(containerId)) {
        updateComponent(containerId, options);
        return;
    }
    
    // Initialize for the first time
    const container = document.getElementById(containerId);
    // ...
    instances.set(containerId, { root, options });
}
```

### Defensive Pattern 3: Error Boundaries Around DotNet Callbacks

When JavaScript calls back into .NET, the .NET method might throw an exception. If this exception is not caught in JavaScript, it becomes an unhandled promise rejection that can crash your application silently:

```javascript
function callDotNet(dotNetRef, methodName, ...args) {
    return dotNetRef.invokeMethodAsync(methodName, ...args)
        .catch(err => {
            console.error(`[ReactWrapper] DotNet callback ${methodName} failed:`, err);
            // Don't rethrow — we don't want the React component to crash
            // because a C# method threw an exception
        });
}
```

### Defensive Pattern 4: Cleanup Before Initialize

If your component can be re-initialized (for example, a grid that refreshes its entire configuration), always clean up the previous instance first:

```javascript
export function initializeGrid(containerId, config, dotNetRef) {
    // Always clean up first
    if (instances.has(containerId)) {
        const old = instances.get(containerId);
        old.root.unmount();
        instances.delete(containerId);
    }
    
    // Now initialize fresh
    const container = document.getElementById(containerId);
    const root = ReactDOM.createRoot(container);
    // ...
    instances.set(containerId, { root, config, dotNetRef });
}
```

## Part 6: Passing Complex JSON Payloads

The serialization format used by Blazor's JSInterop is `System.Text.Json`, with camelCase naming by default. This means your C# property names are camelCased in JSON:

```csharp
// C# record
public record GridRow(int Id, string Name, decimal Value);

// Serializes to JSON as:
// { "id": 1, "name": "Alpha", "value": 1500.00 }
```

Most JavaScript libraries expect camelCase, so this default is usually correct. But there are specific scenarios where serialization causes problems:

### Decimal Precision in JSON

`decimal` in C# is a 128-bit fixed-point type with up to 28 significant digits. JSON has no `decimal` type — only `number`, which is a 64-bit IEEE 754 floating-point. When you serialize `1500.00m` to JSON, it becomes `1500`, which is fine. But `1499.9999999999999` might become `1500` due to floating-point rounding when deserialized in JavaScript.

For financial applications (exactly the scenario where AG Grid is often used), you should serialize monetary values as strings and parse them in JavaScript:

```csharp
// Custom converter for decimal-as-string
public class DecimalAsStringConverter : JsonConverter<decimal>
{
    public override decimal Read(ref Utf8JsonReader reader, Type typeToConvert, JsonSerializerOptions options)
        => decimal.Parse(reader.GetString()!, CultureInfo.InvariantCulture);

    public override void Write(Utf8JsonWriter writer, decimal value, JsonSerializerOptions options)
        => writer.WriteStringValue(value.ToString("F2", CultureInfo.InvariantCulture));
}
```

### DateTime Serialization

C# `DateTime` serializes to ISO 8601 format (`2026-05-20T14:30:00Z`). JavaScript's `Date` object can parse this, but React components often expect string values or specific date formats. Be explicit about your serialization format and document it in your JavaScript wrapper:

```javascript
// Parse the ISO 8601 string from .NET
function parseNetDate(dateString) {
    if (!dateString) return null;
    return new Date(dateString);
}
```

### Avoiding Round-Trip Serialization for Large Datasets

If you have a large dataset (say, 50,000 rows for a financial grid), passing it through the JSInterop boundary on every update is expensive. A better pattern is to load the data on the JavaScript side and expose update APIs that work with row IDs:

```csharp
// Instead of passing 50,000 rows:
await _module.InvokeVoidAsync("updateAllRows", _allRows); // Expensive!

// Pass only the changed row IDs and their new values:
var changedRows = _allRows
    .Where(r => _changedIds.Contains(r.Id))
    .ToList();
await _module.InvokeVoidAsync("updateRows", changedRows); // Cheap
```

## Part 7: When the Hybrid Architecture Is Not Worth It

Everything described in this article is technically feasible. But "feasible" and "advisable" are different things. The hybrid architecture adds complexity. It adds a surface area for bugs at the JS/WASM boundary. It requires maintaining JavaScript code in a primarily .NET project. It makes debugging harder — bugs at the boundary can be traced in either the browser's JavaScript debugger or the .NET debugger, and it is not always obvious which side the bug is on.

Here is a framework for deciding whether the hybrid approach is justified:

**Hybrid is worth it when:**
- The React component you need provides functionality that would take months to replicate in a native Blazor component.
- The component is relatively stable — it will not be changing its API frequently.
- The integration surface is clean — data flows in via props, events flow out via callbacks, and the component does not need deep access to Blazor state.
- The team has JavaScript skills to maintain the wrapper layer.

**Hybrid is not worth it when:**
- A native Blazor component library (MudBlazor, Radzen Blazor, Blazor Fluent UI) has a component that covers 90% of the requirement.
- The React component has complex state that would need deep two-way synchronization with Blazor state.
- The team is primarily .NET developers with limited JavaScript experience.
- The feature is a core part of the application, not a specialized tool.

In many cases, reaching for MudBlazor's `MudTable` or `MudDataGrid` component — which are native Blazor, work within the Blazor rendering model, and are fully testable with bUnit — is the better engineering decision even if it means accepting slightly less functionality than AG Grid provides out of the box.

## Conclusion: The Border as an Architectural Boundary

The JSInterop boundary between Blazor WebAssembly and JavaScript is a real border in your application's architecture. Like all borders, it exists for a reason (separating two different execution environments with different type systems), and crossing it has a cost (serialization, reference management, increased complexity).

Used wisely, JSInterop is the bridge that allows Blazor applications to access the full breadth of the JavaScript ecosystem when needed, while keeping the core application in clean, testable C#. Used carelessly, it becomes a source of memory leaks, synchronization bugs, and architectural complexity that makes the codebase harder to maintain.

The Strangler Fig pattern gives you a disciplined way to migrate from a React application to Blazor (or embed React components in a Blazor app) incrementally, with each step validated before the next begins.

In Part 5 of this series, we turn to one of the most significant conceptual differences between React and Blazor: how they handle global application state. React's journey through prop drilling, Redux, Context API, and atomic state libraries like Zustand and Jotai represents a decade-long search for a solution to a problem that Blazor never had in the first place — because Blazor has `IServiceCollection`.
