## Introduction: The Morning Everyone Gave Up On CSS

Once upon a time — this was real, and it happened to the team that runs this magazine, and it is not exaggerated — a developer spent an entire Monday morning trying to figure out why the `color: red` they had added to a button was not applying. The button stubbornly remained blue. They checked their selector. They checked the spelling. They checked that the file was loading. They even restarted their laptop. They tried `color: red !important`. That worked. They committed it. They went to stand-up.

At stand-up, a more senior engineer asked what they had been working on, and when they admitted that they had had to use `!important`, the senior engineer said — in a tone that was probably kinder than the one we are about to quote — *"You know you can just write a more specific selector, right?"*

The junior developer did not, in fact, know that. They had been working on the codebase for three months. The codebase had 14,000 lines of CSS across 90 files, most of which had been written six years earlier by a team that had since left the company. The code loaded in a different order in production than in development. Nobody knew where the buttons' original styles came from. `grep -r "button"` returned 280 hits.

The senior engineer, who was genuinely trying to help, explained the specificity algorithm. They explained that a selector like `.page .card .button` would beat `.button`. They explained that IDs beat classes. They explained `!important`. They explained inline styles. They explained user-agent styles. They explained inheritance. Forty minutes later the junior dev was back at their desk, staring at the screen, feeling worse than when they had started. They knew more, but the 14,000 lines had not gotten any smaller. The only difference was that now they were afraid of them.

Within a month the team was on a call evaluating Tailwind. Within six months they had started an incremental rewrite. Within a year they were discussing CSS-in-JS. Within two years they had three conventions in the same codebase — original CSS, Tailwind classes, and styled-components — and the maintenance cost was literally higher than it had been before.

We have walked into some version of this story at every enterprise ASP.NET shop we have ever visited. The underlying problem is not that CSS is bad. The underlying problem is that for a generation of developers, **CSS did not ship with the tools necessary to organise a large codebase**. Specificity was a ladder you kept climbing until you fell off. `!important` was the escape hatch. The community responded with frameworks — BEM, SMACSS, OOCSS, CSS Modules, CSS-in-JS, Tailwind — each of which was, at heart, a strategy for *avoiding the cascade*. If the cascade is an unpredictable monster, your framework's job is to wall it off.

In 2026, the cascade has all the tools it was missing. **`@layer`**, which has been Baseline since 2022, lets you declare an explicit ordering of rule groups. **`@scope`**, which [reached Baseline Newly Available in early 2026](https://web.dev/blog/web-platform-12-2025), lets you bound a block of rules to a DOM subtree. **Custom properties** (CSS variables) let you parameterise styles without a preprocessor. **Native nesting** gives you Sass's most-used feature without Sass. And the venerable `:where()` pseudo-class lets you author at zero specificity whenever you want to.

Put together, these features make the cascade a *friend*. You can build a 14,000-line stylesheet in 2026, maintain it, and sleep at night, without any framework, any build step, any Tailwind, any Sass, any CSS-in-JS. Day 3 is about how.

This is a dense article. You may want two cups of coffee. We will cover, in order: how CSS loads and parses, inheritance and the initial/inherit/unset/revert keywords, the cascade algorithm in its entirety, specificity done without memorisation, `:where()` and `:is()`, custom properties in depth, `@property`, native CSS nesting, `@layer`, `@scope`, and a practical architecture for a mid-sized application. Every section has runnable code.

## Part 1: How CSS Actually Gets To The Page

Before the cascade, a few things have to happen. Understanding them clears up about a third of the common confusions.

When a browser loads an HTML document, it:

1. **Parses the HTML** into a DOM tree.
2. **Encounters each `<link rel="stylesheet">` and `<style>`** and fetches / parses it into a **CSSOM** — the CSS Object Model, a tree of stylesheets and rules analogous to the DOM.
3. **Matches each element in the DOM against each selector in the CSSOM** to build a **computed style** for every element — one value for every single CSS property, for every single element.
4. **Lays out** the page using those computed styles.
5. **Paints** pixels.
6. **Composites** the result into what you see on screen.

Every step matters, but the one that causes people grief is step 3 — matching and computing. There are thousands of CSS properties and tens of thousands of elements on a typical page, so the browser cannot afford to be clever. It runs a deterministic algorithm: for each element, for each property, pick exactly one value. That algorithm is **the cascade**. Every other CSS concept you have ever learned — specificity, `!important`, inheritance, `@layer` — is just an input to this algorithm.

A note on source order. When you write:

```html
<link rel="stylesheet" href="/css/reset.css">
<link rel="stylesheet" href="/css/app.css">
<style>
  .button { color: blue; }
</style>
```

the browser reads them in document order. Later rules, all else equal, override earlier ones. This is so fundamental that it is often forgotten, and yet the entire ordering of build-tool outputs is based on getting this right.

CSS is **declarative**. You are not writing instructions; you are making assertions about what styles should apply to what elements. The order of your declarations matters, but only because "last writer wins" is one of the tiebreakers in the cascade. Most of the time, the cascade will resolve conflicts before it gets to source order, which is why selector changes five lines down can change the outcome of a declaration five lines up.

## Part 2: Inheritance — The Other Thing That Is Not Specificity

Before the cascade proper, there is **inheritance**, which is separate and is always run first.

CSS divides properties into two classes:

- **Inherited properties.** By default, the value is inherited from the parent. `color`, `font-family`, `font-size`, `line-height`, `text-align`, `cursor`, `visibility`, and roughly 40 others. These are things that "make sense" to inherit: if the paragraph is dark grey, the `<em>` inside it should probably also be dark grey.
- **Non-inherited properties.** By default, the value is the property's initial value, not the parent's. `margin`, `padding`, `border`, `background`, `display`, `width`, `height`, and roughly 400 others. These do not make sense to inherit: if the paragraph has `padding: 1rem`, the `<em>` inside should not also get `padding: 1rem`.

You can override this behaviour per-property. CSS provides four keywords that are valid for every property:

- **`initial`** — use the property's specified initial value from the CSS specification. For `color`, that's `canvastext` (black-ish in most themes). For `display`, that's `inline`. This is rarely what you want.
- **`inherit`** — explicitly inherit from the parent, even for normally non-inherited properties.
- **`unset`** — behave as `inherit` if the property is inherited, or `initial` otherwise. This is the "reset to default" keyword most people actually want.
- **`revert`** — roll back to the value the user-agent stylesheet or the user stylesheet provided. For `display` on a `<p>`, that's `block`. For `display` on a `<span>`, that's `inline`. This is the magical "make it behave like the native HTML element again" keyword.
- **`revert-layer`** — same as `revert`, but only rolls back the current `@layer` (we will get to this in Part 10).

Practical example. You have a reset stylesheet that sets `all: unset` on buttons (a very aggressive reset). Later, a form button needs to look like a native button again. You can write:

```css
.native-button {
  all: revert;
}
```

That one line undoes the reset and restores the browser default.

The full list of inherited properties is in the [CSS 2 spec, Appendix A](https://www.w3.org/TR/CSS2/propidx.html) — about 40 entries. You do not need to memorise it. You need to know that it exists and that CSS distinguishes between the two categories.

## Part 3: The Cascade — The Complete Algorithm

When the browser computes a value for a property on an element, it runs this algorithm. The [CSS Cascade 5 specification](https://drafts.csswg.org/css-cascade-5/) codifies it precisely; what follows is its plain-English form. Every tiebreaker is applied in order. If a tiebreaker picks a winner, the rest are not consulted.

1. **Origin and importance.** Declarations are grouped into "origins" and either "important" or "normal." The ordering, from most powerful to least, is:
   - Transition declarations (while a CSS transition is active).
   - Important user-agent declarations.
   - Important user declarations.
   - Important author declarations (your `!important` rules).
   - Animation declarations (while a CSS animation is active).
   - Normal author declarations (your regular rules).
   - Normal user declarations.
   - Normal user-agent declarations (browser defaults).
   - At the very bottom: the property's initial value.
2. **Context.** Rules inside a shadow tree vs rules outside (we will cover shadow DOM in Day 10).
3. **Cascade layers** — the `@layer` ordering. Higher layers win. No-layer rules win over layered rules (for non-important declarations). For **important** declarations, the ordering is *reversed* — lower layers win over higher, and no-layer loses. This is one of the more counterintuitive corners of the spec, and we will return to it.
4. **Specificity** — the famous (a, b, c) triple, which we will demystify in Part 5.
5. **Order of appearance** — later wins. This is the last tiebreaker, and the only one most developers know by name.

Notice how far down specificity comes. A rule in a later `@layer` beats a more specific rule in an earlier layer. This is the single most important mental shift of modern CSS: **specificity is no longer your primary organising tool.** Layers are.

An example. Suppose you have:

```css
@layer reset, components, utilities;

@layer reset {
  button { background: black; }
}

@layer utilities {
  button { background: red; }
}
```

Which wins? `utilities`, because it is declared *later* in the `@layer` list — even though both rules have identical specificity. No matter how you rewrite the reset rule (`button`, `.reset button`, `#root button`), `utilities` wins, because layer order is applied before specificity. This is what makes `@layer` transformative. You will see this pattern repeatedly.

## Part 4: Origin and `!important` — The Honest Truth

`!important` has a reputation as "the thing you use when you've given up." That is half true. It is also, in the modern cascade, a precise and occasionally correct tool. Let us think about when.

The four origins:

- **User-agent stylesheet.** The browser's built-in styles. `h1` has a large margin; `a` is underlined and blue. You can see them in Chromium DevTools under "user agent stylesheet" in the Computed tab.
- **User stylesheet.** Styles the user has installed. Largely a relic today, but still used by accessibility tools and browser extensions. Power users override site styles via the user stylesheet for accessibility.
- **Author stylesheet.** Your CSS.
- **Animation.** Active CSS animations inject declarations at this level.
- **Transition.** Active CSS transitions inject declarations at their own, highest level.

When `!important` is applied, the order is inverted. The reasoning is important (pun intended): if a user has an `!important` style in their user stylesheet — say, `body { font-size: 24px !important; }` because they need large type to read — the author (you) should not be able to override that. The cascade spec encodes this by making user `!important` win over author `!important`.

In practice, the cases where `!important` is correct in your own author CSS are:

1. **Overriding a third-party stylesheet you cannot modify.** The analytics widget inject inline `style="color: green"`? You can hit it with `.widget { color: red !important; }`.
2. **Forcing a utility class to win even when the cascade disagrees.** Tailwind uses `!important` selectively for this reason. We will build a utilities layer without `!important` in Part 10.
3. **Debugging.** Use `!important` in DevTools to confirm a selector matches before working out a cleaner approach.

In nearly every other case, there is a better answer — often a better `@layer`.

## Part 5: Specificity — Without Memorisation

Specificity is a three-number tuple: **(a, b, c)**. For any selector, you compute:

- **a** — the number of ID selectors.
- **b** — the number of class selectors, attribute selectors (`[href]`), and pseudo-classes (`:hover`, `:not()`'s argument, etc.).
- **c** — the number of type selectors (`div`) and pseudo-elements (`::before`).

Compare two specificities left-to-right like version numbers. `(1, 0, 0)` beats `(0, 9, 9)`. `(0, 2, 0)` beats `(0, 1, 99)`.

Specific cases:

- `*` (universal selector) has specificity `(0, 0, 0)`.
- `!important` does not change specificity. It's tracked separately in origin-and-importance (Part 3).
- `:not(X)` has the specificity of X. `:not(.a, .b)` has the specificity of the most specific argument. So `:not(.foo)` has specificity `(0, 1, 0)`.
- `:is(X, Y, Z)` has the specificity of its most specific argument.
- `:where(X, Y, Z)` always has specificity `(0, 0, 0)`. This is its superpower.
- `:has(X)` has the specificity of its most specific argument.
- Inline `style="..."` has a higher specificity than any selector, but lower than `!important` declarations.
- `id` selectors beat any number of class selectors. This is the main reason we have the "0,0,1,0" horror story.

Examples, with specificity computed:

```css
a                              /* (0, 0, 1) */
a:hover                        /* (0, 1, 1) */
.nav a                         /* (0, 1, 1) */
.nav a:hover                   /* (0, 2, 1) */
ul.nav > li > a                /* (0, 1, 3) */
#site-header .nav a            /* (1, 1, 1) */
:where(.nav, .menu) a          /* (0, 0, 1) — :where() always 0 */
:is(.nav, .menu) a             /* (0, 1, 1) — :is() takes most specific */
a[href*=example]               /* (0, 1, 1) */
a[href*=example]:hover         /* (0, 2, 1) */
body.home #hero .cta .btn      /* (1, 3, 1) */
```

**The real-world lesson.** You can go your entire career without calculating specificity in your head if you follow three rules:

1. **Do not use `#id` selectors for styling.** IDs are useful in HTML for anchoring and for labelling, but they are too powerful in the cascade. One accidental `#header .button` shuts down every attempt to theme the button elsewhere. Use classes.
2. **Do not nest more than two levels deep.** `.card .title` is fine. `.home .card .meta .title` is asking for trouble.
3. **Wrap your "base styles" in `:where()` to give them zero specificity.** Then your component styles always override them without needing to match the base's specificity.

We will exercise all three rules in Part 10.

Specificity remains the cascade's most famous tiebreaker. It is also, in modern CSS, one of the *least* important. The tools we introduce in Parts 9 and 10 let you bypass it almost entirely.

## Part 6: `:where()` and `:is()` — The Selectors That Changed Everything

These two pseudo-classes were added to the Selectors Level 4 spec for the same purpose: to let you group selectors without writing them out longhand. `:is()` keeps the specificity of its most specific argument; `:where()` always has zero specificity. Both have been Baseline Widely Available since 2022.

Consider a common pattern: "all headings inside an article should have tight letter spacing." You could write:

```css
article h1,
article h2,
article h3,
article h4,
article h5,
article h6 {
  letter-spacing: -0.02em;
}
```

With `:is()`:

```css
article :is(h1, h2, h3, h4, h5, h6) {
  letter-spacing: -0.02em;
}
```

That's a nice-to-have, but the more interesting rewrite uses `:where()`:

```css
article :where(h1, h2, h3, h4, h5, h6) {
  letter-spacing: -0.02em;
}
```

Now the specificity is `(0, 0, 1)` — equivalent to just `h1`. That means any rule that targets `h1` with specificity `(0, 0, 1)` or higher will still win, even if that rule is defined *earlier* in the stylesheet. In practice, this lets you author opinionated "base styles" that yield gracefully to component styles without you having to care about selector construction.

We use `:where()` as the single most powerful tool in modern CSS authoring. A base layer where every selector is wrapped in `:where()` acts as "defaults that can be overridden by anything." This is the foundation of modern style architecture.

### `:is()` in practice

`:is()` is useful when you want specificity to match but want to shorten the selector:

```css
/* Instead of: */
.card .title, .card .subtitle, .card .caption { font-family: serif; }

/* You can write: */
.card :is(.title, .subtitle, .caption) { font-family: serif; }
```

Both forms have specificity `(0, 2, 0)`.

`:is()` is also forgiving: if one of its arguments is an unknown selector, the other arguments still work. In contrast, `.card .title, .invalid::foo, .card .subtitle` would invalidate the whole rule because of `.invalid::foo`. With `:is()`, only the invalid argument is ignored.

## Part 7: Custom Properties — The Variable System You Already Had

CSS custom properties (aka CSS variables) have been [Baseline Widely Available since October 2019](https://developer.mozilla.org/en-US/docs/Web/CSS/Reference/Properties/--*) and are without exaggeration the most architecturally important CSS feature of the last decade. If you ever used Sass variables, you know the shape. The differences — and why custom properties are more powerful — are:

1. **They are live.** Change a custom property in one place and everything using it updates instantly. Sass variables are compile-time; custom properties are runtime.
2. **They cascade and inherit.** Set `--brand` on `body` and every descendant inherits it.
3. **They are scoped.** Set `--brand` on `.card` and it only applies inside `.card`.
4. **They can be overridden by media queries, container queries, `:hover`, `data-theme`, anything.**
5. **They can be read and written from JavaScript.** `getComputedStyle(el).getPropertyValue("--brand")` reads; `el.style.setProperty("--brand", "red")` writes.

The core syntax:

```css
:root {
  --brand: oklch(60% 0.15 250);
  --brand-muted: oklch(from var(--brand) 60% 0.06 h);
  --space-xs: 0.25rem;
  --space-sm: 0.5rem;
  --space-md: 1rem;
  --space-lg: 2rem;
  --space-xl: 4rem;
  --font-body: system-ui, sans-serif;
  --font-heading: Georgia, serif;
  --border-radius: 0.375rem;
  --shadow-sm: 0 1px 2px oklch(0% 0 0 / 0.05);
  --shadow-md: 0 4px 8px oklch(0% 0 0 / 0.1);
}

.button {
  background: var(--brand);
  padding: var(--space-sm) var(--space-md);
  border-radius: var(--border-radius);
  font-family: var(--font-body);
}
```

### Fallbacks

`var()` takes an optional second argument as a fallback:

```css
.button {
  background: var(--brand, blue);
}
```

If `--brand` is not defined, `blue` is used. You can nest `var()` inside the fallback:

```css
color: var(--text-primary, var(--text, black));
```

This is the basis of a graceful theme system — the component uses a specific variable, which falls back to a general one, which falls back to a literal value.

### Scoping

Custom properties inherit. To scope a value to a subtree, set it on that subtree:

```css
.theme-dark {
  --bg: oklch(15% 0 0);
  --text: oklch(95% 0 0);
}

.theme-light {
  --bg: oklch(98% 0 0);
  --text: oklch(15% 0 0);
}

body {
  background: var(--bg);
  color: var(--text);
}
```

Putting `class="theme-dark"` on `<body>` or any ancestor swaps the variables for that subtree. This is the core mechanism for dark-mode support, themed sections, multi-tenant styling, and variant components. We will build a complete dark-mode system in Day 6.

### `@property` — Typed Custom Properties

Plain custom properties are untyped strings. CSS cannot tell whether `--brand: red` is a color or just the word "red." [`@property`](https://developer.mozilla.org/en-US/docs/Web/CSS/Reference/At-rules/@property), [Baseline Newly Available since July 2024](https://web.dev/blog/at-property-baseline), lets you register a custom property with a type, an initial value, and inheritance behaviour.

```css
@property --brand {
  syntax: "<color>";
  inherits: true;
  initial-value: rebeccapurple;
}
```

With this registration, `--brand` is now a `<color>` property. The browser will reject invalid values (falling back to the initial value), and — crucially — it will *animate* the property smoothly. Animating untyped custom properties does not work, because the browser does not know how to interpolate "red" to "blue" without knowing they are colors. Once registered, you can animate `--brand` like any typed property.

A concrete example: animated gradients, which were impossible in plain CSS before `@property`:

```css
@property --gradient-angle {
  syntax: "<angle>";
  inherits: false;
  initial-value: 0deg;
}

.spinner {
  background: conic-gradient(from var(--gradient-angle), red, blue);
  animation: spin 3s linear infinite;
}

@keyframes spin {
  to { --gradient-angle: 360deg; }
}
```

Without `@property`, the angle would jump from `0deg` to `360deg` instantly instead of interpolating. With it, it rotates smoothly. This is the kind of thing that used to need JavaScript.

`@property` supports the following `syntax` values: `<length>`, `<number>`, `<percentage>`, `<length-percentage>`, `<color>`, `<image>`, `<url>`, `<integer>`, `<angle>`, `<time>`, `<resolution>`, `<transform-function>`, `<transform-list>`, `<custom-ident>`, and `*` (any). Combinators `+` (one or more) and `|` (alternation) are supported too.

## Part 8: Native Nesting — Sass In The Browser

CSS nesting is the feature that most former Sass users will reach for first. [It became Baseline Newly Available in August 2023](https://web.dev/baseline) and should be considered safe to use in 2026. The syntax:

```css
.card {
  padding: var(--space-md);
  border-radius: var(--border-radius);

  & .title {
    font-size: 1.25rem;
    font-weight: 600;
  }

  & .meta {
    color: oklch(50% 0 0);
    font-size: 0.875rem;
  }

  &:hover {
    box-shadow: var(--shadow-md);
  }

  @media (min-width: 48rem) {
    padding: var(--space-lg);
  }
}
```

Three things to know:

1. **The `&` is the parent selector**, analogous to Sass. It represents every element that matched the parent rule. `&:hover` is "this element when hovered." `& .child` is "a `.child` inside this."
2. **A nested selector without `&` is treated as a descendant combinator** in modern browsers. Early versions of the spec required `&` explicitly; that was relaxed. In 2026, both forms work, but explicit `&` is clearer.
3. **Nesting compounds specificity.** A nested selector has the same specificity it would have if you had written it out longhand. Nest three levels deep and you have a specificity-3 selector. Keep nesting shallow for the same reason we don't nest deep in Sass.

Nesting inside media/container/supports queries is the killer feature:

```css
.card {
  padding: 1rem;

  @media (min-width: 48rem) {
    padding: 2rem;
  }

  @container (min-width: 30rem) {
    padding: 1.5rem;
  }

  @supports (display: subgrid) {
    display: grid;
  }
}
```

One property, in one place, defined for all its contexts. Much easier to read than the un-nested equivalent scattered across a file.

### Nesting pitfalls

- **Nesting pseudo-elements requires `&`.** `&::before` works; `::before` alone is the descendant combinator applied to pseudo-elements, which is nonsense.
- **Do not nest inside `:root`.** Custom properties inside `:root` should be flat. Nesting provides no benefit and hurts readability.
- **Do not over-nest.** Three levels is usually the practical maximum; beyond that, extract into a sibling class.

## Part 9: `@layer` — Explicit Cascade Order

Here we are. The single most important addition to the cascade in a decade. [`@layer` is Baseline Widely Available](https://web.dev/baseline).

The mental model: layers are named groups of rules, and you declare the order explicitly.

```css
@layer reset, base, tokens, components, utilities;
```

That one-line declaration establishes the order: later wins. Any `@layer reset {...}` block anywhere in the stylesheet (or in subsequently loaded stylesheets) contributes to the `reset` layer. Any rule in `components` wins over any rule in `reset` or `base`, regardless of specificity.

A full example:

```css
/* Declare the order up front. */
@layer reset, base, tokens, components, utilities;

/* The reset layer — aggressive defaults. */
@layer reset {
  *, *::before, *::after {
    box-sizing: border-box;
  }

  body, h1, h2, h3, h4, h5, h6, p, figure, blockquote, dl, dd {
    margin: 0;
  }

  img, picture, video, canvas, svg {
    display: block;
    max-width: 100%;
  }

  button, input, textarea, select {
    font: inherit;
    color: inherit;
  }
}

/* Design tokens. */
@layer tokens {
  :root {
    --brand: oklch(60% 0.15 250);
    --space-md: 1rem;
    --space-lg: 2rem;
    --radius: 0.375rem;
  }
}

/* Base element styles — wrapped in :where() for zero specificity. */
@layer base {
  :where(body) {
    font-family: system-ui, sans-serif;
    line-height: 1.6;
    color: oklch(15% 0 0);
  }

  :where(h1) { font-size: 2rem; font-weight: 700; }
  :where(h2) { font-size: 1.5rem; font-weight: 700; }
  :where(a) { color: var(--brand); }
}

/* Components. */
@layer components {
  .button {
    display: inline-block;
    padding: 0.5rem var(--space-md);
    background: var(--brand);
    color: white;
    border: 0;
    border-radius: var(--radius);
    cursor: pointer;

    &:hover {
      background: oklch(from var(--brand) 50% c h);
    }
  }

  .card {
    padding: var(--space-lg);
    border-radius: var(--radius);
    box-shadow: 0 1px 2px oklch(0% 0 0 / 0.05);
  }
}

/* Utilities — last layer wins. */
@layer utilities {
  .m-0 { margin: 0; }
  .text-center { text-align: center; }
  .sr-only {
    position: absolute;
    width: 1px;
    height: 1px;
    padding: 0;
    margin: -1px;
    overflow: hidden;
    clip-path: inset(50%);
    white-space: nowrap;
    border: 0;
  }
}
```

This is a complete, production-quality style architecture in about 70 lines of plain CSS. Let us make explicit what this achieves:

1. **Reset beats the user-agent stylesheet.** Any non-layered rule by an author also beats user-agent, because the user-agent stylesheet lives in its own origin below all author rules.
2. **Tokens are global defaults.** Defined in `:root`, they inherit everywhere.
3. **Base rules use `:where()` for zero specificity.** Any class-based override can beat them effortlessly.
4. **Components have class-based specificity.** They override base rules automatically.
5. **Utilities in the highest layer beat all components.** No `!important` needed. `class="button text-center"` on a button has the `text-align: center` win even though `.button` comes later in source order, because `utilities` is a higher layer than `components`.

### Nested layers

You can nest layers for sub-grouping:

```css
@layer components {
  @layer typography, layout, interaction;

  @layer typography {
    .heading { font-family: Georgia, serif; }
  }

  @layer layout {
    .stack > * + * { margin-top: 1em; }
  }
}
```

Nested layers follow their parent's order. Everything in `components.typography` loses to everything in `utilities`, because `utilities` wins over the whole `components` block, including its nested layers.

### Anonymous layers

You can create a layer without naming it:

```css
@layer {
  /* rules */
}
```

Anonymous layers are positional — each one creates a new layer at that position in the order. Useful in imports:

```css
@import url("third-party.css") layer;
@import url("our-overrides.css") layer;
```

The two imports go into two distinct anonymous layers, in order. Anything in `our-overrides.css` wins over anything in `third-party.css`, regardless of specificity.

### `@import layer(name)`

You can assign imports to a named layer:

```css
@import url("bootstrap.css") layer(vendor);
@import url("app.css") layer(app);

@layer vendor, app;
```

Bootstrap's `.btn-primary` styles now live in the `vendor` layer; your own `.btn-primary` styles live in `app`. Yours win, because `app` comes later in the layer order — *regardless of Bootstrap's specificity*. This means you can drop a third-party CSS framework in and override it without specificity gymnastics.

### `revert-layer`

Inside a layer, you can use `revert-layer` on a property to roll back to what a lower layer said:

```css
.disabled-button {
  background: revert-layer;
  cursor: not-allowed;
  opacity: 0.5;
}
```

This is useful for state-specific components that want to undo one of their own layer's defaults.

### `!important` and layers — the twist

Remember Part 4: with `!important`, the layer order is *inverted*. An `!important` rule in the `reset` layer beats an `!important` rule in the `utilities` layer, because `reset` is earlier and "earlier wins" for important. The rationale: an `!important` in your reset is the most authoritative baseline, and later layers should not accidentally override it.

In practice, this is exotic, but it means you should rarely mix `!important` and `@layer`. If you find yourself doing so, step back and reconsider.

## Part 10: `@scope` — Style Inside A Box

[`@scope` reached Baseline Newly Available in January 2026 with Firefox 146](https://web.dev/blog/web-platform-12-2025). It is the final piece that lets you abandon CSS Modules, BEM, and CSS-in-JS with a clear conscience.

The syntax:

```css
@scope (.card) {
  .title {
    font-size: 1.25rem;
    font-weight: 600;
  }

  :scope {
    border: 1px solid oklch(80% 0.05 240);
    border-radius: var(--radius);
  }
}
```

Two things happen here:

1. The rules **only match elements inside a `.card`**. So `.title` matches `.title` only when it is a descendant of `.card`. Elsewhere in the document, `.title` is unaffected.
2. `:scope` is a pseudo-class that refers to the scope root itself — the `.card` element.

`@scope` replaces one of the most tedious parts of writing CSS for reusable components. Without it, you had to write `.card .title` everywhere and trust that no other `.title` elsewhere would collide. With it, the collision is impossible by construction.

### Lower bounds — the "donut hole"

`@scope` supports a second selector that bounds the scope from below:

```css
@scope (.card) to (.widget) {
  a {
    color: var(--card-link-color);
  }
}
```

This selects `a` elements inside a `.card`, *unless* they are inside a `.widget` that is itself inside the card. The `to` clause defines a "hole" in the scope. Use case: a card contains a widget (an embedded calendar, a third-party player, an iframe container) whose styles you don't want to touch.

### Proximity cascade

`@scope` introduces a new cascade tiebreaker, higher than specificity: **scope proximity**. When two scoped rules both match an element, the one whose scope root is closer to the element wins.

```css
@scope (.outer) {
  .title { color: blue; }
}
@scope (.inner) {
  .title { color: red; }
}
```

Given the markup:

```html
<div class="outer">
  <div class="inner">
    <h1 class="title">Hi</h1>
  </div>
</div>
```

`.title` is inside both scopes. Specificity is equal. Source order would give blue (assuming the inner rule is declared first). But proximity prefers the closer scope: `red` wins, because `.inner` is a closer ancestor than `.outer`.

This is remarkably useful for theme overrides: an inner section can redefine variables and styles for its subtree without having to invent more specific selectors. It replaces a huge amount of the ad-hoc specificity-climbing in legacy CSS.

### Inline `@scope` in HTML

`@scope` can live inside a `<style>` block in the HTML body, scoped to the parent element:

```html
<article class="post">
  <style>
    @scope {
      .lede { font-size: 1.25rem; }
      .byline { color: oklch(50% 0 0); }
    }
  </style>
  <p class="lede">This is the lede.</p>
  <p class="byline">By My Blazor Team</p>
</article>
```

`@scope` with no argument uses the enclosing element (`.post`) as the scope root. The styles apply only inside this article. Scroll to another article, and `.lede` elsewhere is unaffected.

This pattern is genuinely useful for one-off styled content — a landing page hero, a single marketing section, a one-time dialog. No cascade pollution. No need to invent unique class names. The scope is implicit in the HTML structure.

### `@scope` vs Shadow DOM

Both provide encapsulation. They are not the same.

- **`@scope`** provides *style* encapsulation only. Styles outside the scope do not match; styles inside the scope match only within it. The DOM itself is not isolated.
- **Shadow DOM** (Day 10) provides full DOM isolation: style encapsulation, event encapsulation, focus isolation. `querySelector` from the outside cannot see inside a shadow tree.

For most "scope this component's styles" use cases, `@scope` is simpler and does the job. Shadow DOM is necessary only when you need the DOM itself to be isolated, typically for distributed Web Components.

## Part 11: A Practical Architecture For A Mid-Sized Application

Now we put it all together. The CSS architecture for the capstone we are building over this series. One folder, six files, roughly 1,500 lines total.

```
css/
├── app.css              ← entry point; declares layer order and @imports
├── reset.css            ← reset layer
├── tokens.css           ← design tokens (custom properties)
├── base.css             ← base element styles (wrapped in :where())
├── components.css       ← component styles
└── utilities.css        ← utility classes
```

The entry point:

```css
/* css/app.css */
@layer reset, tokens, base, components, utilities;

@import url("reset.css")      layer(reset);
@import url("tokens.css")     layer(tokens);
@import url("base.css")       layer(base);
@import url("components.css") layer(components);
@import url("utilities.css")  layer(utilities);
```

One file, six lines. The entire load order is declared here. New file? Add another `@import`. Want to insert a vendor layer before components? Add it to the layer list and the `@import` list:

```css
@layer reset, tokens, base, vendor, components, utilities;

@import url("vendor/bootstrap.css") layer(vendor);
```

Bootstrap is now below your components in the cascade, and all specificity arguments become moot.

### `reset.css`

A minimal, modern reset. Not as aggressive as Eric Meyer's classic; much smaller than Normalize.css:

```css
  text-size-adjust: none;
  -webkit-text-size-adjust: none;
  color-scheme: light dark;
}

body,
h1, h2, h3, h4, h5, h6,

  margin: 0;
}

h1, h2, h3, h4, h5, h6 {
  text-wrap: balance;
}

p {
  text-wrap: pretty;
}

img, picture, video, canvas, svg {
  display: block;
  max-width: 100%;
  height: auto;
```

Every one of those rules corrects a historical browser oddity or sets a better default. `text-wrap: balance` on headings and `text-wrap: pretty` on paragraphs — both Baseline as of late 2024 — make text wrapping visibly better with one line each.

### `tokens.css`


Three observations:

1. **`light-dark()`** lets a single token definition cover both modes, as long as `color-scheme` is set. The browser picks the right one based on the user's OS setting. We will cover this in depth on Day 6.

3. **Every token has a name, not just a value.** Downstream components should use `var(--space-md)`, not `1rem` — so that when a designer changes the scale, every component updates.


## Part 12: Answers To The Frameworks

Now that we have a complete architecture, let's address the frameworks we are not using.

### Tailwind

Tailwind is an extraordinarily well-designed framework that popularised the "utility-first" approach. Its core insight — that CSS authoring is often more productive when it feels closer to inline styling — is valuable. In 2026, with the features we have covered, you can get 90 percent of Tailwind's benefits by:

1. A tokens layer with everything Tailwind's config gives you — spacing scale, colour palette, type scale.
2. A utilities layer with the classes you actually use.
3. Native nesting for variant composition.

The result is smaller, closer to the metal, and does not require a build step. The trade-off is that Tailwind has a very large, carefully-curated library of utility classes. You will write some of your own, or adopt one of the [open-source vanilla-CSS equivalents](https://open-props.style/), to approach parity.

### Sass

Sass was essential for years. In 2026, the two features you miss from Sass are mixins (for repeated blocks of declarations) and the `@use`/`@forward` module system. Neither is available in plain CSS yet. In practice, mixins can be replaced by components (a CSS class *is* a reusable block of declarations) and by `:where()`-based utility selectors. If you find yourself wanting a mixin, a component with custom properties is usually the right answer.

### CSS Modules

CSS Modules' value proposition is local scoping. `@scope` is native, browser-level local scoping. CSS Modules are a build-time trick; `@scope` is a runtime fact. If you liked CSS Modules for scope, use `@scope`.

### CSS-in-JS

CSS-in-JS solved two problems: dynamic styling from runtime values, and colocating styles with components. The first is now solvable with custom properties — you set a property in a `style="..."` attribute from JavaScript, and CSS picks it up. The second is solvable with the shadow DOM (Day 10) or with one stylesheet per component imported with `@import`.

If you used CSS-in-JS for runtime theming, custom properties + `data-theme="dark"` on the root is more efficient (no re-rendering, no JavaScript) and equally powerful.

### BEM

BEM (`.block__element--modifier`) is a naming convention invented to simulate local scope. With `@scope`, you do not need a naming convention — you have actual local scope. With layers, you do not need specificity discipline — you have actual layer ordering. BEM was a workaround; both problems it addressed are now solved natively.

## Part 13: Specificity Self-Defence — Tricks For Legacy Code

Some of our readers are not starting fresh. They are walking into 14,000 lines of legacy CSS and need to add one button without collateral damage. Three techniques for this situation:

### Technique 1: Isolate with a layer

Wrap all new CSS in a high-priority layer:

```css
@layer legacy, app-new;

/* Old code goes into the unnamed origin (no layer) */
/* or into the legacy layer via @import. */

@layer app-new {
  /* Your new rules. They automatically win over legacy rules
     regardless of specificity. */
}
```

This is the strongest migration strategy we know. You do not have to match specificity. You do not have to clean up the old code first. You add a layer and you are done.

### Technique 2: Boost specificity with `:is(selector, selector, selector)` repetition

Need one rule to win without rewriting others? Wrap the selector in `:is()` multiple times:

```css
:is(.button):is(.button):is(.button) {
  background: red;
}
```

Specificity becomes `(0, 3, 0)` — enough to beat most common selectors. This is a hack of last resort, but it is more sane than `!important`.

### Technique 3: Reduce specificity with `:where()`

If a legacy rule is *too* specific and you want to author a more normal-looking rule, write your new rule and ensure the old rule's specificity is lowered. In practice you cannot change the old rule's specificity without editing it, but you can often replace the old rule using a layer strategy (Technique 1).

Every feature we have used in this article is Baseline as of April 2026:

- **Custom properties** — Widely since October 2019.
- **CSS nesting** — Widely since 2026 (Newly Aug 2023).
- **`@layer`** — Widely since 2024 (Newly March 2022).
- **`@scope`** — Newly as of early 2026.
- **`@property`** — Newly since July 2024.
- **`:where()` / `:is()` / `:has()`** — Widely.
- **`oklch()`** — Widely since late 2024.
- **`light-dark()`** — Newly since late 2024.
- **`text-wrap: balance`/`pretty`** — Newly since mid-2024.

For any feature you are uncertain about, use the `@supports` at-rule to feature-detect:

```css
@supports (display: subgrid) {
  .grid {
    display: subgrid;
  }
}

@supports not (display: subgrid) {
  .grid {
    /* fallback */
  }
}
```

`@supports` can test any property-value pair, any selector (`@supports selector(:has(a))`), or any `@font-tech` / `@font-format` feature for font loading. Using it liberally lets you adopt new CSS incrementally without waiting for every browser to catch up.

## Part 15: A Case Study — The Button That Would Not Go Red

Let us close by returning to our opening story. The developer who spent a Monday morning fighting a button. In the legacy codebase, the problem was the cascade in the abstract. In 2026 CSS, the fix is two lines:

```css
@layer base, components, overrides;

@layer overrides {
  .button {
    background: red;
  }
}
```

If `.button` is defined in the `components` layer (or anywhere else without a layer), the rule in `overrides` wins, because layer order beats specificity. No `!important`. No specificity climbing. No brittle selectors.

This is the promise of modern CSS: **the cascade is a tool, not a weapon**. With layers and scope, you author without fear.

## Part 16: A Checklist

The CSS organisation checklist for a modern codebase:

- [ ] An explicit `@layer reset, tokens, base, components, utilities;` (or similar) at the top of your entry stylesheet.
- [ ] A reset layer that corrects browser defaults.
- [ ] A tokens layer with all colours, spacing, type sizes, shadows, radii, durations, and easings as custom properties.
- [ ] A base layer where element selectors are wrapped in `:where()` for zero specificity.
- [ ] Components as classes, optionally using `@scope` for local styles.
- [ ] Utilities as small, composable classes in the final layer.
- [ ] No `#id` selectors in your stylesheets.
- [ ] No more than two levels of nesting per rule.
- [ ] No `!important` except on truly last-resort utilities like `.hidden`.
- [ ] Every custom property documented in the tokens file.
- [ ] `@supports` guards around features that are not yet Baseline for your audience.

If your codebase meets those criteria, adding a new component is a three-line task, and removing one is a two-file task.
