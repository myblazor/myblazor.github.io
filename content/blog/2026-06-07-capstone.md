## Introduction: The Pull Request That Said "Migrate To Vite"

We had been writing this series for about three weeks when an email landed from a former colleague. She had been a senior engineer on a team we worked with around 2020. She was now CTO of a different company — a small product team, six engineers, building a customer-portal SaaS. She had been reading the series on her commute. She wrote:

> "Your series is fine in theory but I cannot send my team into the wilderness. We have a Vite + React + TanStack Query + React Router + Tailwind + Vitest + Playwright stack, and it works. Our junior devs were productive in two weeks. The component library is consistent. The tests run in eight seconds. We deploy with one click. What you are describing sounds like 1998 with extra steps. Convince me otherwise, or admit this is just nostalgia."

It was a fair email. It deserved a fair answer. We sat with it for a few days.

The honest answer is that her stack is fine. We are not arguing that everyone should rip out their working tools. The case we have been making across these fifteen days is narrower than that, and we want to put it precisely before we close.

Our argument is: **for any new project starting in 2026, the no-build path is a real option you should evaluate on its merits.** It is not nostalgia. It is not a religion. It is a recognition that the browser has shipped — slowly, over the last decade, then quickly in the last three years — almost all of the infrastructure that justified the build-tool empire of the 2010s. The empire is still good engineering for the cases it was built for. But there are now whole categories of projects (blogs, documentation sites, internal tools, content-driven apps, design-system primitives, marketing sites, anything where the application logic fits in a few thousand lines of code) where the build-tool empire is not the right fit. For those projects, the no-build path is faster to start, cheaper to maintain, easier to understand, and far more durable.

Our colleague's customer-portal SaaS is probably not in that category. It has a lot of state, a lot of interactions, and a team that knows React. Migrating it to vanilla would be expensive and pointless. We replied with that and a question: "What would you build *next* if you were starting today?"

She has not yet answered. We hope this last article helps with that conversation, for her and for anyone in the same position.

Day 15 is the closer. We will assemble the entire series into one working application — a complete blog reader that demonstrates every technique we have covered. We will walk through the file structure, the (small) build pipeline, the deployment story to GitHub Pages, and the Lighthouse audit. We will be honest about what we cheated on. We will give you a decision framework for the next project. And we will say what we have been driving toward for fifteen days: that the web has come a very long way, that the platform is genuinely capable, and that you should stop assuming a build step is the price of entry.

This is Part 15 of 15. The end of the road. Thank you for reading.

## Part 1: The Capstone — What We Are Building

The application: **a complete blog reader for a magazine site, end to end.** Specifically, what you have been reading: this magazine, ObserverMagazine.github.io, hosted on GitHub Pages, served as static files, no runtime backend.

Functional requirements:

- **Article reading.** A list of articles, each with a detail page. Markdown source, rendered to HTML at build time.
- **Search.** Type in a search box, see matching articles in real time.
- **Tag filtering.** Click a tag, see articles with that tag. Combine multiple tags.
- **Dark mode.** Auto-detect OS preference; user can override; preference persists.
- **View Transitions.** Smooth animated navigation between list and detail.
- **Keyboard navigation.** Tab, arrow keys, Enter, Escape. Skip link. Focus management.
- **Accessibility.** WCAG 2.2 AA conformant. Screen-reader-tested.
- **Offline support.** Service worker caches everything; site works on a plane.
- **Installable.** PWA manifest; add to home screen / install on desktop.
- **RSS feed.** `feed.xml` for readers who want it.
- **Comment composer.** A form to add a comment, posting to a third-party service (or queued offline).
- **Responsive.** Looks great from 320px wide to 4K.
- **Fast.** Lighthouse 95+ on all four categories.

Non-functional requirements:

- **Zero npm dependencies in the published app.** External code comes from esm.sh with version pinning and SRI, or is self-hosted.
- **Build step is optional and minimal.** A bash script that runs in CI to convert markdown to HTML and generate the RSS feed.
- **One-`git-push` deployment.** GitHub Actions runs the build and pushes to `gh-pages`.
- **Total source tree under 2,000 lines.** Counting all our HTML, CSS, and JavaScript.

Spoiler: we hit all of these. Let us walk through how.

## Part 3: The Build Pipeline

We have a build step. We promised one might exist. Here is the entire `build/build.sh`:

```bash
#!/usr/bin/env bash
# build.sh — produce the site in dist/ from src/ + content/

set -euo pipefail

ROOT=$(dirname "$(realpath "$0")")/..
DIST="$ROOT/dist"
SRC="$ROOT/src"
CONTENT="$ROOT/content"

# 1. Clean and copy static
rm -rf "$DIST"
mkdir -p "$DIST"
cp -r "$SRC"/* "$DIST/"

# 2. Render each post markdown to a JSON + a static HTML
mkdir -p "$DIST/blog" "$DIST/api/posts"
for md in "$CONTENT/posts/"*.md; do
  slug=$(basename "$md" .md)
  # Use pandoc to render (installed in the GH Actions image)
  html=$(pandoc -f markdown -t html --shift-heading-level-by=1 "$md")
  meta=$(grep -m1 -A20 '^---$' "$md" | sed -n '/^---$/,/^---$/p')

  # Write JSON for the SPA's data API
  jq -n --arg slug "$slug" --arg html "$html" --arg meta "$meta" '
    { slug: $slug, html: $html, meta: $meta }
  ' > "$DIST/api/posts/$slug.json"

  # Write a static HTML page for SEO + first-paint
  cat "$SRC/blog-post-template.html" \
    | sed "s|{{SLUG}}|$slug|g" \
    | sed "s|{{TITLE}}|$(echo "$meta" | grep '^title:' | sed 's/title: *//;s/^"//;s/"$//')|g" \
    | sed "s|{{BODY_PLACEHOLDER}}|$html|g" \
    > "$DIST/blog/$slug.html"
done

# 3. Build the index of all posts (for the archive page and search)
jq -s '.' "$DIST"/api/posts/*.json > "$DIST/api/posts/index.json"

# 4. Generate the RSS feed
node "$ROOT/build/rss.js" > "$DIST/feed.xml"

# 5. Inject content hashes into asset filenames for cache busting
node "$ROOT/build/hash-assets.js" "$DIST"

# 6. Done
echo "Built $(ls "$DIST/blog/"*.html | wc -l) posts to $DIST/"
```

The script:

1. Cleans `dist/`, copies `src/` into it.
2. For each markdown post, runs Pandoc to produce HTML, extracts the YAML front matter, and writes both a JSON (for the SPA to consume) and a static HTML page (for SEO and first-paint).
3. Concatenates all post JSONs into an `index.json` for the archive and search.
4. Generates `feed.xml` (an RSS 2.0 feed) using a small Node script we will show below.
5. Hashes asset filenames (`app.js` → `app.a3f9c2.js`) and rewrites references in HTML.
6. Done.

Total runtime in CI: about 8 seconds for the full ~120 articles we have published to date. The whole build is observable, debuggable, and modifiable. There is no plugin system to learn.

### Wait, you used Node?

We did. Two things — RSS generation and asset hashing — are easier in JavaScript than in Bash. We use Node only at *build time*, not runtime. The deployed site is still pure browser JavaScript with no server. Using Node-the-tool for build automation is fine; the no-build claim is about the runtime.

If you want zero Node even at build, you can do both of those tasks in shell with `xmlstarlet` for the RSS and `find -exec sha256sum` for hashing. We did the Node version because it is shorter. Twenty lines for the RSS generator:

```javascript
// build/rss.js — emit feed.xml from content/posts/*.md
import { readdirSync, readFileSync } from "node:fs";
import { join } from "node:path";

const POSTS_DIR = "content/posts";
const SITE_URL = "https://observermagazine.github.io";
const SITE_TITLE = "My Blazor Magazine";

const posts = readdirSync(POSTS_DIR)
  .filter((f) => f.endsWith(".md"))
  .map((f) => {
    const slug = f.replace(/\.md$/, "");
    const text = readFileSync(join(POSTS_DIR, f), "utf8");
    const fm = text.match(/^---\n([\s\S]+?)\n---/)?.[1] ?? "";
    return {
      slug,
      title: fm.match(/title: *"?(.+?)"?$/m)?.[1] ?? slug,
      date: fm.match(/date: *(\S+)/m)?.[1] ?? "1970-01-01",
      summary: fm.match(/summary: *"?(.+?)"?$/m)?.[1] ?? "",
    };
  })
  .sort((a, b) => b.date.localeCompare(a.date));

const xmlEscape = (s) =>
  s.replace(/[<>&'"]/g, (c) => ({ "<": "&lt;", ">": "&gt;", "&": "&amp;", "'": "&apos;", '"': "&quot;" })[c]);

console.log(`<?xml version="1.0" encoding="UTF-8"?>
<rss version="2.0" xmlns:atom="http://www.w3.org/2005/Atom">
<channel>
  <title>${xmlEscape(SITE_TITLE)}</title>
  <link>${SITE_URL}</link>
  <description>Long-form articles on web development.</description>
  <atom:link href="${SITE_URL}/feed.xml" rel="self" type="application/rss+xml"/>
  ${posts.map((p) => `
  <item>
    <title>${xmlEscape(p.title)}</title>
    <link>${SITE_URL}/blog/${p.slug}/</link>
    <guid>${SITE_URL}/blog/${p.slug}/</guid>
    <pubDate>${new Date(p.date).toUTCString()}</pubDate>
    <description>${xmlEscape(p.summary)}</description>
  </item>`).join("")}
</channel>
</rss>`);
```

Read the markdown files, parse the front matter, sort by date, emit the RSS XML. Twenty lines.

## Part 4: The Deployment Pipeline

`.github/workflows/deploy.yml`:

```yaml
name: Deploy

on:
  push:
    branches: [main]

permissions:
  contents: read
  pages: write
  id-token: write

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install build tools
        run: |
          sudo apt-get update
          sudo apt-get install -y pandoc jq

      - name: Set up Node (for build scripts)
        uses: actions/setup-node@v4
        with:
          node-version: 22

      - name: Build
        run: bash build/build.sh

      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./dist

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

Forty lines. Push to `main`, the action runs, the site deploys. Total CI time: ~2 minutes, of which the actual build is ~8 seconds and the rest is GitHub Actions warming up.

## Part 5: A Walk Through The Application

Let us trace what happens when a user visits the site.

### First visit

1. **User types `observermagazine.github.io` and hits Enter.**
2. **DNS resolves; HTTPS handshake; request to the server.**
3. **GitHub Pages serves `/index.html`** — a 4KB static file. Critical CSS inlined in `<head>`. Module script `<script type="module" src="/js/app.a3f9c2.js">` deferred. Manifest `<link>` and theme bootstrap `<script>` (with SRI hash in CSP).
4. **Browser begins parsing.** Sees the inline theme bootstrap, sets `data-theme` synchronously to avoid flash-of-wrong-theme.
5. **Main content renders.** The home page is a static HTML file generated at build time, so the user sees the welcome copy and links immediately. LCP fires at ~200ms on a fast connection.
6. **`/js/app.a3f9c2.js` loads in parallel.** Module preload hints have warmed the cache.
7. **`app.js` runs.** Initialises router, registers service worker, mounts theme toggle, renders dynamic regions.
8. **Service worker installs in the background.** Pre-caches the app shell on first run. Future visits are offline-capable.

End-to-end: under 500ms on a fast connection. Lighthouse LCP < 1.0s.

### Subsequent navigation

1. **User clicks "Read more" on an article card.**
2. **Navigation API intercepts** (Day 11). The link is same-origin, not cross-origin, not a download — `event.canIntercept` is true.
3. **`document.startViewTransition` is called.** The browser snapshots the current state.
4. **The route handler runs:** dynamic `import("/js/pages/blog-post.js")` (already preloaded by `<link rel="modulepreload">`), which resolves immediately.
5. **`blog-post.js` fetches `/api/posts/some-slug.json`** — already in the service worker's cache from earlier prefetch. Renders the article content into `<main>`.
6. **View Transition animates the change.** Hero image morphs from card position to detail position via `view-transition-name: hero-some-slug` (Day 11, Part 6).
7. **Document title updated; focus moved to `<h1>` of the new content.** Screen reader announces "Navigated to <title>" via the `aria-live` polite region (Day 11, Part 5).
8. **URL is `/blog/some-slug/`** in the address bar. Real, shareable, refresh-safe (the static HTML for that slug exists from build time).

End-to-end: under 100ms in the typical case (everything cached, no network needed).

### Search

1. **User clicks "Search" or presses `Ctrl-K`.**
2. **Custom-element `<command-palette>`** opens — a modal `<dialog>` with a focused input.
3. **User types.** Each keystroke is debounced 150ms (Day 8 / Day 9).
4. **The search runs against the in-memory index** (loaded from `/api/posts/index.json` — about 30KB gzipped for our 120 articles). Uses [Fuse.js](https://fusejs.io/) (one of our genuinely-justified library dependencies, Day 8 Part 16) for fuzzy matching.
5. **Matching results render** in a virtualised list (because we may have a hundred matches; only render what's visible).
6. **Arrow keys navigate; Enter activates.** All keyboard. All `aria-activedescendant` driven for screen readers.

The whole search interface is one custom element, ~150 lines of JavaScript. Inputs to outputs in single-digit milliseconds.

### Tag filtering

We covered this in Day 9, Part 12 — the complete worked example was the tag filter. The capstone uses exactly that code. URL parameters drive the filter; clicking tags toggles them; results filter live; back/forward and bookmarks work.

### Dark mode

Day 6, Part 2's three-state toggle (`auto` / `light` / `dark`), persisted in localStorage. The inline bootstrap script in `<head>` sets `data-theme` before first paint to avoid flash-of-wrong-theme. CSS uses `light-dark()` everywhere for token resolution.

### Comment composer

Day 12's complete form pattern, plus offline support: if the network is unavailable, the comment is saved to IndexedDB (Day 13) and a Background Sync registered. Chromium browsers send it when the network returns; other browsers send it next time the user returns to the tab via the `visibilitychange` listener (Day 13's honest cross-browser fallback).

The actual comment endpoint, since GitHub Pages has no backend, is a third-party form service. We use [Formspree](https://formspree.io/) for ours, but [Netlify Forms](https://www.netlify.com/products/forms/), [GitHub Issues via the API](https://github.com/utterance/utterances), or any equivalent works.

### Service worker

Day 13's complete service worker. Precaches the shell on install. Stale-while-revalidate for content. Cache-first for images. Network-first for everything else. Offline fallback for navigations.

### RSS

`/feed.xml`, generated at build time. ~12KB. Linked from `<head>`:

```html
<link rel="alternate" type="application/rss+xml"
      title="My Blazor Magazine" href="/feed.xml">
```

Feed readers auto-detect it.

## Part 7: The Honest "Where We Cheated" List

We promised honesty about the corners we cut.

**1. We use Pandoc.** Markdown-to-HTML is hard to do well. Pandoc has done it for fifteen years. We use it. Pandoc is a single binary, no dependencies, available everywhere. We do not consider this a bundler in any meaningful sense.

**2. We use `jq` and `sed` in the build.** Both are POSIX tools, ship with every Linux. The build script uses them for JSON manipulation and string replacement. Not a JavaScript thing.

**3. We use Node for two build scripts.** RSS generation and asset hashing. Both are pure-Node, no `node_modules`. We could rewrite in shell or Python; Node was the most concise. The deployed site does not depend on Node.

**4. We use Fuse.js for search.** Genuinely useful. About 4KB from esm.sh. We could implement fuzzy search ourselves but the result would be worse and longer than the library.

**5. We use a third-party form service.** GitHub Pages has no backend. Forms have to post somewhere. We chose Formspree; alternatives exist. The alternative is no comment submission, which is worse than the dependency.

**6. The capstone has 1,840 lines, not 100.** This is what an actually-feature-complete site looks like. The lines come from real features (search, tag filter, theme toggle, dialog, tabs, live clock, comment form, all the components, all the routes, all the stores). A truly minimal site is much smaller.

**7. We test with Playwright.** Playwright is a serious dependency (one of the bigger ones in our `devDependencies` if we had a `package.json`, which we do not). We use it because it is the right tool for browser-based integration testing. We could test with manual scripts and `curl`; the result would be worse.

**8. We host on GitHub Pages, which is free and excellent but has limits.** No HTTP headers customisation (we use `<meta>` for CSP). No server-side anything. No private content. For a different project, Cloudflare Pages, Netlify, or static-S3 would be drop-in replacements with more features.

These caveats matter. We are not pretending the build pipeline is zero-effort. We are pointing out that the *runtime* is zero-build, and that the build is small and comprehensible — about 200 lines of glue compared to a typical Vite + plugin ecosystem, which is often thousands of lines spread across dozens of packages.

## Part 8: A Decision Framework For Your Next Project

For your next project, we suggest considering the no-build path explicitly. Here is a decision framework.

### Definitely no-build

- **A blog or content site.** Like ours. The content is the product; the technology should disappear.
- **Documentation site.** Same as a blog.
- **Marketing site.** Static-or-near-static; the priority is loading fast and being indexable.
- **Internal tools for small teams.** Two engineers, six tools, no one will maintain a build pipeline.
- **Hobby projects.** The tooling tax is the biggest reason hobby projects die.
- **Prototypes you might keep.** Many "quick prototypes" become production. Starting them in the platform means the production version inherits the simplicity.
- **Embedded widgets.** Things you ship to other people's sites — Web Components in particular.
- **Sites with a small team.** One or two developers can hold a no-build site in their head; add framework knowledge requirements and onboarding gets harder.
- **Long-lived sites.** Anything you intend to maintain for ten or more years.

### Probably no-build

- **Dashboards and admin panels** without complex inter-component state.
- **E-commerce sites** of moderate complexity. Most of the work is templating + product data, both well-suited to the no-build pattern. (The checkout flow may want more sophistication.)
- **Portfolios, agency sites, magazine sites.**

### Probably framework

- **SaaS products with rich application UI.** Many concurrent panels, complex state across components, deep undo/redo, real-time collaboration. The framework's component model and state primitives are doing genuinely useful work.
- **Anything with very heavy interactivity.** Spreadsheet apps, design tools, IDEs, video editors. The state-and-rendering coordination is the hard part; frameworks help.
- **Mobile-first apps that should feel native.** Capacitor / Cordova / React Native all appreciate framework conventions.
- **Anything with a large existing team that knows React, Vue, or similar.** Switching costs.

### Definitely framework

- **Multi-developer applications with strict component-API boundaries.** TypeScript + a framework gives you the type-safety surface that is genuinely hard to achieve in plain JS+JSDoc at scale.
- **Server-rendered apps with hydration.** Next.js, Nuxt, SvelteKit, Remix are tools designed for this; reproducing their work in vanilla is unusually expensive.
- **Apps that need a strong opinion about file structure and conventions.** Opinionated frameworks (Next.js App Router, Rails+Hotwire) give a team a shared structure that vanilla does not.

The point of this framework is not to push everyone toward vanilla. It is to **stop assuming framework**, the way the industry has assumed it for a decade, and to make the choice deliberately.

## Part 9: The Reflection

We have written about 95,000 words across these fifteen days. We have covered HTML semantics, the cascade, layout, responsive design, colour and typography, modules, the DOM, state, components, routing, forms, storage, accessibility, performance, and security. We have built four Web Components, three full applications, a service worker, a router, a state library, a design system, an RSS generator, and a comment form. We have done it without a single npm install in the deployed runtime.

The big lesson, we hope, has been this: **the platform is enough.**

Not "the platform is good but you need a framework for real work." The platform — semantic HTML + modern CSS + ES Modules + the Constraint Validation API + Web Components + Service Workers + the Navigation API + View Transitions + the dozen smaller features we have covered — is, in 2026, a complete, production-ready development environment for the vast majority of websites and applications. The frameworks added value because the platform was incomplete. The platform is no longer incomplete. Frameworks still add value in their core use cases (large applications, large teams, server-rendering), but the tax they impose on small projects is no longer justified by an inability to ship without them.

The other lesson, less explicit but threaded through every article, is about **durability**. The HTML you write today using the techniques in this series will work in 2050. Web standards are explicitly designed to be backward-compatible forever. A `<button>` written in 1995 still works. A `<dialog>` written in 2024 will still work in 2050. A custom element registered today will run in browsers we cannot yet imagine. The same cannot be said of any framework — every one of them has had migrations, breaking changes, deprecated APIs, abandoned versions. The maintenance you save by using the platform is paid forward to the future.

The third lesson is about **understanding**. When something breaks in a vanilla site, you can read every line of code that produced the bug. There is no library to blame, no version mismatch to debug, no plugin interaction to untangle. The cost of figuring out a problem is bounded by the size of your code. This is enormously freeing. We have been programming for a long time, and one of the things we have learned is that the best engineers are the ones who insist on understanding what their code does. Frameworks make understanding optional; the platform makes it inevitable.

We started the series with a story about a junior developer at a startup making a hobby blog who reached for Next.js and found himself debugging webpack at midnight. We will end with the alternative: that developer, in the alternative timeline, started with `<!doctype html>` and a stylesheet, learned the platform feature by feature, and shipped a blog in an afternoon that he completely understood, that loaded in 100 milliseconds, that worked offline, that was indexable by every search engine, that he could continue improving for a decade without ever upgrading anything. That developer would have spent the same time he spent on Next.js, and would have ended up with more skill, more understanding, and a better artifact.

That is the hope.

## Part 10: What Next

What now, with this series ended?

If you have been following along: **build something with what you have learned.** A small blog. A documentation site. An internal tool. A weekend project. Use the patterns. Notice what works and what does not. Send us notes — we love hearing from readers.

If you are evaluating the no-build path for a real project: **prototype it before you commit.** A weekend of trying to build the riskiest part of your application in plain HTML/CSS/JS will tell you more than any blog series. If it works, you have your answer. If it does not, you have a clear list of where the platform fell short for your specific case, which is far more useful than a generic decision.

If you are not going to use the no-build path: **use the platform's features anyway.** Semantic HTML. The Constraint Validation API. View Transitions. `<dialog>`. `light-dark()`. `:has()`. These are useful in any project, framework or not. Every one of them you adopt is a small piece of code you no longer have to maintain.

If you are skeptical of everything we have written: **good.** Be skeptical. Build something we are wrong about. We will read it.
