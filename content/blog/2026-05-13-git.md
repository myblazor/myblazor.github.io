## Part 1 — Why You Are Reading This, and Why It Is Going to Be Okay

You have been using Team Foundation Server. Maybe you call it TFS. Maybe your company upgraded to Azure DevOps Server a couple of years ago and you still call it TFS because that is what everyone calls it. You have been checking in code for years. You right-click in Solution Explorer, you click "Check In," you type a comment like "fixed the thing," and you move on with your life. It works. It has always worked. And now someone — your manager, your tech lead, your new coworker who wears a hoodie and uses a Mac — is telling you that you need to learn Git.

This guide is for you. Not the version of you that reads Hacker News and experiments with Rust on weekends. The version of you that opens Visual Studio, writes ASP.NET Web Forms code, checks in when the build works, and goes home. The version of you that has been doing this for ten years and is very good at it, even if you have never once opened a terminal window on purpose.

We are going to start from the very beginning. We are going to explain what a version control system actually is, what TFS does under the hood, what Git does differently, and why. We are going to map every TFS concept you know to its Git equivalent. We are going to walk through every scenario you will encounter in real life, step by step, click by click. We are going to use Visual Studio 2026 because that is the current version and it has excellent Git support built in. And we are not going to rush.

If at any point you feel overwhelmed, that is normal. Git is a different mental model. It is not harder than TFS — it is different. And different takes time. But every developer who has made this transition before you has come out the other side, and you will too.

Let us begin.

### 1.1 — What Is a Version Control System, Really?

Before we compare TFS and Git, we need to agree on what version control actually does. If you have been using TFS for years, you might think of version control as "the thing that saves my code on the server." That is part of it, but not all of it.

A version control system (VCS) does four things:

First, it keeps a history of every change ever made to every file. Not just the current version — every version. If your coworker accidentally deleted the `Login.aspx.cs` file three weeks ago and nobody noticed until today, version control lets you get it back. You can see what the file looked like yesterday, last month, or four years ago.

Second, it lets multiple people work on the same codebase at the same time. Without version control, you would have to take turns. "Hey, I am editing `Default.aspx` right now, do not touch it." Version control handles the logistics of two people editing the same file and helps you combine their changes.

Third, it lets you create branches — parallel copies of the entire codebase where you can make changes without affecting the main code. When you are done, you merge those changes back. If something goes wrong in the branch, you can throw it away and the main code is unharmed.

Fourth, it gives you an audit trail. Who changed what, when, and why. This matters for debugging ("when did this bug get introduced?"), for compliance ("who approved this change?"), and for your sanity ("wait, did I write this code or did someone else?").

Both TFS and Git do all four of these things. They just do them very, very differently.

### 1.2 — A Brief History of TFS

Team Foundation Server was first released by Microsoft in 2005. It was part of the Visual Studio Team System family and was designed as an all-in-one platform: version control, work item tracking, build automation, and reporting, all in one box.

The version control component of TFS is called Team Foundation Version Control, or TFVC. This is a centralized version control system. "Centralized" means there is one server that holds the canonical copy of all your code and all your history. When you check in, you are sending your changes to that server. When you get latest, you are downloading changes from that server. The server is the single source of truth.

TFVC went through several major releases: TFS 2005, 2008, 2010, 2012, 2013, 2015, 2017, and 2018. In 2019, Microsoft rebranded the product to Azure DevOps Server. The latest on-premises version as of May 2026 is Azure DevOps Server 25H2 (with Patch 2 released March 2026). The cloud-hosted version is called Azure DevOps Services.

Starting with TFS 2013, Microsoft added native Git repository support alongside TFVC. This means that even inside Azure DevOps (the TFS successor), you can choose either TFVC or Git for your version control. Most new projects use Git. Microsoft themselves migrated the Windows operating system source code — one of the largest codebases in the world — from a proprietary version control system to Git.

### 1.3 — A Brief History of Git

Git was created in 2005 by Linus Torvalds, the same person who created the Linux kernel. The Linux kernel project had been using a proprietary version control system called BitKeeper, but a licensing dispute led to BitKeeper revoking the Linux project's free license. Torvalds needed a replacement fast, and he wanted something that was distributed, fast, and capable of handling a massive codebase with thousands of contributors.

He wrote the first version of Git in about two weeks. It was not pretty, but it was fast and it was correct. Over the following years, the Git project grew, and by around 2008–2010 it had become the dominant version control system for open-source software. GitHub launched in 2008 and made Git accessible to millions of developers who might never have used it otherwise.

As of May 2026, the current stable version of Git is 2.53.0, released on February 2, 2026. Git 3.0 is being discussed by the Git developers and may arrive by the end of 2026. The big change planned for Git 3.0 is switching the default hash algorithm from SHA-1 to SHA-256 for improved security.

Git is a distributed version control system. "Distributed" means there is no single server. Every developer has a complete copy of the entire repository, including all history, on their own machine. You can work offline, commit changes, create branches, and do almost everything without ever touching a network. When you are ready, you push your changes to a shared location (like GitHub or Azure DevOps) so others can see them.

This is the single biggest conceptual difference between TFS/TFVC and Git, and it is where most of the confusion comes from.

### 1.4 — The Mental Model Shift

In TFS/TFVC, you think of the server as the brain. Your local machine is a dumb terminal that checks files out, edits them, and checks them back in. The history lives on the server. The branches live on the server. The truth lives on the server.

In Git, your local machine IS the brain. You have everything. All the history, all the branches, all the commits — they are on your hard drive, right now, in a hidden folder called `.git` inside your project directory. The remote server (GitHub, Azure DevOps, GitLab, or whatever you use) is just another copy. A very important copy — the one your team agrees to treat as the shared truth — but technically, it is just another copy.

Here is an analogy. TFS is like a library. There is one copy of each book, and it is in the library. You check a book out, read it, write some notes in it, and return it. If the library burns down, the books are gone.

Git is like a world where every patron has a perfect photocopy of every book in the library on their bookshelf at home. You read your copy, write notes in it, and when you go to the library, you show them what you changed and they update the library's copy. If the library burns down, every patron still has every book. You could rebuild the library from any single patron's collection.

This has profound implications for how you work, and we will explore every one of them. But first, let us build a vocabulary.


## Part 2 — The Vocabulary: Every TFS Concept Mapped to Git

One of the most frustrating things about learning Git as a TFS developer is that Git uses different words for similar (but not identical) concepts. This section is a complete Rosetta Stone. We will take every TFS concept you know and explain its Git equivalent.

### 2.1 — Workspace vs. Clone

**In TFS:** A workspace is the mapping between the server and your local machine. You configure a workspace to say "put the code from `$/MyProject/Main` into `C:\Code\MyProject` on my machine." The workspace is a server-side concept — the TFS server knows about your workspace.

To create a workspace in Visual Studio, you go to **Team Explorer** → **Manage Connections** → connect to your TFS server → map a workspace folder. Or you can use the `tf workspace` command in the Developer Command Prompt.

**In Git:** The equivalent is a "clone." When you clone a repository, Git downloads the entire repository — all files, all history, all branches — to a folder on your machine. But unlike a TFS workspace, the Git server does not know or care about your local clone. There is no server-side mapping. You just have a copy.

To clone a repository in Visual Studio 2026, you go to **Git** → **Clone Repository** → paste the URL of the repository → choose a local folder → click **Clone**. Or you can use the command line:

```bash
git clone https://github.com/kusl/learningbydoing.git
```

This creates a folder called `learningbydoing` on your machine with the complete repository inside it.

**Key differences:**
- A TFS workspace can map only certain folders from the server. A Git clone gets everything.
- A TFS workspace has a server-side record. A Git clone is purely local.
- A TFS workspace can be server-based (files are read-only until checked out) or local. A Git clone is always local and files are always writable.

### 2.2 — Get Latest vs. Pull (and Fetch)

**In TFS:** "Get Latest Version" downloads the newest version of files from the server to your workspace. You right-click a file or folder in Solution Explorer and choose **Source Control** → **Get Latest Version**. Or you use the `tf get` command.

**In Git:** The equivalent is "pull." But Git actually does this in two steps:

1. **Fetch** — Download the new data from the remote server, but do not change any of your local files yet. This is like looking at what is new in the library catalog without actually picking up any books.

2. **Merge** (or **Rebase**) — Integrate the downloaded changes into your local working copy. This is like actually putting the new information into your local copy of the book.

When you do a "pull," Git does both steps at once: fetch, then merge. In Visual Studio 2026, you click the **Pull** button in the Git Changes window. Or from the menu: **Git** → **Pull**. Or from the command line:

```bash
git pull
```

You can also do just a fetch if you want to see what is new without changing your files:

```bash
git fetch
```

**Key difference:** In TFS, "Get Latest" is a single operation that always downloads from the server. In Git, you can separate "find out what changed on the server" (fetch) from "merge those changes into my code" (merge). This separation gives you more control. You might fetch, look at the incoming changes, and decide to delay merging them because you are in the middle of something.

### 2.3 — Check Out vs. (Nothing)

**In TFS with server workspaces:** Before you can edit a file, you have to check it out. Files are read-only on disk until you explicitly tell TFS "I want to edit this file." This is the oldest TFS model, going back to its Visual SourceSafe roots.

**In TFS with local workspaces (TFS 2012+):** Files are writable all the time. TFS detects what you changed when you go to check in, similar to how Git works.

**In Git:** There is no concept of checking out a file for editing. All files are writable all the time. You just edit them. Git will figure out what changed when you are ready to commit. The word "checkout" exists in Git, but it means something completely different — it means switching to a different branch or restoring a file to a previous version. This is one of the most confusing terminology overlaps.

### 2.4 — Check In vs. Commit + Push

This is the biggest conceptual difference and the source of most confusion.

**In TFS:** "Check In" is a single operation that takes your local changes, sends them to the server, and creates a new version on the server. One step. Your changes go straight to the shared repository.

In Visual Studio, you go to **Team Explorer** → **Pending Changes** → write a check-in comment → click **Check In**. Or you use `tf checkin` on the command line.

**In Git:** The equivalent is TWO separate operations:

1. **Commit** — Save your changes to your LOCAL repository. This creates a new version, but only on your machine. Nobody else can see it yet. A commit is private until you push it.

2. **Push** — Send your local commits to the remote server (GitHub, Azure DevOps, etc.) so that others can see them.

In Visual Studio 2026, the **Git Changes** window shows your modified files. You write a commit message and click **Commit All** (this is the commit step). Then you click **Push** (or use the up-arrow button) to send your commits to the server.

From the command line:

```bash
# Stage all changed files
git add .

# Commit with a message (this is local only)
git commit -m "Fixed the login page bug"

# Push to the remote server
git push
```

**Why is this two steps?** Because in Git, a commit is a lightweight, private operation. You can commit ten times in an afternoon as you work through a problem, and none of those commits go to the server until you push. This lets you:

- Work offline (on a plane, in a coffee shop with bad wifi, in a data center with no internet)
- Make small, incremental commits without worrying about breaking the build for others
- Rewrite, combine, or reorder your commits before sharing them (more on this later)
- Keep a detailed local history while presenting a clean set of changes to your team

In TFS, every check-in is immediately public. In Git, you choose when to make your work public. This is one of Git's greatest strengths.

### 2.5 — Changeset vs. Commit

**In TFS:** When you check in, TFS creates a changeset. A changeset has a sequential number (changeset 1, changeset 2, changeset 3, and so on). Changesets are numbered globally across the entire server. If you check in and get changeset 12345, the next person who checks in on any project on that server gets changeset 12346.

**In Git:** The equivalent is a commit. But Git does not use sequential numbers. Instead, each commit is identified by a SHA hash — a 40-character hexadecimal string that looks like `a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0`. This hash is computed from the content of the commit (the files, the parent commit, the author, the timestamp, and the message). This means that if two people make identical commits, they get the same hash. And if anything about the commit changes — even one byte — the hash changes completely.

In practice, you rarely type the full 40-character hash. You can use the first 7–12 characters, as long as they are unique in your repository. So you might refer to a commit as `a1b2c3d` instead of the full hash.

```bash
# TFS: Show changeset 12345
tf changeset 12345

# Git: Show commit a1b2c3d
git show a1b2c3d
```

**Key differences:**
- TFS changeset numbers are sequential and meaningful: changeset 100 happened before changeset 200. Git commit hashes are random-looking and do not imply order.
- TFS changeset numbers are global (shared across all projects on the server). Git commit hashes are unique per commit, not per repository. The same commit hash will not appear in a different repository unless the exact same commit was copied there.
- TFS changesets are created on the server. Git commits are created locally.

### 2.6 — Shelve vs. Stash

**In TFS:** Shelving is a way to save your pending changes on the server without checking them in. It is like putting your work in a locker at the gym — it is stored safely, but it is not in the main code. You might shelve changes when you need to switch to a different task, or when you want to share work-in-progress with a coworker for review.

In Visual Studio: **Team Explorer** → **Pending Changes** → **Shelve** → give it a name → click **Shelve**. Later, you unshelve to get the changes back.

**In Git:** The equivalent is a stash. A stash saves your uncommitted changes to a local holding area and reverts your working directory to a clean state. Unlike TFS shelves, stashes are local — they are not sent to any server.

In Visual Studio 2026, you can stash from the **Git Changes** window: click the **…** menu → **Stash All**. Later, you pop the stash to get your changes back.

From the command line:

```bash
# Save current changes to the stash
git stash

# List all stashes
git stash list

# Restore the most recent stash
git stash pop

# Restore a specific stash
git stash apply stash@{2}
```

**Key differences:**
- TFS shelves are on the server and can be shared with others. Git stashes are local only.
- If you need to share work-in-progress in Git, the idiomatic approach is to commit to a branch and push that branch. Stashes are meant for quick, temporary saves.
- TFS shelves have names. Git stashes are identified by index numbers (stash@{0}, stash@{1}, etc.) but you can add a message: `git stash push -m "WIP on login page"`.

### 2.7 — Branch

**In TFS/TFVC:** A branch is a folder-level copy of the source code on the server. When you branch in TFVC, TFS creates a new folder path on the server (e.g., `$/MyProject/Branches/FeatureX`) and copies the files there. This is a heavy operation that involves the server and creates actual folder paths. Branches in TFVC have a parent-child relationship, and the TFS server tracks which branch was branched from which.

To create a branch in TFS from Visual Studio: right-click a folder in **Source Control Explorer** → **Branching and Merging** → **Branch** → specify the target path.

**In Git:** A branch is a lightweight pointer to a commit. That is it. Creating a branch in Git does not copy any files. It creates a tiny file (41 bytes) that says "this branch points to commit X." This is why Git branches are fast — you can create and delete them in milliseconds, no matter how large your codebase is.

In Visual Studio 2026: **Git** → **New Branch** → type a name → choose which commit to branch from → click **Create**. Or from the bottom-left corner of the IDE, click the branch name → **New Branch**.

From the command line:

```bash
# Create a new branch from the current commit
git branch feature-login

# Create a new branch and switch to it
git checkout -b feature-login

# Modern equivalent (Git 2.23+)
git switch -c feature-login
```

**Key differences:**
- TFS branches are server-side folder copies. Git branches are local pointers that weigh 41 bytes.
- TFS branches are heavy and take time to create. Git branches are instant.
- In TFS, you work "in" a branch by mapping your workspace to a branch folder. In Git, you switch your local copy between branches using `checkout` or `switch`.
- TFS can have a small number of branches (creating hundreds would be impractical). Git repositories routinely have dozens or hundreds of branches.

### 2.8 — Merge

**In TFS:** Merging takes changes from one branch folder and applies them to another branch folder. You do this through Source Control Explorer: right-click the target branch → **Branching and Merging** → **Merge** → select the source branch → resolve any conflicts → check in.

**In Git:** Merging takes the commits from one branch and integrates them into another branch. You switch to the target branch and then merge the source branch into it.

In Visual Studio 2026: Switch to the target branch (e.g., `main`) → **Git** → **Manage Branches** → right-click the source branch → **Merge `source` into `target`**. Or from the command line:

```bash
# Switch to the branch you want to merge INTO
git checkout main

# Merge the feature branch into main
git merge feature-login
```

We will cover merge in exhaustive detail in later sections, including fast-forward merges, three-way merges, merge conflicts, and the exact scenario from the `learningbydoing` repository.

### 2.9 — Label vs. Tag

**In TFS:** A label is a marker you can apply to a specific set of file versions. It is like putting a sticky note on a particular state of the code: "This is the Release 2.1 code." Labels are mutable — you can move a label to point to different file versions after creating it.

**In Git:** The equivalent is a tag. A tag is a pointer to a specific commit. There are two kinds:

- **Lightweight tags:** Just a pointer to a commit, like a branch that never moves.
- **Annotated tags:** A full Git object with a tagger name, date, message, and optionally a GPG signature. These are preferred for release versions.

```bash
# Create a lightweight tag
git tag v2.1

# Create an annotated tag
git tag -a v2.1 -m "Release 2.1 - stable"

# Push tags to the remote
git push --tags
```

**Key difference:** TFS labels are mutable (you can change what they point to). Git annotated tags are immutable — once created, they point to the same commit forever. This makes them trustworthy markers for releases. Lightweight tags are technically movable, but the convention is to never move tags.

### 2.10 — History

**In TFS:** You right-click a file or folder → **Source Control** → **View History**. This shows a list of changesets that affected that item, with dates, authors, and comments.

**In Git:** The equivalent command is `git log`. In Visual Studio 2026, you can right-click a file → **Git** → **View History**, or go to **Git** → **View Branch History** for the full commit log.

```bash
# Show commit history
git log

# Show history with a graph (visualizes branches)
git log --oneline --graph --all

# Show history for a specific file
git log -- path/to/file.cs

# Show who changed each line of a file (blame)
git blame path/to/file.cs
```

The `git log --oneline --graph --all` command is incredibly useful. It draws an ASCII art graph of your branch structure, showing you how branches diverge and merge. It looks something like this:

```
* a1b2c3d (HEAD -> main) Merge branch 'feature-login'
|\
| * d4e5f6a (feature-login) Add login validation
| * 7b8c9d0 Create login form
|/
* e1f2a3b Initial commit
```

### 2.11 — Pending Changes vs. Staged Changes (The Staging Area)

This is a concept that has NO equivalent in TFS, and it is one of the most confusing parts of Git for TFS developers.

**In TFS:** You have two states for a file: unchanged and pending. You edit a file, it becomes "pending" (a pending change). When you check in, all your pending changes go to the server together.

**In Git:** There are THREE states:

1. **Modified** — You changed the file, but have not told Git you want to include it in the next commit.
2. **Staged** — You explicitly told Git "include this file in my next commit." This is done with `git add`.
3. **Committed** — The change is saved in a commit.

The staging area (also called the "index") is a middle ground between your working directory and the repository. It lets you choose exactly which changes go into a commit. You might have changed five files, but only want to commit three of them right now. You `git add` those three, commit, and the other two stay modified but uncommitted.

In Visual Studio 2026, the Git Changes window shows your changes. You can right-click individual files and choose **Stage** to add them to the staging area. Then when you click **Commit Staged**, only the staged files are committed. If you click **Commit All**, Visual Studio stages everything and commits it in one step (which is what most people do and is closest to the TFS experience).

```bash
# Stage a specific file
git add Login.aspx.cs

# Stage all changes
git add .

# See what is staged and what is not
git status

# Commit only what is staged
git commit -m "Fix login validation"
```

**Why does this exist?** The staging area lets you craft precise, meaningful commits. Suppose you fixed a bug in `Login.aspx.cs` and also reformatted `Default.aspx`. Those are two unrelated changes. In TFS, you would check them both in together. In Git, you can:

```bash
git add Login.aspx.cs
git commit -m "Fix null reference in login validation"

git add Default.aspx
git commit -m "Reformat Default.aspx to match team style"
```

Now you have two clean commits, each doing one thing. This makes the history easier to understand, easier to review, and easier to revert if something goes wrong.

If this feels like extra work and you just want to commit everything like you did in TFS, that is fine. Just use `git add .` followed by `git commit`, or use Visual Studio's **Commit All** button. You can adopt the staging area later when you are comfortable.

### 2.12 — Lock vs. (Not Really a Thing in Git)

**In TFS/TFVC server workspaces:** You can lock a file, preventing others from checking it out or checking in changes to it. This is useful for binary files (images, PDFs, Word documents) that cannot be merged.

**In Git:** There is no built-in file locking. Git assumes all files are text and can be merged. For binary files, there is an extension called **Git LFS** (Large File Storage) and a separate lock feature in some hosting platforms (GitHub and Azure DevOps support it), but it is not a core Git feature.

If your team works with many binary files that cannot be merged, this is a genuine disadvantage of Git compared to TFVC. For most teams working primarily with source code (`.cs`, `.js`, `.html`, `.css`, `.csproj` files), this is not an issue.


## Part 3 — What Is a Commit? Understand This and You Understand Git

The single most important concept in Git is the commit. If you understand commits, you understand Git. If you do not understand commits, nothing else will make sense.

### 3.1 — A Commit Is a Snapshot, Not a Diff

In TFS, a changeset stores the differences — the delta — between the previous version and the new version. If you changed line 42 of a file, the changeset stores "line 42 changed from X to Y."

In Git, a commit stores a complete snapshot of every file in the repository at that point in time. Not the differences — the complete files. If you have 1,000 files and you change one of them, the commit conceptually records all 1,000 files.

"But wait," you say, "doesn't that waste a huge amount of disk space?" No, because Git is clever. If a file has not changed, Git does not store a new copy. It stores a pointer to the previous version. And Git compresses everything into pack files that are extremely space-efficient. In practice, Git repositories are often smaller than their TFVC equivalents.

But conceptually, you should think of each commit as a complete photograph of your project at a moment in time. Not a description of what changed — a picture of what everything looks like.

### 3.2 — A Commit Has a Parent (or Two)

Every commit (except the very first one) has a pointer to its parent commit — the commit that came immediately before it. This creates a chain:

```
Commit A ← Commit B ← Commit C ← Commit D
```

Commit D's parent is C. C's parent is B. B's parent is A. A has no parent (it is the initial commit, sometimes called the "root commit").

This chain is the history of your project. By following the parent pointers backward from any commit, you can reconstruct the entire history of how the project evolved.

When you merge two branches, the resulting merge commit has TWO parents — one from each branch. This is how Git records that two lines of development were combined:

```
        ← Commit E ← Commit F ←
       /                         \
Commit D                          Commit G (merge commit, two parents: F and H)
       \                         /
        ← Commit H ←            ←
```

Commit G has two parents: F and H. By following both parent chains, you can see the full history of both branches.

### 3.3 — A Commit Is Identified by Its Hash

As mentioned earlier, every commit has a SHA-1 hash (a 40-character hexadecimal string) computed from:
- The content of all the files in the snapshot
- The parent commit's hash
- The author's name and email
- The committer's name and email
- The timestamp
- The commit message

If any of these change, the hash changes. This means that a commit's hash is a fingerprint that uniquely identifies not just the commit, but the entire history leading up to it. If commit C has hash `abc123`, and its parent B has hash `def456`, then C's hash encodes the fact that B is its parent. You cannot change B without also changing C (and every commit after C).

This is why Git is secure. You cannot tamper with history without changing every subsequent hash, which would be immediately obvious.

### 3.4 — Commits Form a Directed Acyclic Graph (DAG)

In TFS, history is a straight line: changeset 1, changeset 2, changeset 3, and so on. It is linear.

In Git, history is a graph. Specifically, a Directed Acyclic Graph (DAG). "Directed" means the edges (parent pointers) go in one direction (from child to parent). "Acyclic" means there are no loops — you can never follow parent pointers and end up back where you started.

The graph looks like this:

```
A --- B --- C --- D --- G --- H    (main)
              \       /
               E --- F              (feature branch)
```

Commit A is the root. B follows A. C follows B. Then the graph splits: D continues on one path, while E branches off. F follows E. Then G brings the two paths back together (it is a merge commit with parents D and F). H follows G.

Understanding this graph is the key to understanding Git. Every Git operation — branch, merge, rebase, cherry-pick, reset — is an operation on this graph.


## Part 4 — What Is a Branch? It Is Not What You Think

### 4.1 — The TFS Mental Model of Branches (Folders)

In TFS/TFVC, a branch is a folder. When you branch `$/MyProject/Main` to `$/MyProject/Branches/FeatureX`, TFS creates a new folder on the server and copies all the files into it. You switch between branches by switching which folder your workspace is mapped to.

This means:
- Branches take up disk space on the server (one copy of all files per branch)
- Creating a branch is a server operation that takes time
- You can have your workspace mapped to only one branch at a time (or you can map multiple branches to different local folders)
- Branches are visible in Source Control Explorer as folders

### 4.2 — The Git Mental Model of Branches (Pointers)

In Git, a branch is a 41-byte text file that contains the hash of a commit. That is it. Really. Let me prove it to you:

```bash
# Show what the 'main' branch actually is
cat .git/refs/heads/main
# Output: a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0
```

The file `.git/refs/heads/main` contains one line: the hash of the commit that the `main` branch currently points to. When you make a new commit on `main`, Git updates this file to point to the new commit. That is all "being on the main branch" means.

When you create a new branch:

```bash
git branch feature-login
```

Git creates a new file `.git/refs/heads/feature-login` containing the same hash that `main` currently points to. Both branches now point to the same commit. No files are copied. No server is contacted. It takes microseconds.

### 4.3 — HEAD: Where You Are Right Now

Git has a special pointer called `HEAD` that indicates which branch you are currently on. It is a file at `.git/HEAD`:

```bash
cat .git/HEAD
# Output: ref: refs/heads/main
```

This says "I am currently on the `main` branch." When you make a commit, Git:
1. Creates a new commit object (with the current commit as its parent)
2. Updates the branch that HEAD points to, so it points to the new commit
3. HEAD continues to point to the same branch name

If you switch to a different branch:

```bash
git switch feature-login
```

Git updates `.git/HEAD` to say `ref: refs/heads/feature-login`, and then updates all the files in your working directory to match the snapshot that `feature-login` points to.

### 4.4 — Why Git Branches Are Revolutionary

Because branches are just pointers, Git enables workflows that would be impractical in TFS:

- **Create a branch for every task.** Working on a bug fix? Create a branch. Starting a new feature? Create a branch. Experimenting with an idea? Create a branch. If the experiment fails, delete the branch. It costs nothing.
- **Switch branches in seconds.** Git updates the files in your working directory to match the target branch. For most projects, this takes under a second.
- **Have hundreds of branches.** Some projects have hundreds or thousands of branches. Each one is just a 41-byte file.
- **Branch locally.** You can create a branch that only exists on your machine. No server communication needed. You can decide later whether to share it.


## Part 5 — The Scenario: Two Branches, One Conflict, and Why It Happens

Now we are going to work through the exact scenario from the `learningbydoing` repository on GitHub (https://github.com/kusl/learningbydoing/). This is a real scenario that demonstrates one of the most common and confusing aspects of Git: merge conflicts from diverged branches.

### 5.1 — Setting the Stage

Imagine you have a repository with a single file. On the `main` branch, the file looks like this:

```
This is the base commit.

This is common for both branches.
In the next line, I will write the branch name.
main
In the line above, I will replace with the name of the current branch
in each of my two branches.
Because each of those two branches are directly from main,
I won't be able to merge one into the other directly without a conflict
or so I think.
Lets find out.
```

Notice line 5: it says `main`. This is the line we are going to change in each branch.

### 5.2 — Creating branch-1

We create `branch-1` from `main`. At this point, `branch-1` points to the exact same commit as `main`. The file is identical.

```bash
git checkout -b branch-1
```

Now we edit the file, changing line 5 from `main` to `branch-1`:

```
This is the base commit.

This is common for both branches.
In the next line, I will write the branch name.
branch-1
In the line above, I will replace with the name of the current branch
in each of my two branches.
Because each of those two branches are directly from main,
I won't be able to merge one into the other directly without a conflict
or so I think.
Lets find out.
```

We commit this change:

```bash
git add .
git commit -m "Replace main with branch-1"
```

Now the graph looks like this:

```
A --- B    (branch-1)
      |
      (main also points here originally, but branch-1 moved forward)
```

Wait, let me be more precise:

```
A    (main)
 \
  B  (branch-1)
```

Commit A is the original content (with "main" on line 5). Commit B is the branch-1 edit (with "branch-1" on line 5). `main` still points to A. `branch-1` points to B.

### 5.3 — Creating branch-2

We go back to `main` and create `branch-2`:

```bash
git checkout main
git checkout -b branch-2
```

Notice: we went back to `main` first. Our working directory now has the file as it was in commit A — with "main" on line 5. `branch-2` starts from commit A, just like `branch-1` did.

We edit line 5 to say `branch-2`:

```
This is the base commit.

This is common for both branches.
In the next line, I will write the branch name.
branch-2
In the line above, I will replace with the name of the current branch
in each of my two branches.
Because each of those two branches are directly from main,
I won't be able to merge one into the other directly without a conflict
or so I think.
Lets find out.
```

We commit:

```bash
git add .
git commit -m "Replace main with branch-2"
```

Now the graph looks like this:

```
  B  (branch-1) — line 5 says "branch-1"
 /
A    (main) — line 5 says "main"
 \
  C  (branch-2) — line 5 says "branch-2"
```

Three commits. Three branches. All three have different content on line 5.

### 5.4 — Merging branch-1 into main (No Conflict)

Let us merge `branch-1` into `main`:

```bash
git checkout main
git merge branch-1
```

What happens? Git looks at the current state of `main` (commit A) and the current state of `branch-1` (commit B). Because B is a direct descendant of A (B's parent IS A), Git can do a **fast-forward merge**. This means Git just moves the `main` pointer from A to B. No actual merging is needed because there is nothing to merge — B already contains everything A had, plus the change.

After the fast-forward:

```
       (main, branch-1)
       |
  B  — line 5 says "branch-1"
 /
A
 \
  C  (branch-2) — line 5 says "branch-2"
```

Now `main` and `branch-1` both point to commit B. Line 5 in `main` now says "branch-1."

### 5.5 — Merging branch-2 into main (Conflict!)

Now let us try to merge `branch-2` into `main`:

```bash
git merge branch-2
```

This time, things are different. `main` is at commit B (line 5 says "branch-1"). `branch-2` is at commit C (line 5 says "branch-2"). B and C both descended from A, but they diverged — they both changed line 5, but they changed it to different things.

Git performs a **three-way merge**. It looks at three versions of the file:

1. **The common ancestor** (commit A): line 5 says `main`
2. **The current branch** (main, commit B): line 5 says `branch-1`
3. **The branch being merged** (branch-2, commit C): line 5 says `branch-2`

Git compares each version to the ancestor:
- `main` changed line 5 from `main` to `branch-1`
- `branch-2` changed line 5 from `main` to `branch-2`

Both branches changed the same line in different ways. Git cannot decide which change to keep. This is a **merge conflict**.

Git marks the conflict in the file:

```
This is the base commit.

This is common for both branches.
In the next line, I will write the branch name.
<<<<<<< HEAD
branch-1
=======
branch-2
>>>>>>> branch-2
In the line above, I will replace with the name of the current branch
in each of my two branches.
Because each of those two branches are directly from main,
I won't be able to merge one into the other directly without a conflict
or so I think.
Lets find out.
```

The conflict markers tell you:
- `<<<<<<< HEAD` — This is the start of the conflict. Everything between here and `=======` is from the current branch (HEAD, which is `main`).
- `=======` — This separates the two versions.
- `>>>>>>> branch-2` — Everything between `=======` and this line is from the branch being merged (`branch-2`).

You, the human, must decide what the correct content should be. Maybe you want "branch-1." Maybe you want "branch-2." Maybe you want something entirely different. You edit the file to remove the conflict markers and keep the content you want:

```
This is the base commit.

This is common for both branches.
In the next line, I will write the branch name.
branch-1
In the line above, I will replace with the name of the current branch
in each of my two branches.
Because each of those two branches are directly from main,
I won't be able to merge one into the other directly without a conflict
or so I think.
Lets find out.
```

Then you stage the resolved file and commit:

```bash
git add .
git commit -m "Merge branch-2 into main, keeping branch-1's line"
```

This creates a merge commit D with two parents (B and C):

```
  B --------- D  (main) — merge commit
 / \         /
A    \      /
 \    -----/
  C  (branch-2)
```

### 5.6 — Why Merging branch-1 into branch-2 Directly Also Conflicts

Now the question: what if, instead of merging both into main, we tried to merge branch-1 directly into branch-2 (or vice versa)?

```bash
git checkout branch-2
git merge branch-1
```

This would also produce a conflict, for exactly the same reason. The common ancestor of branch-1 (commit B) and branch-2 (commit C) is commit A. Both branches changed line 5 relative to their common ancestor, and they changed it to different things.

The only way to merge without conflict is if one branch's changes are a superset of the other's. If branch-1 changed line 5 and branch-2 changed line 20 (different lines), there would be no conflict. But when both branches change the same line relative to their common ancestor, Git has no way to automatically resolve the dispute. A human must intervene.

### 5.7 — But Wait: Why Did branch-1 Merge into main Without a Conflict?

Because when we merged branch-1 into main, main had NOT changed since branch-1 was created. Main was still at commit A. Branch-1 had made changes (commit B) on top of A. There was no divergence — branch-1 was strictly ahead of main. Git handled this with a fast-forward merge: just move the pointer.

The conflict only appeared when we tried to merge branch-2 into main AFTER main had already moved forward to incorporate branch-1's changes. At that point, main (at B) and branch-2 (at C) had both changed the same line relative to their common ancestor (A), and the conflict was inevitable.

## Part 6 — Fast-Forward Merge vs. Three-Way Merge vs. No-Fast-Forward Merge

### 6.1 — Fast-Forward Merge

A fast-forward merge happens when the current branch has not moved since the other branch was created. The other branch is strictly "ahead." Git just moves the pointer forward.

Before:
```
A --- B --- C  (feature)
|
(main)
```

After `git merge feature` from `main`:
```
A --- B --- C  (main, feature)
```

No new commit is created. The history is a straight line.

**When does this happen in TFS terms?** Imagine you shelved some work, nobody else checked in, and then you unshelved and checked in. Your check-in is the only change since your starting point. That is similar to a fast-forward.

### 6.2 — Three-Way Merge

A three-way merge happens when both branches have new commits. Git finds the common ancestor, compares both branches to it, and combines the changes.

Before:
```
A --- B --- C  (main)
      \
       D --- E  (feature)
```

After `git merge feature` from `main`:
```
A --- B --- C --- F  (main, merge commit)
      \         /
       D --- E  (feature)
```

Commit F is a new commit with two parents: C and E. If there are no conflicts (the two branches changed different lines or different files), Git creates the merge commit automatically. If there are conflicts, you resolve them first.

### 6.3 — No-Fast-Forward Merge (--no-ff)

Sometimes you want to create a merge commit even when a fast-forward is possible. This preserves the fact that work was done on a branch.

```bash
git merge --no-ff feature-login
```

Before (fast-forward would be possible):
```
A --- B  (main)
      \
       C --- D  (feature)
```

With `--no-ff`:
```
A --- B ---------- E  (main, merge commit)
      \          /
       C --- D  (feature)
```

Instead of just moving `main` forward to D, Git creates a merge commit E. The advantage is that the history clearly shows "there was a feature branch, and it was merged." If you use fast-forward merges, the branch history is lost — it looks like everything happened on `main`.

Many teams use `--no-ff` as a policy. GitHub and Azure DevOps pull requests use it by default. When you click "Merge" on a pull request in Azure DevOps or GitHub, it creates a merge commit, not a fast-forward.

### 6.4 — How to Configure This in Visual Studio and Azure DevOps

**In Visual Studio 2026:** When you merge from the Manage Branches panel, Visual Studio performs a regular merge. If a fast-forward is possible, it will fast-forward by default. To force a merge commit, you need to use the command line with `--no-ff`.

**In Azure DevOps:** When you complete a pull request, you have three options:
- **Merge (no fast-forward):** Creates a merge commit. This is the default and most common.
- **Squash merge:** Combines all the branch's commits into a single commit on main. The branch history is lost, but main has a clean linear history.
- **Rebase and fast-forward:** Replays the branch's commits on top of main, then fast-forwards. Linear history is preserved, but individual commits are kept.

We will discuss squash and rebase in detail later.


## Part 7 — Rebase: The Most Feared Git Command, Demystified

### 7.1 — What Is Rebase?

Rebase takes a series of commits and "replays" them on top of a different starting point. It rewrites history so that it looks like you made your changes starting from a newer point, even though you did not.

Here is the scenario. You created a feature branch from main last week. Since then, main has moved forward with new commits. Your branch is "behind."

Before rebase:
```
A --- B --- C --- D  (main)
      \
       E --- F  (feature)
```

Your feature branch (`E --- F`) started from B. Main has since moved to D. If you merge, you get a merge commit. If you rebase, you move your branch's starting point:

```bash
git checkout feature
git rebase main
```

After rebase:
```
A --- B --- C --- D  (main)
                  \
                   E' --- F'  (feature)
```

Git took commits E and F, computed the changes they represent, and applied those changes on top of D. The result is two NEW commits (E' and F') that have the same changes as E and F, but with a different starting point and therefore different hashes.

**The old commits E and F still exist** in the Git object database, but no branch points to them anymore. They are "orphaned" and will eventually be garbage-collected.

### 7.2 — Why Rebase?

Rebase produces a linear history. Instead of a merge commit that creates a fork-and-join pattern in the graph, rebase makes it look like you wrote your code on top of the latest main. Many teams prefer this because it is easier to read.

Compare:

**With merge:**
```
A --- B --- C --- D --- G  (main)
      \               /
       E --- F ------
```

**With rebase and fast-forward:**
```
A --- B --- C --- D --- E' --- F'  (main)
```

The second graph is a straight line. Easier to follow, easier to bisect for bugs.

### 7.3 — The Golden Rule of Rebase

**Never rebase commits that have been pushed to a shared branch.**

Rebase rewrites history. It creates new commits with new hashes. If you rebase commits that other people have already based their work on, you will cause chaos. Their commits will reference parent hashes that no longer exist. Git will get confused. People will get angry. There will be duplicate commits, merge conflicts, and tears.

Rebase is safe for:
- Local commits that you have not pushed yet
- Commits on your personal feature branch that nobody else is working on

Rebase is DANGEROUS for:
- Commits on `main` or any shared branch
- Commits that other people have pulled

In TFS terms: it is like if you could go back in time and change a changeset that other people already got. You would never do that, and the server would not let you. Git lets you do it locally, but you should not do it with shared history.

### 7.4 — Interactive Rebase: Crafting Your History

Interactive rebase (`git rebase -i`) lets you edit, reorder, combine, or delete commits before sharing them. This is one of Git's most powerful features.

```bash
git rebase -i main
```

This opens an editor with a list of your commits:

```
pick e1f2a3b Add login form
pick d4e5f6a Fix typo in login form
pick 7b8c9d0 Add login validation
pick a1b2c3d Fix another typo
```

You can change the commands:

- `pick` — Keep the commit as-is
- `reword` — Keep the commit but change the message
- `edit` — Pause at this commit so you can change its content
- `squash` — Combine this commit with the previous one (keep both messages)
- `fixup` — Combine with previous commit, but discard this commit's message
- `drop` — Delete this commit entirely

So you might change it to:

```
pick e1f2a3b Add login form
fixup d4e5f6a Fix typo in login form
pick 7b8c9d0 Add login validation
fixup a1b2c3d Fix another typo
```

This combines the typo fixes with their preceding commits, resulting in just two clean commits: "Add login form" and "Add login validation." The typo fixes are absorbed.

This is why Git developers often say "commit early, commit often." You can always clean up your commits later with interactive rebase before pushing.


## Part 8 — Cherry-Pick: The Surgeon's Scalpel

### 8.1 — What Is Cherry-Pick?

Cherry-pick takes a single commit from one branch and applies it to another branch. It is like saying "I want just this one change, not the entire branch."

**In TFS:** The closest equivalent is a merge of specific changesets. In Source Control Explorer, you can choose to merge only specific changesets instead of all of them.

**In Git:**

```bash
git checkout main
git cherry-pick a1b2c3d
```

This applies the changes from commit `a1b2c3d` to `main`, creating a new commit. The new commit has the same changes but a different hash (because it has a different parent).

In Visual Studio 2026: **Git** → **View Branch History** → right-click a commit on another branch → **Cherry-Pick**.

### 8.2 — When to Use Cherry-Pick

- **Hot fixes:** You fixed a critical bug on a feature branch and need the fix on main immediately, without merging the entire feature branch.
- **Backporting:** You made a change on a newer release branch and need to apply it to an older release branch.
- **Oops, wrong branch:** You committed to the wrong branch. Cherry-pick the commit to the right branch, then remove it from the wrong one.

### 8.3 — When NOT to Use Cherry-Pick

Do not use cherry-pick as a substitute for regular merging. If you cherry-pick commits one by one from a branch instead of merging the branch, you end up with duplicate changes in your history (the original commits on the source branch and the cherry-picked copies on the target branch). This makes future merges confusing because Git may see the same changes appearing from both sides.

Use merge (or rebase) for combining branches. Use cherry-pick for the occasional surgical extraction of individual commits.


## Part 9 — Reset, Revert, and How to Undo Things

One of the most empowering things about Git is that almost nothing is permanent. You can undo almost anything. Here is your complete guide to undoing things.

### 9.1 — Undo an Uncommitted Change (Get Back to the Last Commit)

You edited a file and want to throw away the changes:

```bash
# Discard changes to a specific file
git checkout -- Login.aspx.cs

# Modern equivalent (Git 2.23+)
git restore Login.aspx.cs

# Discard ALL uncommitted changes
git checkout -- .
git restore .
```

In Visual Studio 2026: In the **Git Changes** window, right-click the file → **Undo Changes**. Or for all files: **…** menu → **Undo All Changes**.

**TFS equivalent:** Right-click → **Source Control** → **Undo**.

### 9.2 — Unstage a Staged File

You ran `git add` but want to un-add (unstage) a file before committing:

```bash
git reset HEAD Login.aspx.cs

# Modern equivalent
git restore --staged Login.aspx.cs
```

In Visual Studio 2026: In the Git Changes window, right-click the staged file → **Unstage**.

### 9.3 — Amend the Last Commit

You just committed but forgot to include a file, or your commit message has a typo:

```bash
# Add the forgotten file
git add ForgottenFile.cs

# Amend the last commit (opens editor for the message)
git commit --amend

# Amend with a new message
git commit --amend -m "Corrected commit message"
```

In Visual Studio 2026: In the Git Changes window, check the **Amend** checkbox before committing.

**Warning:** Amending rewrites the last commit (new hash). Do not amend commits that have already been pushed, unless you are the only one on the branch.

### 9.4 — Revert a Commit (Create an "Undo" Commit)

You want to undo a commit that has already been pushed and shared. You cannot rewrite history (because others have it). Instead, you create a new commit that reverses the changes:

```bash
git revert a1b2c3d
```

This creates a new commit that applies the exact opposite of `a1b2c3d`'s changes. If `a1b2c3d` added line 42, the revert commit removes line 42. History is preserved — both the original commit and the revert are in the log.

In Visual Studio 2026: **Git** → **View History** → right-click the commit → **Revert**.

**TFS equivalent:** In TFS, you would do a "rollback" of a changeset.

### 9.5 — Reset: Moving the Branch Pointer

Reset moves the current branch pointer to a different commit. It is more powerful than revert but more dangerous.

There are three modes:

**`git reset --soft`** — Move the branch pointer, but keep all changes staged. Like un-committing but keeping the work.

```bash
git reset --soft HEAD~1
# Moves back one commit, changes are staged
```

**`git reset --mixed`** (default) — Move the branch pointer and unstage changes, but keep them in the working directory.

```bash
git reset HEAD~1
# Moves back one commit, changes are unstaged but still in files
```

**`git reset --hard`** — Move the branch pointer and throw away all changes. This is destructive.

```bash
git reset --hard HEAD~1
# Moves back one commit, all changes are gone
```

**Warning:** `git reset --hard` destroys uncommitted changes. Use with caution. However, even after a hard reset, committed changes are still in the object database for a while. You can find them with `git reflog` (discussed next).

### 9.6 — Reflog: Your Safety Net

Git keeps a log of every time HEAD moves — every commit, checkout, reset, rebase, merge, and more. This is the reflog (reference log).

```bash
git reflog
```

Output looks like:

```
a1b2c3d HEAD@{0}: commit: Fix login bug
e1f2a3b HEAD@{1}: checkout: moving from feature to main
d4e5f6a HEAD@{2}: commit: Add feature
...
```

If you accidentally reset or deleted a branch, you can find the commit hash in the reflog and recover:

```bash
# Oops, I reset too far
git reflog
# Find the commit hash you want
git reset --hard a1b2c3d
```

The reflog is local and is kept for 90 days by default. It is your last line of defense against mistakes.


## Part 10 — Git Configuration: The Settings You Need

### 10.1 — Identity Configuration

The first thing you must do after installing Git is tell it who you are:

```bash
git config --global user.name "Your Name"
git config --global user.email "your.email@company.com"
```

These are embedded in every commit you make. Use the email associated with your Azure DevOps or GitHub account.

### 10.2 — Line Ending Configuration (Critical on Windows)

This is one of the most common sources of problems for Windows developers. Unix/Mac systems use `LF` (line feed) for line endings. Windows uses `CRLF` (carriage return + line feed). If you do not configure Git properly, you will see phantom changes where every line in every file appears modified because the line endings changed.

```bash
# On Windows: convert CRLF to LF when committing, LF to CRLF when checking out
git config --global core.autocrlf true
```

Better yet, create a `.gitattributes` file in the root of your repository:

```
# Set default behavior to automatically normalize line endings
* text=auto

# Force certain files to always use LF
*.cs text eol=lf
*.csproj text eol=lf
*.sln text eol=lf
*.json text eol=lf
*.xml text eol=lf
*.yml text eol=lf
*.yaml text eol=lf
*.md text eol=lf
*.razor text eol=lf
*.css text eol=lf
*.js text eol=lf
*.ts text eol=lf
*.html text eol=lf

# Binary files (do not normalize)
*.png binary
*.jpg binary
*.gif binary
*.ico binary
*.woff binary
*.woff2 binary
*.ttf binary
*.eot binary
*.pdf binary
```

This file is committed to the repository and ensures consistent behavior for everyone on the team, regardless of their OS.

### 10.3 — Default Branch Name

Older Git versions defaulted to `master` as the initial branch name. Modern practice uses `main`:

```bash
git config --global init.defaultBranch main
```

### 10.4 — Credential Helper

To avoid typing your username and password every time you push or pull:

```bash
# On Windows (uses Windows Credential Manager)
git config --global credential.helper manager

# Or specifically for Git Credential Manager
git config --global credential.helper manager-core
```

Visual Studio 2026 handles credentials automatically through its built-in Git integration. If you are connected to Azure DevOps or GitHub in Visual Studio, it handles authentication for you.

### 10.5 — Editor Configuration

When Git needs you to edit text (commit messages, interactive rebase), it opens an editor. By default on Windows this might be Vim, which can be very confusing if you have never used it. Configure a friendlier editor:

```bash
# Use Visual Studio Code
git config --global core.editor "code --wait"

# Use Notepad
git config --global core.editor "notepad"

# Use Notepad++
git config --global core.editor "'C:/Program Files/Notepad++/notepad++.exe' -multiInst -notabbar -nosession -noPlugin"
```

### 10.6 — Aliases

You can create shortcuts for common commands:

```bash
git config --global alias.st status
git config --global alias.co checkout
git config --global alias.br branch
git config --global alias.ci commit
git config --global alias.lg "log --oneline --graph --all --decorate"
```

Now `git st` runs `git status`, and `git lg` shows a beautiful branch graph.

### 10.7 — Viewing All Configuration

```bash
# List all settings
git config --list

# Show where each setting comes from
git config --list --show-origin
```


## Part 11 — The .gitignore File: What Not to Track

### 11.1 — What It Is

A `.gitignore` file tells Git which files and folders to ignore. Git will not track them, will not show them as modified, and will not commit them.

**TFS equivalent:** In TFS/TFVC, you could use the `.tfignore` file for local workspaces, or you simply chose not to add files to source control. But TFS did not have a built-in ignore mechanism until local workspaces were introduced.

### 11.2 — Essential .gitignore for .NET Projects

Every .NET project should have a `.gitignore` that excludes build outputs, NuGet packages, and user-specific files. Here is a minimal but complete example:

```gitignore
## .NET build output
bin/
obj/

## NuGet
**/packages/*
*.nupkg

## Visual Studio
.vs/
*.user
*.suo

## JetBrains Rider
.idea/

## VS Code
.vscode/

## OS files
.DS_Store
Thumbs.db

## Publish output
publish/

## Test results
TestResults/
```

GitHub provides an excellent .gitignore template for Visual Studio projects. When you create a new repository on GitHub, you can choose "Visual Studio" from the .gitignore template dropdown.

### 11.3 — What If You Already Committed a File You Should Have Ignored?

If you committed `bin/` or `.vs/` before adding them to `.gitignore`, the ignore rule does not retroactively remove them. You need to explicitly remove them from tracking:

```bash
# Remove from Git tracking but keep on disk
git rm -r --cached bin/
git rm -r --cached .vs/

# Commit the removal
git commit -m "Remove build outputs from version control"
```

After this, the ignore rules take effect for those paths.


## Part 12 — Real-World Scenarios: A Cookbook

This section covers the most common real-world situations you will encounter, with exact steps for both the command line and Visual Studio.

### 12.1 — Scenario: "I Need to Start Working on a Bug Fix"

**What you did in TFS:** You got the latest code, made your changes directly in the main branch, and checked in when done.

**What you do in Git:**

1. Pull the latest main: `git pull origin main` (or in VS: Git → Pull)
2. Create a branch: `git switch -c bugfix/login-null-ref` (or in VS: bottom-left branch → New Branch)
3. Make your changes
4. Commit: `git add . && git commit -m "Fix null reference in login page"`
5. Push the branch: `git push -u origin bugfix/login-null-ref` (or in VS: Git → Push)
6. Create a Pull Request in Azure DevOps or GitHub
7. After review and approval, merge the PR
8. Delete the branch (Azure DevOps can do this automatically)
9. Switch back to main and pull: `git switch main && git pull`

### 12.2 — Scenario: "Someone Else Changed the Same File I Was Working On"

**What happened in TFS:** You try to check in and TFS says "merge conflict." You resolve in the merge tool and check in.

**What happens in Git:**

1. You try to pull (or merge): `git pull`
2. Git says there are conflicts. It modifies the conflicted files with conflict markers.
3. In Visual Studio 2026, the Git Changes window shows conflicted files with a warning icon.
4. Double-click each conflicted file to open the merge editor.
5. Resolve each conflict by choosing the left side, right side, or editing manually.
6. Click **Accept Merge**.
7. After all conflicts are resolved, commit: `git commit -m "Resolve merge conflicts"`

### 12.3 — Scenario: "I Committed to the Wrong Branch"

This happens more often than anyone wants to admit.

**Solution 1: Move the commit to the right branch**

```bash
# You are on main and just committed something that should be on feature
# Step 1: Note the commit hash
git log -1 --oneline
# Output: a1b2c3d Accidentally committed on main

# Step 2: Undo the commit on main (keep changes staged)
git reset --soft HEAD~1

# Step 3: Stash the changes
git stash

# Step 4: Switch to the correct branch
git switch feature-login

# Step 5: Apply the stash
git stash pop

# Step 6: Commit on the correct branch
git commit -m "Add login validation"
```

**Solution 2: Cherry-pick**

```bash
# You already committed and maybe even pushed on main
# Step 1: Switch to the correct branch
git switch feature-login

# Step 2: Cherry-pick the commit from main
git cherry-pick a1b2c3d

# Step 3: Go back to main and remove the commit
git switch main
git reset --hard HEAD~1
# WARNING: Only do this if you have NOT pushed yet!
```

### 12.4 — Scenario: "I Need to See What My Coworker Changed"

```bash
# Fetch the latest from the server (without merging)
git fetch

# See all remote branches
git branch -r

# Look at a specific remote branch
git log origin/feature-payments --oneline

# See the diff between your branch and theirs
git diff main..origin/feature-payments

# Check out their branch to look at it
git switch feature-payments
```

In Visual Studio 2026: After fetching, go to **Git** → **Manage Branches**. Under **remotes/origin**, you see all remote branches. Double-click one to check it out locally.

### 12.5 — Scenario: "I Messed Everything Up and Want to Start Over"

Everyone has this moment. Here are your options, from least destructive to most:

**Option 1: Undo uncommitted changes**
```bash
git restore .
git clean -fd  # Remove untracked files too
```

**Option 2: Undo the last commit but keep changes**
```bash
git reset --soft HEAD~1
```

**Option 3: Undo the last commit and throw away changes**
```bash
git reset --hard HEAD~1
```

**Option 4: Reset to exactly what the server has**
```bash
git fetch origin
git reset --hard origin/main
```

**Option 5: Nuclear option — throw away everything and clone again**
```bash
cd ..
rm -rf myproject
git clone https://github.com/myorg/myproject.git
```

Nobody will judge you for Option 5. We have all done it.

### 12.6 — Scenario: "I Need to Work on Two Things at Once"

In TFS, you might have created a second workspace mapped to a different branch folder.

In Git, you have several options:

**Option 1: Stash and switch**
```bash
git stash
git switch other-feature
# work on the other feature
git switch original-feature
git stash pop
```

**Option 2: Use Git worktrees**

Git worktrees let you have multiple working copies of the same repository in different folders, each on a different branch:

```bash
git worktree add ../myproject-hotfix hotfix/urgent-bug
cd ../myproject-hotfix
# Work on the hotfix
# When done:
git worktree remove ../myproject-hotfix
```

This is the closest equivalent to having multiple TFS workspaces.

### 12.7 — Scenario: "My Branch Is Out of Date and I Need the Latest main"

**Option 1: Merge main into your branch**
```bash
git switch my-feature
git merge main
```

This creates a merge commit on your branch. It is safe and preserves history.

**Option 2: Rebase your branch on top of main**
```bash
git switch my-feature
git rebase main
```

This replays your commits on top of the latest main. It creates a cleaner history but rewrites your commits (new hashes). Only do this if you have not pushed yet, or if you are the only person working on the branch.

In Visual Studio 2026: **Git** → **Manage Branches** → right-click `main` → **Merge into Current Branch** (for merge) or **Rebase Current Branch onto main** (for rebase).

### 12.8 — Scenario: "I Need to Find When a Bug Was Introduced"

Git has a powerful tool called `bisect` that does a binary search through your commit history to find the exact commit that introduced a bug.

```bash
# Start bisecting
git bisect start

# The current version has the bug
git bisect bad

# Version from last month was good
git bisect good v2.0

# Git checks out a commit halfway between. Test it.
# If the bug is there:
git bisect bad
# If the bug is NOT there:
git bisect good

# Git keeps narrowing down until it finds the exact commit
# When done:
git bisect reset
```

In TFS, you would have to manually check out different changesets and test. Git bisect automates the binary search.

### 12.9 — Scenario: "I Want to Temporarily See the Code as It Was Last Week"

```bash
# Check out a specific commit (detached HEAD)
git checkout a1b2c3d

# Or check out by date
git checkout $(git rev-list -1 --before="2026-05-06" main)

# Look around, test things
# When done, go back to your branch
git switch main
```

The "detached HEAD" state means HEAD is pointing directly at a commit instead of a branch. You are in read-only mode. If you make commits in detached HEAD state, they are orphaned when you switch back to a branch (unless you create a new branch first).


## Part 13 — Pull Requests and Code Review

### 13.1 — What Is a Pull Request?

A pull request (PR) is not a Git feature — it is a feature of hosting platforms like GitHub, Azure DevOps, GitLab, and Bitbucket. It is a request to merge one branch into another, with an opportunity for code review, discussion, and automated checks before the merge happens.

**TFS equivalent:** In TFS, the closest thing is gated check-ins and code review requests. But TFS code reviews were bolted on; in Git workflows, pull requests are the central mechanism for getting code into the main branch.

### 13.2 — The Pull Request Workflow

1. You push your feature branch to the remote
2. You create a pull request (in the Azure DevOps or GitHub web UI, or in Visual Studio)
3. Reviewers look at the changes, leave comments, and request modifications
4. You make changes on your branch, commit, and push — the PR automatically updates
5. When approved, the PR is merged (creating a merge commit on main)
6. The feature branch is deleted

### 13.3 — Creating a PR in Azure DevOps

After pushing your branch, go to **Repos** → **Pull Requests** → **New Pull Request**. Select the source branch (your feature branch) and the target branch (main). Add reviewers. Write a description of what the PR does and why. Azure DevOps will show the diff and run any configured build pipelines.

You can also create a PR directly from Visual Studio 2026: after pushing, a notification appears saying "Create a Pull Request." Click it.

### 13.4 — Creating a PR in GitHub

After pushing your branch to GitHub, go to the repository page. GitHub usually shows a banner saying "[branch name] had recent pushes — Compare & pull request." Click the button. Fill in the title and description. Add reviewers. GitHub will show the diff and run any configured GitHub Actions.

### 13.5 — Merge Options in Pull Requests

**Azure DevOps offers:**
- **Merge (no fast-forward):** Standard merge commit. Preserves all branch commits.
- **Squash commit:** Combines all branch commits into one. Cleaner main history.
- **Rebase and fast-forward:** Replays commits on top of target. Linear history.
- **Semi-linear merge:** Rebase followed by merge commit.

**GitHub offers:**
- **Create a merge commit:** Same as Azure DevOps merge.
- **Squash and merge:** Same as squash.
- **Rebase and merge:** Same as rebase and fast-forward.

Choose based on your team's preference. For most teams new to Git, "Merge (no fast-forward)" is the safest option because it preserves the complete history and is easiest to understand.


## Part 14 — Branching Strategies

### 14.1 — Git Flow

Git Flow is a branching strategy popularized by Vincent Driessen in 2010. It uses multiple long-lived branches:
- `main` (or `master`) — the production code
- `develop` — the integration branch for the next release
- `feature/*` — feature branches created from develop
- `release/*` — release preparation branches
- `hotfix/*` — urgent fix branches from main

Git Flow is comprehensive but complex. It was designed before continuous integration was common. Many teams find it overly complicated.

### 14.2 — GitHub Flow

GitHub Flow is simpler:
- `main` is always deployable
- Create a feature branch from main
- Push to the remote, create a PR
- After review, merge into main
- Deploy from main

This is the most common strategy for small to medium teams.

### 14.3 — Trunk-Based Development

Trunk-based development minimizes branches:
- Everyone commits to `main` (the "trunk") directly or via very short-lived branches (hours, not days)
- Feature flags are used to hide incomplete work
- Continuous integration runs on every commit

This is used by Google, Facebook, and other companies with large engineering teams. The DORA State of DevOps research consistently shows that trunk-based development correlates with higher performance.

### 14.4 — Which Should You Use?

If you are migrating from TFS and your team is used to long-lived branches, start with GitHub Flow. It is simple, it maps well to pull requests, and it does not require any process changes beyond "use branches and PRs instead of checking in directly." You can evolve to trunk-based development later if your team matures their CI/CD practices.


## Part 15 — Migrating from TFVC to Git

### 15.1 — Should You Migrate History?

You have two choices: migrate your TFVC history to Git, or start fresh with a clean Git repository. Both are valid.

**Migrate history** if your team frequently references old changesets, needs to blame specific changes, or has compliance requirements around audit trails.

**Start fresh** if your TFS history is messy, huge, or not frequently referenced. You can always keep the TFS server running read-only for historical lookups.

### 15.2 — Using git-tfs for Migration

`git-tfs` is an open-source tool that creates a Git repository from a TFVC repository, preserving changeset history as Git commits.

```bash
# Clone a TFVC repository into a Git repository
git tfs clone https://tfs.yourcompany.com/tfs/DefaultCollection $/YourProject/Main

# Clone with limited history (last 1000 changesets)
git tfs quick-clone https://tfs.yourcompany.com/tfs/DefaultCollection $/YourProject/Main
```

Each TFVC changeset becomes a Git commit. Branch relationships are preserved where possible.

### 15.3 — Using Azure DevOps Import

Azure DevOps has a built-in import feature:
1. Go to **Repos** → **Import Repository**
2. Select "TFVC" as the source type
3. Choose the TFVC path to import
4. Specify how much history to import (up to 180 days, or all)

This is the easiest method if you are already on Azure DevOps.

### 15.4 — Post-Migration Checklist

After migrating:
1. Create a `.gitignore` file
2. Create a `.gitattributes` file for line endings
3. Verify the history looks correct with `git log`
4. Set up branch policies (require PRs, build validation)
5. Update your CI/CD pipeline
6. Update your team's documentation
7. Provide training (point them to this article)
8. Run parallel with TFS for a transition period if needed


## Part 16 — Common Pitfalls and How to Avoid Them

### 16.1 — Committing Large Binary Files

Git stores every version of every file. If you commit a 50 MB database backup file, every clone of the repository will download that 50 MB file — forever. Even if you delete it later, it is still in the history.

**Prevention:** Use `.gitignore` to exclude large files. Use **Git LFS** for binary files that must be versioned (design assets, etc.). Never commit database backups, log files, or build outputs.

### 16.2 — Committing Secrets

API keys, connection strings, passwords — once committed, they are in the history. Even if you remove them in the next commit, anyone with access to the repository can see the previous version.

**Prevention:** Use `.gitignore` to exclude files with secrets (like `appsettings.Development.json`). Use environment variables or a secret manager. If you accidentally commit a secret, rotate the secret immediately and use tools like `git-filter-repo` or BFG Repo-Cleaner to purge it from history.

### 16.3 — Not Pulling Before Pushing

If you commit locally and try to push, but someone else has pushed in the meantime, Git will reject your push. This is not an error — it is a safety mechanism. Git is telling you "the remote has changes you do not have."

**Solution:** Pull first, resolve any conflicts, then push.

```bash
git pull --rebase  # or just git pull
git push
```

### 16.4 — Fear of Branching

TFS developers often resist creating branches because in TFS, branches were heavy and merging was painful. In Git, branches are nearly free and merging is fast. The recommended practice is to create a branch for every piece of work, no matter how small.

### 16.5 — Messy Commit Messages

TFS check-in comments like "fixed stuff" or "changes" or "." are unfortunately common. Git makes this worse because you commit more often.

Write commit messages that explain WHY you made the change, not WHAT you changed (Git already knows what changed from the diff).

Bad: `fixed bug`
Good: `Fix null reference when user has no email address`

Bad: `updated files`
Good: `Add email validation to the registration form per JIRA-1234`

### 16.6 — Working Directly on main

Resist the temptation to commit directly to main. Always use feature branches and pull requests, even for small changes. This gives you:
- Code review
- A clean revert path (just revert the merge commit)
- A record of why the change was made (the PR description)
- Automated tests run before the merge


## Part 17 — Git Internals: How It Actually Works Under the Hood

### 17.1 — The .git Directory

When you initialize or clone a Git repository, Git creates a hidden `.git` directory in the project root. This directory IS the repository. Everything Git knows is in there. The rest of your project folder is just the "working tree" — a checkout of one version of the files.

Key contents of `.git`:

- `HEAD` — A file pointing to the current branch
- `refs/heads/` — One file per branch, each containing a commit hash
- `refs/tags/` — One file per tag
- `refs/remotes/origin/` — Tracking branches for the remote
- `objects/` — All the data: commits, trees, blobs (file contents)
- `config` — Repository-specific configuration
- `index` — The staging area

### 17.2 — Object Types

Git has four types of objects, all stored as compressed files in `.git/objects/`:

**Blob** — The contents of a file. Just the raw bytes, no filename.

**Tree** — A directory listing. Maps filenames to blob hashes (and sub-trees for subdirectories).

**Commit** — A pointer to a tree (the snapshot), plus metadata (parent, author, committer, message).

**Tag** — A pointer to a commit, plus metadata (tagger, date, message). Used for annotated tags.

Every object is identified by the SHA-1 hash of its content. This is content-addressable storage — if two files have the same content, they produce the same blob hash and are stored only once. This is why Git is space-efficient.

### 17.3 — How a Commit Is Made

When you run `git commit`, Git:

1. Takes the staging area (index) and creates tree objects for every directory
2. Creates a commit object that points to the root tree, the parent commit, the author/committer info, and the message
3. Updates the current branch ref to point to the new commit

This is all happening locally, on disk, in milliseconds. No network. No server. That is why Git commits are so fast compared to TFS check-ins.

## Part 20 — Best Practices for TFS Developers Adopting Git

### 20.1 — Start Simple

Do not try to learn every Git command at once. Start with these six and you can do 95% of your daily work:

1. `git pull` — Get the latest
2. `git switch -c branch-name` — Create a branch
3. `git add .` — Stage all changes
4. `git commit -m "message"` — Commit
5. `git push` — Push to remote
6. `git switch main` — Switch back to main

Everything else can wait until you need it.

### 20.2 — Use Visual Studio's Git Integration

You do not need to use the command line. Visual Studio 2026 has excellent Git integration. But learning the basic commands helps you understand what the buttons do, and the command line is essential for troubleshooting.

### 20.3 — Commit Early, Commit Often

In TFS, you typically accumulated changes over hours or days before checking in. In Git, commit frequently — every 15-30 minutes if you can. Each commit should represent a small, coherent change. If something goes wrong, you can revert to any commit.

### 20.4 — Write Good Commit Messages

Follow the conventional commit format:

```
<type>: <short description>

<longer description if needed>

Refs: <ticket number>
```

Types: `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`

Example:
```
fix: Prevent null reference when user email is empty

The login page crashed when a user had no email address in the database.
Added a null check before accessing the Email property.

Refs: JIRA-1234
```

### 20.5 — Never Force Push to Shared Branches

`git push --force` overwrites the remote branch with your local version. This destroys other people's commits. Never do this on `main` or any branch that others are working on. Force-push is only acceptable on personal feature branches that nobody else has checked out.

### 20.6 — Use Branch Protection Rules

In Azure DevOps: Go to **Repos** → **Branches** → click the `...` next to `main` → **Branch Policies**. Enable:
- Require a minimum number of reviewers
- Require linked work items
- Build validation (require a passing CI build)
- Comment resolution

In GitHub: Go to **Settings** → **Branches** → **Branch protection rules**. Enable:
- Require pull request reviews
- Require status checks to pass
- Require linear history (optional)

These rules prevent anyone from pushing directly to main, ensuring all changes go through a pull request.

### 20.7 — Clean Up After Yourself

After a PR is merged, delete the feature branch. Both Azure DevOps and GitHub can do this automatically. Locally, periodically clean up:

```bash
# Delete merged local branches
git branch --merged main | grep -v main | xargs git branch -d

# Prune remote tracking branches that no longer exist on the server
git fetch --prune
```


## Part 21 — Git in the Command Line: A Complete Reference

This section provides a complete reference of every Git command you are likely to need, organized by task. Think of it as a cheat sheet you can come back to.

### 21.1 — Setup and Configuration

```bash
# Configure your identity
git config --global user.name "Your Name"
git config --global user.email "you@example.com"

# See all configuration
git config --list
git config --list --show-origin

# Set the default editor
git config --global core.editor "code --wait"

# Set the default branch name
git config --global init.defaultBranch main

# Set up credential caching (Windows)
git config --global credential.helper manager
```

### 21.2 — Creating and Cloning Repositories

```bash
# Initialize a new Git repository in the current folder
git init

# Clone an existing repository
git clone https://github.com/user/repo.git

# Clone into a specific folder
git clone https://github.com/user/repo.git my-folder

# Clone a specific branch
git clone -b develop https://github.com/user/repo.git
```

### 21.3 — Day-to-Day Commands

```bash
# See the status of your working directory
git status

# Stage changes
git add file.cs           # Stage a specific file
git add .                 # Stage all changes
git add -p                # Stage changes interactively (hunk by hunk)

# Commit
git commit -m "message"
git commit -am "message"  # Stage tracked files and commit (skips new files)
git commit --amend        # Modify the last commit

# Unstage
git restore --staged file.cs

# Discard changes
git restore file.cs       # Discard changes to a tracked file
git clean -fd             # Remove untracked files and directories
```

### 21.4 — Branching

```bash
# List branches
git branch                # Local branches
git branch -r             # Remote branches
git branch -a             # All branches

# Create a branch
git branch feature-x      # Create without switching
git switch -c feature-x   # Create and switch

# Switch branches
git switch main
git checkout main         # Older syntax, still works

# Delete a branch
git branch -d feature-x   # Safe delete (refuses if unmerged)
git branch -D feature-x   # Force delete

# Rename a branch
git branch -m old-name new-name
```

### 21.5 — Remote Operations

```bash
# List remotes
git remote -v

# Add a remote
git remote add upstream https://github.com/other/repo.git

# Fetch (download without merging)
git fetch
git fetch --all           # Fetch from all remotes
git fetch --prune         # Remove stale remote tracking branches

# Pull (fetch + merge)
git pull
git pull --rebase         # Fetch + rebase instead of merge

# Push
git push
git push -u origin feature-x  # Push and set upstream tracking
git push --tags                # Push tags
git push origin --delete branch-name  # Delete a remote branch
```

### 21.6 — Merging

```bash
# Merge a branch into the current branch
git merge feature-x

# Merge without fast-forward (always create a merge commit)
git merge --no-ff feature-x

# Abort a merge in progress
git merge --abort

# Continue after resolving conflicts
git add .
git commit
```

### 21.7 — Rebasing

```bash
# Rebase current branch onto main
git rebase main

# Interactive rebase (edit, squash, reorder commits)
git rebase -i main

# Abort a rebase
git rebase --abort

# Continue after resolving conflicts
git rebase --continue
```

### 21.8 — Stashing

```bash
# Stash all changes
git stash

# Stash with a message
git stash push -m "WIP on login page"

# List stashes
git stash list

# Apply the most recent stash (keep it in the list)
git stash apply

# Apply and remove the most recent stash
git stash pop

# Apply a specific stash
git stash apply stash@{2}

# Drop a stash
git stash drop stash@{0}

# Clear all stashes
git stash clear
```

### 21.9 — Viewing History

```bash
# View commit log
git log
git log --oneline                    # Compact view
git log --oneline --graph --all      # Graph view of all branches
git log --stat                       # Show files changed in each commit
git log -p                           # Show the actual diff for each commit
git log --author="Name"              # Filter by author
git log --since="2026-01-01"         # Filter by date
git log -- path/to/file.cs           # History of a specific file

# Show a specific commit
git show a1b2c3d

# Show who changed each line of a file
git blame file.cs

# Show the diff between two commits/branches
git diff main..feature-x
git diff HEAD~3..HEAD                # Last 3 commits
git diff --staged                    # Diff of staged changes
```

### 21.10 — Undoing Things

```bash
# Revert a commit (create an undo commit)
git revert a1b2c3d

# Reset to a previous state
git reset --soft HEAD~1    # Keep changes staged
git reset HEAD~1           # Keep changes unstaged
git reset --hard HEAD~1    # Discard changes

# View the reflog (recovery tool)
git reflog

# Cherry-pick a commit from another branch
git cherry-pick a1b2c3d

# Restore a deleted file from a specific commit
git checkout a1b2c3d -- deleted-file.cs
```

### 21.11 — Tags

```bash
# Create a lightweight tag
git tag v1.0

# Create an annotated tag
git tag -a v1.0 -m "Release 1.0"

# List tags
git tag

# Push tags
git push --tags
git push origin v1.0

# Delete a tag
git tag -d v1.0                  # Local
git push origin --delete v1.0    # Remote

# Checkout a tag
git checkout v1.0
```


## Part 22 — Advanced Topics

### 22.1 — Submodules

Git submodules let you include one Git repository inside another as a subdirectory. This is useful for shared libraries or third-party dependencies.

```bash
# Add a submodule
git submodule add https://github.com/other/library.git libs/library

# Clone a repository that has submodules
git clone --recurse-submodules https://github.com/user/repo.git

# Update submodules
git submodule update --init --recursive
```

**TFS equivalent:** In TFS, you might have used workspace mappings to include code from multiple team projects. Submodules serve a similar purpose but are more explicitly managed.

### 22.2 — Git Hooks

Git hooks are scripts that run automatically at certain points in the Git workflow. They live in `.git/hooks/`.

Common hooks:
- `pre-commit` — Run before a commit is created. Use for linting, formatting.
- `commit-msg` — Run after writing the commit message. Use to enforce message format.
- `pre-push` — Run before pushing. Use to run tests.

Example `pre-commit` hook that runs dotnet format:

```bash
#!/bin/sh
dotnet format --verify-no-changes
if [ $? -ne 0 ]; then
    echo "Code formatting issues found. Please run 'dotnet format' first."
    exit 1
fi
```

For team-wide hooks, use a tool like Husky.NET that installs hooks automatically.

### 22.3 — Git LFS (Large File Storage)

Git LFS replaces large files in your repository with lightweight pointers, while storing the actual file content on a separate server. This prevents your repository from growing enormous due to binary files.

```bash
# Install Git LFS (one time)
git lfs install

# Track large file types
git lfs track "*.psd"
git lfs track "*.png"
git lfs track "*.mp4"

# The tracking rules are in .gitattributes
cat .gitattributes
# *.psd filter=lfs diff=lfs merge=lfs -text
```

GitHub and Azure DevOps both support Git LFS. There may be storage limits depending on your plan.

### 22.4 — Signing Commits with GPG

You can cryptographically sign your commits to prove they came from you:

```bash
# Configure GPG key
git config --global user.signingkey YOUR_KEY_ID
git config --global commit.gpgsign true

# Make a signed commit
git commit -S -m "This commit is verified"
```

GitHub and Azure DevOps show a "Verified" badge on signed commits.

### 22.5 — Sparse Checkout

If your repository is enormous and you only need a subset of the files, you can use sparse checkout:

```bash
git clone --filter=blob:none --sparse https://github.com/big/repo.git
cd repo
git sparse-checkout set src/my-project tests/my-project-tests
```

This downloads only the files you specified. Useful for mono-repos with many projects.


## Part 23 — Frequently Asked Questions from TFS Developers

### 23.1 — "Where Is the Server?"

In Git, "the server" is just another copy of the repository called a "remote." When you clone from GitHub, Git automatically sets up a remote called `origin` that points to the GitHub URL. You can see it with:

```bash
git remote -v
# origin  https://github.com/user/repo.git (fetch)
# origin  https://github.com/user/repo.git (push)
```

### 23.2 — "How Do I Know What Is on the Server?"

```bash
git fetch
git log origin/main --oneline
```

This fetches the latest from the server and shows the commit log of the server's `main` branch.

### 23.3 — "What If Two People Edit the Same File?"

Git handles this automatically as long as they edited different parts of the file. If they edited the same lines, you get a merge conflict that must be resolved manually — just like in TFS.

### 23.4 — "Can I Lock a File?"

Core Git does not support file locking. If you need it (for binary files), use Git LFS with file locking enabled, or use an administrative policy in Azure DevOps.

### 23.5 — "Is My Code Safe? What If My Laptop Dies?"

As long as you have pushed your commits to the remote, your code is on the server. If your laptop dies, clone the repository on a new machine and you have everything. Any commits you did NOT push are only on your local machine, so push regularly.

### 23.6 — "How Do I Go Back to a Previous Version?"

For a single file:
```bash
git checkout abc123 -- path/to/file.cs
```

For the entire repository:
```bash
git checkout abc123
```

For reverting a deployed change:
```bash
git revert abc123
git push
```

### 23.7 — "What Is the Equivalent of `tf undo /recursive`?"

```bash
# Discard all uncommitted changes
git checkout -- .
# or
git restore .

# Also remove untracked files
git clean -fd
```

### 23.8 — "Can I Use Git and TFS at the Same Time?"

Yes. Azure DevOps supports both TFVC and Git repositories in the same project. You can migrate one repository at a time. Some teams run TFVC and Git side by side for months during a transition.
