## The Bundle Size Problem

There is a story — possibly apocryphal, definitely instructive — about a team that launched a beautiful React application, got the first user complaint within hours, and traced the problem to a 4.2 MB gzipped JavaScript bundle. The application worked perfectly on the development machine (connected to a fast internal network with a warm CDN cache). But the first user to open it was on a rural cellular connection, and they waited 34 seconds for anything to appear on screen. By that time, they had already navigated away.

Bundle size matters. Initial load time matters. Not just for consumer applications where you are competing for attention spans measured in seconds, but for any application where your users are not on a fast local network. And even on fast networks, large bundles mean more JavaScript parsing time, more JIT compilation time, and a worse experience on lower-end devices.

Both React applications and Blazor WebAssembly applications have bundle size concerns. They have different characters — React's problems are JavaScript-specific (unused code in npm packages, duplicate dependencies, unoptimized images bundled as imports), while Blazor's problems are .NET-specific (the .NET runtime itself, unreferenced library code) — but the principles of solving them are similar: figure out what you actually need, include only that, and compress everything as much as possible.

This article is about the tools that make this possible on both sides.

## Part 1: The JavaScript Build Tool Landscape in 2026

The JavaScript build tool landscape has undergone remarkable changes over the past five years. Understanding where we are requires a brief historical survey.

### The Webpack Era (2012–2022)

Webpack (released 2012, first stable version 2014) was the dominant JavaScript bundler for about a decade. It introduced a graph-based model: every `import` statement in your code is an edge in a dependency graph, and Webpack traverses this graph starting from your entry point, bundling everything into one or more output files.

Webpack is powerful and highly configurable, but its configuration is notoriously complex (the `webpack.config.js` file can grow to hundreds of lines for a real project), and its build speed is slow — especially for large codebases. For a mid-size React application, development builds could take 30–60 seconds, and production builds could take several minutes.

### Vite and the Native ESM Era (2020–present)

Vite (French for "fast," created by Evan You, creator of Vue.js, released in 2020) took a completely different approach for development mode: instead of bundling all your source code before serving it, Vite serves your source files directly as native ES modules and lets the browser handle the imports. The browser fetches each module individually over HTTP as it is needed.

This eliminates the development build step entirely. When you start `vite dev`, it starts almost instantly regardless of how large your project is. Hot Module Replacement (HMR) — updating a module in the browser when you save a file — takes tens of milliseconds rather than several seconds, because Vite only needs to re-transform the changed module, not re-bundle the entire application.

For production builds, Vite uses Rollup (another bundler) under the hood to produce optimized, bundled output. This two-phase approach (native ESM for dev, Rollup for production) is Vite's defining characteristic.

As of April 2026, Vite's latest version is 8.0.8. Vite 6 (released November 2024) introduced the Environment API, targeting framework authors who need to support multiple runtime environments (browser, Node.js, edge, Cloudflare Workers). Vite 7 (released mid-2025) began integrating Rolldown as an optional bundler. Vite 8 (released early 2026) made Rolldown the default production bundler for new projects, replacing Rollup.

### esbuild — The Speed Benchmark

esbuild (created by Evan Wallace at Figma, released 2020) is a JavaScript/TypeScript bundler written in Go. It is extremely fast — benchmarks consistently show esbuild building large projects in milliseconds that Webpack takes minutes to build. Vite uses esbuild internally for transforming TypeScript, JSX, and for dependency pre-bundling.

esbuild's speed comes from its architecture: it is written in Go (which is fast), uses parallelism aggressively (all CPU cores), avoids string allocation where possible, and skips features that slower bundlers support (like complicated plugin ecosystems). For most bundling tasks, esbuild is the fastest option available.

### Rolldown — The New Contender (2024–2026)

Rolldown is a new bundler built by VoidZero (the company created around Evan You and the Vite ecosystem). It is written in Rust, aims to be compatible with the Rollup plugin ecosystem, and is dramatically faster than Rollup. Rolldown was integrated into Vite 7 as an experimental option and became the default in Vite 8.

In Vite 8, Rolldown handles all production builds. The developer experience remains the same (native ESM dev server), but production builds are now significantly faster — often 3–10x faster than Rollup-based Vite builds, comparable to esbuild but with better Rollup plugin compatibility.

## Part 2: Setting Up a React Project With Vite for Static Deployment

Let us build a complete, production-ready React project setup targeting GitHub Pages — a purely static deployment with no server:

```bash
npm create vite@latest my-react-app -- --template react-ts
cd my-react-app
npm install
```

The generated `vite.config.ts`:

```typescript
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';

// https://vite.dev/config/
export default defineConfig({
    plugins: [react()],
    
    // For GitHub Pages deployment at https://username.github.io/repo-name/
    // The base path must match the repository name
    base: '/my-react-app/',
    
    build: {
        // Output directory (GitHub Actions deploys from here)
        outDir: 'dist',
        
        // Generate source maps for debugging production issues
        // Consider 'hidden' to avoid exposing source to users
        sourcemap: true,
        
        // Rolldown (Vite 8 default) configuration
        rollupOptions: {
            output: {
                // Manual chunk splitting to improve caching
                manualChunks: {
                    // Vendor chunk: React itself (changes rarely → long-lived cache)
                    vendor: ['react', 'react-dom'],
                    
                    // Router chunk: React Router (separate chunk for caching)
                    router: ['react-router-dom'],
                }
            }
        },
        
        // Warn when any chunk exceeds this size (in KB)
        chunkSizeWarningLimit: 500,
        
        // Enable CSS code splitting
        cssCodeSplit: true,
    },
    
    // Development server configuration
    server: {
        port: 5173,
        
        // Proxy API requests during development to avoid CORS issues
        proxy: {
            '/api': {
                target: 'https://localhost:7001',
                changeOrigin: true,
                secure: false,
            }
        }
    }
});
```

### Tree-Shaking in JavaScript

Tree-shaking is the process of removing unused code from your bundle. Modern bundlers (Vite with Rolldown, Webpack 5, esbuild) all support tree-shaking for ES module code.

Tree-shaking works by analyzing the `import` and `export` statements in your code. If you import a named export but never use it, the bundler can determine that the export (and all code that only exists to support that export) is dead code and exclude it from the bundle.

```javascript
// math.js
export function add(a, b) { return a + b; }
export function subtract(a, b) { return a - b; }
export function multiply(a, b) { return a * b; }
export function complexOperation(a, b) {
    // This uses subtract, so if it's included, subtract must be included too
    return subtract(a * b, a + b);
}

// main.js
import { add } from './math.js';
// Only 'add' is used. Tree-shaking will eliminate:
// - 'subtract' (only used by complexOperation, which is not used)
// - 'multiply' (not imported)
// - 'complexOperation' (not imported)
```

For tree-shaking to work correctly, the code must use ES module syntax (`import`/`export`) rather than CommonJS (`require`/`module.exports`). CommonJS modules cannot be statically analyzed — the `require()` call can be conditional or dynamic — so tree-shaking cannot safely remove any CommonJS exports.

This is why you will occasionally encounter situations where importing a library that uses CommonJS pulls in its entire codebase rather than just the functions you use.

### Analyzing Your Bundle

When bundle size matters, use the Rollup/Rolldown visualizer plugin to see what is taking up space:

```bash
npm install --save-dev rollup-plugin-visualizer
```

```typescript
// vite.config.ts
import { visualizer } from 'rollup-plugin-visualizer';

export default defineConfig({
    plugins: [
        react(),
        visualizer({
            open: true,          // Open the report in browser after build
            gzipSize: true,      // Show gzipped sizes (what users actually download)
            brotliSize: true,    // Show Brotli-compressed sizes
            filename: 'dist/bundle-stats.html',
        })
    ],
    // ...
});
```

After running `npm run build`, a visual treemap opens in your browser showing the size of every module in your bundle. This is invaluable for identifying unexpectedly large dependencies.

### Common Bundle Size Culprits and Their Fixes

**Problem**: Importing an entire library when you only need a few functions.

```javascript
// BAD — imports the entire lodash library (70+ KB minified)
import _ from 'lodash';
const result = _.uniq(array);

// GOOD — imports only the specific function (a few KB)
import uniq from 'lodash/uniq';

// BETTER — use native JavaScript for simple operations
const result = [...new Set(array)];
```

**Problem**: Moment.js locale files (adds hundreds of KB for all locale data).

```javascript
// moment.js bundles ALL locale data by default
import moment from 'moment'; // ~290 KB minified

// Solution 1: Use date-fns instead (tree-shakeable)
import { format, parseISO } from 'date-fns'; // Only the functions you use

// Solution 2: Use the browser's built-in Intl API (zero bundle size)
const formatted = new Intl.DateTimeFormat('en-US', { 
    year: 'numeric', month: 'long', day: 'numeric' 
}).format(date);
```

**Problem**: Large icon libraries (importing one icon pulls in all icons).

```javascript
// BAD — pulls in thousands of icons if the library isn't tree-shakeable
import { FaUser, FaHome } from 'react-icons/fa';

// GOOD — import from the specific icon pack subpath
import FaUser from 'react-icons/fa/FaUser';
import FaHome from 'react-icons/fa/FaHome';

// BEST — use Lucide React (well tree-shakeable, MIT license)
import { User, Home } from 'lucide-react';
```

## Part 3: Vite vs. `dotnet watch` — Development Experience Comparison

One of the practical differences between React and Blazor development is the hot reload experience. Let us compare.

### Vite's Hot Module Replacement

Vite's development server uses native ES modules and implements HMR (Hot Module Replacement) — when you save a file, Vite sends a WebSocket message to the browser with the updated module. The browser executes the updated module and (for React) uses React's component reconciliation to update the running application state without a full page reload.

For React, this means:
- Edit a component → save → the component updates in the browser in typically 50–200ms
- Application state is preserved (the counter value, form inputs, etc. survive the update)
- Only the changed module is updated — other modules are not re-executed

This is genuinely excellent. For CSS changes in particular, Vite's HMR injects the new CSS directly without any JavaScript re-execution, giving instant visual feedback.

### `dotnet watch` and Blazor Hot Reload

.NET's equivalent is `dotnet watch` with Blazor's hot reload support. When you save a `.razor` file, the .NET runtime patches the running application with the new component code. Since .NET 6, hot reload supports a wide range of changes without requiring an application restart.

Blazor hot reload for WebAssembly works similarly but has additional considerations: the WebAssembly bytecode runs in the browser, so patches must be transmitted to the browser and applied to the running WASM instance. This is slightly slower than Vite's pure JavaScript HMR — typically 500ms to a few seconds — but it works and preserves the application state for supported change types.

Changes that require a full restart in Blazor (as of .NET 10):
- Adding or removing method parameters
- Adding a new virtual method
- Adding or removing an interface from a type
- Changing a type from class to struct or vice versa

Changes that are supported (hot reload):
- Modifying method bodies
- Adding or modifying lambda expressions
- Adding new methods
- Modifying event handlers
- Modifying Razor template content

In practice, most routine development changes (tweaking component layout, modifying event handler logic, updating CSS) hot reload successfully in Blazor.

## Part 4: Serving Static React on GitHub Pages — The Complete Setup

GitHub Pages serves static files from a repository. For a React SPA, there is a well-known problem: GitHub Pages does not support client-side routing out of the box. If a user navigates to `https://yourname.github.io/app/dashboard` and refreshes, GitHub Pages serves a 404 because there is no actual `dashboard` directory or file — the routing is client-side only.

The solution is the 404.html redirect trick:

```html
<!-- public/404.html -->
<!DOCTYPE html>
<html>
<head>
    <meta charset="utf-8" />
    <script>
        // SPA redirect trick for GitHub Pages
        // Stores the requested path in sessionStorage and redirects to index.html
        var path = location.pathname + location.search + location.hash;
        // Replace forward slashes to encode the path safely in the redirect URL
        sessionStorage.setItem('redirect', path);
        location.replace(location.origin + location.pathname.split('/').slice(0, 2).join('/') + '/?p=1');
    </script>
</head>
</html>
```

```html
<!-- index.html — add this script BEFORE the main app script -->
<script>
    (function() {
        var redirect = sessionStorage.getItem('redirect');
        sessionStorage.removeItem('redirect');
        if (redirect && redirect !== location.href) {
            history.replaceState(null, null, redirect);
        }
    })();
</script>
```

This approach:
1. When GitHub Pages serves a 404 for a deep link, it serves `404.html` instead.
2. `404.html` saves the requested URL to `sessionStorage` and redirects to the app root (`index.html`).
3. `index.html` reads the saved URL from `sessionStorage` and uses `history.replaceState` to restore the correct URL.
4. React Router reads the URL and renders the correct route.

### GitHub Actions Workflow for React

```yaml
# .github/workflows/deploy.yml
name: Deploy React App

on:
  push:
    branches: [main]

permissions:
  contents: read
  pages: write
  id-token: write

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    
    steps:
      - uses: actions/checkout@v4
      
      - uses: actions/setup-node@v4
        with:
          node-version: '22'
          cache: 'npm'
      
      - run: npm ci
      
      - run: npm run build
        env:
          NODE_ENV: production
      
      # GitHub Pages requires a .nojekyll file to serve files with underscores
      - run: touch dist/.nojekyll
      
      - uses: actions/configure-pages@v4
      
      - uses: actions/upload-pages-artifact@v3
        with:
          path: dist
      
      - uses: actions/deploy-pages@v4
```

## Part 5: The Blazor WebAssembly Publish Pipeline

Blazor WebAssembly's build pipeline is `dotnet publish`. Understanding what it does helps you understand what levers you have to pull for performance.

```bash
dotnet publish -c Release -p:RunAOTCompilation=true
```

### What `dotnet publish` Does for Blazor WASM

**Step 1: C# Compilation**. The Razor compiler processes `.razor` files and generates C# code. The C# compiler compiles your code to IL (Intermediate Language) — the same bytecode format used by all .NET applications.

**Step 2: Linking (IL Trimming)**. The .NET linker analyzes the entire application — your code, the .NET libraries it references, and the .NET runtime libraries — and removes any code that is never called. This is dead-code elimination, and it is aggressive. For a typical Blazor application, the linker can remove 60–80% of the .NET runtime code that you never use.

The trimmer is configured in your `.csproj` file:

```xml
<PropertyGroup>
    <!-- Enable aggressive trimming -->
    <PublishTrimmed>true</PublishTrimmed>
    
    <!-- AOT compilation — compile IL to WebAssembly bytecode at publish time -->
    <RunAOTCompilation>true</RunAOTCompilation>
    
    <!-- Optimize for size rather than speed (good for first-load performance) -->
    <OptimizationPreference>Size</OptimizationPreference>
</PropertyGroup>
```

**Step 3: AOT Compilation (if enabled)**. If `RunAOTCompilation=true`, the LLVM compiler (via the `wasm-tools` workload) compiles your IL code to native WebAssembly bytecode. This step takes significantly longer (minutes for a large application) but produces code that executes faster in the browser because the browser doesn't need to JIT-compile anything at runtime.

**Step 4: Compression**. The published files are Brotli and Gzip compressed. Blazor's bundled files have excellent compression ratios because they are either:
- WebAssembly bytecode (which compresses very well)
- .NET runtime binaries (ditto)
- Your application DLLs

**Step 5: Output**. The result is a directory containing:
- `index.html` — the entry point
- `_framework/` — the .NET runtime, your compiled application, and metadata
- `css/` — compiled and bundled CSS
- `images/` and other static assets

### Typical File Sizes

For a minimal Blazor WASM application with AOT enabled and aggressive trimming:
- Uncompressed: 10–15 MB (mostly WebAssembly binaries)
- Brotli-compressed: 4–7 MB
- On subsequent visits: 0 bytes (fully cached)

Compare to a minimal React application:
- Uncompressed: 200–400 KB
- Gzip-compressed: 60–120 KB

The Blazor payload is larger. There is no getting around this. The .NET runtime — even heavily trimmed — is bigger than React's runtime. This is the fundamental trade-off: you pay a higher initial download cost to get the ability to write C# in the browser.

For most business applications deployed internally (where users load the application once a day and the initial download is cached after the first visit), this trade-off is acceptable. For consumer applications where first-load performance is critical for SEO and user retention, this trade-off requires careful consideration.

### Lazy Loading Assemblies

Blazor WASM supports lazy loading assemblies — you can configure specific assemblies to be downloaded only when the user navigates to a part of the application that needs them:

```csharp
// Program.cs
builder.Services.AddLazyAssemblyLoader();
```

```csharp
// In a component that navigates to the reports section
@inject LazyAssemblyLoader AssemblyLoader

@code {
    protected override async Task OnInitializedAsync()
    {
        // Only load the reports assembly when the user navigates here
        await AssemblyLoader.LoadAssembliesAsync(new[]
        {
            "MyApp.Reports.dll",
            "SomeHeavyChartingLibrary.dll"
        });
    }
}
```

```xml
<!-- In .csproj, mark assemblies for lazy loading -->
<ItemGroup>
    <BlazorWebAssemblyLazyLoad Include="MyApp.Reports.dll" />
    <BlazorWebAssemblyLazyLoad Include="SomeHeavyChartingLibrary.dll" />
</ItemGroup>
```

Lazy loading is the Blazor equivalent of React's dynamic import for code splitting:

```javascript
// React code splitting with React.lazy
const ReportsPage = React.lazy(() => import('./pages/ReportsPage'));

function App() {
    return (
        <Suspense fallback={<p>Loading reports...</p>}>
            <Routes>
                <Route path="/reports" element={<ReportsPage />} />
            </Routes>
        </Suspense>
    );
}
```

Both approaches split the application code into smaller chunks that are downloaded only when needed.

## Part 6: Serving Blazor WASM on GitHub Pages

Blazor WASM has the same client-side routing problem as React for GitHub Pages deployment. The `404.html` trick works for Blazor too, but Blazor has its own built-in solution: when you publish, a `404.html` that redirects to `index.html` is automatically generated (or you can configure it).

The bigger challenge with Blazor on GitHub Pages is the `.wasm` file extension. GitHub Pages (which runs on Jekyll) does not serve `.wasm` files with the correct MIME type (`application/wasm`) by default. The fix is a `.nojekyll` file in the repository root, which disables Jekyll processing and causes GitHub Pages to serve all files directly:

```bash
# Create the .nojekyll file
touch .nojekyll
git add .nojekyll
git commit -m "Disable Jekyll for Blazor WASM deployment"
```

The Blazor WASM files also use double extension for the compressed variants (`.wasm.br` for Brotli-compressed WebAssembly). Blazor's JavaScript loader automatically detects whether the browser supports Brotli and requests the compressed variant if so, falling back to the uncompressed version.

### GitHub Actions Workflow for Blazor WASM

```yaml
# .github/workflows/deploy.yml
name: Deploy Blazor WebAssembly

on:
  push:
    branches: [main]

permissions:
  contents: read
  pages: write
  id-token: write

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    
    steps:
      - uses: actions/checkout@v4
      
      - uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '10.0.x'
      
      # Install the WASM tools workload for AOT compilation
      - run: dotnet workload install wasm-tools
      
      # Build and publish with AOT
      - run: dotnet publish src/MyApp.Web/MyApp.Web.csproj 
          -c Release 
          -p:RunAOTCompilation=true
          -p:PublishTrimmed=true
          --output publish
      
      # The published output is in publish/wwwroot
      - uses: actions/configure-pages@v4
      
      - uses: actions/upload-pages-artifact@v3
        with:
          path: publish/wwwroot
      
      - uses: actions/deploy-pages@v4
```

## Part 7: Performance Comparison — Build Pipeline Outcomes

Let us put actual numbers on the comparison. These are approximate figures based on typical applications, not benchmarks from a specific controlled test:

| Metric | React (Vite 8 + React 19) | Blazor WASM (.NET 10, AOT) |
|---|---|---|
| Development start time | < 1 second (Vite native ESM) | 3–10 seconds (first build) |
| Development HMR time | 50–200ms | 500ms–3 seconds |
| Production build time (cold) | 10–60 seconds | 3–10 minutes (AOT) |
| Production build time (CI cache) | 15–30 seconds | 1–3 minutes |
| Initial download (uncompressed) | 200–800 KB | 10–20 MB |
| Initial download (Brotli-compressed) | 60–250 KB | 3–8 MB |
| Cached subsequent loads | Near 0 | Near 0 |
| Tree-shaking effectiveness | Excellent (JS/TS ESM) | Excellent (.NET trimming) |
| Code splitting support | Yes (dynamic import, React.lazy) | Yes (lazy assembly loading) |
| Static hosting compatible | Yes | Yes |

The development experience strongly favors React/Vite for iteration speed. The AOT compilation step in Blazor is the biggest practical friction point for development workflows — if you are doing frequent full republish cycles (for example, testing the exact production behavior), the wait time adds up.

However, for day-to-day feature development where you use `dotnet watch` with hot reload rather than `dotnet publish`, the development experience is much more comparable.

## Conclusion: The Build Pipeline Reflects the Runtime

The differences in build pipelines between React and Blazor are a reflection of their runtime differences. React is a JavaScript library — JavaScript is natively understood by browsers, so bundling it for production is relatively fast, and the output is compact. Blazor is a .NET runtime running in WebAssembly — WebAssembly requires compilation of your C# code and the .NET runtime itself, which takes longer and produces more output.

Neither pipeline is wrong. They are optimized for different goals. React's pipeline is optimized for fast iteration and small initial download. Blazor's pipeline is optimized for enabling a rich, statically typed, full-stack programming model that happens to run in the browser.

The right choice for your team depends on which tradeoff is more painful: the larger initial download of Blazor, or the JavaScript mental model and tooling complexity of React.

In Part 7 — the conclusion of this series — we stop analyzing and start deciding. We give you a concrete framework for making the actual choice: React or Blazor, for your team, for your project, right now.
