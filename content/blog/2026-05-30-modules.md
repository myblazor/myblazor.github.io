## Introduction: The Build Step We Never Needed

Some years ago — this was 2018, probably — the team at this magazine sat down to modernise a legacy ASP.NET Web Forms project. We had about ninety thousand lines of inherited JavaScript, none of it modular, all of it in a folder called `Scripts/` that was served by bundle configs defined in a twenty-thousand-line `BundleConfig.cs` whose last substantive edit was in 2014. Everybody on the team agreed the status quo was untenable. We all wanted "modules." We all wanted to be able to write `import foo from "./foo.js"` and have it work.

So we sat down and picked a bundler.

Three hours later we were still in the meeting. Webpack, Rollup, Parcel, or the new thing called esbuild? TypeScript or Flow? Babel or swc? ts-loader or babel-loader or the native TypeScript transpilation? CommonJS output or ES Modules? Which target — ES5, ES2015, ES2020? Source maps or no source maps? Tree shaking — does it work with our CommonJS legacy code? What about polyfills for the twelve customers still on Internet Explorer 11? Did we need a separate build for the legacy bundle and the modern bundle? Who was going to maintain this configuration?

Six weeks later we had shipped the first "modern" bundle. It used Webpack 4, Babel with thirty-seven loader plugins, a 600-line `webpack.config.js`, a 400-line `tsconfig.json`, a 200-line `.babelrc`, and a collection of postinstall scripts whose combined output nobody on the team fully understood. The bundle was 1.4 megabytes. Loading time on mid-range mobile was about four seconds on 4G. We shipped it anyway, because it was better than what we had before.

Over the next five years we did this three more times. Webpack 4 to Webpack 5. Webpack 5 to Vite. Vite to a frozen snapshot of Vite because we could not keep up with the churn. We spent, if we are being honest, more developer-weeks on tooling in those five years than we did on features.

In 2018, when we made that first bundler decision, our options were: set up a build step, or do not have modules at all. We picked the build step. It was the right call. There was no other option.

**In 2026, there is another option.**

Today, in every evergreen browser, a single HTML tag — `<script type="module">` — gives you the entirety of the module system the bundler was for. You can write `import` and `export` and they just work. The browser fetches, parses, evaluates, and caches modules natively, in a way that is indistinguishable from what Webpack shipped you at runtime. And two additional primitives — the **import map** and the **dynamic `import()` expression** — cover the two remaining cases (dependency aliasing and code splitting) that used to justify a build step.

If you are a seasoned web developer, you have heard this claim before. It has been partially true since 2018 and fully true since about 2023. The question is not whether it is true; the question is whether you have internalised what it means for the architecture of a new project. Day 7 is our attempt to make the case precisely, comprehensively, and with code you can run.

This is Part 7 of 15 in our no-build web series. Today marks a hinge. The first six articles covered HTML and CSS — the surface of a web application. Today we start on the architecture, and every remaining article builds on the conclusions here. If you read only one post in this series, we humbly submit it should be this one.

## Part 1: The Short History Of "Modules"

Quick history, because the context matters.

**1995–2005.** JavaScript had no module system. A script was a flat file loaded with `<script src="foo.js">`. Every variable was global. Collision was managed by convention — prefix everything with your library name (`jQuery`, `YUI`, `Ext`) — and by the IIFE pattern, wrapping your code in `(function() { ... })()` to create a private scope.

**2009.** Node.js shipped with CommonJS, a module system designed for server-side use. `require("fs")` and `module.exports = ...`. Synchronous. File-based. Perfect for a server with a fast filesystem. Wrong for a browser, because synchronous loading blocks the UI.

**2011.** AMD (Asynchronous Module Definition) shipped with RequireJS. Browser-compatible, asynchronous. Ugly syntax. Not adopted widely.

**2014.** Browserify ships. It bundles CommonJS modules into a single file that runs in the browser. First browser-compatible taste of Node-style modules. A wave of tooling follows: Webpack, Rollup, Parcel, esbuild, each building on the same basic idea of "consume module source code, produce a browser-runnable bundle."

**2015.** [ES6 / ES2015](https://tc39.es/ecma262/) is finalised. It includes `import` and `export` as first-class language syntax. The spec is there; no browser implements it yet.

**2017–2018.** Browsers start shipping ES module support. Safari in 2017, Chrome and Firefox in 2018. `<script type="module">` becomes available. The implementation is honest — the browser actually fetches and parses modules at runtime.

**2019.** Node.js adds stable ES module support. Both major runtimes now natively understand the syntax.

**2021.** Chrome 89 ships **import maps**. You can now resolve bare specifiers (`import { something } from "lodash"`) without a bundler. Firefox and Safari follow in 2022 and 2023.

**2023.** [Import maps reach Baseline Newly Available in March](https://web.dev/blog/import-maps-in-all-modern-browsers). Every evergreen browser handles them.

**2025.** Import maps [cross into Baseline Widely Available](https://web-platform-dx.github.io/web-features-explorer/features/import-maps/).

**2026.** You are reading this article.

The significance: **the things we added build tools to solve have one by one become features the browser ships natively.** The JavaScript community assembled a tower of tooling because, for fifteen years, there was no alternative. Now there is. The tower is still useful for some applications, but for many others — probably most small-to-medium projects — it is superfluous.

## Part 2: The Mental Model — A Module, Plainly

Let us start from zero and build the mental model.

A **module** is a file containing JavaScript. What makes it a module, rather than a script, is:

- It has `export` statements that expose values.
- It can have `import` statements that reference other modules.
- Its top-level scope is private — nothing leaks to `window` or to the global scope.
- `this` at the top level is `undefined`, not `globalThis`.
- It runs in strict mode automatically.
- It executes exactly once per URL, regardless of how many times it is imported.

Here is the smallest possible useful module:

```javascript
// math.js
export function add(a, b) {
  return a + b;
}

export function multiply(a, b) {
  return a * b;
}
```

And the smallest possible consumer:

```javascript
// app.js
import { add, multiply } from "./math.js";

console.log(add(2, 3));         // 5
console.log(multiply(4, 5));    // 20
```

To run this in a browser, you need one HTML file:

```html
<!doctype html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <title>Hello modules</title>
</head>
<body>
  <script type="module" src="./app.js"></script>
</body>
</html>
```

Three files. Zero configuration. Double-click (or serve with a local server — see Part 7 below), and in the console you see `5` and `20`.

That is the whole thing. There is no build step. There is no bundler. There is no `package.json`. There is no `node_modules`. Every byte of that setup is yours. You understand every character. This is a working application.

Expand it to ten files, a hundred files, a thousand files — the mental model does not change. The browser fetches, parses, and evaluates each module exactly once. Modules can import from modules, which can import from more modules, and the browser handles the graph transparently.

## Part 3: `export` And `import` — Every Variant

The syntax has a few forms, each with a purpose.

### Named exports

```javascript
// utils.js
export function log(msg) { console.log(msg); }
export const PI = 3.14159;
export class Widget { /* ... */ }

// alternative: export declarations separately
function double(n) { return n * 2; }
const GOLDEN_RATIO = 1.618;
export { double, GOLDEN_RATIO };
```

### Named imports

```javascript
import { log, PI } from "./utils.js";
```

### Renaming on import

```javascript
import { log as debugLog } from "./utils.js";
debugLog("hi");
```

### Renaming on export

```javascript
function internal() { /* ... */ }
export { internal as publicApi };
```

### Default export

```javascript
// button.js
export default class Button {
  /* ... */
}
```

### Default import

```javascript
import Button from "./button.js";
```

The default import does not need curly braces. You can name it whatever you want. This is useful for "the module's primary thing." It is also, in our opinion, somewhat overused — named exports are usually clearer.

### Combining defaults and named

```javascript
// button.js
export default class Button { /* ... */ }
export const BUTTON_VARIANTS = ["primary", "secondary"];

// consumer.js
import Button, { BUTTON_VARIANTS } from "./button.js";
```

### Namespace import

```javascript
import * as Utils from "./utils.js";

Utils.log("hi");
console.log(Utils.PI);
```

Everything the module exports is available on the `Utils` object. Useful when you want to import many things from a module without listing them, or when you want the module to feel like a "library."

### Re-export

```javascript
// barrel.js
export { log, PI } from "./utils.js";
export { default as Button } from "./button.js";
export * from "./widgets.js";
```

A "barrel" file that re-exports from other modules. Useful for making a folder feel like a single package. `consumer.js` can `import { log, PI, Button } from "./barrel.js"` without caring about the internal file structure.

### Side-effect import

```javascript
import "./init.js";
```

Execute the module for its side effects but import nothing. Rarely ideal — prefer explicit exports — but useful for setup code.

### Dynamic import

```javascript
const { log } = await import("./utils.js");
log("dynamically imported");
```

A function-call form that returns a promise. The critical difference: this is evaluated at runtime, not at module load time. It is the key to code splitting, which we cover in Part 6.

### Import attributes

A newer feature — [Baseline Newly Available in 2025](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Statements/import/with) — lets you import JSON directly:

```javascript
import config from "./config.json" with { type: "json" };
console.log(config.apiUrl);
```

Without the `with { type: "json" }`, the browser would try to parse the file as JavaScript and fail. Import attributes tell it how to handle the non-JS content. CSS, HTML, and other content types will follow.

## Part 4: `<script type="module">` — The Tag That Unlocks Everything

A script with `type="module"` has different rules from a classic script. The differences are subtle but matter.

### Module scripts are deferred by default

Classic scripts block parsing unless you add `defer` or `async`. Module scripts behave as if they always had `defer` — they are fetched in parallel with HTML parsing, then executed in order after the document is parsed. This means you can put `<script type="module">` in the `<head>` without worrying about blocking.

```html
<head>
  <script type="module" src="./app.js"></script>
</head>
```

That is fine. It does not block the page. The module fetches in parallel, executes after parse, and by the time it runs, `document.body` exists.

### Module scripts are strict mode

No `var` collisions, no assignment to globals, no this-binding surprises. Just like if you had written `"use strict"` at the top.

### `this` at the top level is `undefined`

In a classic script, `this === window`. In a module, `this === undefined`. Code that accidentally relies on the former breaks. Code that uses `this` intentionally at module top level was probably broken anyway.

### Modules run exactly once per URL

Import the same module fifteen times from fifteen different places; its code executes once. The first import initialises it, subsequent imports get the cached exports.

### `import` statements only work in modules

```javascript
// This is a SyntaxError in a classic <script>
import { x } from "./foo.js";
```

You cannot `import` from a classic script. That is the primary reason to use `type="module"` in the first place.

### CORS requirements

Modules are always fetched with CORS. A `<script type="module" src="https://cdn.example.com/foo.js">` requires the CDN to send `Access-Control-Allow-Origin` — otherwise the browser refuses to load it. This is the opposite of classic scripts, which had no CORS requirement. Most reputable CDNs send the right headers; still, it is a gotcha.

### Module workers

```javascript
new Worker("./worker.js", { type: "module" });
```

Web workers can use the same module syntax. Worth knowing if you do heavy computation off the main thread.

### Inline modules

```html
<script type="module">
  import { log } from "./utils.js";
  log("inline modules work too");
</script>
```

Inline module scripts work like external ones. They cannot be cached, so reserve them for small bootstrap code.

### `nomodule` fallback

If you absolutely must support a legacy browser that does not understand modules, `<script nomodule>` is the escape hatch. That script loads only in browsers that do not understand `type="module"`. In 2026, if you are targeting evergreen browsers, you do not need this. But it is a compatible way to ship a fallback bundle for long-tail users.

## Part 5: Import Maps — Dependency Resolution Without A Package Manager

This is where the rubber hits the road. In a bundler-based project, you write:

```javascript
import { Temporal } from "temporal-polyfill";
```

The bundler knows to resolve `"temporal-polyfill"` to whatever file `node_modules/temporal-polyfill` contains. Without a bundler, the browser has no idea what `"temporal-polyfill"` means — it is not a relative path.

**Import maps fix this.** You declare a mapping in your HTML:

```html
<script type="importmap">
{
  "imports": {
    "temporal-polyfill": "https://esm.sh/temporal-polyfill@0.2.5"
  }
}
</script>

<script type="module">
  import { Temporal } from "temporal-polyfill";
  console.log(Temporal.Now.plainDateISO().toString());
</script>
```

When the browser encounters `import from "temporal-polyfill"`, it looks in the import map, finds the mapping, and fetches the URL. That is it.

### CDN options

Several services serve npm packages as ES modules directly, letting you use them without installing anything:

- **esm.sh** (<https://esm.sh/>) — the current community favourite. Caches aggressively, supports pinned versions, handles CommonJS-to-ESM transformation.
- **unpkg** (<https://unpkg.com/>) — older; less reliable for CommonJS-to-ESM.
- **jsdelivr** (<https://www.jsdelivr.com/>) — also supports npm, reliable, offers statistics.
- **skypack** (<https://www.skypack.dev/>) — alternative to esm.sh; maintenance has slowed.

For a tiny project, any will do. For something serious, **pin your versions** — never use `@latest`:

```json
{
  "imports": {
    "preact": "https://esm.sh/preact@10.26.0",
    "preact/hooks": "https://esm.sh/preact@10.26.0/hooks"
  }
}
```

With versions pinned, the behaviour never changes under you. `@latest` is an invitation to random breakage.

### The `scopes` feature

Import maps have a `scopes` field that lets different modules resolve the same specifier differently. Useful when two libraries want different versions of a shared dependency:

```json
{
  "imports": {
    "lodash": "https://esm.sh/lodash@4.17.21"
  },
  "scopes": {
    "/vendor/old-library/": {
      "lodash": "https://esm.sh/lodash@3.10.1"
    }
  }
}
```

Inside `/vendor/old-library/`, `import from "lodash"` gets v3.10.1. Everywhere else, v4.17.21. This is the same dependency-isolation trick Webpack and Yarn do with `node_modules` nesting — natively, at the browser level, without the nested-folder structure.

### One import map per document

You get exactly one `<script type="importmap">` per HTML document. It must appear *before* the first module import. A second one is ignored (with a console warning). This is the main limitation of import maps: you cannot dynamically update them after initial load.

There is a proposal to allow multiple import maps, and Chrome has shipped early support, but it is not yet standard. For today, plan one big import map per page.

### Bare specifiers vs paths

Without an import map, only relative paths (`./foo.js`, `../bar.js`) and absolute URLs (`https://cdn.example.com/lib.js`) work. With an import map, you can use **bare specifiers** — names without `./` or `/` prefixes. These are resolved via the map.

A common pattern is to map your own project structure to aliases:

```json
{
  "imports": {
    "@app/": "/js/",
    "@components/": "/js/components/",
    "@utils": "/js/utils/index.js"
  }
}
```

Now anywhere in your code:

```javascript
import { debounce } from "@utils";
import { Button } from "@components/button.js";
import { router } from "@app/router.js";
```

Feels like Webpack's `resolve.alias`. Same mechanism, native to the browser. The trailing slash on `@app/` means "any path under this prefix", not a single file.

### Integrity and the Subresource Integrity feature

You can assert that a CDN-served file has a specific hash:

```html
<script type="importmap">
{
  "imports": {
    "preact": "https://esm.sh/preact@10.26.0"
  },
  "integrity": {
    "https://esm.sh/preact@10.26.0": "sha384-..."
  }
}
</script>
```

If the file at that URL has a different hash when the browser fetches it, the load fails. This is your protection against the CDN being compromised. For anything more than a personal site, pin versions *and* add integrity hashes.

### A real-world example

Here is what a practical import map for a non-trivial project looks like:

```html
<script type="importmap">
{
  "imports": {
    "@app/": "/js/",
    "@components/": "/js/components/",
    "preact": "https://esm.sh/preact@10.26.0",
    "preact/hooks": "https://esm.sh/preact@10.26.0/hooks",
    "htm": "https://esm.sh/htm@3.1.1",
    "marked": "https://esm.sh/marked@15.0.3",
    "dompurify": "https://esm.sh/dompurify@3.2.3",
    "fuse.js": "https://esm.sh/fuse.js@7.0.0"
  }
}
</script>
```

That is your `package.json`, in one `<script>` tag. Seven libraries, no install, no lockfile, no `node_modules`, and every version is pinned. Every reader of the project can see at a glance what dependencies exist. In six years, when you come back to this project, this file still works exactly the same way it does today, because the URLs are immutable and the browsers still understand them.

## Part 6: Dynamic Import — Code Splitting For Free

Static imports must be at the top of the module and are resolved immediately. Sometimes you do not want that. You want a chunk of code to load only when the user clicks a specific button, or visits a specific route, or enters fullscreen mode.

Dynamic imports solve this:

```javascript
button.addEventListener("click", async () => {
  const { openEditor } = await import("./editor.js");
  openEditor();
});
```

`import(...)` returns a promise. When the promise resolves, the module is loaded, parsed, evaluated, and its exports are available. Until that button is clicked, `editor.js` and its entire dependency graph are never fetched. This is **code splitting**. Webpack has a whole subsystem to do this. The browser has it natively.

### Use cases

**Route-level splitting.** A page with multiple routes loads each route's code only when navigated to:

```javascript
const routes = {
  "/": () => import("./pages/home.js"),
  "/blog": () => import("./pages/blog.js"),
  "/settings": () => import("./pages/settings.js"),
};

async function navigate(path) {
  const load = routes[path];
  if (load) {
    const module = await load();
    module.default(document.getElementById("app"));
  }
}
```

Each page module is fetched only when its path is requested. We will build this pattern out fully in Day 11.

**Feature flags.** Ship a feature to 1% of users? Dynamic import based on a flag:

```javascript
if (user.features.includes("new-editor")) {
  const { NewEditor } = await import("./new-editor.js");
  new NewEditor().mount(document.getElementById("editor"));
}
```

Users without the flag never download the new editor's code.

**Conditional polyfills.** Load a polyfill only if the browser needs it:

```javascript
if (!("Temporal" in globalThis)) {
  await import("https://esm.sh/temporal-polyfill@0.2.5");
}

// Now Temporal is available globally or via import
```

**Lazy-load heavy dependencies.** A Markdown editor you use on only one page:

```javascript
const editButton = document.getElementById("edit");
editButton?.addEventListener("click", async () => {
  const { MarkdownEditor } = await import("./vendor/markdown-editor.js");
  new MarkdownEditor().mount(document.getElementById("content"));
});
```

The 300KB markdown editor loads only when the user clicks Edit. Everyone else pays nothing.

### Preload hints

The browser only starts loading a dynamically imported module when the `import()` call runs. If you know ahead of time that a module will likely be needed, you can hint the browser to pre-fetch it:

```html
<link rel="modulepreload" href="/js/pages/blog.js">
```

The browser fetches `blog.js` in parallel with the rest of the page, without executing it. When `import("/js/pages/blog.js")` later runs, the response is already warm. This is the same idea as `<link rel="preload">` but for modules — it also prefetches the module's dependency graph, so one `modulepreload` can warm dozens of files.

`modulepreload` is Baseline Widely Available. Use it for high-probability navigation targets (hover-intent hints, predictive preloading).

### The trade-off

Dynamic imports have one cost: each one is a separate HTTP request. In a bundler world, you often merged many small modules into larger chunks for HTTP/1.1 efficiency. In HTTP/2 and HTTP/3, which every modern CDN speaks, the cost of many small requests is much lower — but not zero.

Rule of thumb: **up to a few hundred modules, native imports are fine.** Past that, you may benefit from bundling for aggregate transfer size. Most apps are well under the threshold.

## Part 7: The Local Development Loop

You cannot open module-based pages directly from the filesystem in most browsers — they enforce a same-origin policy that blocks `file://` modules. You need a local server. The smallest ones:

**Python 3** (already installed almost everywhere):

```bash
python3 -m http.server 8080
```

**Node.js** (if you happen to have it):

```bash
npx serve
```

**.NET 10** (the one you already have as an ASP.NET dev):

```bash
dotnet tool install --global dotnet-serve
dotnet serve -p 8080
```

**VS Code** has an extension called **Live Server** that does this and auto-reloads on save.

Pick one, run it in the project folder, open `http://localhost:8080`. That is your development environment. No configuration. No `dev:server` script in a `package.json`. No Vite config. Edit the file, refresh the browser, the change is live.

For auto-reload on save — the thing that HMR gives you, minus the hot-module-replacement precision — any of these works:

```bash
# Python with auto-reload (requires a lightweight helper)
pip install livereload
python3 -m livereload

# Or just a plain reload-on-save — the browser caches aggressively, so just:
# Hit Ctrl-Shift-R (Cmd-Shift-R on Mac) to hard reload.
```

On a laptop you are rarely waiting more than 50ms between save and reload. HMR's precision is nice but not essential for a development experience that is already this fast.

## Part 8: What You Lose Without A Bundler (Honestly)

We do not want to pretend the trade-off is imaginary. Here is the honest list.

### 1. Tree shaking

If a library exports 50 functions and you only use two, a bundler can remove the other 48 from the output. ES modules in the browser cannot. The browser fetches the entire module, parses it, and keeps it all in memory.

**Impact in practice:** modest. ES modules are typically much smaller than the bundled artifacts they used to produce, because they do not need the runtime wrapper. A Lodash import from esm.sh is smaller than its bundled equivalent. For small-to-medium apps, you do not notice.

**Mitigation:** Use libraries that ship as ES modules with fine-grained exports. Instead of `import _ from "lodash"`, use `import debounce from "lodash/debounce"`. The module resolver fetches only what you import.

### 2. Automatic minification

Your source files ship as-is. No space removal, no identifier shortening.

**Impact in practice:** 30–40% larger file sizes, which is meaningful over 3G. Gzip compression in HTTP already removes most of the whitespace cost, so the practical impact is usually 10–20%.

**Mitigation:** If it matters, minify your critical-path JavaScript as a post-step before upload. `terser` takes one line:

```bash
npx terser app.js -c -m -o app.min.js
```

You can run this once manually or as part of your deploy. It is not a "build step" in the Vite sense — it is a one-off optimisation, like compressing images before upload. You still write `import` and `export`. You still have no `webpack.config.js`.

We do not do this on this magazine. Our un-minified JS is small enough that the overhead is noise. If our bundle ever grew past 100KB pre-minified, we would add a deploy-time minify step.

### 3. Compile-to-JavaScript languages

TypeScript, Babel, PostCSS, JSX. None of these run in the browser. If you want TypeScript, you need a compile step.

**Mitigation:** use **JSDoc comments** for type annotations in plain JavaScript (see Part 11). You get 85% of TypeScript's editor experience with 0% of the build step. Alternatively, use a runtime-JSX library like [htm](https://esm.sh/htm) that lets you write JSX-like syntax in template literals, which requires no compile step.

We use plain JavaScript with JSDoc in every module of the capstone. It has caught actual type errors for us. We have not missed TypeScript.

### 4. Dependency vendoring

With a bundler, you have a `node_modules` directory full of everything you depend on, offline. With import maps from a CDN, you are depending on a third party to stay online.

**Mitigation:** for production, **self-host your dependencies**. Download each URL once, commit the files to your repo under `/vendor/`, and point the import map there:

```json
{
  "imports": {
    "preact": "/vendor/preact-10.26.0.js",
    "htm": "/vendor/htm-3.1.1.js"
  }
}
```

Now every user fetches from *your* domain, which has the same SLA as the rest of your site. You have removed the CDN dependency entirely. Your repo is slightly larger but entirely self-contained.

We do this for the capstone. The entire "build process" is `curl` followed by `git commit`.

### 5. Hot Module Replacement

HMR lets you edit a component and see the change without losing application state. This is genuinely nice. Native modules do not support it.

**Mitigation:** full page reload. It is fast (<100ms) and usually unambiguous. Modern browsers' DevTools preserve some state (scroll position, form fields, open panels) across reload, so the actual loss is smaller than it sounds.

If you genuinely need HMR, tools like [Vite](https://vitejs.dev/) offer it *without* producing a bundled output — Vite in dev mode serves native ES modules with HMR injected, and in production produces a bundle. You get the best of both. We are not using Vite, but if a large team wanted HMR while preserving a no-build production architecture, it is a reasonable hybrid.

### 6. Single-file deployment

A bundle is one file. Upload one file, you are done. Native modules fan out into many files.

**Mitigation:** none needed. HTTP/2 and HTTP/3 make many small files nearly as fast as one big one. Your static host (Cloudflare Pages, GitHub Pages, Netlify, S3+CloudFront) handles the fanout fine.

### Summary of trade-offs

You give up: automatic minification, tree shaking, compile-to-JS languages, dependency vendoring (unless you self-host), and HMR.

You gain: massive tooling simplicity, instant onboarding, long-term stability, no supply-chain risk, smaller total cognitive surface, transparent debugging, no lockfile drift, no "it works on my machine" ever.

For small-to-medium projects — blogs, internal tools, personal sites, showcase apps like ours — the gains dwarf the losses. For a large enterprise app with hundreds of routes and strict performance budgets, the calculation shifts.

## Part 9: Module Caching And Invalidation

When the browser fetches a module, it caches it — both in its module registry (for the current page) and in the HTTP cache (across pages). Understanding this is the difference between "why is my update not showing up?" and "ah, cache."

### Module registry

Within one page, each URL is loaded exactly once. If `a.js` and `b.js` both `import { x } from "./c.js"`, the browser fetches `c.js` once and shares it. Side effects in `c.js` run exactly once. Export values are singletons.

### HTTP cache

Across pages, module files are cached via standard HTTP cache rules. If your server sends `Cache-Control: max-age=3600`, the browser reuses the cached version for an hour. If you change the file on the server, the browser still uses the old version until the cache expires.

This is the normal web caching story. The module system does not change it. **In production, version your module URLs** to invalidate the cache on change:

```html
<script type="module" src="/js/app.js?v=2026-05-30"></script>
```

When you update the file, update the query string. Users get the new version immediately.

Or use content-addressable file names generated at deploy time:

```html
<script type="module" src="/js/app.a1b2c3d4.js"></script>
```

Each deploy produces a new filename based on content hash. The browser's cache is never stale because the URL changes with the content.

For the capstone, we will use the second pattern. The deploy script hashes each file, renames, updates references. Twenty lines of Bash. Much smaller than Webpack's caching configuration, and more explicit.

## Part 10: Module Graph And Circular Dependencies

Modules can import each other. A imports B. B imports A. What happens?

```javascript
// a.js
import { b } from "./b.js";
export const a = "a";
console.log("a.js loaded, b =", b);

// b.js
import { a } from "./a.js";
export const b = "b";
console.log("b.js loaded, a =", a);
```

The behaviour depends on which one is loaded first. The first one evaluates, sees that its import is not yet initialised (`a` is in a "temporal dead zone"), and often throws a `ReferenceError`. The second one completes normally. This is called a **cyclic dependency**, and it is almost always a design smell.

In practice, cyclic dependencies usually resolve through accidents of ordering — but the resolution is brittle and tends to break at the worst time. If you encounter one, extract the shared code into a third module that both originals import.

The module system correctly handles cycles in the sense that it does not infinite-loop. But the runtime behaviour is subtle, and the answer to "did I just create a cycle?" is usually "try not to."

## Part 11: JSDoc — Types Without A Compiler

The biggest thing you lose by not using TypeScript is the type system. Most of what that type system gives you, you can get in plain JavaScript with JSDoc comments.

### Basic JSDoc

```javascript
/**
 * Format a date as YYYY-MM-DD.
 * @param {Date} date - The date to format.
 * @returns {string} The ISO date string.
 */
export function formatDate(date) {
  return date.toISOString().slice(0, 10);
}
```

VS Code reads this and:

- Shows the description when you hover over the function.
- Type-checks callers. `formatDate("not a date")` gets flagged as an error.
- Provides auto-completion for the return value (string methods).

Install nothing. Configure nothing. VS Code just does it — it uses the same TypeScript language server TypeScript-the-language uses, applied to plain JavaScript with JSDoc hints.

### More complex types

```javascript
/** @typedef {{ id: string, title: string, author: string, published: Date }} Post */

/**
 * Filter posts by author.
 * @param {Post[]} posts
 * @param {string} author
 * @returns {Post[]}
 */
export function byAuthor(posts, author) {
  return posts.filter(p => p.author === author);
}
```

### Generics

```javascript
/**
 * @template T
 * @param {T[]} items
 * @param {(item: T) => boolean} predicate
 * @returns {T | undefined}
 */
export function findFirst(items, predicate) {
  for (const item of items) {
    if (predicate(item)) return item;
  }
  return undefined;
}
```

### Optional parameters, defaults

```javascript
/**
 * @param {string} base
 * @param {number} [count=3]
 * @returns {string}
 */
export function repeat(base, count = 3) {
  return base.repeat(count);
}
```

The `[count=3]` syntax marks the parameter as optional with a default.

### Union types and nullability

```javascript
/**
 * @param {string | number | null} value
 * @returns {string}
 */
export function toString(value) {
  return value === null ? "" : String(value);
}
```

### Typing your own module exports

```javascript
/** @type {number} */
export const PORT = 8080;

/** @type {Map<string, number>} */
export const counts = new Map();
```

### Turning on type checking

VS Code will show inferred types on hover without any configuration. To actively type-check the file — with red squiggles on errors — add a `// @ts-check` comment at the top:

```javascript
// @ts-check

/** @param {number} n */
export function double(n) {
  return n * 2;
}

double("not a number");  // Error: Argument of type 'string' is not assignable...
```

Or project-wide, add a minimal `jsconfig.json`:

```json
{
  "compilerOptions": {
    "checkJs": true,
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "node"
  },
  "include": ["js/**/*.js"]
}
```

That is a ten-line config. Your plain JavaScript files now get TypeScript-level type checking in the editor. No compile step. No emit. Just squiggles.

### Where JSDoc falls short

JSDoc cannot express a few things TypeScript can, most notably:

- **Branded types.** `type UserId = string & { __brand: "UserId" }` — this idiom works in TS, not quite in JSDoc.
- **Some advanced conditional types** — `Exclude<T, U>`, `ReturnType<F>`, etc. Some are available, some not.
- **Private class fields with types** — use `@private` on the JSDoc comment; JSDoc private is a convention, not enforced.

For 95% of code, JSDoc is enough. For the remaining 5%, either accept the limitation or use TypeScript the language with a compile step. You are allowed to have a compile step on *one* project and no compile step on *another*. We are not absolutists.

Let us finish with something you can paste into a folder and run. A minimal multi-page app with native modules, an import map, and dynamic-imported routes.

**`index.html`:**

```html

<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Hello Modules</title>
  <link rel="stylesheet" href="/css/app.css">

  <script type="importmap">
  {
    "imports": {
      "@app/": "/js/",
      "@utils": "/js/utils.js"
    }
  }
  </script>

  <link rel="modulepreload" href="/js/app.js">
</head>
<body>
  <nav aria-label="Primary">
    <a href="/">Home</a>
    <a href="/about">About</a>
    <a href="/settings">Settings</a>
  </nav>
  <main id="app"></main>

  <script type="module" src="/js/app.js"></script>
</body>
</html>
```

**`js/app.js`:**

```javascript
// @ts-check
import { log } from "@utils";

/** @type {Record<string, () => Promise<{ default: (el: HTMLElement) => void }>>} */
const routes = {
  "/":         () => import("@app/pages/home.js"),
  "/about":    () => import("@app/pages/about.js"),
  "/settings": () => import("@app/pages/settings.js"),
};

async function navigate(path) {
  const load = routes[path] ?? routes["/"];
  const module = await load();
  const app = /** @type {HTMLElement} */ (document.getElementById("app"));
  module.default(app);
  log(`Navigated to ${path}`);
}

// Intercept link clicks
document.addEventListener("click", (event) => {
  const link = /** @type {HTMLAnchorElement | null} */ (
    event.target instanceof Element ? event.target.closest("a") : null
  );
  if (!link) return;
  const url = new URL(link.href, location.href);
  if (url.origin !== location.origin) return;
  event.preventDefault();
  history.pushState({}, "", url.pathname);
  navigate(url.pathname);
});

window.addEventListener("popstate", () => navigate(location.pathname));

// Boot
navigate(location.pathname);
```

**`js/utils.js`:**

```javascript
// @ts-check
/**
 * Log a message with a prefix.
 * @param {string} msg
 */
export function log(msg) {
  console.log(`[app] ${msg}`);
}
```

**`js/pages/home.js`:**

```javascript
// @ts-check
/** @param {HTMLElement} el */
export default function home(el) {
  el.innerHTML = `
    <h1>Home</h1>

```

**`js/pages/about.js`:**


## Part 14: When A Bundler Is Still The Right Answer

We want to be fair here. There are cases where a bundler remains the right tool:

1. **Hundreds of routes, large application.** Transfer efficiency over the wire starts to matter past a certain scale. Bundling into a few dozen chunks outperforms fanning out to hundreds of modules, especially on HTTP/1.1 or high-latency networks.

2. **Heavy use of TypeScript-only features.** Decorators, advanced conditional types, branded types. If you need these, you need the TS compiler. A TS-powered bundler is not a bad choice.

3. **Large teams with existing Webpack / Vite expertise.** Migrating away from a working bundler for philosophical reasons is a bad trade. Keep what works; apply these techniques to new projects.

4. **Sophisticated optimisations.** Aggressive tree-shaking, deep dead-code elimination, CSS-in-JS extraction, route-aware prefetching. Modern bundlers do these, native ES modules do not.

5. **Server-side rendering.** A bundler that can run both in the browser and in Node.js for SSR handles the isomorphic case better than raw ES modules.

For a blog, a documentation site, a showcase app, an admin dashboard for a small team, a hobby project — native modules are simpler and sufficient. For a production-grade SaaS front-end with a large team and a budget, a bundler still earns its keep.

**The key is to stop reaching for the bundler as the default.** Start with nothing. Add a bundler only when you have a specific, identifiable need that modules do not meet. Most projects, we predict, will never reach that threshold.
