## The Problem With "It Just Updates the Screen"

Every UI framework ultimately has one job: it has to take some state — some description of what your application currently knows — and translate that into actual pixels on a screen. Simple enough, right? You change the state, the screen updates, everyone is happy.

Except that updating a screen is not simple at all. Between your C# or JavaScript code and the pixels on a monitor, there are approximately seventeen layers of software doing very complicated things. And the choices made at the layer where your framework lives — the layer where it decides what to change and how to change it — have profound consequences for performance, correctness, and the mental model you need to maintain as a developer.

React and Blazor WebAssembly make radically different choices at this layer, built on different underlying technologies and different philosophical assumptions about what a user interface is. Understanding those choices is not just academically interesting — it explains why React needs `useMemo` and `useCallback`, why Blazor components have the lifecycle methods they have, and why specific performance problems in each framework feel the way they do.

This article is going to get fairly technical. We are going to look at actual algorithms, actual data structures, and actual C# and JavaScript code. If some of it feels unfamiliar, sit with it — we will explain every piece before we use it.

## Part 1: The Document Object Model — The Common Enemy

Before we can talk about what React and Blazor do differently, we need to talk about what they both have to deal with: the DOM.

The DOM — Document Object Model — is the browser's tree-structured representation of an HTML document. When you load a web page, the browser parses the HTML, constructs a tree of nodes, and that tree is what gets rendered to the screen. Each element in your HTML (`<div>`, `<p>`, `<button>`, etc.) becomes a node in this tree. Nodes have parent-child relationships. The root node is the `<html>` element.

The DOM is accessed from JavaScript through a browser API. You can call `document.getElementById('my-button')` to get a reference to a DOM node, then modify its properties (`node.textContent = 'New text'`), add event listeners (`node.addEventListener('click', handler)`), or restructure the tree (`parent.appendChild(newChild)`).

Here is the critical thing to understand: **DOM operations are expensive.** Not in the sense of "this takes a millisecond," but in the sense of "this triggers a cascade of work in the browser's rendering pipeline." When you change a DOM node, the browser may need to:

1. **Recalculate styles** — figuring out which CSS rules apply to the changed element and all affected elements.
2. **Layout (reflow)** — recalculating the positions and sizes of elements on the page.
3. **Paint** — actually drawing the pixels for the changed content.
4. **Composite** — assembling the drawn layers into a final frame.

Not every DOM change triggers all four steps — minor property changes might skip layout, and some changes only affect the compositing step. But the more DOM operations you perform, especially in rapid succession, the more likely you are to trigger expensive layout recalculations. And doing layout work on the main thread (the only thread that can manipulate the DOM in a browser) means that if layout takes too long, your user interface freezes. The user sees a janky frame, or worse, a completely frozen tab.

This is the problem that both React's Virtual DOM and Blazor's Render Tree are trying to solve: how do you update the DOM as efficiently as possible, touching the minimum number of nodes and triggering the minimum amount of re-layout?

## Part 2: The Virtual DOM — React's Approach

React's answer to the DOM performance problem is to keep a virtual copy of the DOM in memory — a JavaScript object tree that mirrors the structure of the real DOM — and use a diffing algorithm to compute the minimum set of real DOM changes needed when your application state changes.

The concept is elegant:

1. Your component function returns a description of the UI (a "virtual DOM tree" — a plain JavaScript object, not a real DOM node).
2. React keeps the previous virtual DOM tree.
3. When state changes, React calls your component function again, gets a new virtual DOM tree.
4. React runs a "diffing" algorithm that compares the old tree to the new tree.
5. React generates a minimal list of real DOM operations (patches) based on the diff.
6. React applies those patches to the actual DOM.

Steps 1-4 happen entirely in JavaScript memory, without touching the DOM at all. Only step 6 actually touches the DOM, and it only touches the nodes that actually changed.

### Building a Tiny React Clone in 200 Lines of JavaScript

The best way to understand how the Virtual DOM and diffing work is to build a miniature version ourselves. What follows is a simplified but real implementation of the core concepts. This is not production React — it is missing error handling, the Fiber architecture, portals, context, and a hundred other things. But it is enough to understand the fundamental algorithm.

```javascript
// ============================================================
// PART 1: The Virtual DOM representation
// ============================================================

// A virtual DOM node (vNode) is just a plain JavaScript object.
// It can be:
//   - A string (text node)
//   - An object with { type, props, children }
//     where type is a string ('div', 'p', 'button')
//     or a function (a component)
//
// This is essentially what JSX compiles to.

function createElement(type, props, ...children) {
    return {
        type,
        props: props || {},
        children: children.flat().map(child =>
            typeof child === 'string' || typeof child === 'number'
                ? String(child)  // normalize text to strings
                : child
        )
    };
}

// ============================================================
// PART 2: Rendering a virtual DOM tree to real DOM
// ============================================================

// This takes a vNode and creates a real DOM element.
// On first render, we call this for every node in the tree.

function createDomNode(vNode) {
    if (typeof vNode === 'string') {
        // Text nodes are just browser text nodes
        return document.createTextNode(vNode);
    }

    if (typeof vNode.type === 'function') {
        // A component — call it to get its rendered output
        const rendered = vNode.type(vNode.props);
        return createDomNode(rendered);
    }

    // An HTML element
    const element = document.createElement(vNode.type);

    // Apply props (event listeners, attributes, etc.)
    applyProps(element, {}, vNode.props);

    // Recursively create and append children
    for (const child of vNode.children) {
        element.appendChild(createDomNode(child));
    }

    return element;
}

function applyProps(element, oldProps, newProps) {
    // Remove old event listeners and attributes
    for (const key of Object.keys(oldProps)) {
        if (key.startsWith('on')) {
            const eventName = key.slice(2).toLowerCase();
            element.removeEventListener(eventName, oldProps[key]);
        } else if (key !== 'children') {
            element.removeAttribute(key);
        }
    }

    // Apply new event listeners and attributes
    for (const key of Object.keys(newProps)) {
        if (key.startsWith('on')) {
            const eventName = key.slice(2).toLowerCase();
            element.addEventListener(eventName, newProps[key]);
        } else if (key === 'className') {
            element.setAttribute('class', newProps[key]);
        } else if (key !== 'children') {
            element.setAttribute(key, newProps[key]);
        }
    }
}

// ============================================================
// PART 3: The Diffing Algorithm
// ============================================================

// This is the heart of the Virtual DOM approach.
// Given an old vNode tree, a new vNode tree, and the real DOM node,
// this function updates the real DOM to match the new vNode tree
// with the minimum number of operations.

function diff(parentElement, oldVNode, newVNode, index = 0) {
    const existingChild = parentElement.childNodes[index];

    // Case 1: New node exists, old node doesn't → add it
    if (!existingChild && newVNode !== undefined) {
        parentElement.appendChild(createDomNode(newVNode));
        return;
    }

    // Case 2: Old node exists, new node doesn't → remove it
    if (existingChild && newVNode === undefined) {
        parentElement.removeChild(existingChild);
        return;
    }

    // Case 3: Both are text nodes
    if (typeof newVNode === 'string' && typeof oldVNode === 'string') {
        if (newVNode !== oldVNode) {
            // Only touch the DOM if the text actually changed
            existingChild.textContent = newVNode;
        }
        return;
    }

    // Case 4: Type changed → replace entirely
    if (oldVNode.type !== newVNode.type) {
        parentElement.replaceChild(createDomNode(newVNode), existingChild);
        return;
    }

    // Case 5: Same type → update props and recurse into children
    if (typeof newVNode.type !== 'function') {
        applyProps(existingChild, oldVNode.props, newVNode.props);

        // Diff children
        const maxLength = Math.max(
            oldVNode.children.length,
            newVNode.children.length
        );
        for (let i = 0; i < maxLength; i++) {
            diff(existingChild, oldVNode.children[i], newVNode.children[i], i);
        }
    } else {
        // It's a component — call it and diff the result
        const oldRendered = oldVNode.type(oldVNode.props);
        const newRendered = newVNode.type(newVNode.props);
        diff(parentElement, oldRendered, newRendered, index);
    }
}

// ============================================================
// PART 4: State management — the simple version
// ============================================================

// In real React, useState is managed by a Fiber node.
// Here, we keep global state per "render cycle" for simplicity.
// This is obviously not production-quality — it shows the concept.

let currentRoot = null;
let currentVDomTree = null;

function createApp(rootElement, componentFn, initialProps = {}) {
    function render() {
        const newVDomTree = componentFn(initialProps);

        if (!currentVDomTree) {
            // First render — no diff, just create
            const dom = createDomNode(newVDomTree);
            rootElement.appendChild(dom);
        } else {
            // Subsequent renders — diff the trees
            diff(rootElement, currentVDomTree, newVDomTree);
        }

        currentVDomTree = newVDomTree;
    }

    render(); // Initial render
    return { rerender: render };
}

// ============================================================
// PART 5: A tiny working example
// ============================================================

// Let's build a simple counter using our mini-React clone.

let count = 0;

function Counter() {
    return createElement(
        'div',
        { className: 'counter' },
        createElement('p', {}, `Count: ${count}`),
        createElement(
            'button',
            {
                onClick: () => {
                    count++;
                    app.rerender();
                }
            },
            'Increment'
        )
    );
}

const appRoot = document.getElementById('app');
const app = createApp(appRoot, Counter);
```

Let us walk through what this code does, because every line matters.

**`createElement(type, props, ...children)`**: This is the function that JSX compiles to. When you write `<div className="counter"><p>Count: {count}</p></div>` in JSX, the Babel or SWC compiler transforms it into `createElement('div', { className: 'counter' }, createElement('p', {}, 'Count: ' + count))`. The result is a plain JavaScript object — no DOM manipulation, no browser API calls. Just a nested object.

**`createDomNode(vNode)`**: This translates a virtual DOM node into a real DOM node. It is called on the very first render, when we need to create the entire DOM structure from scratch.

**`diff(parentElement, oldVNode, newVNode, index)`**: This is the diffing algorithm. Notice the structure: it handles five different cases, and only in Case 5 (same element type, same position) does it update props in place and recurse into children. In all other cases where something fundamental has changed (different type, node removed, node added), it creates or destroys DOM nodes. The critical optimization is in the text node case (Case 3): it only calls `existingChild.textContent = newVNode` if the text actually changed. Every avoided DOM operation is a win.

**The applyProps function**: This handles a subtle but important issue — event listeners. In the browser, you attach an event listener with `addEventListener`. If you call `addEventListener` again with the same function reference, you get two event listeners. Our `applyProps` function removes old event listeners before adding new ones to avoid this accumulation. Real React handles this more efficiently by using event delegation — rather than attaching an event listener to each individual element, React attaches a single listener at the document root and uses event bubbling to route events to the correct handler.

### The Key Insight About the Diffing Algorithm

Look at the diff function again. It operates under one critical assumption that real React also makes: **elements at the same position in the tree with the same type are considered the same element and are updated in place.** This is why React's `key` prop exists.

Imagine you have a list of items:

```jsx
// Before state change:
<ul>
  <li>Alice</li>
  <li>Bob</li>
  <li>Charlie</li>
</ul>

// After state change (Alice removed from beginning):
<ul>
  <li>Bob</li>
  <li>Charlie</li>
</ul>
```

Without `key` props, a naive diffing algorithm compares position by position. Position 0: was `<li>Alice</li>`, now `<li>Bob</li>` — same type, update text content. Position 1: was `<li>Bob</li>`, now `<li>Charlie</li>` — same type, update text content. Position 2: was `<li>Charlie</li>`, now nothing — remove. This results in two DOM text updates and one removal, when the semantically correct operation would be one removal at the beginning.

More importantly: if your list items had state (checkbox checked/unchecked, focused input values, etc.), the naive approach would appear to move state between items incorrectly — the checkmark you put on "Alice" would visually move to "Bob" because React thinks position 0 is the same element as before.

This is why React requires `key` props on list items:

```jsx
items.map(item => <li key={item.id}>{item.name}</li>)
```

The `key` tells the diffing algorithm "this element's identity is tied to this specific ID, not to its position." React can then correctly identify that the element with `key="alice"` has been removed, not that the element at position 0 has changed.

This `key` prop requirement is something .NET developers often find confusing at first, but once you understand the diffing algorithm, it becomes completely obvious why it exists.

## Part 3: React Fiber — Breaking Work into Small Pieces

Our tiny React clone above has a significant problem that real React 18+ fixes with the Fiber architecture: it is synchronous. When `rerender()` is called, it diffs the entire tree and applies all DOM patches before returning. If the component tree is large and the diff is expensive, this can block the main thread for long enough to cause the UI to feel unresponsive.

React 16 introduced the Fiber architecture, which is a complete rewrite of React's internal reconciler to make rendering work interruptible. Understanding Fiber at a high level is important for any developer working with React, because it is the foundation of all of React's concurrent features.

### What is a Fiber?

A Fiber is a JavaScript object that represents a unit of work — specifically, a component instance at a specific point in the component tree. Each rendered component has a corresponding Fiber node. The Fiber node stores:

- The component type (a function or class)
- The component's props
- The component's state
- The computed output (the rendered JSX)
- Links to the parent Fiber, first child Fiber, and next sibling Fiber
- A reference to the corresponding DOM node (when it exists)

This linked tree structure (parent-child-sibling links rather than a traditional array-of-children tree) allows React to walk the tree incrementally — process one Fiber node, do a tiny bit of work, check whether there is something more urgent to do (like handling a user input event), and if so, pause and come back later.

### The Two-Phase Commit

React Fiber splits rendering into two distinct phases:

**Phase 1 — Render (a.k.a. "Reconcile")**: React walks the Fiber tree, calling each component function, computing the new output, and comparing it to the previous output. This phase is interruptible — React can pause it at any point and resume later. During this phase, no DOM mutations happen. This phase can happen asynchronously, across multiple event loop frames.

**Phase 2 — Commit**: Once all reconciliation work is complete, React applies the changes to the actual DOM in a single, synchronous operation. The commit phase cannot be interrupted, because interrupting it would leave the DOM in an inconsistent state. However, by the time React commits, it knows exactly what needs to change, so this phase is usually fast.

The render phase can be paused and resumed. This is what allows React to yield to the browser between heavy rendering work, preventing the UI from freezing. In React 18's Concurrent Mode, if a lower-priority re-render is in progress and a high-priority event (like a user typing in an input field) arrives, React can pause the in-progress render, handle the input event, and then resume the deferred render later.

### How Fiber Affects Your Code

The two-phase commit model has a practical consequence that surprises many developers: **your component function may be called more than once for a single visible update.** In Concurrent Mode, React might start rendering your component, pause, and then discard the work and start over. This means side effects (things that affect the outside world — API calls, subscriptions, timers) must not be run during the render phase.

This is exactly why `useEffect` exists. Effects run after commit, not during render. They are React's way of saying "here is code that has side effects, please run it after the DOM is stable."

And this is why `useEffect` causes so many bugs — a fact we will explore in exhaustive detail in Part 3 of this series.

## Part 4: The Render Tree — Blazor's Approach

Blazor WebAssembly uses a fundamentally different mechanism for the same goal: efficiently updating the browser DOM when application state changes. Instead of the Virtual DOM (a complete JavaScript object representation of the entire DOM), Blazor uses a more compact structure called the Render Tree, represented by a sequence of `RenderTreeFrame` structs.

### What Is a RenderTreeFrame?

A `RenderTreeFrame` is a C# struct (a value type, not a reference type — this is important for memory reasons) that represents a single element in the rendered output. It is a discriminated union of sorts — a single struct type that can represent different kinds of UI elements:

- **Component frames**: A child component (equivalent to a React component instance)
- **Element frames**: An HTML element (like `<div>`, `<button>`)
- **Attribute frames**: An attribute on the immediately preceding element (`class="foo"`, `onclick=...`)
- **Text frames**: Text content
- **Markup frames**: Raw HTML markup
- **Region frames**: A logical group of frames (used for conditional rendering)
- **ElementReferenceCapture frames**: For capturing a reference to a real DOM element (like a React `ref`)
- **ComponentReferenceCapture frames**: For capturing a reference to a child component

The Blazor runtime maintains a pair of these frame arrays for each rendered component: the "current" frames (what the browser currently shows) and the "previous" frames (what was shown in the last render). When a re-render happens, Blazor compares these two arrays — this is Blazor's version of the "diff."

### The RenderTreeBuilder in Action

You normally never interact with `RenderTreeBuilder` directly when writing Blazor components — the Razor compiler generates this code for you. But understanding what the Razor compiler generates helps you understand what Blazor is actually doing.

Consider this simple Blazor component:

```razor
<div class="counter">
    <p>Count: @count</p>
    <button @onclick="Increment">Increment</button>
</div>

@code {
    private int count = 0;
    
    private void Increment()
    {
        count++;
    }
}
```

The Razor compiler (which runs at build time, not at runtime) generates something approximately like this C# code:

```csharp
protected override void BuildRenderTree(RenderTreeBuilder builder)
{
    builder.OpenElement(0, "div");
    builder.AddAttribute(1, "class", "counter");
    
    builder.OpenElement(2, "p");
    builder.AddContent(3, "Count: ");
    builder.AddContent(4, count);  // ← This is where the dynamic value goes
    builder.CloseElement();
    
    builder.OpenElement(5, "button");
    builder.AddAttribute(6, "onclick", EventCallback.Factory.Create(this, Increment));
    builder.AddContent(7, "Increment");
    builder.CloseElement();
    
    builder.CloseElement();
}
```

The integer arguments (0, 1, 2, 3...) are **sequence numbers** — a critical feature of Blazor's diffing algorithm. They are hardcoded at compile time and tell the diffing algorithm the source-code position of each frame. This is different from React, which infers the relative positions from the tree structure. Blazor's sequence numbers allow a very efficient diffing algorithm because the algorithm knows the static structure of the template and only needs to check the dynamic parts.

### Why Sequence Numbers Are Clever

Here is why sequence numbers matter. Consider a component that conditionally renders some content:

```razor
@if (showExtraContent)
{
    <p class="extra">Extra content here</p>
}
<p>This always shows</p>
```

The compiled output generates sequence numbers for each frame that correspond to the source code position:

```csharp
// sequence number 0 → the conditional block (if statement)
if (showExtraContent)
{
    builder.OpenElement(1, "p");  // sequence 1
    builder.AddAttribute(2, "class", "extra");  // sequence 2
    builder.AddContent(3, "Extra content here");  // sequence 3
    builder.CloseElement();
}
builder.OpenElement(4, "p");  // sequence 4 — this is always present
builder.AddContent(5, "This always shows");  // sequence 5
builder.CloseElement();
```

When `showExtraContent` changes from `true` to `false`, the diff between the previous frames (which included sequence 1, 2, 3) and the new frames (which skip directly from nothing to sequence 4) is unambiguous. The algorithm knows from the sequence numbers that the frame at position 4 in the new output corresponds to the `<p>This always shows</p>` element — not the conditional `<p class="extra">` — even though the conditional block's frames have been removed.

Without sequence numbers, if you just compared by array position, you would see an element at position 0 that was previously `<p class="extra">` and is now `<p>` (no class) with different text content, and an element at position 1 that was `<p>This always shows</p>` and is now absent — leading to a completely wrong diff and incorrect DOM updates.

The sequence number approach is what allows Blazor's diffing to be faster than a naive positional approach for components with conditional rendering.

### Comparing the Two Approaches

Let us put the two approaches side by side:

| Concern | React Virtual DOM | Blazor Render Tree |
|---|---|---|
| Data structure | Full JavaScript object tree | Array of RenderTreeFrame structs |
| Memory allocation | New object tree on every render | Struct array (value types, stack-friendly) |
| Diffing strategy | Tree-walk with type and position comparison | Sequence number comparison |
| Key mechanism | `key` prop on list items | `@key` directive on list items |
| Threading model | Main thread only (Fiber can yield but not truly parallel) | Main thread only in WASM |
| Render interruption | Yes (Fiber / Concurrent Mode) | No (synchronous per component) |

The Blazor Render Tree approach has some memory advantages over the Virtual DOM approach. C# structs are value types, which means they can be allocated on the stack or in pooled arrays rather than on the heap. This reduces garbage collection pressure. In React, every render creates new JavaScript objects for every element in the tree, and those objects eventually need to be garbage collected by the JavaScript engine.

## Part 5: The Garbage Collection Problem

Both JavaScript and C# use automatic memory management through garbage collection. But the details matter enormously for UI performance.

### JavaScript Garbage Collection in a React Application

In a React application with a Virtual DOM, every re-render creates a new set of JavaScript objects representing the new virtual tree. The old virtual tree objects become garbage — they are no longer referenced and will eventually be collected by the garbage collector.

For small, infrequent re-renders, this is not a problem. The garbage collector runs occasionally in the background, collects the old objects, and everything is fine. But for applications that re-render frequently — components that update on every animation frame, real-time data streams, or interactive grids with many cells — the constant allocation and collection of virtual DOM objects can cause "GC pauses" — moments where the JavaScript garbage collector stops your application code and does collection work, causing a frame to drop or a stutter in animations.

This is one of the reasons React introduced `useMemo` and `useCallback`. By memoizing the output of expensive computations and the identity of callback functions across renders, you reduce the number of new JavaScript objects created on each render, which reduces GC pressure.

```jsx
// Without memoization: a new object is created on every render
const expensiveData = computeExpensiveData(rawData);

// With useMemo: the computation only re-runs when rawData changes
// The result object is reused across renders
const expensiveData = useMemo(
    () => computeExpensiveData(rawData),
    [rawData]
);
```

```jsx
// Without useCallback: a new function object is created on every render
// If this function is passed as a prop to a child component,
// the child will see a new prop value every render and re-render unnecessarily
const handleClick = () => { doSomething(value); };

// With useCallback: the function identity is stable across renders
// as long as 'value' does not change
const handleClick = useCallback(
    () => { doSomething(value); },
    [value]
);
```

These hooks exist because of the Virtual DOM's allocation model. Every re-render creates new objects. If those new objects are used as props for child components, the child components see "new" props even when the values are semantically identical, and they re-render unnecessarily. `useMemo` and `useCallback` are escape hatches that let you preserve object identity across renders.

### C# and WebAssembly Garbage Collection

The garbage collection story in Blazor WebAssembly is more nuanced. The .NET runtime running in WebAssembly has its own garbage collector — this is the same .NET GC that runs on the server, just compiled to WebAssembly bytecode.

However, the `RenderTreeFrame` structs that represent Blazor's render tree are value types (C# structs). Value types in .NET do not participate in garbage collection — they are either allocated on the stack (and freed when the stack frame is popped) or embedded inline in arrays. When Blazor's renderer allocates a new array of `RenderTreeFrame` for a component re-render, it reuses an existing pooled array rather than allocating a new one. This is a deliberate design choice to minimize GC pressure in the hot rendering path.

The practical consequence is that simple Blazor components with no dynamic collections tend to produce very little garbage during re-renders. The GC pressure tends to come from dynamic list rendering (where `@foreach` over a collection produces some allocations) and from string formatting.

### .NET AOT Compilation — A Different Kind of Optimization

When you publish a Blazor WebAssembly application with AOT compilation enabled (`dotnet publish -c Release --runtime browser-wasm`), the .NET compiler performs Ahead-of-Time compilation of your C# code to native WebAssembly bytecode, rather than leaving it as IL (Intermediate Language) to be JIT-compiled at runtime.

AOT compilation enables a class of optimizations that React with JavaScript cannot benefit from, because they depend on complete knowledge of the code at compile time:

**Dead code elimination (Trimming)**: The .NET trimmer analyzes your application's entire call graph and removes any code that is never called. If you reference `System.Text.Json` but only use its serialization features, the deserialization code is trimmed out. This can dramatically reduce the download size of your Blazor application.

**Inlining**: The AOT compiler can inline small methods at their call sites, eliminating function call overhead for hot paths.

**Struct optimization**: Value types that are small enough can be passed in registers rather than on the heap, avoiding allocation entirely.

**Devirtualization**: When the compiler knows the concrete type of an object at compile time, it can replace virtual dispatch (which requires a vtable lookup) with a direct call, which is faster and more friendly to CPU branch prediction.

These optimizations happen automatically at compile time. You do not need to add `useMemo` around them or annotate your code with memoization attributes. The compiler finds them.

React, being a JavaScript library, runs in a JavaScript engine that does JIT compilation. V8 (Chrome's JavaScript engine), SpiderMonkey (Firefox's), and JavaScriptCore (Safari's) are all extremely sophisticated JIT compilers and perform many of the same optimizations. But they do so at runtime, based on profiling the code as it executes, rather than at compile time with complete knowledge of the entire program. The React Compiler (released in stable form in 2025 as part of the React 19 ecosystem) moves some of this optimization work to build time for React — it analyzes your components and automatically inserts `useMemo` and `useCallback` calls where they would help. This closes some of the gap.

## Part 6: How This Affects Component Re-Rendering

Understanding the Virtual DOM and Render Tree mechanics helps explain one of the most common sources of React performance problems: unnecessary re-renders.

### React's Re-Rendering Rules

In React, when a component's state changes (via `setState` or a state setter from `useState`), that component and all of its descendants in the component tree will re-render by default. This is the design decision that makes React's programming model simple — you update state, the UI reflects the new state, full stop.

But "re-render" in React means "call the component function again, produce a new virtual DOM tree, and diff it against the old tree." For most components, this is fast. But if a component has many children, or if producing the virtual DOM tree for those children is expensive (computationally intensive filtering, sorting, or formatting operations), unnecessary re-renders can add up.

React's solution is `React.memo`, `useMemo`, and `useCallback`. Let us see a concrete example.

```jsx
// Parent component
function BlogList({ posts, onPostSelected }) {
    const [filterText, setFilterText] = useState('');
    
    // Every time filterText changes, this entire component re-renders.
    // BlogPostCard is a child component. Without memoization,
    // it will also re-render for every filterText change,
    // even if the posts and onPostSelected haven't changed.
    
    const filteredPosts = posts.filter(p => 
        p.title.toLowerCase().includes(filterText.toLowerCase())
    );
    
    return (
        <div>
            <input value={filterText} onChange={e => setFilterText(e.target.value)} />
            {filteredPosts.map(post => (
                <BlogPostCard 
                    key={post.id} 
                    post={post} 
                    onSelect={onPostSelected}  // ← new function reference every render!
                />
            ))}
        </div>
    );
}

// Child component — this will re-render every time BlogList re-renders
// unless we wrap it with React.memo
const BlogPostCard = React.memo(function BlogPostCard({ post, onSelect }) {
    return (
        <div className="card" onClick={() => onSelect(post.id)}>
            <h2>{post.title}</h2>
            <p>{post.summary}</p>
        </div>
    );
});
```

There are two problems here:

1. `filteredPosts` is recomputed on every render, even if `posts` and `filterText` haven't changed. This can be fixed with `useMemo`.

2. `onPostSelected` is passed as a prop. Even with `React.memo`, if `onPostSelected` is a new function reference on every parent render, `BlogPostCard` will still re-render because its `onSelect` prop appears to have changed. This is the problem `useCallback` solves.

Here is the correctly memoized version:

```jsx
function BlogList({ posts, onPostSelected }) {
    const [filterText, setFilterText] = useState('');
    
    // Memoize the filtered results — only recompute when posts or filterText changes
    const filteredPosts = useMemo(
        () => posts.filter(p => 
            p.title.toLowerCase().includes(filterText.toLowerCase())
        ),
        [posts, filterText]  // ← dependency array
    );
    
    // Memoize the callback — stable function reference across renders
    const handleSelect = useCallback(
        (postId) => { onPostSelected(postId); },
        [onPostSelected]  // ← only new function when onPostSelected changes
    );
    
    return (
        <div>
            <input value={filterText} onChange={e => setFilterText(e.target.value)} />
            {filteredPosts.map(post => (
                <BlogPostCard 
                    key={post.id} 
                    post={post} 
                    onSelect={handleSelect}  // ← stable reference
                />
            ))}
        </div>
    );
}
```

This is correct and performant. But notice how much cognitive overhead is required: you have to understand object identity in JavaScript, understand how React's shallow comparison for `React.memo` works, understand that functions are objects and have identity, and keep track of which things in your component produce new references on every render. This is the burden that React places on developers in exchange for its simple "re-render everything" mental model.

### Blazor's Re-Rendering Rules

Blazor's re-rendering behavior is different and, in some ways, simpler for .NET developers to reason about.

A Blazor component re-renders in these situations:
1. `StateHasChanged()` is called (explicitly requesting a re-render)
2. A parameter value changes (Blazor does a shallow comparison of parameter values using `ShouldRender()`)
3. An event handler is invoked (Blazor automatically calls `StateHasChanged()` after event handlers)
4. `await` returns within an event handler or lifecycle method

By default, when a parent component re-renders, it will re-render all its children. But Blazor has `ShouldRender()` — an overridable method where you can provide your own logic to determine whether a component needs to update:

```csharp
@code {
    [Parameter] public BlogPost Post { get; set; } = default!;
    private BlogPost? _previousPost;
    
    protected override bool ShouldRender()
    {
        // Only re-render if the post has actually changed
        if (_previousPost?.Id == Post.Id && _previousPost?.UpdatedAt == Post.UpdatedAt)
        {
            return false;
        }
        _previousPost = Post;
        return true;
    }
}
```

This is Blazor's equivalent of `React.memo`. It is more explicit (you write the comparison logic yourself) but also more flexible — you can use any criteria you want.

For parameter-receiving components, you can also use the `LayoutComponentBase` or mark your component as a "pure" component with the `[ShouldRender]` attribute approach. But most performance-sensitive Blazor applications get good results just by being careful about when they call `StateHasChanged()` and using `ShouldRender()` override for expensive components.

The critical difference from React is that Blazor does not create a new "render tree" object on every re-render — it updates the existing render tree frames in place. This is more memory-efficient and puts less pressure on the garbage collector.

## Part 7: A Concrete Performance Comparison

Theory is fine, but let us make this tangible with a concrete scenario.

Imagine you have a component that renders a list of 1,000 blog post cards, each showing a title, author, date, and reading time. Every five seconds, the reading time of the most recently viewed post updates in real time.

### In React (naive implementation)

Every five-second update causes the parent component to call `setState`, which triggers a re-render of the entire parent. Without memoization, all 1,000 `BlogPostCard` child components re-render (their functions are called). The React reconciler diffs 1,000 virtual DOM trees. It discovers that only one card's reading time actually changed, and it applies one DOM text update.

The work done: 1,000 function calls, 1,000 object tree allocations (some of which will be GC'd), 1,000 tree diffs, 1 DOM update.

### In React (with React.memo on BlogPostCard)

Only the one card whose props actually changed re-renders. The other 999 cards are skipped by `React.memo`'s shallow comparison.

The work done: 1 function call, 1 object tree allocation, 1 tree diff, 1 DOM update. Plus 999 shallow comparison operations (checking whether any prop references changed).

### In Blazor WebAssembly

By default, updating state in the parent causes all child components to re-render. But if `BlogPostCard` overrides `ShouldRender()` to check whether the specific post's reading time changed, 999 out of 1,000 components short-circuit immediately without even calling `BuildRenderTree`.

The work done: 1 call to `BuildRenderTree`, comparison of the rendered frames for that one card, 1 DOM update. Plus 1,000 calls to `ShouldRender()`.

### What This Tells Us

The performance profile of both frameworks is dominated by the same concern: avoiding work for components whose output has not changed. React does this through `React.memo`, `useMemo`, and `useCallback` (explicit memoization). Blazor does this through `ShouldRender()` overrides (explicit shouldUpdate logic). The concepts are the same, just expressed differently.

The advantage of Blazor's approach is that the comparison in `ShouldRender()` operates on C# values using C#'s value equality, which is natural and explicit. In React, the comparison in `React.memo` uses JavaScript reference equality, which is confusing until you fully understand that JavaScript objects are compared by reference, not by content.

## Part 8: What the React Compiler Changes

In 2025, React shipped the React Compiler as a stable tool (distributed as the `babel-plugin-react-compiler` package). The React Compiler is a build-time tool that analyzes your React component code and automatically inserts `useMemo` and `useCallback` calls where they would help performance — without you having to write them.

```jsx
// What you write:
function BlogList({ posts, onPostSelected }) {
    const filteredPosts = posts.filter(p => p.active);
    return filteredPosts.map(post => (
        <BlogPostCard key={post.id} post={post} onSelect={onPostSelected} />
    ));
}

// What the React Compiler generates (approximately):
function BlogList({ posts, onPostSelected }) {
    const filteredPosts = useMemo(
        () => posts.filter(p => p.active),
        [posts]
    );
    return filteredPosts.map(post => (
        <BlogPostCard key={post.id} post={post} onSelect={onPostSelected} />
    ));
}
```

The compiler can only do this correctly if your components follow React's rules (no direct mutation of state, etc.). When it works, it eliminates the need for manual memoization and reduces the burden on developers to reason about object identity.

This is, conceptually, moving React closer to how .NET's AOT compiler works — doing optimization work at build time rather than requiring developers to annotate their code manually. The gap between the two approaches is narrowing.

## Conclusion: What Have We Learned?

Both React and Blazor WebAssembly solve the same fundamental problem: efficiently update the browser DOM when application state changes. They solve it differently.

React uses a Virtual DOM — a JavaScript object tree that mirrors the UI structure — combined with a diffing algorithm that computes the minimum set of DOM changes. The Fiber architecture makes this process interruptible, enabling concurrent rendering. The cost is per-render memory allocation and the need for explicit memoization to avoid unnecessary re-renders.

Blazor uses a compact array of `RenderTreeFrame` structs with sequence numbers, enabling a highly efficient diff that leverages compile-time knowledge of template structure. With AOT compilation, the C# code is compiled to native WebAssembly with aggressive dead-code elimination and optimization. The programming model is more familiar to .NET developers and avoids some of the JavaScript reference-equality pitfalls that make `useMemo` necessary.

Neither approach is unambiguously superior. For applications with massive lists and frequent updates, the memoization model matters a lot. For typical business applications with moderate complexity, both frameworks perform well out of the box.
