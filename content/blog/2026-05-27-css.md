## Introduction: The Fifty-Seven-Line Centering Attempt

This story is nine years old but the thread is still open, and we check it whenever we need cheering up. The question, on Stack Overflow: *"How do I center a div vertically inside another div?"* The top answer, as of 2017, ran to fifty-seven lines of CSS, used three nested wrappers, required the outer div to have an explicit height, and was annotated with disclaimers about Internet Explorer 10. There were six alternate answers beneath it, each with their own nested wrappers, each bickering with the others about edge cases.

In 2026, the same question has a six-character answer:

```css
display: grid;
place-items: center;
```

Two declarations. Zero wrappers. Works everywhere. It is not that our problems are different — people still need to centre a div inside another div — it is that the tools available to us are staggeringly better than they were. An entire generation of developers who learned CSS by learning "five ways to center," all of which were workarounds for missing features, are now in the position of learning new layout models that largely obsolete those workarounds.

Today we are going to cover those models, comprehensively, from first principles. Flexbox. Grid. Subgrid. Container queries. The whole modern layout toolkit. By the end of this article you will be able to build essentially any layout you can imagine without a framework, without a preprocessor, without a single `float: left` declaration, and — for most layouts — without a single media query either.

## Part 1: The Foundation — The Box Model And Layout Modes

Before Flexbox, before Grid, there is **the box model**. Every element on a page is a rectangular box with four zones, from innermost to outermost:

1. **Content** — the text or nested elements. Sized by `width` and `height`.
2. **Padding** — transparent space inside the border. Sized by `padding-*`.
3. **Border** — the visible edge. Sized by `border-*`.
4. **Margin** — transparent space outside the border. Sized by `margin-*`.

The confusing historical default is that `width` refers to the **content width only** — so an element with `width: 300px; padding: 20px; border: 1px solid` actually takes up 342 pixels of horizontal space. This is "content-box" sizing. A generation of web developers shipped buggy layouts because of this default. Every modern CSS reset therefore includes:

```css
*, *::before, *::after {
  box-sizing: border-box;
}
```

With `border-box`, `width` includes padding and border. Write `width: 300px` and the element is 300 pixels total. No surprise arithmetic. We do this in our `reset.css` in Day 3 and we will never think about it again.

**Logical properties.** Modern CSS replaces `margin-left`, `padding-right`, etc. with direction-agnostic equivalents: `margin-inline-start`, `padding-inline-end`. The "inline" axis is the direction text runs (left-to-right in English, right-to-left in Arabic). The "block" axis is perpendicular (top-to-bottom in Latin scripts, right-to-left in some vertical scripts). Use logical properties and your layouts work for international audiences without modification:

- `margin-inline: 1rem` — left and right margin in LTR.
- `padding-block: 1rem` — top and bottom padding.
- `border-inline-start: 1px solid` — left border in LTR, right border in RTL.

For a monolingual English-only site, the difference is invisible. For anything shipped globally, logical properties are the right default. We use them throughout this series.

### The four layout modes

CSS has four main layout modes, any of which you opt into by setting `display`:

- **Block and Inline** — the default, inherited from the original HTML. Block elements stack vertically; inline elements flow horizontally. Useful for prose, largely superseded by the below for layouts.
- **Flexbox** — `display: flex`. One-dimensional layout along a single axis (row or column).
- **Grid** — `display: grid`. Two-dimensional layout across rows and columns simultaneously.
- **Multi-column** — `columns: 3`. Newspaper-style column flow. Niche.

There are also `table`, `inline-block`, `flow-root`, and a handful of legacy modes. We will not cover them here; they are rarely the right choice in 2026.

The rule of thumb: **Flexbox for one-dimensional layouts, Grid for two-dimensional layouts.** That is not quite precise — you can do one-dimensional layouts in Grid and they often work better there — but it is the right starting point. Navigation bars, button groups, form rows: Flexbox. Page layouts, card grids, complex dashboards: Grid.

## Part 2: Flexbox — Complete Reference

Flexbox has been [Baseline Widely Available since 2020](https://web.dev/baseline). It is the simplest of the modern layout modes and the one you will use most. You turn on Flexbox on a *container*, and the container's *direct children* become flex items.

```html
<div class="toolbar">
  <button>Save</button>
  <button>Cancel</button>
  <span class="spacer"></span>
  <button>Help</button>
</div>
```

```css
.toolbar {
  display: flex;
  gap: 0.5rem;
  align-items: center;
}
.spacer {
  flex: 1;
}
```

That is a toolbar with two buttons on the left, a flexible gap, and a Help button pushed to the right. Seven lines of CSS. Let us unpack every property Flexbox exposes.

### Container properties

- **`display: flex`** — enables Flexbox, creates a row by default.
- **`display: inline-flex`** — same but the container itself is inline-level.
- **`flex-direction`** — the main axis. `row` (default), `row-reverse`, `column`, `column-reverse`.
- **`flex-wrap`** — `nowrap` (default, one line), `wrap` (wrap onto new lines), `wrap-reverse`.
- **`flex-flow`** — shorthand for `flex-direction` and `flex-wrap`.
- **`justify-content`** — alignment along the main axis. Values: `flex-start`, `flex-end`, `center`, `space-between`, `space-around`, `space-evenly`, `start`, `end`, `left`, `right`.
- **`align-items`** — alignment along the cross axis for each line. Values: `stretch` (default), `flex-start`, `flex-end`, `center`, `baseline`, `first baseline`, `last baseline`, `start`, `end`.
- **`align-content`** — alignment of multiple lines along the cross axis when `flex-wrap: wrap` is set. Similar values to `justify-content`.
- **`gap`** (and `row-gap`, `column-gap`) — space between items. Baseline Widely Available since 2022. Before this, we used `margin: 0 0.5rem` hacks and had to fight the first-item problem.

### Item properties

- **`flex-grow`** — how much spare space to absorb, relative to siblings. Default `0` (don't grow).
- **`flex-shrink`** — how much to shrink when there is not enough space. Default `1`.
- **`flex-basis`** — the item's starting size, before `grow` and `shrink` apply. Default `auto` (its natural content size).
- **`flex`** — shorthand for `flex-grow flex-shrink flex-basis`. `flex: 1` means `1 1 0%` — grow, shrink, and start at zero — which is the "fill available space" idiom.
- **`align-self`** — override the container's `align-items` for one item.
- **`order`** — a number controlling visual order. Changes the visual order *without changing the DOM order*. Use sparingly; see the accessibility note below.

### Common patterns

**Centering.** A flex container with a single child:

```css
.centered {
  display: flex;
  justify-content: center;
  align-items: center;
  min-height: 100dvh;
}
```

That centres the child both horizontally and vertically. `100dvh` is "100% of the dynamic viewport height," which accounts for mobile browser toolbars that appear and disappear. `100vh` is the old unit; prefer `dvh` on mobile.

**The holy grail: two items, spacer between.**

```css
.bar {
  display: flex;
  justify-content: space-between;
  align-items: center;
}
```

Item one left, item two right. No spacer div. No absolute positioning.

**Equal-width columns.**

```css
.row {
  display: flex;
  gap: 1rem;
}
.row > * {
  flex: 1;
}
```

Each child takes equal width. Add a second child, they split evenly. Add a fifth, they split into fifths.

**Wrapping with minimum width.**

```css
.cards {
  display: flex;
  flex-wrap: wrap;
  gap: 1rem;
}
.cards > * {
  flex: 1 1 250px;
}
```

Children are 250px wide minimum; extra space is distributed equally; they wrap onto new lines when the row is full. This is usually close enough to what you want for a responsive card grid. Grid can do better (see Part 4), but this pattern is fine for quick prototypes.

**Toolbar with pushed-right actions.**

```html
<div class="toolbar">
  <button>Save</button>
  <button>Cancel</button>
  <button style="margin-inline-start: auto">Help</button>
</div>
```

`margin-inline-start: auto` consumes all spare space before the item. Another way: add a spacer `<span>` with `flex: 1`, as in our opening example.

### The `order` property: the accessibility warning

`order` changes visual order without changing DOM order. That is a feature. It is also a footgun. Screen readers read in DOM order. Keyboard tab focus follows DOM order. If `order` creates a mismatch between visual and DOM order, keyboard users will experience the page as chaotic. The rule: **use `order` only for purely cosmetic reshuffling, never to put essential content out of reading order.** If you genuinely need to rearrange for responsive purposes, change DOM order with JavaScript or use Grid (which has better tools for this).

### The gotcha every Flexbox beginner hits

`align-items: stretch` is the default on the cross axis. If your flex container is a row (`flex-direction: row`, the default), the children will stretch to equal height — usually what you want. But if your items have `height: auto` and happen to have different content, you get visually uneven cards because each item stretches to the tallest *after* being sized.

If you want items to stretch, leave `align-items: stretch`. If you want them to shrink to content, set `align-items: start`. If you want all items to be the same visual height regardless of content, wrap them in a Grid (`display: grid; grid-auto-rows: 1fr`) instead — Flexbox cannot equal-height items independently of their content.

### Baseline alignment

The `align-items: baseline` value aligns items by their text baselines. Incredibly useful for form rows with labels and inputs, or for icons next to headings:

```css
.field {
  display: flex;
  align-items: baseline;
  gap: 0.5rem;
}
```

Now the label and the input text sit on the same baseline regardless of their individual heights, which looks infinitely more professional than naive middle alignment.

## Part 3: Grid — The Real Layout Engine

If Flexbox is a one-dimensional dance, Grid is a two-dimensional choreography. You declare a grid of rows and columns, and you place elements into named or numbered cells. The end.

[CSS Grid has been Baseline Widely Available since 2020](https://web.dev/baseline). It is the most powerful layout primitive in CSS, period. Here is the simplest possible Grid:

```css
.gallery {
  display: grid;
  grid-template-columns: 1fr 1fr 1fr;
  gap: 1rem;
}
```

Three equal columns. The `1fr` unit means "one fractional part of the free space." If the container is 900px wide and the gap is 16px, each column is `(900 - 32) / 3 = 289` pixels. Children of `.gallery` automatically arrange into the grid, three per row, wrapping as needed.

### Defining tracks

**`grid-template-columns`** and **`grid-template-rows`** accept a space-separated list of track sizes:

- **Fixed**: `100px`, `20rem`, `5vw`.
- **Fractional**: `1fr`, `2fr` — distributes free space.
- **Intrinsic**: `auto` (takes what the content needs), `min-content`, `max-content`.
- **Minimax**: `minmax(200px, 1fr)` — at least 200px, at most the full fraction.
- **Named lines**: `[start] 100px [middle] 1fr [end]` — named grid lines for placement.
- **Repeat**: `repeat(3, 1fr)`, `repeat(auto-fill, minmax(250px, 1fr))`.

The `auto-fill` / `auto-fit` + `minmax` pattern is one of the most useful constructions in modern CSS. It says "make as many columns as will fit, each at least 250 pixels wide, each stretching to fill free space." Apply it:

```css
.gallery {
  display: grid;
  grid-template-columns: repeat(auto-fill, minmax(250px, 1fr));
  gap: 1rem;
}
```

Now the gallery is fully responsive, with no media queries. At 1200px wide it shows four columns. At 800px wide, three. At 500px wide, two. At 300px wide, one. The browser figures out the right number based on the content.

The difference between `auto-fill` and `auto-fit`: `auto-fill` creates empty tracks if there are not enough items to fill; `auto-fit` collapses empty tracks. For most cases, either is fine. Use `auto-fit` if you want items to stretch when there are few of them; use `auto-fill` if you want a consistent grid size regardless of item count.

### Placing items

By default, items fill the grid in source order, starting at the top left, wrapping onto new rows as they exceed the column count. You can place individual items explicitly using `grid-column` and `grid-row`:

```css
.hero {
  grid-column: 1 / 4;     /* from column line 1 to column line 4 — i.e. spans three columns */
  grid-row: 1 / 3;        /* spans two rows */
}
```

Or using named lines:

```css
.container {
  grid-template-columns: [start] 1fr [main-start] 2fr [main-end] 1fr [end];
}
.main {
  grid-column: main-start / main-end;
}
```

Named lines make complex layouts readable. `grid-column: start / end` is clearer than `grid-column: 1 / 4`.

### `grid-template-areas` — the most designer-friendly layout syntax

`grid-template-areas` lets you draw your layout in ASCII art:

```css
.layout {
  display: grid;
  grid-template-columns: 200px 1fr 300px;
  grid-template-rows: auto 1fr auto;
  grid-template-areas:
    "header header header"
    "sidebar main aside"
    "footer footer footer";
  gap: 1rem;
  min-height: 100dvh;
}

.site-header { grid-area: header; }
.site-sidebar { grid-area: sidebar; }
.site-main { grid-area: main; }
.site-aside { grid-area: aside; }
.site-footer { grid-area: footer; }
```

That is a full page layout — header, three-column body, footer — in twelve readable lines. A designer can read it. A non-CSS-expert can understand it. You can redraw it in ten seconds when the design changes. The empty-string or `.` character represents empty cells.

We use this pattern for the capstone's main page layout in Day 15.

### Implicit tracks

If you place an item explicitly at `grid-row: 5`, but you have only declared three rows, the grid adds two implicit rows to accommodate it. Their size is controlled by `grid-auto-rows`:

```css
.container {
  grid-auto-rows: minmax(100px, auto);
}
```

This makes any implicit rows at least 100 pixels tall, growing with their content. Useful for grids where you know the column count but not the row count.

### `grid-auto-flow`

Controls how items auto-place. Values:

- `row` (default) — fill row by row, left to right.
- `column` — fill column by column, top to bottom.
- `dense` — allow later items to fill earlier gaps. Breaks source order; use with care.

`row dense` is powerful for image galleries where some items span multiple cells — it prevents holes in the layout. But like `order` in Flexbox, it decouples visual order from DOM order. Test keyboard navigation.

### Alignment

Grid supports the same alignment properties as Flexbox, but they apply across both axes independently:

- **`justify-items`** — align items within their cells along the inline (row) axis.
- **`align-items`** — along the block (column) axis.
- **`justify-content`** — position the whole grid within the container along the inline axis (matters when the grid is smaller than the container).
- **`align-content`** — along the block axis.
- **`place-items`** — shorthand: `place-items: center` is `align-items: center; justify-items: center`.
- **`place-content`** — shorthand for content alignment.

And on individual items:

- **`justify-self`**, **`align-self`**, **`place-self`** — override alignment for one item.

Combined:

```css
.centered {
  display: grid;
  place-items: center;
  min-height: 100dvh;
}
```

Two lines. Fully centered. No wrapper divs. This is the fifty-seven-line answer from 2017, condensed.

### Spanning across cells

```css
.hero {
  grid-column: span 2;  /* spans two columns starting from auto-placement */
  grid-row: span 3;
}
```

Or explicitly:

```css
.hero {
  grid-column: 1 / -1;  /* from first column line to last — full width */
}
```

The `-1` means "the last line." `1 / -1` is the idiom for "full width across all columns" regardless of the column count.

## Part 4: Subgrid — When A Child Grid Inherits Parent Tracks

By default, a grid item that itself has `display: grid` creates an entirely new grid, independent of its parent. This often creates alignment problems when you have nested content that needs to line up with the outer grid. Consider a common card layout:

```html
<ul class="cards">
  <li class="card">
    <img src="photo-1.jpg">
    <h3>A long title that might wrap</h3>
    <p>Body text.</p>
    <p class="price">$10</p>
  </li>
  <li class="card">
    <img src="photo-2.jpg">
    <h3>Short title</h3>
    <p>Body text.</p>
    <p class="price">$20</p>
  </li>
</ul>
```

You want the images to align, the titles to align (even when one wraps to two lines), the body to align, and the prices to align across cards. Without Subgrid, you have to carefully set fixed heights on each region and hope nothing overflows. With Subgrid, each card inherits its rows from the parent grid:

```css
.cards {
  display: grid;
  grid-template-columns: repeat(auto-fill, minmax(250px, 1fr));
  gap: 1rem;
}

.card {
  display: grid;
  grid-template-rows: subgrid;
  grid-row: span 4;          /* this card takes 4 rows of the parent grid */
  gap: 0.5rem;
}
```

Now every card spans the same four rows of the parent, and those four rows size to the tallest content across all cards. Row 1 fits the tallest image, row 2 fits the tallest title, etc. Every element aligns.

[Subgrid reached Baseline Newly Available in September 2023](https://web.dev/articles/css-subgrid). Browser support is universal in 2026 (Chrome 117+, Safari 16+, Firefox 71+). For the small fraction of users on older browsers, use `@supports` to fall back:

```css
@supports not (grid-template-rows: subgrid) {
  .card {
    display: flex;
    flex-direction: column;
    /* plain fallback */
  }
}
```

### What `subgrid` actually means

When you write `grid-template-rows: subgrid`, the element:

1. Passes the track definitions through from the parent.
2. Inherits the named lines from the parent.
3. Does *not* pass `gap` through — you can override it on the subgrid.
4. Re-numbers line numbers starting at 1 inside the subgrid.

Subgrid applies to one axis or both:

- `grid-template-rows: subgrid` — rows are subgridded, columns are not.
- `grid-template-columns: subgrid` — columns are subgridded.
- `grid: subgrid / subgrid` — both.

For the card layout above, subgridding rows is what we want. The column is defined by the card's placement in the outer grid (one cell wide), so there is only one column to share.

### The "full bleed" pattern

A common article layout is "constrained prose with occasional full-width media." Subgrid makes this trivial:

```css
.article {
  display: grid;
  grid-template-columns:
    [full-start] minmax(1rem, 1fr)
    [content-start] minmax(0, 65ch)
    [content-end] minmax(1rem, 1fr) [full-end];
}

.article > * {
  grid-column: content;
}

.article .fullbleed {
  grid-column: full;
}
```

Normal paragraphs sit in the `content` column (max 65 characters wide, centered). Images marked `class="fullbleed"` span from `full-start` to `full-end` — the entire container width. No negative margins. No absolute positioning. This is the layout pattern this very magazine uses for article bodies.

## Part 5: Reimplementing Bootstrap's Grid In Nine Lines

Bootstrap's twelve-column grid is one of the most recognisable CSS APIs in the world. Millions of developers know the `col-md-4` / `col-lg-6` vocabulary. Bootstrap's grid CSS is about 450 lines of generated declarations covering every combination of breakpoints, column counts, and offsets. Here is the entire thing, distilled:

```css
.row {
  display: grid;
  grid-template-columns: repeat(12, 1fr);
  gap: 1.5rem;
}

.col-1  { grid-column: span 1; }
.col-2  { grid-column: span 2; }
.col-3  { grid-column: span 3; }
.col-4  { grid-column: span 4; }
.col-5  { grid-column: span 5; }
.col-6  { grid-column: span 6; }
.col-7  { grid-column: span 7; }
.col-8  { grid-column: span 8; }
.col-9  { grid-column: span 9; }
.col-10 { grid-column: span 10; }
.col-11 { grid-column: span 11; }
.col-12 { grid-column: span 12; }
```

Nine lines plus twelve trivially-generated classes. You now have Bootstrap's grid. Add responsive variants if you need them:

```css
@media (min-width: 768px) {
  .col-md-6 { grid-column: span 6; }
  .col-md-4 { grid-column: span 4; }
}
```

But here is the bigger point: in 2026 **you probably do not want a twelve-column grid.** A twelve-column grid is a workaround for the days when you had to specify breakpoints manually and wanted a flexible mental model of "how many columns." With `repeat(auto-fill, minmax(Xpx, 1fr))`, you describe the intent ("items are at least X pixels wide, fill the row as much as fits") and the browser figures out the columns. That is usually better than "give me three columns on medium screens and four on large."

The twelve-column grid still has its uses — particularly for editorial layouts where the designer has strong opinions about column counts. But for data grids, card galleries, and dashboards, intrinsic responsive sizing with `minmax` is more robust.

## Part 6: Container Queries — The Breakthrough

Media queries have a flaw that no amount of discipline can paper over: **they only know about the viewport, not about the component's context.** If a card is placed in a narrow sidebar on a wide screen, media queries cannot adjust its layout; they only see that the browser window is 1400 pixels wide. The result: components styled for "large screens" that look cramped because they happened to land in a narrow container.

[Container queries reached Baseline Widely Available in August 2025](https://web.dev/baseline), and they fix this. The basic shape:

```css
.card-host {
  container-type: inline-size;
  container-name: card;
}

@container card (min-width: 500px) {
  .card {
    display: grid;
    grid-template-columns: 150px 1fr;
    gap: 1rem;
  }
}

@container card (max-width: 499px) {
  .card {
    display: flex;
    flex-direction: column;
  }
}
```

The `.card-host` declares itself as a query container. The `@container` rules then respond to **the container's width**, not the viewport's. Drop the same card into a narrow sidebar and it lays out vertically. Drop it into a wide main column and it lays out horizontally. Same HTML. Same CSS. Context-aware.

### Container types

- **`container-type: inline-size`** — you can query the container's inline size (width in LTR). The container becomes a query container on the inline axis only. Most common.
- **`container-type: size`** — query both axes. Requires the container to have a defined size on both axes, which means any contained element with auto-sizing will collapse. Use sparingly.
- **`container-type: normal`** — the element is not a size query container, but it can still be a style query container (see below).

### Container units

Container queries bring new length units that resolve relative to the query container:

- **`cqw`** — 1% of the container's inline size.
- **`cqh`** — 1% of the container's block size.
- **`cqi`** — 1% of the container's inline size (same as `cqw` for horizontal writing modes).
- **`cqb`** — 1% of the container's block size.
- **`cqmin`** — the smaller of `cqi` and `cqb`.
- **`cqmax`** — the larger.

Use case: fluid typography inside a card:

```css
.card-title {
  font-size: clamp(1rem, 5cqi, 2rem);
}
```

The title scales with the container, not the viewport. In a narrow sidebar the title is 1rem; in a wide column it is 2rem. No media queries.

### Style queries

Container queries also support `style()` tests, letting you branch on a custom property:

```css
@container style(--variant: loud) {
  .card {
    background: var(--brand);
    color: white;
    font-weight: 700;
  }
}
```

Set `--variant: loud` on an ancestor and the card becomes loud. Set it elsewhere and it is quiet. This is a lightweight way to theme subtrees without class-based variants. [Style queries shipped in Safari 18.0 in 2024 and are in Interop 2026](https://webkit.org/blog/17818/announcing-interop-2026/) for full cross-browser consistency.

### The "container-less" gotcha

A container query only affects an element that is *inside* the query container. Put the `@container` rule in your stylesheet but forget to set `container-type` on an ancestor, and the rule silently has no effect. Make sure your query container is correctly established.

### Replacing most of your media queries

Rules of thumb in 2026:

- **Media queries** for viewport-level things: dark mode (`prefers-color-scheme`), reduced motion (`prefers-reduced-motion`), print (`print`), orientation (`orientation`).
- **Container queries** for component-level layout.
- **Intrinsic sizing** (`minmax`, `clamp`) for things that should be fluid regardless of context.

Most of your CSS will not need a `@media` rule at all. That is the goal.

## Part 7: The Few Cases Where Media Queries Still Matter

Not every responsive concern is about size. Media queries remain the right tool for:

### `prefers-color-scheme`

```css
@media (prefers-color-scheme: dark) {
  :root { --bg: #111; --text: #eee; }
}
```

Though in 2026 we usually use `light-dark()` in a declaration instead, which is simpler. We will cover this in Day 6.

### `prefers-reduced-motion`

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
  }
}
```

The user has asked their OS to reduce motion; respect it. This is one of the two legitimate `!important` uses in our entire stylesheet.

### `prefers-contrast`

`no-preference`, `more`, `less`, `custom`. High-contrast users benefit from sturdier borders and stronger colour differences. Check the user's preference and adjust.

### `prefers-reduced-transparency`

For users who find frosted-glass effects disorienting. `backdrop-filter: blur()` can be disabled when this media feature is `reduce`.

### `print`

The rules for printing the page. Hide navigation, expand collapsed content, flatten layouts. This is often forgotten, but a printed page is often the "final form" of an article.

```css
@media print {
  .no-print { display: none; }
  body { max-width: none; }
  a::after { content: " (" attr(href) ")"; }
}
```

The last line is clever: for print, it appends the URL after every link. This is how a printed webpage should behave.

### `orientation`

`portrait` or `landscape`. Useful for a handful of cases — photo galleries, video players — but rarely the right breakpoint.

### `hover` and `pointer`

`hover: hover` targets devices where hovering is possible (a mouse). `hover: none` is a touch device. `pointer: fine` is a precise pointer (mouse); `pointer: coarse` is a touch finger. These let you only show hover effects on mouse-capable devices:

```css
@media (hover: hover) {
  .button:hover {
    background: var(--brand-hover);
  }
}
```

On a touch device, the hover state is never shown because it is triggered on tap, which is disruptive.

### Responsive viewport queries

If you need them — and increasingly you do not — use logical property variants: `(min-width: 48em)` rather than `(min-width: 768px)`, so that zooming (which changes rem and em but not px) scales the breakpoint.

## Part 8: Gap, And The End Of Margin-Based Spacing

`gap` is the one feature that does more than any other to clean up a stylesheet. Before `gap`, spacing between flex or grid children meant either `margin` on every child (with a "strip the last child's margin" hack), or padding on the container with collapsing margins on the children. Both were fragile.

`gap` works in Grid (since forever), Flexbox ([since 2022](https://web.dev/baseline)), and multi-column layout:

```css
.toolbar { display: flex; gap: 0.5rem; }
.grid    { display: grid; gap: 1rem 2rem; }   /* row-gap column-gap */
```

`gap` is margin-collapsing-free, which is to say it just works. No extra first-child or last-child rules. We write `margin: 0` on everything in our reset and use `gap` everywhere for spacing inside Flex and Grid containers.

For spacing in block layout (prose, articles), the "owl selector" is the right tool:

```css
.flow > * + * {
  margin-block-start: 1rem;
}
```

This adds a top margin to every child *except the first*. It is Heydon Pickering's classic pattern and it still works in 2026. `.flow` is a class you apply to anything whose children should be spaced vertically.

Combine these two patterns — `gap` in Flex/Grid, owl selector in block flow — and you will rarely touch `margin` on individual elements again.

## Part 9: The Masonry Saga

If you have been around the modern CSS scene, you have heard about **masonry** — the Pinterest-style layout where items pack tightly without row alignment, filling gaps organically. The browsers have been arguing about how to add this to CSS for five years, and as of April 2026, the feature is still not Baseline.

The short version: WebKit proposed extending Grid with `grid-template-rows: masonry`. Chrome objected on performance grounds and proposed `display: masonry` as a separate layout mode. After years of back-and-forth, [the CSS Working Group resolved in January 2025 to reuse grid template properties](https://css-tricks.com/masonry-layout-is-now-grid-lanes/) with a new value called `grid-lanes`, with `display: grid-lanes` as the opt-in. [Safari Technology Preview 234 shipped `display: grid-lanes` in December 2025](https://webkit.org/blog/17660/introducing-css-grid-lanes/), and mainline Safari 26 has it. Chrome and Firefox are behind flags as of early 2026.

What this means for you, today:

- **Do not ship masonry as a production-critical layout.** Most of your users do not have it yet.
- **You can write it with progressive enhancement.** Use `@supports` to detect and enhance:

```css
.gallery {
  display: grid;
  grid-template-columns: repeat(auto-fill, minmax(200px, 1fr));
  gap: 1rem;
}

@supports (grid-template-rows: masonry) {
  .gallery {
    display: grid-lanes;
    grid-template-rows: masonry;
  }
}
```

- **Avoid the CSS Columns hack for new projects.** `columns: 3` gives you masonry-like packing, but it orders items column-by-column (top of column 1, then top of column 2...), which breaks both DOM order and keyboard navigation.

We will revisit masonry once it is Baseline. Probably next year.

## Part 10: The Pattern Library

Here are the layouts you will actually build on a real site. We cover each as a complete, working snippet. Paste them into a stylesheet.

### Centered page with max width

```css
.page {
  max-width: min(80rem, 100% - 2rem);
  margin-inline: auto;
}
```

`min()` gives you "80rem OR the full width minus 2rem of breathing room, whichever is smaller." Works at every viewport. Replaces three media queries.

### Sidebar + main layout

```css
.layout {
  display: grid;
  grid-template-columns: minmax(15rem, 20rem) 1fr;
  gap: 2rem;
}

@container (max-width: 40rem) {
  .layout {
    grid-template-columns: 1fr;
  }
}
```

Sidebar between 15rem and 20rem wide; main fills the rest. On narrow containers, stack. Note we used a container query, not a media query — the layout responds to its container, not the viewport.

### Holy grail (header, sidebar, main, aside, footer)

```css
.holy-grail {
  display: grid;
  grid-template-columns: 15rem 1fr 15rem;
  grid-template-rows: auto 1fr auto;
  grid-template-areas:
    "header header header"
    "sidebar main aside"
    "footer footer footer";
  min-height: 100dvh;
  gap: 1rem;
}

.hg-header { grid-area: header; }
.hg-sidebar { grid-area: sidebar; }
.hg-main { grid-area: main; }
.hg-aside { grid-area: aside; }
.hg-footer { grid-area: footer; }

@container (max-width: 50rem) {
  .holy-grail {
    grid-template-columns: 1fr;
    grid-template-areas:
      "header"
      "main"
      "sidebar"
      "aside"
      "footer";
  }
}
```

A decade ago this pattern had its own name because it was famously hard to implement. Now it is twenty lines of CSS.

### Auto-responsive card grid

```css
.cards {
  display: grid;
  grid-template-columns: repeat(auto-fill, minmax(min(100%, 16rem), 1fr));
  gap: 1.5rem;
}
```

No media queries. The `min(100%, 16rem)` guards against overflow when the container is narrower than 16rem — a subtle detail that prevents horizontal scroll on mobile.

### Media object (image-plus-text)

```css
.media {
  display: grid;
  grid-template-columns: auto 1fr;
  gap: 1rem;
  align-items: start;
}
```

Image on the left, text on the right. `auto` for the image means "as wide as the image needs"; `1fr` for the text means "take the rest."

### Cluster (wrapping inline items)

```css
.cluster {
  display: flex;
  flex-wrap: wrap;
  gap: 0.5rem;
  align-items: center;
}
```

For tag lists, button groups, pill collections. Items sit in a row until they don't fit, then wrap.

### Stack (vertical flow with consistent spacing)

```css
.stack {
  display: flex;
  flex-direction: column;
}
.stack > * + * {
  margin-block-start: var(--stack-space, 1rem);
}
```

A vertical rhythm component. Override the spacing per-instance with `style="--stack-space: 2rem"`. This is Every Layout's Stack pattern and it is a work of art.

### Switcher (horizontal until too narrow, then vertical)

```css
.switcher {
  display: flex;
  flex-wrap: wrap;
  gap: 1rem;
}
.switcher > * {
  flex-grow: 1;
  flex-basis: calc((40rem - 100%) * 999);
}
```

This is black magic. The `calc((40rem - 100%) * 999)` computes a huge positive `flex-basis` (items prefer being huge, they wrap) when the container is narrower than 40rem, and a huge negative `flex-basis` (items prefer being zero, they stay in one row) when wider. The result: the layout switches from horizontal to vertical at a single threshold, without a media query, without a container query. From [Every Layout](https://every-layout.dev/layouts/switcher/), and we still have to stop and re-read the code every time.

### Sidebar (pushed to right, main stretches)

```css
.with-sidebar {
  display: flex;
  flex-wrap: wrap;
  gap: 1rem;
}
.with-sidebar > :first-child {
  flex-basis: 15rem;
  flex-grow: 1;
}
.with-sidebar > :last-child {
  flex-basis: 0;
  flex-grow: 999;
  min-inline-size: 50%;
}
```

The main content (second child) has `flex-grow: 999`, so it takes almost all the space. It also has `min-inline-size: 50%`, which forces it to wrap to a new row if it would be less than half the container. Sidebar goes next to it on wide screens; below it on narrow ones.

### Full-bleed content in a constrained article

```css
.article {
  display: grid;
  grid-template-columns:
    [full-start] minmax(1rem, 1fr)
    [content-start] min(65ch, 100% - 2rem)
    [content-end] minmax(1rem, 1fr) [full-end];
}
.article > * {
  grid-column: content;
}
.article .fullbleed {
  grid-column: full;
}
```

Standard prose sits in a 65ch-max column, centered. Elements with `.fullbleed` (typically hero images, quotes, code blocks) span the entire container width.

### Sticky header

```css
.site-header {
  position: sticky;
  top: 0;
  z-index: 10;
  background: var(--bg);
  backdrop-filter: blur(8px);
}
```

Sticky positioning — `position: sticky` — is Grid-and-Flexbox-compatible and works inside any scrollable container. The `backdrop-filter` gives the frosted-glass effect when content scrolls behind. Respect `prefers-reduced-transparency` in users' accessibility settings.

## Part 11: The Viewport Units You Need To Know

Viewport units have multiplied since the original `vw` / `vh`. In 2026, use:

- **`vw`**, **`vh`** — 1% of the viewport width/height, **assuming full viewport size**. On mobile, when the browser chrome (address bar) shows and hides dynamically, `vh` represents the "large viewport" size, meaning content may be hidden under the chrome.
- **`dvw`**, **`dvh`** — "dynamic" viewport. Accounts for the chrome being shown or hidden. `100dvh` always matches the actual visible viewport.
- **`svw`**, **`svh`** — "small" viewport. Assumes the chrome is always visible (maximum chrome, minimum content).
- **`lvw`**, **`lvh`** — "large" viewport. Assumes chrome is hidden (identical to `vw`/`vh`).

**Use `dvh` for "full screen" sections.** `100vh` on mobile causes a nasty jumping effect when the address bar hides. `100dvh` prevents this.

- **`vmin`** — the smaller of `vw` and `vh`. Useful for elements that should scale with the smaller dimension.
- **`vmax`** — the larger.

## Part 12: Aspect Ratio — Goodbye Padding-Bottom Hack

`aspect-ratio` has been Baseline Widely Available since 2023. It replaces the notorious "padding-bottom: 56.25%" hack for maintaining aspect ratios:

```css
.video-wrapper {
  aspect-ratio: 16 / 9;
}
```

That is it. The element maintains a 16:9 aspect ratio regardless of its width. Combine with `<iframe>` or `<video>` for responsive embeds:

```html
<div class="embed">
  <iframe src="..." title="..."></iframe>
</div>
```

```css
.embed {
  aspect-ratio: 16 / 9;
  width: 100%;
}
.embed > iframe {
  width: 100%;
  height: 100%;
  border: 0;
}
```

For images, set `aspect-ratio` to prevent layout shift while the image loads:

```css
img {
  aspect-ratio: attr(width) / attr(height);
}
```

This relies on `attr()` in values, which is rolling out but not fully Baseline. A safer version is to write the aspect ratio in CSS:

```css
.portrait-img { aspect-ratio: 2 / 3; }
.landscape-img { aspect-ratio: 3 / 2; }
```

Or, the simplest: always set `width` and `height` on `<img>` tags, which modern browsers use to compute aspect-ratio automatically. (We covered this in Day 2.)

## Part 13: Z-Index And Stacking Contexts

If there is one part of CSS where ASP.NET developers still panic, it is z-index. The rules:

1. `z-index` only applies to **positioned** elements (`position: relative`, `absolute`, `fixed`, `sticky`) or flex/grid items with an explicit `z-index`.
2. `z-index` creates a new **stacking context**. Children of a stacking context are ordered relative to each other, not globally.
3. `z-index: 9999` does not reliably put an element on top, because it only affects ordering within its parent's stacking context. If a parent has `z-index: 1`, your `9999` child is still below a sibling of the parent that has `z-index: 2`.

Modern alternative: **the top layer**. `<dialog>` with `showModal()` and elements with `popover` attribute render in the top layer, which is above everything regardless of z-index. For modals, menus, tooltips, and dropdowns, use the platform primitives and you can usually avoid z-index altogether. We covered these in Day 2.

When you do need z-index, keep them in a narrow range (1 through 10) and document what each value means in your tokens file:

```css
:root {
  --z-base: 0;
  --z-sticky: 10;
  --z-overlay: 100;
  --z-modal: 1000;  /* rarely needed since <dialog> uses top layer */
}
```

## Part 14: Debugging Layout

Three tools, free, in every browser's DevTools:

1. **Grid overlay.** Chrome, Edge, Firefox, and Safari all have a "Grid inspector" that draws the grid lines and cells on top of your page. Find your grid container in Elements, click the grid badge next to it, and the overlay appears. Essential when debugging `grid-template-areas` or missing items.

2. **Flexbox overlay.** Same idea for Flexbox. Click the "flex" badge next to a flex container to see axes, alignment, and gaps.

3. **Container query simulator.** In Chrome and Edge, you can resize individual containers to test container queries without resizing the whole viewport. Right-click a container in Elements → "Force container width" (or similar wording).

A lesser-known trick: **add a red outline to every element** temporarily to see the full layout:

```css
* { outline: 1px solid red !important; }
```

Paste it in DevTools' "Styles" panel as a scratch rule. You see every box. Remove when done.

## Part 15: Performance

CSS layout performance is not usually a problem. But a few things to know:

- **`will-change: transform`** hints to the browser to promote an element to its own compositor layer. Use on elements that *will* animate, not on everything. Overuse causes memory pressure.
- **`contain: layout`** tells the browser that the element's layout does not affect elements outside. Can be a significant optimisation for large grids. `contain: content` is a stronger version.
- **`content-visibility: auto`** — tells the browser to skip rendering work for offscreen content. [Baseline Widely Available since 2024](https://dev.to/marianocodes/baseline-web-features-you-can-safely-use-today-to-boost-performance-4bnb). Add it to long-list items and you can shave hundreds of milliseconds off initial rendering time. One caveat: search-in-page may not find content inside `content-visibility: auto` elements until they scroll into view, unless you use `hidden="until-found"` instead.
- **Avoid `display: none` for elements that animate in and out.** Use `display: grid` and a `display: none` that toggles with an animation via `@starting-style` and `display: allow-discrete`. Or use `<dialog>`, `popover`, or `hidden="until-found"`.

## Part 16: What You Should Do Now

Rewrite one layout. Not a new feature — an existing one you know well. Take a card grid, a sidebar, a form row, and rewrite it using the techniques in this article. Delete media queries where container queries would be better. Replace margin-based spacing with `gap`. Use `minmax` with `repeat(auto-fill, ...)` wherever you have a responsive grid with explicit breakpoints. Measure how many lines of CSS you delete.

When we did this on this magazine's codebase, we deleted more lines than we added, and the resulting layout was demonstrably better on unusual viewport sizes. You will likely have a similar experience.
