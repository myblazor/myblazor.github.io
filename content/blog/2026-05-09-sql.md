## Part 1: The World Before Relational Databases

### 1.1 The Data Problem

Long before you were worried about SQL injection, long before there were web applications, long before there was SQL at all, human beings had a data problem. Not a small data problem. Not a "my spreadsheet got big" data problem. A profound, civilization-scale data problem that threatened to make it impossible to run large organizations at all.

The core of the problem is this: any organization above a certain size needs to store, retrieve, and update enormous amounts of structured information. A bank needs to know who owns which account, what transactions have happened, what balances are current, and how those balances relate to each other. A railway company needs to know which trains run on which tracks, at what times, carrying which cargo, for which customers, with which freight rates, subject to which contracts. The United States Army in World War II needed to track millions of personnel records, equipment manifests, supply chains, and operational plans — all simultaneously, all accurately, all retrievably.

In the 1950s and 1960s, the dominant approach to this problem was to build custom file systems. Every application kept its own files, in its own format, organized in ways that made sense for that specific application's most common operations. If you needed to do something the file format wasn't designed for, you wrote new code. If you needed to combine information from two different applications — say, match payroll records with timekeeping records — you wrote an entire program to do it, because neither system had been designed to talk to the other.

This worked, after a fashion, in the way that duct tape works. It held things together. But it had catastrophic weaknesses.

**Data redundancy**: The same fact — say, a customer's address — might be stored in ten different files by ten different applications. When the customer moved, someone had to remember to update all ten. They never did. The applications then disagreed with each other forever.

**Data inconsistency**: Because there was no shared source of truth, the same information could exist in multiple contradictory versions simultaneously. Which version was correct? Nobody knew.

**Program-data dependence**: Every application contained hardcoded knowledge about the physical structure of the files it used. If you changed the file structure — adding a field, reorganizing a record — every application that touched that file broke, and had to be rewritten.

**Physical access complexity**: Retrieving anything non-trivial required writing programs that navigated the physical structure of the data. To find all customers in Texas who had purchased more than $500 of goods in the last quarter, you might have to write hundreds of lines of code to traverse linked records, follow pointer chains, and accumulate results.

This last problem is particularly important because it captures something fundamental: these early systems required programmers to be *navigators*. When Don Chamberlin — one of the two people who would later co-create SQL — received the ACM Turing Award in 1973, a database pioneer named Charles Bachman gave an address titled "The Programmer as Navigator." That phrase was celebratory. He was praising database programming as an art of skilled navigation through complex data structures.

Edgar Codd read this speech and found it appalling. The programmer should not have to navigate. The programmer should simply describe *what* they want, and the machine should figure out *how* to get it. That insight was the seed of everything that follows.

### 1.2 The Two Dominant Models Before SQL

To understand what Codd was reacting against, you need to understand the two models that dominated data storage in the late 1960s.

**The Hierarchical Model**

IBM's Information Management System (IMS), first deployed in 1968 to support the Apollo space program, used a hierarchical model. Data was organized in trees: parent records contained child records, which could contain their own children. This maps naturally to some real-world structures — a company has departments, departments have employees, employees have skills — but it falls apart immediately when reality refuses to be a tree.

Consider a hospital database. A patient has many doctors. A doctor has many patients. This is a many-to-many relationship — the fundamental structure of almost everything interesting in the real world — and it does not fit in a tree. In IMS, you handled this by duplicating data: each patient's record contained a copy of each doctor's record, and each doctor's record contained a copy of each patient's record. Update one side and forget to update the other, and your database lies.

Retrieval was navigation: to find all patients of Dr. Smith who had been admitted in January, you started at the root, found Dr. Smith's record, then traversed all child records looking for the ones with January admission dates. If you wanted to ask the same question from the opposite direction — "which doctors treated patients admitted in January?" — you rewrote the traversal.

**The Network Model**

The CODASYL (Committee on Data Systems Languages) network model, championed by Charles Bachman, was more flexible. Instead of strict tree hierarchies, records could participate in multiple "sets" — named relationships between record types. This allowed many-to-many relationships directly, which was an improvement.

But the complexity exploded. Programs navigating a CODASYL database had to explicitly manage set membership, follow pointers between records, and maintain awareness of the physical layout. Writing a database program meant understanding not just what data you wanted, but the exact sequence of pointer traversals needed to get it. A single, seemingly simple question like "how many orders were placed by customers in California last month?" might require pages of COBOL.

The programmer was, indeed, a navigator.

### 1.3 Codd's Insight: Relations and Algebra

Edgar Frank Codd was born on the Isle of Portland, Dorset, England, on August 19, 1923. He studied mathematics and chemistry at Oxford, flew with the Royal Air Force Coastal Command during World War II, and eventually joined IBM in 1948. By the mid-1960s he was working in computer science, and by 1967 he had transferred to IBM's San Jose Research Laboratory where he began seriously developing his ideas about data.

In June 1970, Codd published "A Relational Model of Data for Large Shared Data Banks" in *Communications of the ACM* (Volume 13, Number 6, pages 377–387). It is one of the most important computer science papers ever written, and its practical impact dwarfs virtually every other theoretical paper in the field.

The core ideas were these:

**All data should be represented as relations (tables)**. A relation is a mathematical concept from set theory: it is a set of tuples, where each tuple has the same attributes. In practical terms, this means a table of rows and columns. Every row represents one "fact" of a certain type. Every column has a consistent data type and meaning.

**Data independence**: Applications should interact with a logical view of the data, completely insulated from how that data is physically stored. If the database reorganizes its internal file structures for performance reasons, the application should not need to change. Codd called this "data independence" and it was, at the time, completely radical.

**Querying should be declarative**: Instead of writing programs that navigate data structures, users should be able to express *what* they want in a high-level language, and the database management system should figure out *how* to retrieve it. This is the property that makes SQL feel almost like English — you say what you want, not how to get it.

**Relations should be manipulated with algebra**: Codd grounded his model in relational algebra — a set of operations (selection, projection, join, union, difference, etc.) that take relations as input and produce relations as output. This gave the whole system a mathematical foundation that could be reasoned about formally.

Codd did not invent SQL. He proposed the theoretical framework. The actual language came later, from Don Chamberlin and Raymond Boyce at IBM's System R project (which began in 1974), and it was originally called SEQUEL (Structured English Query Language), later renamed SQL to avoid a trademark conflict. Larry Ellison, having read conference papers about System R before it shipped, built Oracle and actually shipped the first commercial relational database in 1979. IBM's DB2 followed in 1983.

In 1981, Codd received the ACM Turing Award — often called the Nobel Prize of Computer Science — for this work. In 2002, Forbes magazine listed the relational model as one of the most important innovations of the previous 85 years. In 2003, Codd died at the age of 79, having given the world perhaps the single most economically impactful idea in the history of computing.

The point of this history is not trivia. It is to show you that relational databases were born from a genuine intellectual crisis — a recognition that the dominant way of managing data was fundamentally broken — and that the solution came from mathematical first principles. When we talk about why certain SQL patterns are bad and others are good, we are echoing arguments that were settled, carefully and rigorously, half a century ago. The wisdom is old. The wounds from ignoring it are fresh.

## Part 2: The Relational Model From First Principles

### 2.1 What Is a Relation?

Before we write a single line of SQL, we need to understand what a *relation* is. Not the English-language meaning ("things relate to each other"), but the mathematical meaning.

In set theory, a **relation** between sets A and B is a subset of the Cartesian product A × B. The Cartesian product of two sets is the set of all possible ordered pairs from those sets. A relation picks some subset of those pairs — the ones where the relationship holds.

For example, if A = {Alice, Bob, Carol} and B = {Math, History, Science}, then the "studies" relation might be {(Alice, Math), (Alice, Science), (Bob, History), (Carol, Math), (Carol, History), (Carol, Science)}. Alice studies Math and Science; Bob studies History; Carol studies all three.

Codd generalized this beyond two sets. An **n-ary relation** is a subset of A₁ × A₂ × ... × Aₙ. Each Aᵢ is called a **domain** — a set of possible values for that attribute. An element of the relation is called a **tuple** (from the mathematical suffix for ordered groups: 2-tuple, 3-tuple, n-tuple).

Critically, a relation in Codd's model is a *set* of tuples. Sets have no order and no duplicates. This matters: a table in a relational database has no inherent row order (unless you explicitly sort it), and no two rows should be identical.

In practical terms:

- A **table** represents a relation.
- A **column** represents an attribute, with its domain (data type).
- A **row** represents a tuple — one instance of the relationship.
- A **cell** holds a single value from the column's domain.

This is a simple and beautiful structure. Its simplicity is not a limitation; it is a feature. By reducing everything to tables of typed values, the model becomes uniformly queryable, combinable, and understandable.

### 2.2 Keys: How We Identify Things

Because a relation is a set of tuples, and sets have no duplicates, every tuple in a relation must be distinguishable from every other tuple. The mechanism for this is the **key**.

**Superkey**: Any set of attributes whose values uniquely identify a tuple. If no two rows have the same combination of values for those attributes, the attribute set is a superkey.

**Candidate key**: A minimal superkey — a superkey with no redundant attributes. If {Name, BirthDate} is a superkey but {Name} alone is not (because two people can have the same name), then {Name, BirthDate} might be a candidate key. But if {BirthDate} alone were a superkey (every person has a unique birthdate — unlikely but hypothetical), then {Name, BirthDate} would not be a candidate key because it is not minimal.

**Primary key**: One chosen candidate key, which we designate as the official identifier for rows in the table. The primary key must never be NULL and must never change (or change very rarely). In practice, we often use a surrogate key — an artificial integer or GUID that has no real-world meaning — as the primary key, to avoid all the complications that arise from natural keys (names change, IDs get reassigned, etc.).

**Foreign key**: An attribute (or set of attributes) in one table that references the primary key of another table. Foreign keys enforce **referential integrity** — they ensure that the reference actually goes somewhere. You cannot have a `CustomerID` in an Orders table that does not correspond to any row in the Customers table, if that column is a foreign key.

Let's make this concrete:

```sql
-- Customers table
CREATE TABLE Customers (
    CustomerID   INT           NOT NULL,  -- Primary key
    Name         NVARCHAR(200) NOT NULL,
    Email        NVARCHAR(200) NOT NULL,
    CreatedAt    DATETIME2     NOT NULL DEFAULT GETUTCDATE(),
    CONSTRAINT PK_Customers PRIMARY KEY (CustomerID),
    CONSTRAINT UQ_Customers_Email UNIQUE (Email)
);

-- Orders table
CREATE TABLE Orders (
    OrderID     INT       NOT NULL,    -- Primary key
    CustomerID  INT       NOT NULL,    -- Foreign key → Customers.CustomerID
    OrderDate   DATETIME2 NOT NULL DEFAULT GETUTCDATE(),
    TotalAmount DECIMAL(18, 2) NOT NULL,
    CONSTRAINT PK_Orders PRIMARY KEY (OrderID),
    CONSTRAINT FK_Orders_Customers FOREIGN KEY (CustomerID)
        REFERENCES Customers (CustomerID)
);
```

The `CustomerID` in `Orders` is a foreign key. The database will refuse to insert an order with a `CustomerID` that does not exist in `Customers`. It will also refuse to delete a customer if there are orders referencing them (unless you configure cascade rules). This is the database *enforcing your business rules* — not your application code, not a service layer, not a prayer — the database itself will reject invalid data.

This enforcement is one of the greatest advantages of relational databases over application-level data management. Applications have bugs. Deployments get messy. People write ad-hoc SQL scripts against production. The database's constraints are the last line of defense, and they hold regardless of how the data gets there.

### 2.3 What Are Domains?

A domain in Codd's model is the set of possible values an attribute can take. In SQL databases, domains are approximately equivalent to data types, plus any constraints you apply.

Choosing the right data types is not aesthetic. It is functional. Let's look at some common mistakes:

**Using VARCHAR for everything**

```sql
-- BAD
CREATE TABLE Users (
    UserID       VARCHAR(50),
    Age          VARCHAR(10),   -- Age is a number. Store it as a number.
    IsActive     VARCHAR(5),    -- "true", "True", "TRUE", "yes", "1", "Yes"... chaos.
    CreatedDate  VARCHAR(30)    -- "2024-01-15", "Jan 15 2024", "15/01/2024"... nightmare.
);
```

When you store ages as strings, you cannot do `WHERE Age > 21` without converting — and the conversion will fail for rows where someone stored "unknown" or "N/A". When you store booleans as strings, you get five different representations in the same column by the end of the first week. When you store dates as strings, sorting by date gives you alphabetical order, not chronological order. "2024-02-01" sorts before "2024-10-01" alphabetically — but it also happens to sort correctly there *by accident*. "Jan 15 2024" does not sort before "Mar 15 2024" alphabetically. You will be debugging this at 2am.

```sql
-- GOOD
CREATE TABLE Users (
    UserID       INT            NOT NULL,
    Age          TINYINT        NULL,       -- or calculate from BirthDate
    IsActive     BIT            NOT NULL DEFAULT 1,
    CreatedDate  DATETIME2(7)   NOT NULL DEFAULT GETUTCDATE()
);
```

**Using FLOAT for money**

```sql
-- TERRIBLE
TotalPrice FLOAT  -- Floating point numbers cannot represent 0.10 exactly in binary
```

Floating-point arithmetic is approximate. IEEE 754 floating-point cannot represent most decimal fractions exactly. 0.1 + 0.2 in floating point is 0.30000000000000004. For scientific computing where you care about the magnitude of a result, this is fine. For money, where you care about exact cent values, it is an accounting error waiting to happen.

```sql
-- CORRECT
TotalPrice DECIMAL(18, 2)  -- Exact decimal arithmetic, 2 decimal places
```

`DECIMAL(18, 2)` means "a number with up to 18 digits total, 2 of which are after the decimal point." 0.1 + 0.2 is exactly 0.30. Always.

**Not using NOT NULL**

Every column should be `NOT NULL` unless you have a specific, defensible reason for it to be nullable. NULL in SQL is not zero, not empty string, not false. NULL means "unknown" or "inapplicable." NULL arithmetic gives NULL. NULL comparisons give neither true nor false but unknown. Three-valued logic is a source of subtle bugs that are genuinely difficult to reason about.

When you encounter a NULL, it means "we don't know." That is sometimes the right answer. A customer's middle name might genuinely be unknown. A ship's arrival date is NULL until the ship arrives. But when in doubt, use NOT NULL with a reasonable default. The discipline of being explicit about which fields can genuinely be unknown will save you countless hours of debugging queries that "should" return rows but don't.

### 2.4 Normalization: The Art of Not Repeating Yourself

**Normalization** is the process of structuring a relational database schema to reduce data redundancy and improve data integrity. It was formalized by Codd himself, who introduced the concept of **normal forms** — progressively stricter criteria that a table design must satisfy.

The motivation for normalization is straightforward: if the same fact is stored in multiple places, it must be updated in multiple places when it changes. If any update misses even one copy, the database now contains contradictory information. Normalization moves toward a schema where each fact is stored in exactly one place.

Let's walk through the normal forms with a concrete example.

**The Starting Mess**

Imagine you are building a system for a small software consultancy. You start by dumping everything into one table:

```sql
-- Unnormalized table — everything in one place
CREATE TABLE ProjectData (
    RecordID         INT,
    ProjectName      NVARCHAR(200),
    ClientName       NVARCHAR(200),
    ClientAddress    NVARCHAR(500),
    ClientPhone      NVARCHAR(50),
    DeveloperName    NVARCHAR(200),
    DeveloperEmail   NVARCHAR(200),
    DeveloperSkill1  NVARCHAR(100),
    DeveloperSkill2  NVARCHAR(100),
    DeveloperSkill3  NVARCHAR(100),
    HoursWorked      INT,
    BillingRate      DECIMAL(10, 2),
    InvoiceDate      DATE
);
```

This table is a disaster waiting to happen. Here is why:

- If a client changes their phone number, you have to update every row that mentions that client. Miss one, and now your database disagrees with itself.
- If a developer has four skills, you either add a fourth column or lose the fourth skill. Your schema is fighting your data.
- If a project has zero invoice records yet, you cannot add the project at all (it would have NULL everywhere), or you add a row full of NULLs that you have to handle everywhere.
- Deleting an invoice might delete the last reference to a project, destroying project information.

These are known as **update anomalies**, **insertion anomalies**, and **deletion anomalies**, and they are the canonical symptoms of an unnormalized schema.

#### First Normal Form (1NF)

A table is in **First Normal Form** if:
1. Every column contains atomic (indivisible) values — no arrays, no comma-separated lists, no repeating groups.
2. Every row is uniquely identifiable (there is a primary key).
3. There are no repeating groups of columns.

Our table violates 1NF with `DeveloperSkill1`, `DeveloperSkill2`, `DeveloperSkill3`. These are a repeating group. We move to 1NF by splitting skills into a separate table:

```sql
-- After 1NF
CREATE TABLE Projects (
    ProjectID     INT           NOT NULL,
    ProjectName   NVARCHAR(200) NOT NULL,
    ClientName    NVARCHAR(200) NOT NULL,
    ClientAddress NVARCHAR(500) NOT NULL,
    ClientPhone   NVARCHAR(50)  NOT NULL,
    CONSTRAINT PK_Projects PRIMARY KEY (ProjectID)
);

CREATE TABLE ProjectDevelopers (
    ProjectID       INT           NOT NULL,
    DeveloperName   NVARCHAR(200) NOT NULL,
    DeveloperEmail  NVARCHAR(200) NOT NULL,
    HoursWorked     INT           NOT NULL,
    BillingRate     DECIMAL(10,2) NOT NULL,
    InvoiceDate     DATE          NOT NULL,
    CONSTRAINT PK_ProjectDevelopers PRIMARY KEY (ProjectID, DeveloperName)
);

CREATE TABLE DeveloperSkills (
    DeveloperName NVARCHAR(200) NOT NULL,
    Skill         NVARCHAR(100) NOT NULL,
    CONSTRAINT PK_DeveloperSkills PRIMARY KEY (DeveloperName, Skill)
);
```

Better — but we still have the `ClientName`/`ClientAddress`/`ClientPhone` repeated in every project row. And `DeveloperName`/`DeveloperEmail` are repeated across all projects a developer works on.

#### Second Normal Form (2NF)

A table is in **Second Normal Form** if it is already in 1NF and every non-key attribute is **fully functionally dependent** on the entire primary key.

This concept of "functional dependency" is central to normalization. Attribute B is *functionally dependent* on attribute A if knowing A's value tells you B's value exactly. In a properly designed Employees table, knowing the EmployeeID tells you the employee's name, because each employee ID refers to exactly one person.

In our `ProjectDevelopers` table, the primary key is `(ProjectID, DeveloperName)`. But `DeveloperEmail` depends only on `DeveloperName`, not on the full composite key. Knowing a developer's name tells you their email regardless of which project they're on. This is a **partial dependency**, and it violates 2NF.

Fix: extract entities into their own tables.

```sql
CREATE TABLE Clients (
    ClientID    INT           NOT NULL,
    ClientName  NVARCHAR(200) NOT NULL,
    Address     NVARCHAR(500) NOT NULL,
    Phone       NVARCHAR(50)  NOT NULL,
    CONSTRAINT PK_Clients PRIMARY KEY (ClientID)
);

CREATE TABLE Developers (
    DeveloperID   INT           NOT NULL,
    DeveloperName NVARCHAR(200) NOT NULL,
    Email         NVARCHAR(200) NOT NULL,
    CONSTRAINT PK_Developers PRIMARY KEY (DeveloperID)
);

CREATE TABLE Projects (
    ProjectID   INT           NOT NULL,
    ClientID    INT           NOT NULL,
    ProjectName NVARCHAR(200) NOT NULL,
    CONSTRAINT PK_Projects PRIMARY KEY (ProjectID),
    CONSTRAINT FK_Projects_Clients FOREIGN KEY (ClientID) REFERENCES Clients(ClientID)
);

CREATE TABLE ProjectAssignments (
    ProjectID   INT           NOT NULL,
    DeveloperID INT           NOT NULL,
    HoursWorked INT           NOT NULL,
    BillingRate DECIMAL(10,2) NOT NULL,
    InvoiceDate DATE          NOT NULL,
    CONSTRAINT PK_ProjectAssignments PRIMARY KEY (ProjectID, DeveloperID),
    CONSTRAINT FK_PA_Projects    FOREIGN KEY (ProjectID)   REFERENCES Projects(ProjectID),
    CONSTRAINT FK_PA_Developers  FOREIGN KEY (DeveloperID) REFERENCES Developers(DeveloperID)
);

CREATE TABLE DeveloperSkills (
    DeveloperID INT           NOT NULL,
    Skill       NVARCHAR(100) NOT NULL,
    CONSTRAINT PK_DeveloperSkills PRIMARY KEY (DeveloperID, Skill),
    CONSTRAINT FK_DS_Developers FOREIGN KEY (DeveloperID) REFERENCES Developers(DeveloperID)
);
```

This is already much better. Client information is stored once. Developer information is stored once. If a client's phone number changes, you update one row. If a developer changes their email, you update one row.

#### Third Normal Form (3NF)

A table is in **Third Normal Form** if it is in 2NF and there are no **transitive dependencies** — no non-key attribute depends on another non-key attribute.

A classic example: suppose we add a `DepartmentID` and `DepartmentName` to an Employees table:

```sql
-- Violates 3NF
CREATE TABLE Employees (
    EmployeeID     INT           NOT NULL,
    EmployeeName   NVARCHAR(200) NOT NULL,
    DepartmentID   INT           NOT NULL,
    DepartmentName NVARCHAR(200) NOT NULL,  -- depends on DepartmentID, not EmployeeID
    CONSTRAINT PK_Employees PRIMARY KEY (EmployeeID)
);
```

`DepartmentName` depends on `DepartmentID`, not directly on `EmployeeID`. This is a transitive dependency: `EmployeeID` → `DepartmentID` → `DepartmentName`. If the department renames itself, you must update every employee row in that department.

Fix: separate tables.

```sql
CREATE TABLE Departments (
    DepartmentID   INT           NOT NULL,
    DepartmentName NVARCHAR(200) NOT NULL,
    CONSTRAINT PK_Departments PRIMARY KEY (DepartmentID)
);

CREATE TABLE Employees (
    EmployeeID   INT           NOT NULL,
    EmployeeName NVARCHAR(200) NOT NULL,
    DepartmentID INT           NOT NULL,
    CONSTRAINT PK_Employees PRIMARY KEY (EmployeeID),
    CONSTRAINT FK_Employees_Departments FOREIGN KEY (DepartmentID) REFERENCES Departments(DepartmentID)
);
```

Now `DepartmentName` is stored once, in `Departments`. Rename the department, update one row.

#### Higher Normal Forms

There are higher normal forms — Boyce-Codd Normal Form (BCNF), Fourth Normal Form (4NF), Fifth Normal Form (5NF) — that address increasingly exotic types of anomalies. For the vast majority of practical database design, reaching 3NF is sufficient and appropriate. BCNF and beyond are primarily relevant in academic contexts or highly specialized schema designs.

A useful heuristic credited to Bill Kent (and sometimes attributed to others in various forms): **"Every non-key attribute must provide a fact about the key, the whole key, and nothing but the key."** This covers 1NF, 2NF, and 3NF in one memorable sentence.

#### When to Denormalize

Normalization is not an absolute law. High-read, low-write systems (analytics databases, reporting systems, data warehouses) frequently use **denormalized** schemas — deliberately redundant data — to improve query performance by reducing the number of joins required.

The OLAP (Online Analytical Processing) world uses schemas like **star schema** and **snowflake schema** that are intentionally denormalized. When you are running a query that aggregates billions of rows, joining multiple normalized tables is expensive, and keeping redundant data pre-joined can be the difference between a query that runs in seconds and one that runs in hours.

But this is a performance optimization made with full understanding of the tradeoffs. The starting point should always be normalization. You denormalize *deliberately*, with documentation, after profiling demonstrates the need. You do not start denormalized and add more redundancy by accident.

## Part 3: SQL — The Language That Runs the World

### 3.1 What SQL Is and What It Is Not

SQL (Structured Query Language, pronounced either "S-Q-L" or "sequel" depending on who you ask and how much you want to start a minor argument) is a domain-specific language for managing data held in a relational database management system (RDBMS). It was standardized by ANSI in 1986 and ISO in 1987, and has been updated many times since: SQL-89, SQL-92, SQL:1999, SQL:2003, SQL:2006, SQL:2008, SQL:2011, SQL:2016, SQL:2019, and SQL:2023.

That said, no database implements the SQL standard perfectly or completely. Every major RDBMS — Microsoft SQL Server, PostgreSQL, MySQL, Oracle, SQLite, MariaDB — has its own dialect of SQL with proprietary extensions, quirks, and deviations from the standard. The core SELECT/INSERT/UPDATE/DELETE plus JOINs and WHERE clauses are widely compatible. The moment you start using window functions, recursive CTEs, date functions, or string manipulation, you are in dialect territory.

This matters for .NET developers because:
- Development often uses SQLite (lightweight, file-based, great for testing).
- Small production deployments often use PostgreSQL.
- Enterprise deployments often use SQL Server.
- Legacy systems might use MySQL or Oracle.

Your SQL will need to be slightly different for each. Entity Framework Core handles most of this through provider-specific translations. If you write raw SQL, be aware of which system you are targeting.

SQL has several sub-languages:

**DDL (Data Definition Language)**: Creates, alters, and drops schema objects.
- `CREATE TABLE`, `ALTER TABLE`, `DROP TABLE`
- `CREATE INDEX`, `DROP INDEX`
- `CREATE VIEW`, `DROP VIEW`
- `CREATE PROCEDURE`, `DROP PROCEDURE`

**DML (Data Manipulation Language)**: Reads and writes data.
- `SELECT` — queries data
- `INSERT` — adds new rows
- `UPDATE` — modifies existing rows
- `DELETE` — removes rows

**DCL (Data Control Language)**: Manages permissions.
- `GRANT` — gives a permission to a user or role
- `REVOKE` — removes a permission

**TCL (Transaction Control Language)**: Manages transactions.
- `BEGIN TRANSACTION`
- `COMMIT`
- `ROLLBACK`
- `SAVEPOINT`

Most day-to-day application development deals with DML and TCL. DDL happens during migrations. DCL is infrastructure/ops. But understanding all four is important for a complete picture.

### 3.2 The SELECT Statement — Reading Data

`SELECT` is how you ask a relational database for information. It is, in the spirit of Codd's declarative vision, a statement of *what you want*, not *how to get it*.

The basic syntax:

```sql
SELECT column1, column2, ...
FROM   table_name
WHERE  condition
ORDER BY column1 [ASC|DESC]
```

Let's build up from simple to complex.

**The simplest select:**

```sql
SELECT * FROM Products;
```

This returns all columns and all rows from the Products table. The `*` means "all columns." This is fine for exploration in a query tool. In application code, you should almost never use `SELECT *`. Here is why:

1. You transfer data you don't need, wasting bandwidth and memory.
2. If someone adds a column to the table, your query suddenly returns that column, potentially breaking code that expected a specific number of columns.
3. In queries with joins, `SELECT *` returns columns from all joined tables, potentially including columns with the same name from different tables — only the last one survives in many frameworks.
4. The optimizer cannot always use covering indexes effectively when you ask for all columns.

**Select specific columns:**

```sql
SELECT ProductID, ProductName, Price
FROM   Products;
```

This returns exactly three columns. If we later add a Description column to Products, this query still returns exactly three columns. Nothing breaks.

**Filtering with WHERE:**

```sql
SELECT ProductID, ProductName, Price
FROM   Products
WHERE  Price > 100.00;
```

The `WHERE` clause is a predicate — a condition that evaluates to true, false, or NULL for each row. Only rows where the condition evaluates to TRUE are returned. (Rows where it evaluates to NULL are *not* returned — this is the three-valued logic mentioned earlier.)

**Multiple conditions:**

```sql
SELECT ProductID, ProductName, Price
FROM   Products
WHERE  Price > 100.00
  AND  CategoryID = 5;
```

```sql
SELECT ProductID, ProductName, Price
FROM   Products
WHERE  Price < 10.00
    OR Price > 500.00;
```

**NULL handling:**

```sql
-- This DOES NOT work:
SELECT * FROM Products WHERE Description = NULL;

-- This DOES work:
SELECT * FROM Products WHERE Description IS NULL;
SELECT * FROM Products WHERE Description IS NOT NULL;
```

Because NULL means "unknown," asking whether NULL equals NULL returns "unknown" (NULL), not true. SQL requires special syntax for NULL comparisons.

**Sorting:**

```sql
SELECT ProductID, ProductName, Price
FROM   Products
ORDER BY Price DESC, ProductName ASC;
```

`DESC` is descending (highest first). `ASC` is ascending (lowest first, the default). You can sort by multiple columns — the first column is the primary sort, the second is used as a tiebreaker, and so on.

**Limiting results:**

Different databases have different syntax for this:

```sql
-- SQL Server / T-SQL
SELECT TOP 10 ProductID, ProductName, Price
FROM Products
ORDER BY Price DESC;

-- SQL Server (also supports this ANSI syntax from SQL:2008)
SELECT ProductID, ProductName, Price
FROM Products
ORDER BY Price DESC
OFFSET 0 ROWS FETCH NEXT 10 ROWS ONLY;

-- PostgreSQL / MySQL / SQLite
SELECT product_id, product_name, price
FROM products
ORDER BY price DESC
LIMIT 10;

-- PostgreSQL / MySQL (with paging)
SELECT product_id, product_name, price
FROM products
ORDER BY price DESC
LIMIT 10 OFFSET 20;  -- Skip first 20 rows, return next 10
```

### 3.3 The JOIN — The Heart of Relational Querying

The JOIN is what makes the relational model powerful. Normalization spreads data across multiple tables; JOINs bring it back together for querying without requiring you to store it redundantly.

The concept: a JOIN combines rows from two tables based on a condition, typically a foreign key relationship.

**INNER JOIN** (most common):

```sql
SELECT o.OrderID, c.Name, o.OrderDate, o.TotalAmount
FROM   Orders  AS o
INNER JOIN Customers AS c ON o.CustomerID = c.CustomerID;
```

This returns all rows where there is a matching customer for the order. If an order has a CustomerID that doesn't exist in Customers — which shouldn't happen with proper foreign keys, but might happen in legacy systems — that order row is omitted.

The `AS` keyword creates an alias. `o` is now an alias for `Orders`, `c` for `Customers`. This is mandatory for readability once you have more than one table. Without aliases, you would write `Orders.OrderID`, `Customers.Name`, etc. — verbose and repetitive.

**LEFT JOIN** (or LEFT OUTER JOIN):

```sql
SELECT c.Name, COUNT(o.OrderID) AS OrderCount
FROM   Customers AS c
LEFT JOIN Orders AS o ON c.CustomerID = o.CustomerID
GROUP BY c.CustomerID, c.Name;
```

Returns all rows from the left table (Customers), and matching rows from the right table (Orders) where they exist. If a customer has no orders, they appear in the results with NULL for all Orders columns (here, `o.OrderID` is NULL, so `COUNT` returns 0 for them).

This is how you answer "show me all customers, including those with no orders."

**RIGHT JOIN**: Opposite of LEFT JOIN — returns all rows from the right table plus matching rows from the left. Less commonly used because you can always rewrite a RIGHT JOIN as a LEFT JOIN by swapping the table order.

**FULL OUTER JOIN**: Returns all rows from both tables, with NULLs where there is no match on either side. Useful for finding all mismatches between two tables. Not supported in all databases (MySQL didn't support it until version 8.0, and it still requires awkward syntax in some contexts).

**CROSS JOIN**: Returns the Cartesian product — every possible combination of rows from both tables. Usually accidental and catastrophic if your tables are large. But occasionally useful for generating combinations.

```sql
-- CROSS JOIN: intentional use case
-- Generate a calendar table for a month (all combinations of week numbers and day names)
SELECT week_numbers.num, day_names.name
FROM (VALUES (1),(2),(3),(4),(5)) AS week_numbers(num)
CROSS JOIN (VALUES ('Mon'),('Tue'),('Wed'),('Thu'),('Fri')) AS day_names(name);
```

**Multiple joins:**

```sql
SELECT 
    o.OrderID,
    c.Name        AS CustomerName,
    p.ProductName,
    oi.Quantity,
    oi.UnitPrice
FROM Orders AS o
INNER JOIN Customers    AS c  ON o.CustomerID = c.CustomerID
INNER JOIN OrderItems   AS oi ON oi.OrderID   = o.OrderID
INNER JOIN Products     AS p  ON p.ProductID  = oi.ProductID
WHERE o.OrderDate >= '2024-01-01'
ORDER BY o.OrderDate DESC, o.OrderID, oi.OrderItemID;
```

This query spans four tables and produces a flat result showing customer name, product name, quantity, and unit price for all orders in 2024. This is perfectly idiomatic SQL. The database's query optimizer will determine the most efficient execution plan — you just said what you want.

### 3.4 Aggregation

Aggregate functions operate on groups of rows and return a single value per group:

- `COUNT(*)` — number of rows
- `COUNT(column)` — number of non-NULL values in column
- `SUM(column)` — total
- `AVG(column)` — average
- `MIN(column)` — minimum value
- `MAX(column)` — maximum value

They are used with `GROUP BY`:

```sql
SELECT 
    c.Name,
    COUNT(o.OrderID)      AS NumberOfOrders,
    SUM(o.TotalAmount)    AS TotalSpend,
    AVG(o.TotalAmount)    AS AverageOrderValue,
    MIN(o.OrderDate)      AS FirstOrderDate,
    MAX(o.OrderDate)      AS LastOrderDate
FROM Customers AS c
INNER JOIN Orders AS o ON o.CustomerID = c.CustomerID
GROUP BY c.CustomerID, c.Name
ORDER BY TotalSpend DESC;
```

This gives you a customer analytics report: every customer, how many orders they placed, what they spent in total, what their average order was, and when their first and last orders were.

`HAVING` is like `WHERE` but for groups — it filters *after* aggregation:

```sql
-- Only customers who spent more than $1000 total
SELECT 
    c.Name,
    SUM(o.TotalAmount) AS TotalSpend
FROM Customers AS c
INNER JOIN Orders AS o ON o.CustomerID = c.CustomerID
GROUP BY c.CustomerID, c.Name
HAVING SUM(o.TotalAmount) > 1000.00
ORDER BY TotalSpend DESC;
```

A common mistake: putting aggregate conditions in `WHERE` instead of `HAVING`. `WHERE` filters rows before grouping. `HAVING` filters groups after aggregation. You cannot reference aggregates in a `WHERE` clause.

```sql
-- WRONG — cannot use aggregate in WHERE
SELECT c.Name, SUM(o.TotalAmount) AS TotalSpend
FROM Customers c JOIN Orders o ON o.CustomerID = c.CustomerID
WHERE SUM(o.TotalAmount) > 1000  -- ERROR
GROUP BY c.CustomerID, c.Name;

-- RIGHT — use HAVING for aggregate filters
SELECT c.Name, SUM(o.TotalAmount) AS TotalSpend
FROM Customers c JOIN Orders o ON o.CustomerID = c.CustomerID
GROUP BY c.CustomerID, c.Name
HAVING SUM(o.TotalAmount) > 1000;
```

### 3.5 Subqueries and CTEs

Sometimes you need to build a query on top of another query. SQL provides two primary mechanisms.

**Subqueries** (inline queries):

```sql
-- Find customers who have placed orders with total > $500 each
SELECT DISTINCT c.Name
FROM Customers AS c
WHERE c.CustomerID IN (
    SELECT o.CustomerID
    FROM Orders AS o
    WHERE o.TotalAmount > 500
);
```

The inner query runs first, produces a set of CustomerIDs, and then the outer query filters by that set. This is called a **non-correlated subquery** — it runs independently of the outer query.

A **correlated subquery** references the outer query:

```sql
-- Find customers whose latest order was in 2024
SELECT c.Name
FROM Customers AS c
WHERE (
    SELECT MAX(o.OrderDate)
    FROM Orders AS o
    WHERE o.CustomerID = c.CustomerID  -- reference to outer query
) >= '2024-01-01';
```

This runs the inner query once *per row* of the outer query. For large tables, correlated subqueries can be catastrophically slow. A JOIN-based rewrite is almost always faster.

**Common Table Expressions (CTEs)** are named temporary result sets that you define at the beginning of a query:

```sql
WITH CustomerTotals AS (
    SELECT 
        c.CustomerID,
        c.Name,
        SUM(o.TotalAmount) AS TotalSpend
    FROM Customers AS c
    JOIN Orders AS o ON o.CustomerID = c.CustomerID
    GROUP BY c.CustomerID, c.Name
),
TopCustomers AS (
    SELECT CustomerID, Name, TotalSpend
    FROM CustomerTotals
    WHERE TotalSpend > 1000
)
SELECT *
FROM TopCustomers
ORDER BY TotalSpend DESC;
```

CTEs make complex queries readable by breaking them into named pieces. You can reference a CTE multiple times in the same query. They also enable **recursive CTEs** for querying hierarchical data:

```sql
-- Hierarchical employee org chart: find all reports under a given manager
WITH OrgChart AS (
    -- Anchor: start with the top manager
    SELECT EmployeeID, Name, ManagerID, 0 AS Level
    FROM Employees
    WHERE ManagerID IS NULL  -- top of the hierarchy
    
    UNION ALL
    
    -- Recursive: find direct reports of each found employee
    SELECT e.EmployeeID, e.Name, e.ManagerID, oc.Level + 1
    FROM Employees AS e
    INNER JOIN OrgChart AS oc ON e.ManagerID = oc.EmployeeID
)
SELECT EmployeeID, Name, Level
FROM OrgChart
ORDER BY Level, Name;
```

This recursively walks the entire organizational hierarchy without you having to know its depth.

### 3.6 INSERT, UPDATE, DELETE

**INSERT**:

```sql
-- Single row
INSERT INTO Products (ProductName, Price, CategoryID)
VALUES ('Widget Pro', 49.99, 3);

-- Multiple rows
INSERT INTO Products (ProductName, Price, CategoryID)
VALUES 
    ('Gadget Plus', 29.99, 2),
    ('Thing Supreme', 99.99, 1),
    ('Doohickey Deluxe', 14.99, 4);

-- Insert from a query
INSERT INTO OrderArchive (OrderID, CustomerID, OrderDate, TotalAmount)
SELECT OrderID, CustomerID, OrderDate, TotalAmount
FROM Orders
WHERE OrderDate < '2020-01-01';
```

**UPDATE**:

```sql
-- Update specific rows (ALWAYS use WHERE unless you really mean to update all rows)
UPDATE Products
SET Price = Price * 1.10  -- 10% price increase
WHERE CategoryID = 5;

-- Update with a join (SQL Server syntax)
UPDATE p
SET p.Price = p.Price * 1.10
FROM Products AS p
INNER JOIN Categories AS c ON c.CategoryID = p.CategoryID
WHERE c.CategoryName = 'Premium';
```

**Warning about UPDATE without WHERE**: If you forget the WHERE clause on an UPDATE, you update every single row in the table. This is one of the most common and most devastating database accidents. Always double-check your WHERE clause before executing an UPDATE or DELETE in production. Many DBAs wrap their UPDATE/DELETE statements in transactions so they can check the row count before committing:

```sql
BEGIN TRANSACTION;

UPDATE Products
SET IsDiscontinued = 1
WHERE CategoryID = 7;  -- Did I get the right CategoryID?

-- Check how many rows were affected
SELECT @@ROWCOUNT;  -- SQL Server
-- If the number looks wrong, ROLLBACK; otherwise COMMIT;

COMMIT;
```

**DELETE**:

```sql
-- Delete specific rows
DELETE FROM Products
WHERE IsDiscontinued = 1 AND LastSoldDate < '2020-01-01';

-- Delete all rows (clears the table but preserves structure)
DELETE FROM Products;  -- Slow if table is large; logs each deletion

-- Faster alternative for clearing a table:
TRUNCATE TABLE Products;  -- Cannot be rolled back in some databases; resets identity counter
```

### 3.7 Transactions

A **transaction** is a sequence of database operations treated as a single unit of work. A transaction must satisfy the ACID properties:

**A — Atomicity**: Either all operations in the transaction succeed, or none of them do. If the transaction fails halfway through, all changes are rolled back to the starting state.

**C — Consistency**: A transaction takes the database from one valid state to another valid state. All constraints (foreign keys, NOT NULL, CHECK constraints, unique constraints) must hold true after the transaction completes.

**I — Isolation**: Concurrent transactions are isolated from each other. Partially completed transactions are not visible to other transactions (the exact behavior depends on the isolation level).

**D — Durability**: Once a transaction is committed, its changes are permanent — they survive system failures, power outages, and crashes.

The canonical example is a bank transfer:

```sql
BEGIN TRANSACTION;

-- Debit from source account
UPDATE Accounts
SET Balance = Balance - 100.00
WHERE AccountID = 1001;

-- Check we didn't overdraw
IF (SELECT Balance FROM Accounts WHERE AccountID = 1001) < 0
BEGIN
    ROLLBACK;
    RAISERROR('Insufficient funds', 16, 1);
    RETURN;
END;

-- Credit to destination account
UPDATE Accounts
SET Balance = Balance + 100.00
WHERE AccountID = 1002;

COMMIT;
```

If the server crashes after the debit but before the credit, the transaction is not committed. On restart, the database rolls back the incomplete transaction. Account 1001 never lost the $100. This is atomicity in action.

In C# with `System.Data.SqlClient` or the modern `Microsoft.Data.SqlClient`:

```csharp
using var connection = new SqlConnection(connectionString);
await connection.OpenAsync();

using var transaction = connection.BeginTransaction();
try
{
    using var debitCmd = new SqlCommand(
        "UPDATE Accounts SET Balance = Balance - @Amount WHERE AccountID = @Source",
        connection, transaction);
    debitCmd.Parameters.AddWithValue("@Amount", amount);
    debitCmd.Parameters.AddWithValue("@Source", sourceAccountId);
    await debitCmd.ExecuteNonQueryAsync();

    using var creditCmd = new SqlCommand(
        "UPDATE Accounts SET Balance = Balance + @Amount WHERE AccountID = @Dest",
        connection, transaction);
    creditCmd.Parameters.AddWithValue("@Amount", amount);
    creditCmd.Parameters.AddWithValue("@Dest", destAccountId);
    await creditCmd.ExecuteNonQueryAsync();

    await transaction.CommitAsync();
}
catch
{
    await transaction.RollbackAsync();
    throw;
}
```

## Part 4: Talking to a Database from ASP.NET — The Wrong Ways

This is where we get to the part of the article that will make some readers uncomfortable, because they will recognize their own code. That is fine. The goal is not to embarrass — it is to show the progression from "works on my machine" to "works in production without destroying everything."

### 4.1 The Horror of String Concatenation

Let us paint a picture. It is 2009. You have just discovered ASP.NET. You have a login form. You need to check if a username and password match a user in the database. You write this code. You are proud of it.

```csharp
// ❌ DO NOT DO THIS — EVER
protected void LoginButton_Click(object sender, EventArgs e)
{
    string username = UsernameTextBox.Text;
    string password = PasswordTextBox.Text;
    
    string connectionString = ConfigurationManager.ConnectionStrings["DefaultConnection"].ConnectionString;
    
    using (SqlConnection conn = new SqlConnection(connectionString))
    {
        conn.Open();
        
        // Build the query by concatenating strings
        string query = "SELECT COUNT(*) FROM Users WHERE Username = '" 
                       + username 
                       + "' AND Password = '" 
                       + password 
                       + "'";
        
        using (SqlCommand cmd = new SqlCommand(query, conn))
        {
            int count = (int)cmd.ExecuteScalar();
            if (count > 0)
            {
                // Logged in!
                Session["Username"] = username;
                Response.Redirect("~/Dashboard.aspx");
            }
            else
            {
                ErrorLabel.Text = "Invalid username or password.";
            }
        }
    }
}
```

This code "works." It will log people in. It will pass every happy-path test you can think of. You ship it.

And then someone types this in the username field:

```
' OR '1'='1
```

With any password at all, the SQL query becomes:

```sql
SELECT COUNT(*) FROM Users WHERE Username = '' OR '1'='1' AND Password = 'anything'
```

Wait — there's operator precedence to consider. AND binds more tightly than OR, so this is actually:

```sql
SELECT COUNT(*) WHERE Username = '' OR ('1'='1' AND Password = 'anything')
```

Hmm, that's not quite "works as expected either." Let's try the classic full bypass:

Username: `' OR 1=1 --`
Password: (anything)

The `--` is a SQL comment — everything after it is ignored. The query becomes:

```sql
SELECT COUNT(*) FROM Users WHERE Username = '' OR 1=1 --' AND Password = 'anything'
```

Which resolves to:

```sql
SELECT COUNT(*) FROM Users WHERE Username = '' OR 1=1
```

Since `1=1` is always true, this returns the count of ALL users in the table. If `count > 0` — and it will be, unless your users table is empty — the login succeeds. You are now logged in as the first user in the database, which is often the admin account.

Every single user account in your system is now compromised without the attacker knowing a single password.

### 4.2 Making It Worse — Information Extraction

A clever attacker is not just trying to log in. They want to steal your data. Let's say you have a product search:

```csharp
// ❌ VULNERABLE
string search = Request.QueryString["q"];
string query = $"SELECT ProductID, ProductName, Price FROM Products WHERE ProductName LIKE '%{search}%'";
```

The attacker visits: `https://yoursite.com/search?q=x' UNION SELECT 1, Username, Password FROM Users --`

The query becomes:

```sql
SELECT ProductID, ProductName, Price FROM Products WHERE ProductName LIKE '%x' 
UNION 
SELECT 1, Username, Password FROM Users --'%'
```

The `UNION` operator combines two SELECT results. The attacker's injected query returns the username and password of every user in your database, displayed on the product search results page alongside your products.

If your passwords are stored in plain text (another enormous mistake we will discuss), the attacker now has every username and password in your system. Your users probably reuse passwords. The attacker now has access to their email accounts, their banks, their everything.

And your code *worked perfectly* — it returned exactly what the SQL said to return. The problem is not a bug in the code's logic. The problem is that you treated untrusted user input as part of your SQL command, blurring the boundary between code and data.

### 4.3 The Sanitization Illusion

When developers discover SQL injection, the first instinct is often to "sanitize" the input — filter out dangerous characters. This approach is fundamentally flawed.

```csharp
// ❌ INSUFFICIENT — "Sanitization" is not the answer
string SanitizeInput(string input)
{
    // Remove or escape single quotes
    return input.Replace("'", "''");
}

string query = $"SELECT * FROM Users WHERE Username = '{SanitizeInput(username)}'";
```

Why is this wrong?

1. **You will always miss something.** SQL has many ways to express the same attack: Unicode variants of apostrophes, escape sequences, encoding tricks, alternate comment syntax (`/**/` vs `--`), stacked queries, numeric injections (where no quotes are needed at all).

2. **It only works for strings.** If you have a numeric parameter:

```csharp
// ❌ STILL VULNERABLE — no quotes to escape, but the attack still works
int id = int.Parse(Request.QueryString["id"]);  // Fine, it's parsed... but:
string query = $"SELECT * FROM Products WHERE ProductID = {id}";
```

If the URL is `?id=1 OR 1=1`, `int.Parse("1 OR 1=1")` throws an exception — but if you wrote:

```csharp
string idStr = Request.QueryString["id"];  // "1 OR 1=1"
string query = $"SELECT * FROM Products WHERE ProductID = {idStr}";
```

No quotes needed. No sanitization helps. The attack lands.

3. **Context varies.** The escaping rules that are correct for a string literal in a WHERE clause are different from the escaping rules for a table name, a column name, a LIKE pattern, or a dynamic ORDER BY clause. There is no single escaping function that is correct for all contexts.

4. **You will make mistakes.** Sanitization is manual, case-by-case, and error-prone. Parameterized queries are systematic, automatic, and context-aware. A systematic solution to a systematic problem is always more reliable than a manual one.

### 4.4 The Slightly-Less-Wrong But Still Wrong: String.Format and Interpolation

Some developers, aware that concatenation is dangerous, switch to `string.Format` or C# string interpolation, believing these are somehow safer. They are not. They are the same operation with different syntax.

```csharp
// ❌ STILL WRONG — same problem, different syntax
string query1 = string.Format(
    "SELECT * FROM Users WHERE Username = '{0}' AND Password = '{1}'",
    username, password);

string query2 = $"SELECT * FROM Users WHERE Username = '{username}' AND Password = '{password}'";
```

The SQL query produced by these statements is byte-for-byte identical to the concatenated version. The attacker's `' OR '1'='1` payload works identically in all three cases.

String interpolation looks modern and clean, which makes it especially dangerous — it feels like you are doing something deliberate and careful. You are not. You are building an injectable SQL string with prettier syntax.

## Part 5: Parameterized Queries — The Right Way

### 5.1 The Fundamental Insight

Parameterized queries (also called prepared statements in some contexts) solve the SQL injection problem at its root by fundamentally separating the SQL command from the data values.

Here is the key insight: SQL injection is possible because when you build a SQL query by concatenating user input into a string, the database receives *one string* and must figure out which parts are commands and which parts are data. And because user input can contain SQL syntax, it can "escape" from the data role and become part of the command.

With parameterized queries, you send two things to the database:

1. The SQL command template, with placeholders (`@ParamName` in SQL Server / T-SQL, `$1` or `@name` in PostgreSQL/Npgsql, `?` in ODBC/MySQL style).
2. The parameter values, separately.

The database receives these as two distinct entities. It parses the SQL template first, builds a query plan, and then substitutes the parameter values as *data* — never as SQL syntax. No matter what value a parameter contains, it cannot change the structure of the SQL command. The command template is set in stone.

```csharp
// ✅ CORRECT — parameterized query
string query = "SELECT COUNT(*) FROM Users WHERE Username = @Username AND Password = @Password";

using var connection = new SqlConnection(connectionString);
await connection.OpenAsync();

using var command = new SqlCommand(query, connection);
command.Parameters.AddWithValue("@Username", username);
command.Parameters.AddWithValue("@Password", password);

int count = (int)await command.ExecuteScalarAsync();
```

Now if the attacker sends `' OR '1'='1` as the username, here is what the database sees:

- SQL: `SELECT COUNT(*) FROM Users WHERE Username = @Username AND Password = @Password`
- Parameter `@Username`: value is the string `' OR '1'='1`
- Parameter `@Password`: value is whatever they typed

The database looks for a row where `Username` is literally the string `' OR '1'='1` — a username that almost certainly does not exist. There is no injection because the parameter value is never interpreted as SQL syntax.

### 5.2 Parameters in Detail: SQL Server / ADO.NET

Let us look at parameterized queries in depth using ADO.NET with SQL Server, the foundational API that everything in .NET data access builds on.

**The basic pattern:**

```csharp
public async Task<User?> GetUserByIdAsync(int userId)
{
    const string sql = @"
        SELECT UserID, Username, Email, CreatedAt
        FROM   Users
        WHERE  UserID = @UserID";

    await using var connection = new SqlConnection(_connectionString);
    await connection.OpenAsync();
    
    await using var command = new SqlCommand(sql, connection);
    command.Parameters.Add(new SqlParameter("@UserID", SqlDbType.Int) { Value = userId });
    
    await using var reader = await command.ExecuteReaderAsync();
    if (!await reader.ReadAsync())
        return null;
    
    return new User
    {
        UserID    = reader.GetInt32(0),
        Username  = reader.GetString(1),
        Email     = reader.GetString(2),
        CreatedAt = reader.GetDateTime(3)
    };
}
```

Several things to notice:

**The `@` prefix**: SQL Server uses `@ParameterName` syntax. The parameter names can be anything, but they must match between the SQL string and the parameter collection you add to the command.

**`SqlDbType` specification**: When using `new SqlParameter("@UserID", SqlDbType.Int)`, you are explicitly telling the database the data type of the parameter. This matters for two reasons: it prevents type coercion surprises (a string value being coerced into an integer in unexpected ways), and it allows the query plan to be efficiently cached by the database.

**`AddWithValue` vs explicit parameter creation**: You will see many examples using `command.Parameters.AddWithValue("@Username", value)`. This is convenient but has a gotcha: it infers the `SqlDbType` from the C# type at runtime, which can cause problems:

```csharp
// AddWithValue infers SqlDbType.NVarChar for string parameters, 
// but with the length of the actual string, not the column's length.
// This can cause index key length mismatches and poor plan caching.
command.Parameters.AddWithValue("@Username", "alice");  
// Actual: SqlDbType.NVarChar, size 5 — doesn't match column size of 200

// Better: specify the type and size explicitly
command.Parameters.Add(new SqlParameter("@Username", SqlDbType.NVarChar, 200) { Value = "alice" });
```

For high-performance applications, explicit parameter types improve query plan caching significantly. For simple CRUD applications, `AddWithValue` is adequate.

**Reader column access**: `reader.GetString(1)` accesses column by ordinal position (0-indexed). This is fast but fragile — if someone reorders the SELECT columns, your column indices are wrong. An alternative is `reader.GetString(reader.GetOrdinal("Username"))`, which looks up by name. Slower, but robust to column reordering.

### 5.3 Handling Multiple Parameters

```csharp
public async Task<int> CreateProductAsync(string name, decimal price, int categoryId)
{
    const string sql = @"
        INSERT INTO Products (ProductName, Price, CategoryID, CreatedAt)
        VALUES (@Name, @Price, @CategoryID, GETUTCDATE());
        SELECT SCOPE_IDENTITY();";

    await using var connection = new SqlConnection(_connectionString);
    await connection.OpenAsync();
    
    await using var command = new SqlCommand(sql, connection);
    command.Parameters.Add(new SqlParameter("@Name",       SqlDbType.NVarChar, 200) { Value = name });
    command.Parameters.Add(new SqlParameter("@Price",      SqlDbType.Decimal)       { Value = price, Precision = 18, Scale = 2 });
    command.Parameters.Add(new SqlParameter("@CategoryID", SqlDbType.Int)            { Value = categoryId });
    
    // ExecuteScalar returns the first column of the first row — our new identity value
    var result = await command.ExecuteScalarAsync();
    return Convert.ToInt32(result);
}
```

`SCOPE_IDENTITY()` returns the last identity value inserted in the current scope. This is how you get the auto-generated primary key back after an insert. (Do not use `@@IDENTITY` — it is affected by triggers and can return unexpected values. `SCOPE_IDENTITY()` is scoped to the current batch and trigger level.)

In PostgreSQL/Npgsql, the equivalent is:

```csharp
// PostgreSQL
const string sql = @"
    INSERT INTO products (product_name, price, category_id, created_at)
    VALUES (@Name, @Price, @CategoryID, NOW())
    RETURNING product_id";

await using var command = new NpgsqlCommand(sql, connection);
command.Parameters.AddWithValue("@Name",       NpgsqlDbType.Varchar,  name);
command.Parameters.AddWithValue("@Price",      NpgsqlDbType.Numeric,  price);
command.Parameters.AddWithValue("@CategoryID", NpgsqlDbType.Integer,  categoryId);

var result = await command.ExecuteScalarAsync();
return (int)result!;
```

### 5.4 UPDATE and DELETE with Parameters

```csharp
public async Task<bool> UpdateProductPriceAsync(int productId, decimal newPrice)
{
    const string sql = @"
        UPDATE Products
        SET    Price     = @NewPrice,
               UpdatedAt = GETUTCDATE()
        WHERE  ProductID = @ProductID";

    await using var connection = new SqlConnection(_connectionString);
    await connection.OpenAsync();
    
    await using var command = new SqlCommand(sql, connection);
    command.Parameters.Add(new SqlParameter("@NewPrice",   SqlDbType.Decimal) { Value = newPrice, Precision = 18, Scale = 2 });
    command.Parameters.Add(new SqlParameter("@ProductID",  SqlDbType.Int)     { Value = productId });
    
    int rowsAffected = await command.ExecuteNonQueryAsync();
    return rowsAffected == 1;  // We expect exactly one row to be updated
}

public async Task<bool> DeleteProductAsync(int productId)
{
    const string sql = "DELETE FROM Products WHERE ProductID = @ProductID";

    await using var connection = new SqlConnection(_connectionString);
    await connection.OpenAsync();
    
    await using var command = new SqlCommand(sql, connection);
    command.Parameters.Add(new SqlParameter("@ProductID", SqlDbType.Int) { Value = productId });
    
    int rowsAffected = await command.ExecuteNonQueryAsync();
    return rowsAffected == 1;
}
```

`ExecuteNonQueryAsync` returns the number of rows affected. For UPDATE and DELETE, checking this count lets you detect cases where no rows matched (the record didn't exist) or where too many rows were affected (your WHERE clause was wrong).

### 5.5 Reading Multiple Rows

```csharp
public async Task<List<Product>> GetProductsByCategoryAsync(int categoryId)
{
    const string sql = @"
        SELECT ProductID, ProductName, Price, CreatedAt
        FROM   Products
        WHERE  CategoryID = @CategoryID
        ORDER BY ProductName";

    var products = new List<Product>();
    
    await using var connection = new SqlConnection(_connectionString);
    await connection.OpenAsync();
    
    await using var command = new SqlCommand(sql, connection);
    command.Parameters.Add(new SqlParameter("@CategoryID", SqlDbType.Int) { Value = categoryId });
    
    await using var reader = await command.ExecuteReaderAsync();
    while (await reader.ReadAsync())
    {
        products.Add(new Product
        {
            ProductID   = reader.GetInt32(0),
            ProductName = reader.GetString(1),
            Price       = reader.GetDecimal(2),
            CreatedAt   = reader.GetDateTime(3)
        });
    }
    
    return products;
}
```

The `while (await reader.ReadAsync())` pattern is the idiomatic way to iterate over a SQL result set. `ReadAsync` moves to the next row and returns `true` until there are no more rows.

### 5.6 Handling NULLs in Parameters

NULL requires special treatment in ADO.NET:

```csharp
// ❌ WRONG — passing a null C# value to a SqlParameter
string? middleName = null;
command.Parameters.Add(new SqlParameter("@MiddleName", SqlDbType.NVarChar, 100) 
{ 
    Value = middleName  // C# null is NOT the same as SQL NULL
});
// This throws a SqlException when executed — the parameter has no value

// ✅ CORRECT — use DBNull.Value for SQL NULL
command.Parameters.Add(new SqlParameter("@MiddleName", SqlDbType.NVarChar, 100)
{
    Value = (object?)middleName ?? DBNull.Value
});

// Or with a helper extension method (define once, use everywhere):
public static object ToDbValue(this object? value) => value ?? DBNull.Value;

// Usage:
command.Parameters.Add(new SqlParameter("@MiddleName", SqlDbType.NVarChar, 100)
{
    Value = middleName.ToDbValue()
});
```

When *reading* potentially null values:

```csharp
// ❌ WRONG — throws InvalidCastException if value is NULL
string middleName = reader.GetString(3);

// ✅ CORRECT — check for DBNull before accessing
string? middleName = reader.IsDBNull(3) ? null : reader.GetString(3);

// Or use the reader's generic method that handles null:
string? middleName = reader[3] as string;  // returns null if DBNull
```

### 5.7 The IN Clause Problem

SQL parameterized queries have a well-known limitation: you cannot parameterize a list of values for an `IN` clause the way you might expect.

```csharp
// ❌ WRONG — this does NOT work
var ids = new[] { 1, 2, 3, 4, 5 };
string query = "SELECT * FROM Products WHERE ProductID IN (@IDs)";
command.Parameters.AddWithValue("@IDs", ids);  // Will NOT expand to IN (1,2,3,4,5)
```

This sends the parameter as a single value — some databases will interpret it as a comma-separated string, which will not match integer IDs at all.

The solutions:

**Option 1: Build the IN clause dynamically with numbered parameters:**

```csharp
public async Task<List<Product>> GetProductsByIdsAsync(IEnumerable<int> productIds)
{
    var idList = productIds.ToList();
    if (idList.Count == 0) return [];

    // Build: WHERE ProductID IN (@p0, @p1, @p2, ...)
    var paramNames = idList.Select((_, i) => $"@p{i}").ToList();
    string sql = $"SELECT ProductID, ProductName, Price FROM Products WHERE ProductID IN ({string.Join(", ", paramNames)})";

    await using var connection = new SqlConnection(_connectionString);
    await connection.OpenAsync();
    
    await using var command = new SqlCommand(sql, connection);
    for (int i = 0; i < idList.Count; i++)
    {
        command.Parameters.Add(new SqlParameter($"@p{i}", SqlDbType.Int) { Value = idList[i] });
    }
    
    var products = new List<Product>();
    await using var reader = await command.ExecuteReaderAsync();
    while (await reader.ReadAsync())
    {
        products.Add(MapProduct(reader));
    }
    return products;
}
```

This is safe. The query structure (`IN (@p0, @p1, @p2)`) is fixed at build time — the number of parameters is determined by your code, not by user input. Each value is separately parameterized.

**Option 2: Table-valued parameter (SQL Server):**

For large lists, building hundreds of parameters is inefficient. SQL Server supports Table-Valued Parameters (TVPs), which allow passing a table of values as a single parameter:

```sql
-- SQL Server: create a user-defined table type
CREATE TYPE IntList AS TABLE (Value INT NOT NULL);
```

```csharp
public async Task<List<Product>> GetProductsByIdsAsync(IEnumerable<int> productIds)
{
    var idList = productIds.ToList();
    if (idList.Count == 0) return [];

    // Build a DataTable to pass as a TVP
    var tvp = new DataTable();
    tvp.Columns.Add("Value", typeof(int));
    foreach (var id in idList)
        tvp.Rows.Add(id);

    const string sql = @"
        SELECT p.ProductID, p.ProductName, p.Price
        FROM   Products AS p
        INNER JOIN @IDs AS ids ON p.ProductID = ids.Value";

    await using var connection = new SqlConnection(_connectionString);
    await connection.OpenAsync();
    
    await using var command = new SqlCommand(sql, connection);
    var param = command.Parameters.AddWithValue("@IDs", tvp);
    param.SqlDbType = SqlDbType.Structured;
    param.TypeName  = "dbo.IntList";

    var products = new List<Product>();
    await using var reader = await command.ExecuteReaderAsync();
    while (await reader.ReadAsync())
    {
        products.Add(MapProduct(reader));
    }
    return products;
}
```

**Option 3: Use Dapper (the smart micro-ORM):**

Dapper handles IN-clause expansion transparently:

```csharp
// Dapper automatically expands IEnumerable to IN clause parameters
var products = await connection.QueryAsync<Product>(
    "SELECT * FROM Products WHERE ProductID IN @IDs",
    new { IDs = productIds }
);
```

Behind the scenes, Dapper generates `WHERE ProductID IN (@IDs0, @IDs1, @IDs2, ...)` with properly typed parameters — safe and correct.

## Part 6: The Case for Dapper

### 6.1 What Is Dapper?

ADO.NET parameterized queries work, but they are verbose. For every query, you write: open connection, create command, add parameters one by one, execute, read rows, map columns to properties, close everything. It is correct but tedious, and tedium leads to shortcuts which lead to bugs.

Dapper is a micro-ORM — a thin layer over ADO.NET that eliminates the boilerplate while keeping you in control of your SQL. It was created by the Stack Overflow engineering team and is open-source under the Apache 2.0 license.

Dapper does three things:

1. **Maps query results to C# objects automatically**, by matching column names to property names.
2. **Handles parameter binding automatically**, converting anonymous objects (`new { Username = "alice" }`) to properly typed parameters.
3. **Manages connections** — you provide the connection, Dapper handles opening/closing if needed.

It does *not* generate SQL for you. You write SQL. Dapper just removes the mapping code.

### 6.2 Dapper Basics

```xml
<!-- In your .csproj -->
<PackageReference Include="Dapper" Version="2.1.35" />
```

```csharp
using Dapper;

public class UserRepository
{
    private readonly string _connectionString;
    
    public UserRepository(string connectionString)
    {
        _connectionString = connectionString;
    }
    
    // Create a new connection for each operation
    private SqlConnection CreateConnection() => new(_connectionString);

    // Query single object
    public async Task<User?> GetByIdAsync(int userId)
    {
        using var conn = CreateConnection();
        return await conn.QuerySingleOrDefaultAsync<User>(
            "SELECT UserID, Username, Email, CreatedAt FROM Users WHERE UserID = @UserID",
            new { UserID = userId }
        );
    }
    
    // Query multiple objects
    public async Task<IEnumerable<User>> GetAllAsync()
    {
        using var conn = CreateConnection();
        return await conn.QueryAsync<User>(
            "SELECT UserID, Username, Email, CreatedAt FROM Users ORDER BY Username"
        );
    }
    
    // Query with parameters
    public async Task<IEnumerable<User>> SearchAsync(string searchTerm)
    {
        using var conn = CreateConnection();
        return await conn.QueryAsync<User>(
            "SELECT UserID, Username, Email FROM Users WHERE Username LIKE @Pattern OR Email LIKE @Pattern",
            new { Pattern = $"%{searchTerm}%" }  // The % is data, not structure — this is safe
        );
    }
    
    // Insert and get new ID
    public async Task<int> CreateAsync(string username, string email, string passwordHash)
    {
        using var conn = CreateConnection();
        return await conn.ExecuteScalarAsync<int>(
            @"INSERT INTO Users (Username, Email, PasswordHash, CreatedAt)
              VALUES (@Username, @Email, @PasswordHash, GETUTCDATE());
              SELECT CAST(SCOPE_IDENTITY() AS INT);",
            new { Username = username, Email = email, PasswordHash = passwordHash }
        );
    }
    
    // Update
    public async Task<int> UpdateEmailAsync(int userId, string newEmail)
    {
        using var conn = CreateConnection();
        return await conn.ExecuteAsync(
            "UPDATE Users SET Email = @Email, UpdatedAt = GETUTCDATE() WHERE UserID = @UserID",
            new { Email = newEmail, UserID = userId }
        );
    }
    
    // Delete
    public async Task<int> DeleteAsync(int userId)
    {
        using var conn = CreateConnection();
        return await conn.ExecuteAsync(
            "DELETE FROM Users WHERE UserID = @UserID",
            new { UserID = userId }
        );
    }
}
```

The key differences from raw ADO.NET:

- No `SqlCommand` construction or `SqlParameter` management.
- No `DataReader` loop — Dapper handles mapping automatically.
- Anonymous objects (`new { UserID = userId }`) serve as the parameter bag.
- Column names in SQL results map to property names in your C# class (case-insensitive by default).

Every Dapper query is still fully parameterized. The anonymous object approach is not string interpolation — Dapper converts the object's properties to properly typed `SqlParameter` objects. The protection is identical to hand-written ADO.NET parameters.

### 6.3 Dapper's Multi-Mapping

One of Dapper's powerful features is multi-mapping — populating multiple objects from a single join query:

```csharp
public async Task<IEnumerable<OrderWithCustomer>> GetOrdersWithCustomersAsync()
{
    using var conn = CreateConnection();
    
    const string sql = @"
        SELECT o.OrderID, o.OrderDate, o.TotalAmount,
               c.CustomerID, c.Name, c.Email
        FROM Orders AS o
        INNER JOIN Customers AS c ON c.CustomerID = o.CustomerID
        ORDER BY o.OrderDate DESC";
    
    return await conn.QueryAsync<Order, Customer, OrderWithCustomer>(
        sql,
        (order, customer) => new OrderWithCustomer(order, customer),
        splitOn: "CustomerID"  // The column where the second object starts
    );
}
```

Dapper splits the result row at the `CustomerID` column: columns before it map to `Order`, columns starting at `CustomerID` map to `Customer`. The lambda combines them.

### 6.4 Transactions with Dapper

```csharp
public async Task TransferFundsAsync(int fromAccountId, int toAccountId, decimal amount)
{
    using var conn = CreateConnection();
    await conn.OpenAsync();
    
    using var transaction = conn.BeginTransaction();
    try
    {
        await conn.ExecuteAsync(
            "UPDATE Accounts SET Balance = Balance - @Amount WHERE AccountID = @ID",
            new { Amount = amount, ID = fromAccountId },
            transaction
        );
        
        await conn.ExecuteAsync(
            "UPDATE Accounts SET Balance = Balance + @Amount WHERE AccountID = @ID",
            new { Amount = amount, ID = toAccountId },
            transaction
        );
        
        transaction.Commit();
    }
    catch
    {
        transaction.Rollback();
        throw;
    }
}
```

Pass the transaction as the `transaction` parameter to each Dapper call, and all operations participate in the same transaction.

## Part 7: The Case for Entity Framework Core

### 7.1 What is EF Core?

Entity Framework Core (EF Core) is the official .NET ORM — Object-Relational Mapper. It generates SQL for you based on your C# object models and LINQ queries. For most CRUD applications, you never write SQL at all.

EF Core sits at the opposite end of the abstraction spectrum from raw ADO.NET. With ADO.NET, you write SQL and ADO.NET executes it. With EF Core, you define your data model in C# and EF Core generates and executes the SQL.

EF Core handles parameterization automatically and completely. When you write:

```csharp
var user = await context.Users
    .Where(u => u.Username == username)
    .FirstOrDefaultAsync();
```

EF Core generates:

```sql
SELECT TOP(1) [u].[UserID], [u].[Username], [u].[Email], [u].[CreatedAt]
FROM [Users] AS [u]
WHERE [u].[Username] = @__username_0
```

Where `@__username_0` is a properly parameterized value. There is no concatenation. There is no injection risk.

### 7.4 Raw SQL in EF Core

Sometimes EF Core's LINQ translation is not efficient enough, or you need database-specific features. EF Core supports raw SQL — and when used correctly, it is still safe:

```csharp
// Safe raw SQL — parameters are still used
var minPrice = 10.00m;
var maxPrice = 100.00m;

var products = await _context.Products
    .FromSqlInterpolated(
        $"SELECT * FROM Products WHERE Price BETWEEN {minPrice} AND {maxPrice}")
    .ToListAsync();
```

`FromSqlInterpolated` uses C# string interpolation syntax but is **not** string concatenation. EF Core intercepts the interpolated values and converts them to SQL parameters. The generated SQL is:

```sql
SELECT * FROM Products WHERE Price BETWEEN @p0 AND @p1
```

With `@p0 = 10.00` and `@p1 = 100.00` as properly typed parameters.

**Never use `FromSqlRaw` with unparameterized user input:**

```csharp
// ✅ Safe — parameters used
var products = await _context.Products
    .FromSqlRaw("SELECT * FROM Products WHERE Price > {0}", minPrice)
    .ToListAsync();

// ❌ DANGEROUS — raw string interpolation into FromSqlRaw bypasses parameter protection
string userInput = Request.Query["search"];
var products = await _context.Products
    .FromSqlRaw($"SELECT * FROM Products WHERE ProductName = '{userInput}'")  // INJECTABLE
    .ToListAsync();
```

The distinction: `FromSqlInterpolated` takes an interpolated string and converts each interpolated value to a parameter. `FromSqlRaw` takes a plain string and format arguments — the string template itself must have `{0}`, `{1}` placeholders, not embedded C# values. Using string interpolation with `FromSqlRaw` bypasses the protection entirely.

## Part 8: Indexing — The Difference Between Fast and Unacceptable

### 8.1 Why Indexes Matter

A table with no indexes is a heap — rows stored in no particular order. To find all orders placed by customer 42, the database reads every single row in the Orders table and checks whether `CustomerID = 42`. This is called a **full table scan**, and for a small table it is fine. For a table with ten million rows, it is catastrophic.

An index is a separate data structure — typically a B-tree — that the database maintains in sorted order on one or more columns. With an index on `Orders.CustomerID`, finding all orders for customer 42 is a logarithmic operation (binary search through the B-tree), then a direct lookup. The difference between a table scan and an index seek can be the difference between seconds and milliseconds.

Here is a concrete illustration. Imagine your Orders table has 50 million rows. A full table scan reads all 50 million rows — if each row is 200 bytes, that is 10GB of data read from disk. A single index seek on `CustomerID`, which has 100,000 unique values, traverses maybe 17 levels of a B-tree and reads a few hundred rows. The ratio is 5,000,000:1.

### 8.2 Types of Indexes

**Clustered index**: Determines the physical order of rows in the table. A table can have only one clustered index (because the physical data can only be in one order). In SQL Server, the primary key is the clustered index by default. In MySQL/InnoDB, it is also the primary key. Rows are stored sorted by the clustered index key.

**Non-clustered index**: A separate structure sorted by the indexed columns, containing the index key values plus a pointer back to the actual row (via the clustered index key or a row ID). A table can have many non-clustered indexes.

**Unique index**: Enforces that no two rows have the same value(s) in the indexed columns. `UNIQUE` constraints on columns are implemented as unique indexes.

**Composite index**: An index on multiple columns. The order matters: a composite index on (LastName, FirstName) supports queries filtering by LastName alone, or by both LastName and FirstName, but does NOT help with queries filtering by FirstName alone (because the B-tree is sorted first by LastName).

**Covering index**: A non-clustered index that "covers" a query — it contains all the columns the query needs in the index itself, so the database can satisfy the query entirely from the index without touching the base table. This is the most powerful performance optimization you can make with indexing.

```sql
-- This query:
SELECT Username, Email FROM Users WHERE Username LIKE 'j%'

-- Is covered by this index:
CREATE INDEX IX_Users_Username_Email ON Users (Username) INCLUDE (Email);
```

The `INCLUDE` clause adds Email to the index's leaf pages without making it a sort key. The index is sorted by Username (enabling the LIKE 'j%' seek), and Email is right there in the same index, so no lookup to the base table is needed.

### 8.3 When to Create Indexes

Create an index when you frequently:

- Filter by a column in a WHERE clause
- Join on a column (foreign keys should almost always be indexed)
- Sort by a column in an ORDER BY clause
- Look up a column in a GROUP BY clause

Do not create an index on:

- Tables with very few rows (the optimizer will prefer a table scan anyway)
- Columns with very low cardinality (a `Gender` column with values 'M'/'F' is a poor index candidate — you are still scanning half the table)
- Columns that are written to very frequently (indexes have a write cost — every INSERT, UPDATE, DELETE must also update all indexes on the table)

The correct answer to "should I add an index?" is always: measure first. Enable query execution plan analysis in your database, identify slow queries, and add indexes to address specific bottlenecks. Adding indexes speculatively can hurt performance by slowing down writes.

```sql
-- SQL Server: check index usage
SELECT 
    OBJECT_NAME(ix.object_id) AS TableName,
    ix.name AS IndexName,
    ixu.user_seeks,
    ixu.user_scans,
    ixu.user_lookups,
    ixu.user_updates
FROM sys.dm_db_index_usage_stats AS ixu
JOIN sys.indexes AS ix ON ixu.index_id = ix.index_id AND ixu.object_id = ix.object_id
WHERE OBJECTPROPERTY(ix.object_id, 'IsUserTable') = 1
ORDER BY ixu.user_seeks + ixu.user_scans DESC;
```

An index with zero seeks and thousands of updates is costing you write performance with no read benefit. Drop it.

### 8.4 The N+1 Problem in ORM Queries

A common and devastating performance issue with ORMs is the N+1 problem. Here is how it happens:

```csharp
// ❌ N+1 Problem — loads all orders, then makes one query per order for the customer
var orders = await _context.Orders.ToListAsync();  // 1 query: SELECT * FROM Orders

foreach (var order in orders)
{
    Console.WriteLine($"Order {order.OrderID} by {order.User.Username}");
    // ↑ Each access to order.User triggers ANOTHER query:
    // SELECT * FROM Users WHERE UserID = X
}
```

If you have 1000 orders, this makes 1001 queries. For 10,000 orders, it makes 10,001 queries. The application appears to work fine on small test datasets and then falls apart in production.

The fix is **eager loading** with `.Include()`:

```csharp
// ✅ Correct — loads orders and users in a single JOIN query
var orders = await _context.Orders
    .Include(o => o.User)
    .ToListAsync();

// Now order.User is already loaded — no additional queries
foreach (var order in orders)
{
    Console.WriteLine($"Order {order.OrderID} by {order.User.Username}");
}
```

EF Core generates:

```sql
SELECT o.OrderID, o.OrderDate, o.TotalAmount, 
       u.UserID, u.Username, u.Email
FROM Orders AS o
INNER JOIN Users AS u ON u.UserID = o.UserID
```

One query. One round trip. Correct data.

Or use **explicit loading** for selective loading:

```csharp
// Load only the user if we actually need it
var order = await _context.Orders.FirstOrDefaultAsync(o => o.OrderID == orderId);
if (order is not null && needsUserDetails)
{
    await _context.Entry(order).Reference(o => o.User).LoadAsync();
}
```

Or use **projection** to select only what you need:

```csharp
// ✅ Best performance — no tracking, only required columns
var summaries = await _context.Orders
    .Select(o => new OrderSummary
    {
        OrderID      = o.OrderID,
        CustomerName = o.User.Username,  // EF Core handles the join
        OrderDate    = o.OrderDate,
        TotalAmount  = o.TotalAmount
    })
    .ToListAsync();
```

This generates an optimized JOIN query with only the needed columns, and no tracking overhead. For read-only operations that aggregate or present data, projection is almost always the right choice.

## Part 9: Passwords — Storing Them Properly

### 9.1 The Crime of Plain-Text Passwords

We need to talk about passwords because they are deeply entangled with the parameterized query story — SQL injection frequently targets the Users table, and if passwords are stored in plain text, a successful injection exposes your users' most sensitive secret.

Let us be absolutely clear about something: storing passwords in plain text is an act of negligence toward your users. Full stop. It is not a matter of judgment. It is not a tradeoff. It is wrong.

Your users reuse passwords. Research consistently shows that somewhere between 50% and 80% of people reuse the same password across multiple sites. When your database leaks their password, the attacker tries it on their Gmail, their bank, their other accounts. The harm propagates far beyond your site.

```sql
-- What you should NEVER see in production
SELECT Username, Password FROM Users WHERE Username = 'alice'
-- Returns: alice | supersecretpassword123
```

Never. This is never acceptable.

### 9.2 Hashing — The Minimum

The minimum acceptable practice is to store a **cryptographic hash** of the password, not the password itself. A hash function takes arbitrary input and produces a fixed-size output. Critically, it is a one-way function — given the hash, you cannot (practically) recover the original input.

To verify a password: hash the input, compare to the stored hash.

However, not all hash functions are appropriate for passwords. MD5 and SHA-1 are broken (collision attacks). SHA-256 and SHA-512 are cryptographically strong but *too fast* — they can be computed millions or billions of times per second on modern hardware, enabling brute-force and dictionary attacks.

Password hashing algorithms are designed to be slow — they incorporate a computational cost factor that makes each hash computation take, say, 100 milliseconds. When logging in, 100ms is imperceptible to a user. For an attacker trying to guess passwords at scale, 100ms per attempt means they can only try 10 passwords per second per CPU core — instead of millions.

### 9.3 bcrypt, scrypt, Argon2 — The Right Tools

The appropriate password hashing algorithms are:

**bcrypt**: Developed in 1999, still widely used. Has a configurable work factor. The de facto standard for a long time.

**scrypt**: Memory-hard — requires significant RAM in addition to CPU, making it expensive to run on GPU farms (which have much RAM per die).

**Argon2**: Winner of the Password Hashing Competition (2015). Has three variants: Argon2i (side-channel resistant), Argon2d (GPU-resistant), and Argon2id (hybrid, recommended). The current gold standard.

In .NET:

```csharp
// Install: dotnet add package BCrypt.Net-Next
using BCrypt.Net;

// Hash a password
string passwordHash = BCrypt.HashPassword(plainTextPassword, workFactor: 12);
// Work factor 12 means 2^12 = 4096 iterations — takes ~100-300ms on modern hardware

// Verify a password
bool isMatch = BCrypt.Verify(plainTextPassword, storedHash);
```

The `BCrypt.HashPassword` function automatically generates a random salt (a random value added to the input before hashing, preventing two users with the same password from having the same hash) and embeds it in the output hash string. You store the output of `HashPassword` directly. When verifying, `Verify` extracts the embedded salt and work factor automatically.

```csharp
// Example in ASP.NET Core Identity (which handles all this for you)
// ASP.NET Core Identity uses Argon2 by default in .NET 8+

var result = await _userManager.CreateAsync(user, plainTextPassword);
// No hash code required — Identity handles it

var signInResult = await _signInManager.PasswordSignInAsync(username, plainTextPassword, isPersistent: false, lockoutOnFailure: true);
// No hash code required — Identity handles verification
```

If you are building a serious authentication system, use **ASP.NET Core Identity**. It handles password hashing (with Argon2 by default), salt management, account lockout, two-factor authentication, token generation, and dozens of other security concerns that you do not want to implement yourself. Roll your own authentication only if you have a deep understanding of cryptographic protocols and a compelling reason to deviate from the standard.

## Part 10: Advanced SQL Patterns Every ASP.NET Developer Should Know

### 10.1 Window Functions

Window functions perform calculations across sets of rows related to the current row, without collapsing those rows into groups (unlike `GROUP BY` + aggregate functions).

```sql
-- Rank products by price within each category
SELECT 
    p.ProductName,
    c.CategoryName,
    p.Price,
    RANK() OVER (PARTITION BY p.CategoryID ORDER BY p.Price DESC) AS PriceRankInCategory,
    ROW_NUMBER() OVER (ORDER BY p.Price DESC) AS OverallPriceRank,
    AVG(p.Price) OVER (PARTITION BY p.CategoryID) AS AvgCategoryPrice,
    p.Price - AVG(p.Price) OVER (PARTITION BY p.CategoryID) AS DifferenceFromCategoryAvg
FROM Products AS p
JOIN Categories AS c ON c.CategoryID = p.CategoryID;
```

`OVER (PARTITION BY ... ORDER BY ...)` is the window specification — it defines which rows each calculation operates over.

`RANK()` — rank with gaps (1, 2, 2, 4 if two rows tie for 2nd)
`DENSE_RANK()` — rank without gaps (1, 2, 2, 3)
`ROW_NUMBER()` — unique sequential number regardless of ties
`LEAD(column, n)` / `LAG(column, n)` — value from n rows ahead/behind
`FIRST_VALUE(column)` / `LAST_VALUE(column)` — first/last value in the window

**Running totals:**

```sql
SELECT 
    o.OrderDate,
    o.TotalAmount,
    SUM(o.TotalAmount) OVER (
        PARTITION BY YEAR(o.OrderDate), MONTH(o.OrderDate) 
        ORDER BY o.OrderDate
        ROWS UNBOUNDED PRECEDING
    ) AS MonthlyRunningTotal
FROM Orders AS o
ORDER BY o.OrderDate;
```

Window functions are extraordinarily useful for analytics queries — things that used to require complex subqueries or application-side computation can often be expressed cleanly with window functions.

### 10.2 MERGE (Upsert)

`MERGE` allows you to insert, update, or delete rows in a target table based on a join with a source, all in a single statement:

```sql
-- Upsert: insert if not exists, update if exists
MERGE Products AS target
USING (VALUES (@ProductID, @ProductName, @Price, @CategoryID)) 
      AS source (ProductID, ProductName, Price, CategoryID)
ON target.ProductID = source.ProductID
WHEN MATCHED THEN
    UPDATE SET 
        target.ProductName = source.ProductName,
        target.Price       = source.Price,
        target.UpdatedAt   = GETUTCDATE()
WHEN NOT MATCHED THEN
    INSERT (ProductID, ProductName, Price, CategoryID, CreatedAt)
    VALUES (source.ProductID, source.ProductName, source.Price, source.CategoryID, GETUTCDATE());
```

This is particularly useful for synchronization scenarios: you have a feed of product data from a supplier, and you want to update existing products and insert new ones in one pass.

In PostgreSQL, the equivalent is `INSERT ... ON CONFLICT`:

```sql
-- PostgreSQL upsert
INSERT INTO products (product_id, product_name, price, category_id, created_at)
VALUES (@ProductID, @ProductName, @Price, @CategoryID, NOW())
ON CONFLICT (product_id) DO UPDATE SET
    product_name = EXCLUDED.product_name,
    price        = EXCLUDED.price,
    updated_at   = NOW();
```

### 10.3 Common Table Expressions for Complex Logic

CTEs can be chained and stacked to build very complex queries in a readable, step-by-step manner:

```csharp
const string sql = @"
    WITH 
    -- Step 1: Calculate total spend per customer in the last year
    CustomerYearlySpend AS (
        SELECT 
            c.CustomerID,
            c.Name,
            c.Email,
            SUM(o.TotalAmount) AS TotalSpendLastYear,
            COUNT(o.OrderID)   AS OrderCount
        FROM Customers AS c
        LEFT JOIN Orders AS o ON o.CustomerID = c.CustomerID
            AND o.OrderDate >= DATEADD(YEAR, -1, GETUTCDATE())
        GROUP BY c.CustomerID, c.Name, c.Email
    ),
    -- Step 2: Rank customers by spend
    RankedCustomers AS (
        SELECT 
            CustomerID, Name, Email, 
            TotalSpendLastYear, OrderCount,
            NTILE(4) OVER (ORDER BY TotalSpendLastYear DESC) AS SpendQuartile
        FROM CustomerYearlySpend
    )
    -- Final: Return the top quartile (high-value customers)
    SELECT CustomerID, Name, Email, TotalSpendLastYear, OrderCount
    FROM   RankedCustomers
    WHERE  SpendQuartile = 1
    ORDER BY TotalSpendLastYear DESC;";
```

This query breaks a complex business question ("who are our top-quartile customers by spend in the last year?") into readable, named steps. Without CTEs, this would require nested subqueries that are very difficult to follow.

### 10.4 Stored Procedures

A stored procedure is a named, pre-compiled SQL program stored in the database. There are legitimate reasons to use them (security, encapsulation, pre-compilation, atomic operations that should not be expressible in ad-hoc SQL), and there are cults that use them for everything as a misguided form of security theater.

The legitimate case for stored procedures:

```sql
CREATE PROCEDURE GetOrdersForCustomer
    @CustomerID INT,
    @StartDate  DATE,
    @EndDate    DATE
AS
BEGIN
    SET NOCOUNT ON;
    
    SELECT 
        o.OrderID,
        o.OrderDate,
        o.TotalAmount,
        COUNT(oi.OrderItemID) AS ItemCount
    FROM Orders AS o
    JOIN OrderItems AS oi ON oi.OrderID = o.OrderID
    WHERE o.CustomerID = @CustomerID
      AND o.OrderDate BETWEEN @StartDate AND @EndDate
    GROUP BY o.OrderID, o.OrderDate, o.TotalAmount
    ORDER BY o.OrderDate DESC;
END;
```

Calling from C# with parameters:

```csharp
using var command = new SqlCommand("GetOrdersForCustomer", connection);
command.CommandType = CommandType.StoredProcedure;
command.Parameters.Add(new SqlParameter("@CustomerID", SqlDbType.Int)  { Value = customerId });
command.Parameters.Add(new SqlParameter("@StartDate",  SqlDbType.Date) { Value = startDate });
command.Parameters.Add(new SqlParameter("@EndDate",    SqlDbType.Date) { Value = endDate });

await using var reader = await command.ExecuteReaderAsync();
// ... read results
```

Stored procedures are still parameterized — the `@CustomerID`, `@StartDate`, `@EndDate` are parameters, not embedded in a SQL string. The injection protection is the same.

The security argument for stored procedures is that you can grant an application user permission to EXECUTE a stored procedure without granting direct SELECT/INSERT/UPDATE/DELETE permissions on the underlying tables. The application can only do what the stored procedures allow. This is a real security benefit in environments where the database is shared by multiple applications and strict least-privilege policies are required.

## Part 11: Patterns and Anti-Patterns in the Repository Layer

### 11.1 The Repository Pattern

The Repository pattern wraps data access behind an interface, hiding the details of how data is retrieved from the rest of the application. This serves several purposes: it makes the application easier to test (you can swap the real repository for a fake), it centralizes query logic, and it decouples your domain model from your database technology.

A basic repository:

```csharp
// Interface — what the application sees
public interface IProductRepository
{
    Task<Product?>          GetByIdAsync(int productId);
    Task<List<Product>>     GetByCategoryAsync(int categoryId);
    Task<List<Product>>     SearchAsync(string searchTerm);
    Task<Product>           CreateAsync(Product product);
    Task<bool>              UpdateAsync(Product product);
    Task<bool>              DeleteAsync(int productId);
}

// Dapper implementation
public class DapperProductRepository : IProductRepository
{
    private readonly string _connectionString;
    
    public DapperProductRepository(string connectionString) 
        => _connectionString = connectionString;
    
    private SqlConnection CreateConnection() => new(_connectionString);
    
    public async Task<Product?> GetByIdAsync(int productId)
    {
        using var conn = CreateConnection();
        return await conn.QuerySingleOrDefaultAsync<Product>(
            "SELECT ProductID, ProductName, Price, CategoryID FROM Products WHERE ProductID = @ID",
            new { ID = productId }
        );
    }
    
    public async Task<List<Product>> GetByCategoryAsync(int categoryId)
    {
        using var conn = CreateConnection();
        var results = await conn.QueryAsync<Product>(
            "SELECT ProductID, ProductName, Price FROM Products WHERE CategoryID = @CategoryID ORDER BY ProductName",
            new { CategoryID = categoryId }
        );
        return results.ToList();
    }
    
    public async Task<List<Product>> SearchAsync(string searchTerm)
    {
        using var conn = CreateConnection();
        var results = await conn.QueryAsync<Product>(
            "SELECT ProductID, ProductName, Price FROM Products WHERE ProductName LIKE @Pattern",
            new { Pattern = $"%{searchTerm}%" }
        );
        return results.ToList();
    }
    
    public async Task<Product> CreateAsync(Product product)
    {
        using var conn = CreateConnection();
        int newId = await conn.ExecuteScalarAsync<int>(
            @"INSERT INTO Products (ProductName, Price, CategoryID)
              VALUES (@ProductName, @Price, @CategoryID);
              SELECT CAST(SCOPE_IDENTITY() AS INT);",
            product
        );
        product.ProductID = newId;
        return product;
    }
    
    public async Task<bool> UpdateAsync(Product product)
    {
        using var conn = CreateConnection();
        int rows = await conn.ExecuteAsync(
            "UPDATE Products SET ProductName = @ProductName, Price = @Price WHERE ProductID = @ProductID",
            product
        );
        return rows > 0;
    }
    
    public async Task<bool> DeleteAsync(int productId)
    {
        using var conn = CreateConnection();
        int rows = await conn.ExecuteAsync(
            "DELETE FROM Products WHERE ProductID = @ID",
            new { ID = productId }
        );
        return rows > 0;
    }
}

// EF Core implementation of the same interface
public class EfProductRepository : IProductRepository
{
    private readonly AppDbContext _context;
    
    public EfProductRepository(AppDbContext context) => _context = context;
    
    public async Task<Product?> GetByIdAsync(int productId)
        => await _context.Products.FindAsync(productId);
    
    public async Task<List<Product>> GetByCategoryAsync(int categoryId)
        => await _context.Products
            .Where(p => p.CategoryID == categoryId)
            .OrderBy(p => p.ProductName)
            .ToListAsync();
    
    public async Task<List<Product>> SearchAsync(string searchTerm)
        => await _context.Products
            .Where(p => EF.Functions.Like(p.ProductName, $"%{searchTerm}%"))
            .ToListAsync();
    
    public async Task<Product> CreateAsync(Product product)
    {
        _context.Products.Add(product);
        await _context.SaveChangesAsync();
        return product;
    }
    
    public async Task<bool> UpdateAsync(Product product)
    {
        _context.Products.Update(product);
        int rows = await _context.SaveChangesAsync();
        return rows > 0;
    }
    
    public async Task<bool> DeleteAsync(int productId)
    {
        var product = await _context.Products.FindAsync(productId);
        if (product is null) return false;
        _context.Products.Remove(product);
        await _context.SaveChangesAsync();
        return true;
    }
}
```

Registration in DI:

```csharp
// Can swap between implementations based on configuration
builder.Services.AddScoped<IProductRepository, DapperProductRepository>();
// or:
builder.Services.AddScoped<IProductRepository, EfProductRepository>();
```

The application code that uses `IProductRepository` does not care which implementation it gets. This is the Dependency Inversion Principle (the D in SOLID) in action.

### 11.2 Unit Testing with In-Memory / SQLite

Because our repository is behind an interface, we can test the service layer with a fake:

```csharp
// Fake repository for unit tests
public class FakeProductRepository : IProductRepository
{
    private readonly Dictionary<int, Product> _products = new();
    private int _nextId = 1;
    
    public Task<Product?> GetByIdAsync(int id)
        => Task.FromResult(_products.GetValueOrDefault(id));
    
    public Task<List<Product>> GetByCategoryAsync(int categoryId)
        => Task.FromResult(_products.Values.Where(p => p.CategoryID == categoryId).ToList());
    
    public Task<List<Product>> SearchAsync(string term)
        => Task.FromResult(_products.Values
            .Where(p => p.ProductName.Contains(term, StringComparison.OrdinalIgnoreCase))
            .ToList());
    
    public Task<Product> CreateAsync(Product product)
    {
        product.ProductID = _nextId++;
        _products[product.ProductID] = product;
        return Task.FromResult(product);
    }
    
    public Task<bool> UpdateAsync(Product product)
    {
        if (!_products.ContainsKey(product.ProductID)) return Task.FromResult(false);
        _products[product.ProductID] = product;
        return Task.FromResult(true);
    }
    
    public Task<bool> DeleteAsync(int id)
    {
        bool removed = _products.Remove(id);
        return Task.FromResult(removed);
    }
}
```

For integration tests that need to test actual SQL — the query correctness, the parameterization, the schema — use SQLite in-memory mode:

```csharp
// Integration test using SQLite in-memory database
```

## Part 12: Connection Management, Pooling, and Performance

### 12.1 Connection Pooling

Creating a database connection is expensive — it involves network TCP handshake, authentication, session establishment on the server, and allocation of server-side resources. On a loaded server, this can take 20-100ms or more. For a web application handling 100 requests per second, creating and destroying a connection per request would spend most of its time just on connection overhead.

**Connection pooling** solves this by reusing connections. ADO.NET includes a built-in connection pool: when you "open" a connection, ADO.NET first checks the pool for an available idle connection with the same connection string. If one exists, it is returned immediately. When you "close" a connection, it goes back to the pool rather than actually being destroyed.

The pool is transparent: you write the same code, but connections are reused under the hood.

```csharp
// This looks like it creates and destroys a connection, but actually uses the pool
await using var connection = new SqlConnection(connectionString);
await connection.OpenAsync();  // Gets a connection from the pool (or creates new)
// ... do work ...
// Dispose returns the connection to the pool
```

This is why the "open late, close early" pattern matters: the longer a connection is held, the more it denies that pooled connection to other requests.

```csharp
// ❌ BAD — holds connection open for entire request lifetime
public class OrderService
{
    private readonly SqlConnection _conn;
    
    public OrderService(IConfiguration config)
    {
        _conn = new SqlConnection(config.GetConnectionString("DefaultConnection"));
        _conn.Open();  // Opened at construction time
    }
    
    // ... methods that use _conn ...
    // Connection is held open for as long as this service instance lives
    // In ASP.NET with Singleton lifetime, this is forever
}

// ✅ GOOD — open connection only when needed, close immediately after
public class OrderService
{
    private readonly string _connectionString;
    
    public OrderService(IConfiguration config)
    {
        _connectionString = config.GetConnectionString("DefaultConnection")!;
    }
    
    public async Task<Order?> GetOrderAsync(int orderId)
    {
        await using var conn = new SqlConnection(_connectionString);
        await conn.OpenAsync();  // Get from pool
        // ... execute query ...
        // Dispose at end of method — return to pool
    }
}
```

In EF Core, the `DbContext` manages this automatically. Each `DbContext` instance uses its own connection, and the connection is opened and closed for each operation. Because `DbContext` is registered as `Scoped` in ASP.NET Core, it lives for the duration of one HTTP request and is disposed at the end.

### 12.2 The Connection String and Configuration

Connection strings contain sensitive credentials — server address, database name, username, password. They must never be hardcoded in source code or committed to version control.

```json
// appsettings.json — committed to version control (no secrets here)
{
  "ConnectionStrings": {
    "DefaultConnection": "Data Source=:memory:"
  }
}

// appsettings.Development.json — local development, NOT committed
{
  "ConnectionStrings": {
    "DefaultConnection": "Server=(localdb)\\mssqllocaldb;Database=MyApp;Trusted_Connection=True"
  }
}
```

```bash
# For production: use environment variables or a secrets manager
export ConnectionStrings__DefaultConnection="Server=prod-db.example.com;Database=MyApp;User Id=appuser;Password=SUPER_SECRET_VALUE"
```

In Docker/Kubernetes, inject secrets as environment variables. In Azure, use Azure Key Vault with managed identity. In AWS, use Secrets Manager. Never hard-code production credentials in any file.

For local development secrets, use the .NET User Secrets tool:

```bash
dotnet user-secrets init
dotnet user-secrets set "ConnectionStrings:DefaultConnection" "Server=localhost;Database=MyApp;User Id=dev;Password=devpassword"
```

User secrets are stored in a file outside the project directory (in `~/.microsoft/usersecrets/` on Linux/macOS), so they are never accidentally committed.

### 12.3 Least Privilege Database Accounts

Your application should not connect to the database as a superuser (sa, postgres, root). It should connect as a purpose-specific user with the minimum permissions it needs.

For a typical web application:

```sql
-- SQL Server
CREATE LOGIN appuser WITH PASSWORD = 'A_secure_random_password_123!';
CREATE USER appuser FOR LOGIN appuser;

-- Grant only what the app needs
GRANT SELECT, INSERT, UPDATE, DELETE ON SCHEMA::dbo TO appuser;
-- If using stored procedures, grant EXECUTE instead of direct table access:
GRANT EXECUTE ON SCHEMA::dbo TO appuser;
DENY SELECT, INSERT, UPDATE, DELETE ON SCHEMA::dbo TO appuser;  -- Only procs
```

With a least-privilege account:
- Even if your application has a SQL injection vulnerability, the attacker is limited to what the database user can do.
- The attacker cannot `DROP TABLE`, cannot read system tables, cannot create backdoors.
- The blast radius of a successful attack is minimized.

This is defense in depth: parameterized queries are the first defense (prevent injection), least privilege is the second defense (limit damage if injection somehow succeeds anyway).

## Part 13: SQLite — The Surprisingly Powerful Option

### 13.1 What SQLite Is

SQLite is a self-contained, serverless, zero-configuration relational database engine. It stores an entire database in a single file. It requires no installation, no service running in the background, and no network connection. The entire library is a single C file of about 150,000 lines.

SQLite is the most widely deployed database engine in the world — it is embedded in virtually every smartphone (Android, iOS), all major web browsers (Chrome, Firefox, Safari store their history/bookmarks/settings in SQLite), many desktop applications, and countless embedded systems.

For .NET web applications, SQLite is an excellent choice for:

- **Development and testing**: Zero setup, works on any machine, creates/destroys instantly.
- **Small applications**: Single-server applications with low to moderate write concurrency (SQLite uses file-level locking for writes).
- **Desktop applications**: Particularly Blazor Hybrid / MAUI applications.

SQLite is not appropriate for:
- High-write-concurrency web applications (only one writer at a time).
- Multi-server deployments (the database is a local file — it cannot be shared across servers without additional infrastructure like Litestream or rqlite).

### 13.2 SQLite with .NET

```bash
dotnet add package Microsoft.Data.Sqlite
# Or with EF Core:
dotnet add package Microsoft.EntityFrameworkCore.Sqlite
```

```csharp
// Direct SQLite with parameterized query
using Microsoft.Data.Sqlite;

await using var connection = new SqliteConnection("Data Source=app.db");
await connection.OpenAsync();

await using var command = connection.CreateCommand();
command.CommandText = "SELECT * FROM Users WHERE Username = @Username";
command.Parameters.AddWithValue("@Username", username);

await using var reader = await command.ExecuteReaderAsync();
while (await reader.ReadAsync())
{
    Console.WriteLine($"{reader["UserID"]}: {reader["Username"]}");
}
```

SQLite parameters use the same `@Name` syntax as SQL Server in `Microsoft.Data.Sqlite`. The same parameterization patterns apply.

**SQLite-specific performance tips:**

```sql
-- WAL mode: allows concurrent reads alongside a single writer
PRAGMA journal_mode = WAL;

-- Synchronous = NORMAL: safer than OFF but faster than FULL
PRAGMA synchronous = NORMAL;

-- Increase cache size (default is very small)
PRAGMA cache_size = -64000;  -- 64MB cache

-- Enable foreign key enforcement (OFF by default in SQLite!)
PRAGMA foreign_keys = ON;
```

The foreign key pragma is critically important: SQLite does not enforce foreign keys by default for historical compatibility reasons. You must explicitly enable it per connection. In EF Core, the SQLite provider enables this automatically. In raw ADO.NET, you must add it to your connection initialization.

## Part 14: PostgreSQL and Npgsql

### 14.1 PostgreSQL: The Professional's Choice

PostgreSQL (often called "Postgres") is a free, open-source, full-featured relational database. It has been in active development since 1986, with version 1 released in 1989. As of 2024/2025, the current stable version is PostgreSQL 17.

PostgreSQL is the preferred database for serious production work in the .NET open-source world:
- Fully free software (PostgreSQL License, BSD-like)
- No row limits, no size limits, no commercial restrictions
- Excellent performance
- Rich feature set: advanced indexing (B-tree, GiST, GIN, BRIN, Hash), JSON/JSONB support, full-text search, geospatial data (PostGIS), time-series, partitioning, and more
- ACID compliant with excellent multiversion concurrency control (MVCC)

For .NET, the PostgreSQL driver is **Npgsql** (NuGet: `Npgsql` or `Npgsql.EntityFrameworkCore.PostgreSQL`).

### 14.2 Npgsql Parameterized Queries

Npgsql uses positional (`$1`, `$2`, ...) or named (`@Name`) parameters:

```csharp
using Npgsql;

await using var connection = new NpgsqlConnection(connectionString);
await connection.OpenAsync();

// Named parameters (preferred in most cases)
await using var command = new NpgsqlCommand(
    "SELECT user_id, username, email FROM users WHERE username = @Username",
    connection);
command.Parameters.AddWithValue("@Username", username);

await using var reader = await command.ExecuteReaderAsync();
while (await reader.ReadAsync())
{
    Console.WriteLine($"{reader.GetInt32(0)}: {reader.GetString(1)}");
}
```

Npgsql has strong type mapping for PostgreSQL's native types:

```csharp
// Mapping PostgreSQL-specific types
command.Parameters.AddWithValue("@Data", NpgsqlDbType.Jsonb, jsonString);
command.Parameters.AddWithValue("@Tags", NpgsqlDbType.Array | NpgsqlDbType.Text, new string[] { "tag1", "tag2" });
command.Parameters.AddWithValue("@Point", NpgsqlDbType.Point, new NpgsqlPoint(1.5, 2.3));
```

### 14.3 PostgreSQL-Specific Features Worth Knowing

**JSONB for semi-structured data:**

```sql
-- Store and query JSON data natively
CREATE TABLE Events (
    EventID   SERIAL PRIMARY KEY,
    EventType VARCHAR(100) NOT NULL,
    Payload   JSONB,
    CreatedAt TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Index a JSON field
CREATE INDEX IX_Events_UserId ON Events ((Payload->>'userId'));

-- Query JSON fields
SELECT * FROM Events 
WHERE EventType = 'Login' 
  AND Payload->>'userId' = '42'
  AND (Payload->>'success')::BOOLEAN = TRUE;
```

**Array columns:**

```sql
-- A column that is an array of text values
CREATE TABLE Articles (
    ArticleID SERIAL PRIMARY KEY,
    Title     TEXT NOT NULL,
    Tags      TEXT[] NOT NULL DEFAULT '{}'
);

-- Query: find articles with a specific tag
SELECT * FROM Articles WHERE 'blazor' = ANY(Tags);

-- Query: find articles with ALL of these tags
SELECT * FROM Articles WHERE Tags @> ARRAY['blazor', 'dotnet'];
```

**Full-text search:**

```sql
-- Create a full-text search index
CREATE INDEX IX_Articles_FTS ON Articles USING GIN(to_tsvector('english', Title || ' ' || Content));

-- Search
SELECT ArticleID, Title, ts_rank(to_tsvector('english', Title || ' ' || Content), query) AS Rank
FROM Articles, to_tsquery('english', 'blazor & webassembly') AS query
WHERE to_tsvector('english', Title || ' ' || Content) @@ query
ORDER BY Rank DESC;
```

## Part 15: Production Considerations

### 15.1 Error Handling and Logging

In production, database errors need careful handling. You must never expose raw SQL error messages to end users — they contain schema information that aids attackers.

```csharp
public class ProductRepository
{
    private readonly string _connectionString;
    private readonly ILogger<ProductRepository> _logger;
    
    public ProductRepository(string connectionString, ILogger<ProductRepository> logger)
    {
        _connectionString = connectionString;
        _logger = logger;
    }
    
    public async Task<Product?> GetByIdAsync(int productId)
    {
        try
        {
            await using var conn = new SqlConnection(_connectionString);
            return await conn.QuerySingleOrDefaultAsync<Product>(
                "SELECT ProductID, ProductName, Price FROM Products WHERE ProductID = @ID",
                new { ID = productId }
            );
        }
        catch (SqlException ex)
        {
            // Log the full exception internally (has technical details)
            _logger.LogError(ex, "Database error retrieving product {ProductId}", productId);
            
            // Throw a generic exception to the application layer — NO SQL details
            throw new DataAccessException($"Failed to retrieve product {productId}", ex);
        }
    }
}

// Custom exception — no SQL details
public class DataAccessException : Exception
{
    public DataAccessException(string message, Exception inner) : base(message, inner) { }
}
```

In your controller or minimal API:

```csharp
app.UseExceptionHandler(errorApp =>
{
    errorApp.Run(async context =>
    {
        context.Response.StatusCode = 500;
        context.Response.ContentType = "application/json";
        
        // Generic error to the client
        await context.Response.WriteAsJsonAsync(new { error = "An error occurred processing your request." });
        
        // The full error was already logged internally
    });
});
```

Never, under any circumstances, send SQL error text, stack traces, or connection strings in HTTP responses.

### 15.2 Query Timeouts

Every database query should have a timeout:

```csharp
using var command = new SqlCommand(sql, connection);
command.CommandTimeout = 30;  // 30 seconds (default is 30 in ADO.NET)

// Dapper:
await conn.QueryAsync<Product>(sql, param, commandTimeout: 30);

// EF Core:
using var cts = new CancellationTokenSource(TimeSpan.FromSeconds(30));
var results = await _context.Products.ToListAsync(cts.Token);
```

Without timeouts, a slow query can hold a connection and block other requests indefinitely. In a connection pool with a limited size, a handful of timed-out queries can exhaust the pool and bring down the entire application.

### 15.3 Database Migrations in Production

Never run `dotnet ef database update` against a production database during a live deployment. The proper production workflow:

```bash
# Generate a migration SQL script
dotnet ef migrations script PreviousMigrationName LatestMigrationName --output migration.sql

# Review the script
cat migration.sql

# Apply during a maintenance window or as part of a blue-green deployment
# Apply with your DBA's approval and with a backup in hand
sqlcmd -S prod-server -d MyDatabase -i migration.sql
```

For zero-downtime migrations, design schema changes to be backward-compatible:
- Add new columns as nullable or with defaults (old code ignores new columns).
- Keep old columns alongside new ones during the transition period (old code uses old columns, new code uses new columns).
- Remove old columns only after all code using them has been deployed.
- Never rename columns in a single migration; add the new column, migrate data, drop the old column in separate deployments.

### 16.1 The Scenario

Let us build a small but complete order management API for an e-commerce application. We will start with a terrible implementation and progressively make it correct.

### 16.3 Version 2: Correct Implementation

```csharp
// ✅ CORRECT VERSION

// Models
public record OrderSummary(int OrderID, DateTime OrderDate, string Status, int ItemCount);
public record CreateOrderRequest(int CustomerID, List<OrderItemRequest> Items);
public record OrderItemRequest(int ProductID, int Quantity);
public record CreateOrderResponse(int OrderID);

// Repository interface
public interface IOrderRepository
{
    Task<List<OrderSummary>> GetOrdersByCustomerAsync(int customerId, CancellationToken ct = default);
    Task<int> CreateOrderAsync(int customerId, List<OrderItemRequest> items, CancellationToken ct = default);
}

// Dapper repository
public class OrderRepository : IOrderRepository
{
    private readonly string _connectionString;
    private readonly ILogger<OrderRepository> _logger;
    
    public OrderRepository(IConfiguration config, ILogger<OrderRepository> logger)
    {
        _connectionString = config.GetConnectionString("DefaultConnection")
            ?? throw new InvalidOperationException("Connection string 'DefaultConnection' not configured");
        _logger = logger;
    }
    
    public async Task<List<OrderSummary>> GetOrdersByCustomerAsync(int customerId, CancellationToken ct = default)
    {
        const string sql = @"
            SELECT 
                o.OrderID,
                o.OrderDate,
                o.Status,
                COUNT(oi.OrderItemID) AS ItemCount
            FROM Orders AS o
            LEFT JOIN OrderItems AS oi ON oi.OrderID = o.OrderID
            WHERE o.CustomerID = @CustomerID
            GROUP BY o.OrderID, o.OrderDate, o.Status
            ORDER BY o.OrderDate DESC";
        
        await using var conn = new SqlConnection(_connectionString);
        var results = await conn.QueryAsync<OrderSummary>(
            new CommandDefinition(sql, new { CustomerID = customerId }, cancellationToken: ct)
        );
        return results.ToList();
    }
    
    public async Task<int> CreateOrderAsync(int customerId, List<OrderItemRequest> items, CancellationToken ct = default)
    {
        // Validate we have items before touching the database
        if (items.Count == 0)
            throw new ArgumentException("Order must have at least one item", nameof(items));
        
        await using var conn = new SqlConnection(_connectionString);
        await conn.OpenAsync(ct);
        await using var transaction = conn.BeginTransaction();
        
        try
        {
            // 1. Verify customer exists
            int? customerExists = await conn.QuerySingleOrDefaultAsync<int?>(
                new CommandDefinition(
                    "SELECT CustomerID FROM Customers WHERE CustomerID = @ID",
                    new { ID = customerId },
                    transaction: transaction,
                    cancellationToken: ct
                )
            );
            
            if (customerExists is null)
                throw new InvalidOperationException($"Customer {customerId} not found");
            
            // 2. Fetch product prices (verify products exist)
            var productIds = items.Select(i => i.ProductID).Distinct().ToList();
            var paramNames = productIds.Select((_, i) => $"@p{i}").ToList();
            var productSql  = $"SELECT ProductID, Price, StockQty FROM Products WHERE ProductID IN ({string.Join(", ", paramNames)})";
            
            var productCmd = new SqlCommand(productSql, conn, transaction);
            for (int i = 0; i < productIds.Count; i++)
                productCmd.Parameters.Add(new SqlParameter($"@p{i}", SqlDbType.Int) { Value = productIds[i] });
            
            var productPrices = new Dictionary<int, (decimal Price, int Stock)>();
            await using (var reader = await productCmd.ExecuteReaderAsync(ct))
            {
                while (await reader.ReadAsync(ct))
                {
                    productPrices[reader.GetInt32(0)] = (reader.GetDecimal(1), reader.GetInt32(2));
                }
            }
            
            // Verify all products exist and have sufficient stock
            foreach (var item in items)
            {
                if (!productPrices.TryGetValue(item.ProductID, out var productInfo))
                    throw new InvalidOperationException($"Product {item.ProductID} not found");
                if (productInfo.Stock < item.Quantity)
                    throw new InvalidOperationException($"Insufficient stock for product {item.ProductID}");
            }
            
            // 3. Create the order header
            int orderId = await conn.ExecuteScalarAsync<int>(
                new CommandDefinition(
                    @"INSERT INTO Orders (CustomerID) VALUES (@CustomerID);
                      SELECT CAST(SCOPE_IDENTITY() AS INT);",
                    new { CustomerID = customerId },
                    transaction: transaction,
                    cancellationToken: ct
                )
            );
            
            // 4. Insert order items and decrement stock
            foreach (var item in items)
            {
                var (price, _) = productPrices[item.ProductID];
                
                await conn.ExecuteAsync(
                    new CommandDefinition(
                        @"INSERT INTO OrderItems (OrderID, ProductID, Quantity, UnitPrice)
                          VALUES (@OrderID, @ProductID, @Qty, @Price)",
                        new { OrderID = orderId, ProductID = item.ProductID, Qty = item.Quantity, Price = price },
                        transaction: transaction,
                        cancellationToken: ct
                    )
                );
                
                // Decrement stock atomically
                await conn.ExecuteAsync(
                    new CommandDefinition(
                        "UPDATE Products SET StockQty = StockQty - @Qty WHERE ProductID = @ProductID",
                        new { Qty = item.Quantity, ProductID = item.ProductID },
                        transaction: transaction,
                        cancellationToken: ct
                    )
                );
            }
            
            await transaction.CommitAsync(ct);
            
            _logger.LogInformation(
                "Created order {OrderId} for customer {CustomerId} with {ItemCount} items",
                orderId, customerId, items.Count
            );
            
            return orderId;
        }
        catch
        {
            await transaction.RollbackAsync(ct);
            throw;
        }
    }
}

// Controller — thin, clean
[ApiController]
[Route("api/orders")]
public class OrdersController : ControllerBase
{
    private readonly IOrderRepository _repo;
    
    public OrdersController(IOrderRepository repo) => _repo = repo;
    
    [HttpGet("customer/{customerId:int}")]
    public async Task<IActionResult> GetCustomerOrders(int customerId, CancellationToken ct)
    {
        var orders = await _repo.GetOrdersByCustomerAsync(customerId, ct);
        return Ok(orders);
    }
    
    [HttpPost]
    public async Task<IActionResult> CreateOrder(CreateOrderRequest request, CancellationToken ct)
    {
        try
        {
            int orderId = await _repo.CreateOrderAsync(request.CustomerID, request.Items, ct);
            return CreatedAtAction(nameof(GetCustomerOrders), new { customerId = request.CustomerID },
                new CreateOrderResponse(orderId));
        }
        catch (InvalidOperationException ex)
        {
            return BadRequest(new { error = ex.Message });
        }
    }
}
```

Notice what the correct version does:

1. **Connection string from configuration, not hardcoded** — uses `IConfiguration`, throws a clear error if not configured.
2. **Parameterized queries throughout** — every value is a parameter. Never a concatenated string.
3. **Transaction wrapping multi-step operations** — rollback on any failure.
4. **Async everywhere** — proper `await` throughout.
5. **Stock decrement within the same transaction as order creation** — atomicity.
6. **Validation before database operations** — check for empty items list first.
7. **Business errors (invalid customer, out of stock) return 400 Bad Request** — not 500.
8. **Structured logging** with named parameters (not string concatenation in the log message).
9. **CancellationToken propagation** — respects request cancellation.
10. **`int customerId` in the route constraint (`:int`)** — the route constraint prevents a non-integer value from even reaching the method.

## Part 17: OWASP, Security Checklists, and Going Further

### 17.1 The OWASP Top 10 and SQL Injection

The Open Web Application Security Project (OWASP) maintains the OWASP Top 10 — a regularly updated list of the most critical web application security risks. SQL injection has appeared on every edition of this list since its inception.

In the 2021 edition, injection vulnerabilities (including SQL injection) are listed under "A03:2021 – Injection." OWASP states that injection vulnerabilities were found in 94% of analyzed applications, with SQL injection being the most common specific type.

OWASP's recommended defenses, in order of preference:

1. **Use a safe API** that avoids the interpreter entirely, or provides a parameterized interface. (EF Core, Dapper, ADO.NET parameterized queries.)
2. **Use positive server-side input validation**. Validate the type, length, format, and range of all inputs. This does not replace parameterization — it is a complementary defense.
3. **Escape special characters** using the specific escape syntax for that interpreter. OWASP ranks this as "frail" compared to parameterization, but better than nothing when parameterization is not possible (e.g., table names in dynamic queries).
4. **Use LIMIT and other SQL controls** inside queries to prevent mass disclosure of records.

Note the explicit ranking: parameterized queries first, validation second, escaping third. This is not an endorsement of a mix-and-match approach — it is recognition that different situations have different constraints, and these defenses should be layered.

### 17.2 Beyond Injection: Other Database Security Concerns

**Excessive privileges**: Your application's database account should have only the permissions it needs. If your application is read-only, its database account should have only SELECT permissions.

**Encryption at rest**: Sensitive data — Social Security numbers, credit card numbers, health information — should be encrypted in the database, not just in transit. SQL Server's Transparent Data Encryption (TDE) encrypts the entire database file at the storage level. Column-level encryption protects specific sensitive columns. Application-level encryption (encrypting before storing) provides the strongest guarantee because even database administrators cannot read the data without the encryption key.

**Encryption in transit**: All connections to the database should use TLS. In SQL Server connection strings: `Encrypt=True;TrustServerCertificate=False`. In PostgreSQL: `sslmode=require`.

**Audit logging**: For applications handling sensitive data, enable database audit logging to record who accessed or modified what data and when. SQL Server has built-in auditing. PostgreSQL has `pgaudit`. This is critical for compliance (GDPR, HIPAA, SOX) and for incident response.

**Regular backups and testing**: A database without tested backups is a catastrophe waiting to happen. Test your restore process regularly — not just that the backup file exists, but that you can actually restore from it in a reasonable time.

### 17.3 Dynamic SQL: When You Genuinely Cannot Avoid It

Sometimes you need to build SQL dynamically — for example, when the columns to sort by or the table name itself are determined at runtime. Parameters cannot be used for these because column names and table names are not data values; they are parts of the SQL syntax.

The correct approach for unavoidable dynamic SQL is **allowlisting**:

```csharp
// Sort column determined by user input — cannot use a parameter for a column name
public async Task<List<Product>> GetProductsSortedAsync(string sortColumn, string sortDirection)
{
    // Allowlist: only accept known valid column names
    var allowedColumns = new HashSet<string>(StringComparer.OrdinalIgnoreCase)
    {
        "ProductName", "Price", "CreatedAt"
    };
    
    // Reject anything not on the allowlist
    if (!allowedColumns.Contains(sortColumn))
        throw new ArgumentException($"Invalid sort column: {sortColumn}");
    
    // Direction must be ASC or DESC
    string direction = sortDirection.Equals("DESC", StringComparison.OrdinalIgnoreCase) ? "DESC" : "ASC";
    
    // Now we can safely embed the validated column name and direction
    string sql = $"SELECT ProductID, ProductName, Price FROM Products ORDER BY [{sortColumn}] {direction}";
    
    using var conn = new SqlConnection(_connectionString);
    return (await conn.QueryAsync<Product>(sql)).ToList();
}
```

The bracket-quoting (`[{sortColumn}]`) is an additional protection in SQL Server — it prevents the column name from being interpreted as SQL keywords even if something unexpected slipped through the allowlist. PostgreSQL uses double-quotes instead: `"{columnName}"`.

Never, under any circumstances, allow arbitrary user input into a column name, table name, or SQL keyword position without explicit allowlisting.

## Part 18: Final Summary — The Ten Commandments of Database Access

After everything we have covered, let us distill it to core principles.

**I. Thou shalt use parameterized queries for every value passed to a SQL query.**

No exceptions. Not for "trusted" inputs. Not for integers. Not for things that "can't possibly contain SQL." Every single value that enters a SQL query must go through a parameter.

**II. Thou shalt never build SQL strings by concatenating or interpolating user input.**

Not `+`. Not `$"..."`. Not `string.Format`. Not `StringBuilder.Append`. If you are combining user-provided values with SQL syntax, you are doing it wrong.

**III. Thou shalt store passwords using a purpose-designed password hashing algorithm.**

bcrypt, scrypt, or Argon2id. Never MD5, SHA-1, or plain text. If in doubt, use ASP.NET Core Identity.

**IV. Thou shalt use a least-privilege database account.**

Your application user should not be able to drop tables, create users, or do anything the application does not need to do. Test this: try to `DROP TABLE` with your application credentials and make sure it fails.

**V. Thou shalt never expose database error messages to end users.**

Log them. Keep them. Analyze them. Never send them in HTTP responses. A SQL error message is a roadmap for an attacker.

**VI. Thou shalt normalize your schema to at least Third Normal Form.**

Unless you have a demonstrated, measured performance reason to denormalize, your schema should eliminate redundancy. Redundancy breeds inconsistency. Inconsistency is a defect factory.

**VII. Thou shalt use transactions for all multi-step operations.**

If two or more database writes must succeed or fail together, wrap them in a transaction. The atomicity guarantee is worth the overhead.

**VIII. Thou shalt index foreign keys and columns used in WHERE, JOIN, and ORDER BY.**

Measure query performance. Add indexes to address measured bottlenecks. Remove indexes that have no seeks. Always measure; never index speculatively.

**IX. Thou shalt keep connection strings in configuration, not in code.**

Connection strings contain credentials. Credentials in code end up in version control. Version control is visible. Use environment variables, user secrets, or a secrets manager.

**X. Thou shalt test your data access layer with queries against a real (test) database.**

Unit tests with fake repositories are valuable. Integration tests against SQLite in-memory or a Testcontainers Docker instance prove that your queries are actually correct SQL. Include a test that attempts a SQL injection payload and verifies it has no effect.
