## A Story About a Friday Afternoon

It is 4:47 PM on a Friday. You are already mentally in weekend mode. One of the sales team asks if you can take a quick look at something weird in the customer dashboard. The live orders counter — the number in the top right that shows pending orders — sometimes shows a number that is clearly wrong. It shows 14 orders, but when you refresh the page, it shows 9. Or it shows 47 orders and then jumps down to 3. It is inconsistent and nobody can reproduce it reliably.

You look at the component. It seems fine. It fetches data. It updates state. What could possibly go wrong?

You are about to meet one of the most subtle bugs in React development: the stale closure problem inside `useEffect`. It is not a bug in your application logic. It is not a bug in the data. It is a fundamental mismatch between how closures work in JavaScript and how developers who are accustomed to object-oriented state management tend to think about them.

Understanding this bug, and the `useEffect` model more broadly, is essential to writing correct React applications. And once you understand it, you will appreciate why Blazor's lifecycle model — which maps to patterns you already know from ASP.NET — feels so much more natural for .NET developers.

## Part 1: The Bug — A Forensic Breakdown

Here is the simplified version of the broken orders component:

```jsx
import { useState, useEffect } from 'react';

function LiveOrdersCounter() {
    const [orderCount, setOrderCount] = useState(0);
    const [userId, setUserId] = useState(null);

    // Fetch the user ID when the component mounts
    useEffect(() => {
        fetchCurrentUser().then(user => setUserId(user.id));
    }, []); // ← empty dependency array: runs once on mount

    // Set up a polling interval to refresh the order count
    useEffect(() => {
        const intervalId = setInterval(() => {
            fetchOrderCount(userId).then(count => {
                setOrderCount(count);
            });
        }, 5000); // poll every 5 seconds

        return () => clearInterval(intervalId); // cleanup on unmount
    }, []); // ← empty dependency array: runs once on mount
    
    return <div className="counter">Pending Orders: {orderCount}</div>;
}
```

Do you see the bug? If you are a React expert, you spotted it immediately. If you come from a C# background, it might not be obvious at all — in fact, the code probably looks completely reasonable.

Let us trace through exactly what happens at runtime.

**Time 0**: Component mounts. React calls the component function. `orderCount` is 0, `userId` is `null`. Both `useEffect` callbacks are registered.

**Time 0+ (microtask)**: First `useEffect` runs. It calls `fetchCurrentUser()`, gets the user, and calls `setUserId(user.id)`. Let us say the user ID is 42.

**Time 0+ (after state update)**: React sees that `userId` changed. It re-renders the component. The component function is called again. A new function object is created for each `useEffect` callback. But here is the thing: **React does NOT re-run the effect with the empty dependency array.** The `[]` dependency array told React "this effect does not depend on any reactive values, so only run it once — on mount." React takes you at your word.

**Time 0+ (state after re-render)**: The component has re-rendered. The new render's closure captures `userId = 42`. But the `setInterval` callback was created in the FIRST render's closure, where `userId` was `null`. The `setInterval` callback is still alive, still running every 5 seconds, and it still captures the `userId` from the first render: `null`.

**Time 5 seconds**: `setInterval` fires. It calls `fetchOrderCount(null)`. Your server returns... something. Maybe 0. Maybe an error that gets swallowed. Maybe a count for a default/anonymous user. Whatever it returns, it is wrong.

**Time 10 seconds**: Same thing happens again. `fetchOrderCount(null)` is called again. The counter continues to show incorrect values, or fluctuates based on whatever your server returns for a `null` user ID.

**When the user refreshes**: A fresh mount creates new closures. This time `userId` is set before the interval starts... or is it? Actually, no — the same problem exists on fresh mount, just with slightly different timing. The interval starts with `userId = null`, and the user fetch also starts with `userId = null`. If the user fetch completes quickly and the interval fires afterward, you might get correct behavior. If the interval fires before the user fetch completes, you get the wrong behavior. This is why the bug is intermittent.

This is the stale closure problem.

## Part 2: Closures — A Primer for Object-Oriented Developers

To understand why this happens, you need to understand JavaScript closures. This concept has no direct equivalent in C# class-based thinking, and that disconnect is exactly why the bug is so easy to write and so hard to see.

### What Is a Closure?

In JavaScript (and in many functional languages), a function "closes over" the variables in its surrounding scope. This means the function carries with it a reference to those variables — not just their values at the time of creation, but the variables themselves.

Here is a simple example:

```javascript
function makeCounter() {
    let count = 0; // ← this variable lives in makeCounter's scope
    
    function increment() {
        count++; // ← increment closes over 'count'
        return count;
    }
    
    return increment;
}

const counter = makeCounter();
console.log(counter()); // 1
console.log(counter()); // 2
console.log(counter()); // 3
```

Each call to `counter()` increments the same `count` variable, even though that variable is not accessible from outside `makeCounter`. The `increment` function keeps the variable alive through the closure.

### What Closures Look Like in React

In a React functional component, the component function body IS the "render method." Every time the component re-renders, the function is called again. Each call creates a new set of local variables (state values from `useState`, derived values, etc.) and a new set of closures over those variables.

```jsx
function Counter() {
    const [count, setCount] = useState(0);
    
    // This function is created fresh on EVERY render
    // Each version closes over the 'count' value of that specific render
    const handleDoubleIncrement = () => {
        setCount(count + 1);
        setCount(count + 1); // ← This does NOT increment twice! 
                              // Both setCount calls see the same stale 'count'
    };
    
    return (
        <button onClick={handleDoubleIncrement}>
            Double Increment (count: {count})
        </button>
    );
}
```

When `handleDoubleIncrement` is called and the count is 5:
- The first `setCount(count + 1)` calls `setCount(6)` — because `count` in this closure is 5.
- The second `setCount(count + 1)` also calls `setCount(6)` — because `count` is still 5 in this same closure.
- React batches both state updates and applies the second one, resulting in count being 6, not 7.

If you are thinking "that is bizarre, why would you not want it to increment twice," you are experiencing the object-oriented mental model collision. In C#, if you write:

```csharp
private int _count = 0;

private void HandleDoubleIncrement()
{
    _count++;
    _count++;
    // _count is now 2 more than it was
}
```

`_count++` modifies the field, and the second `_count++` reads the updated field. The increment is genuinely additive. But in React, `count` is not a mutable field — it is a local variable in this render's closure, representing the state value at the time this render happened. Calling `setCount` does not update `count` — it schedules a re-render where the new `count` value will be the one you passed.

To correctly double-increment, you use the functional update form:

```jsx
const handleDoubleIncrement = () => {
    // Functional form: React passes you the most recent pending state value
    setCount(prevCount => prevCount + 1);
    setCount(prevCount => prevCount + 1); // ← this sees the result of the previous setCount
};
```

### Why This Is Different From C# Classes

In a C# class, instance fields are shared state that exists independently of any particular method call. When you call `this._count++` in one method, the change is immediately visible to any other method that reads `this._count`. The object has identity and mutable state.

In a React functional component, there is no `this`. There are no instance fields. Each render is a fresh function call with its own local variables. The `useState` hook is a bridge between the pure function world (each render is independent) and the need for persistent state (state must survive across renders). It does this by storing state outside the component function in React's internal fiber node and returning the current value as a local variable on each render call.

This design is the source of React's entire hook system — and the source of most of its subtle bugs.

## Part 3: The `useEffect` Dependency Array — The Contract You Are Making With React

Now that you understand closures, you can understand what the dependency array in `useEffect` actually means.

```jsx
useEffect(() => {
    // This callback is a closure. It captures:
    // - Any state values referenced inside it
    // - Any props referenced inside it
    // - Any other variables from the component's scope
    
    const intervalId = setInterval(() => {
        fetchOrderCount(userId); // ← This captures 'userId'
    }, 5000);
    
    return () => clearInterval(intervalId);
}, [userId]); // ← The dependency array
```

The dependency array is a **contract**. It says: "I promise that this effect callback only depends on the values listed here. Run the effect when any of these values change. Clean up the previous effect before running the new one."

When you put `[]` (empty array), you are saying: "I promise this effect has no dependencies — it never reads any state or props, so it only needs to run once." If that statement is false — if the effect DOES read state or props — you have a stale closure bug.

When you put `[userId]`, you are saying: "This effect depends on `userId`. Every time `userId` changes, clean up the old effect (by calling the returned cleanup function) and run the new effect with the new closure that captures the new `userId`."

Here is the fixed version of the orders component:

```jsx
function LiveOrdersCounter() {
    const [orderCount, setOrderCount] = useState(0);
    const [userId, setUserId] = useState(null);

    useEffect(() => {
        fetchCurrentUser().then(user => setUserId(user.id));
    }, []); // Correct: genuinely no dependencies

    useEffect(() => {
        if (userId === null) return; // Don't poll until we have a user ID
        
        const intervalId = setInterval(() => {
            // Now the callback captures 'userId' which is in the dependency array
            fetchOrderCount(userId).then(count => {
                setOrderCount(count);
            });
        }, 5000);

        return () => clearInterval(intervalId);
    }, [userId]); // ← userId is listed because it is used inside the effect
    
    return <div className="counter">Pending Orders: {orderCount}</div>;
}
```

Now when `userId` changes from `null` to 42:
1. The previous effect's cleanup function runs: `clearInterval(oldIntervalId)`. The null-user interval is destroyed.
2. The new effect runs with a closure that captures `userId = 42`. A new interval is created.
3. Subsequent polls correctly call `fetchOrderCount(42)`.

The bug is fixed. But we only fixed it because we understood the dependency array contract.

### The Linter's Role

ESLint's `eslint-plugin-react-hooks` rule (`react-hooks/exhaustive-deps`) will warn you when your `useEffect` dependency array is incomplete. This is an invaluable tool that catches most stale closure bugs at lint time. If you are working on a React project and this rule is disabled or generating warnings that are being suppressed with `// eslint-disable` comments, that is a serious red flag.

```jsx
useEffect(() => {
    fetchOrderCount(userId); // ← linter knows: 'userId' is read here but not in deps
}, []); // ← linter warning: 'userId' missing from deps
```

When you see a lint warning like this, never suppress it without understanding why — the warning is almost always telling you about a real potential bug.

## Part 4: `useEffect` and `useLayoutEffect` — When They Run

React has two effect hooks with subtly different timing:

**`useEffect`**: Runs after the browser has painted the new frame. This is the "normal" effect hook. It does not block the browser from updating the display. Side effects in `useEffect` run after the user sees the updated UI.

**`useLayoutEffect`**: Runs synchronously after all DOM mutations but before the browser paints. This is for effects that need to read or write DOM layout information (element sizes, positions) and make immediate visual adjustments based on what they find. Running after paint would cause a flash of incorrect layout.

In most cases, `useEffect` is correct. You reach for `useLayoutEffect` only when you are doing specific DOM measurement work — for example, positioning a tooltip relative to a button's bounding box.

Here is when each runs relative to a component lifecycle event:

```
State change detected
    ↓
Component function called (render)
    ↓  
React commits changes to DOM
    ↓
useLayoutEffect cleanup runs (from previous render)
    ↓
useLayoutEffect runs (synchronous, blocks paint)
    ↓
Browser paints the frame (user sees update)
    ↓
useEffect cleanup runs (from previous render, async/deferred)
    ↓
useEffect runs (async/deferred)
```

## Part 5: The Complete React Lifecycle Map

React's functional component lifecycle, expressed through hooks, can be confusing because there is no explicit lifecycle. There are no methods called `componentDidMount` or `componentWillUnmount`. Instead, lifecycle events are expressed through the timing of effect cleanup and execution. Let us map the full lifecycle:

### Mount (First Render)

```jsx
function MyComponent({ postId }) {
    const [data, setData] = useState(null);
    
    // Runs once after mount
    useEffect(() => {
        console.log('Component mounted!');
        fetchData(postId).then(setData);
        
        // The cleanup returned here runs on UNMOUNT
        return () => {
            console.log('Component unmounted!');
        };
    }, [postId]);
    
    return <div>{data?.title}</div>;
}
```

### Update (State or Props Change)

```jsx
useEffect(() => {
    console.log(`postId changed to: ${postId}`);
    fetchData(postId).then(setData);
    
    // Cleanup runs BEFORE the next execution of this effect
    // (i.e., before the next time postId changes)
    return () => {
        console.log(`Cleaning up for postId: ${postId}`);
        // Cancel any pending requests for the old postId
    };
}, [postId]); // ← Runs when postId changes
```

### Unmount (Component Removed From Tree)

The cleanup function returned from `useEffect` runs on unmount. This is where you cancel subscriptions, clear timers, and abort fetch requests.

```jsx
useEffect(() => {
    const controller = new AbortController();
    
    fetch(`/api/posts/${postId}`, { signal: controller.signal })
        .then(res => res.json())
        .then(setData)
        .catch(err => {
            if (err.name === 'AbortError') return; // Expected, not an error
            console.error(err);
        });
    
    return () => {
        controller.abort(); // Cancel the in-flight request on unmount
    };
}, [postId]);
```

### The "After Render" Hook

If you need to do something after every render (with no specific dependency), you can use `useEffect` with no dependency array:

```jsx
useEffect(() => {
    // Runs after EVERY render
    document.title = `Orders: ${orderCount}`;
}); // ← No dependency array means "run after every render"
```

This is different from `[]` (empty array, run once on mount). The absence of a dependency array means "no conditional, run every time."

## Part 6: Mapping React Hooks to .NET/Blazor Equivalents

Now that we have built a complete mental model of React's lifecycle system, let us map it directly to the Blazor equivalents that .NET developers already know.

### `useState` → Private Fields + `StateHasChanged()`

In React, `useState` is how you store state that persists across renders and triggers re-renders when it changes.

```jsx
// React
const [count, setCount] = useState(0);
const handleClick = () => setCount(count + 1);
```

In Blazor, you use regular private fields. Calling `StateHasChanged()` tells Blazor to re-render the component. But for event handlers triggered by the UI, Blazor automatically calls `StateHasChanged()` after the handler returns, so you often don't need to call it explicitly.

```csharp
// Blazor
@code {
    private int count = 0;
    
    private void HandleClick()
    {
        count++;
        // StateHasChanged() is called automatically after this event handler returns
    }
}
```

### `useEffect` with `[]` → `OnInitializedAsync`

Running code once when the component mounts, like setting up initial data:

```jsx
// React
useEffect(() => {
    fetchPosts().then(setPosts);
}, []); // Run once on mount
```

```csharp
// Blazor
protected override async Task OnInitializedAsync()
{
    _posts = await BlogService.GetAllPostsAsync();
    // StateHasChanged() is called automatically after lifecycle methods
}
```

`OnInitializedAsync` runs once when the component is first initialized. It is the direct equivalent of `useEffect(() => { ... }, [])`.

### `useEffect` with Dependencies → `OnParametersSetAsync`

Running code when props/parameters change:

```jsx
// React
useEffect(() => {
    fetchPost(postId).then(setPost);
}, [postId]); // Run when postId changes
```

```csharp
// Blazor
[Parameter] public string PostId { get; set; } = "";
private BlogPost? _post;

protected override async Task OnParametersSetAsync()
{
    // This runs every time any parameter changes
    _post = await BlogService.GetPostAsync(PostId);
}
```

If you only want to fetch when `PostId` specifically changes (not when other parameters change), you can cache the previous value:

```csharp
private string? _previousPostId;

protected override async Task OnParametersSetAsync()
{
    if (_previousPostId == PostId) return; // No change, skip
    _previousPostId = PostId;
    _post = await BlogService.GetPostAsync(PostId);
}
```

### `useLayoutEffect` → `OnAfterRenderAsync`

Code that needs to run after the DOM is updated (for DOM measurements, JavaScript interop that reads element sizes, etc.):

```jsx
// React
useLayoutEffect(() => {
    const height = divRef.current.getBoundingClientRect().height;
    setSomeState(height);
}, [someValue]);
```

```csharp
// Blazor
protected override async Task OnAfterRenderAsync(bool firstRender)
{
    if (firstRender)
    {
        // Runs after the first render — equivalent to useLayoutEffect with []
        var height = await JS.InvokeAsync<double>(
            "getElementHeight", _elementRef);
        _measuredHeight = height;
        StateHasChanged(); // ← Required if you change state here
    }
}
```

`OnAfterRenderAsync(firstRender)` passes a boolean indicating whether this is the first render or a subsequent one. This is extremely common: most JSInterop calls belong in `OnAfterRenderAsync` because JavaScript can only interact with DOM elements that actually exist in the browser, and elements only exist after they have been rendered.

### Cleanup (Unmount) → `IAsyncDisposable` / `IDisposable`

In React, cleanup happens in the function returned from `useEffect`. In Blazor, cleanup happens by implementing `IDisposable` or `IAsyncDisposable`.

```jsx
// React
useEffect(() => {
    const subscription = eventBus.subscribe('update', handleUpdate);
    return () => subscription.unsubscribe(); // cleanup
}, []);
```

```csharp
// Blazor — implement IAsyncDisposable
@implements IAsyncDisposable

@code {
    private IDisposable? _subscription;
    
    protected override void OnInitialized()
    {
        _subscription = EventBus.Subscribe("update", HandleUpdate);
    }
    
    public async ValueTask DisposeAsync()
    {
        _subscription?.Dispose();
    }
}
```

Or implement `IDisposable` if your cleanup is synchronous:

```csharp
@implements IDisposable

@code {
    private Timer? _pollingTimer;
    
    protected override void OnInitialized()
    {
        _pollingTimer = new Timer(async _ =>
        {
            await LoadData();
            await InvokeAsync(StateHasChanged);
        }, null, TimeSpan.Zero, TimeSpan.FromSeconds(5));
    }
    
    public void Dispose()
    {
        _pollingTimer?.Dispose();
    }
}
```

Note the `InvokeAsync(StateHasChanged)` call in the timer callback. Because the timer callback runs on a thread pool thread, not the Blazor synchronization context, you need to marshal the `StateHasChanged` call back to the correct context. This is the Blazor equivalent of making sure you update UI from the main thread in a Windows Forms or WPF application — a pattern .NET developers are very familiar with.

### The Complete Lifecycle Comparison Table

| Event | React | Blazor |
|---|---|---|
| Component created (sync) | Body of component function | `OnInitialized()` |
| Component created (async) | `useEffect(() => {...}, [])` | `OnInitializedAsync()` |
| Parameters changed (sync) | Component function body (re-render) | `OnParametersSet()` |
| Parameters changed (async) | `useEffect(() => {...}, [dep])` | `OnParametersSetAsync()` |
| After DOM updated (sync) | `useLayoutEffect` | `OnAfterRender(firstRender)` |
| After DOM updated (async) | `useLayoutEffect` + async logic | `OnAfterRenderAsync(firstRender)` |
| Component destroyed | Cleanup function from `useEffect` | `Dispose()` / `DisposeAsync()` |
| Explicit re-render trigger | `setStateSetter(newValue)` | `StateHasChanged()` |
| Should component update? | `React.memo` comparison function | `ShouldRender()` override |

## Part 7: The Stale Closure Problem — Why It Does Not Exist in Blazor

One of the most important things to understand about Blazor's lifecycle model is that the stale closure problem — the entire class of bugs described at the beginning of this article — simply does not exist in Blazor in the same way.

In Blazor, your component is a C# class. Instance methods on that class share the same instance fields. When `OnInitialized` sets `_userId = 42`, and later a timer fires and calls `FetchOrders`, the `FetchOrders` method reads `this._userId` from the actual field, which is always the current value. There is no closure capturing an old snapshot of a variable.

Let us rewrite the broken React orders component in Blazor:

```razor
@implements IAsyncDisposable

<div class="counter">Pending Orders: @_orderCount</div>

@code {
    private int _orderCount = 0;
    private string? _userId;
    private PeriodicTimer? _pollingTimer;
    
    protected override async Task OnInitializedAsync()
    {
        var user = await UserService.GetCurrentUserAsync();
        _userId = user.Id;
        
        // Start polling — _userId is read from the field on each poll tick
        _pollingTimer = new PeriodicTimer(TimeSpan.FromSeconds(5));
        _ = PollOrdersAsync();
    }
    
    private async Task PollOrdersAsync()
    {
        try
        {
            while (await _pollingTimer!.WaitForNextTickAsync())
            {
                // _userId is read from the instance field here
                // It reflects the current value, not a snapshot from initialization
                var count = await OrderService.GetOrderCountAsync(_userId!);
                _orderCount = count;
                await InvokeAsync(StateHasChanged);
            }
        }
        catch (OperationCanceledException)
        {
            // Timer was disposed — expected shutdown
        }
    }
    
    public async ValueTask DisposeAsync()
    {
        _pollingTimer?.Dispose();
    }
}
```

Notice what is different:
- `_userId` is an instance field, not a local variable in a closure.
- `PollOrdersAsync` is an instance method that reads `_userId` when it runs, not when the method was created.
- When `_userId` changes (hypothetically, if you allowed it to change after initialization), the poll method would automatically pick up the new value because it reads from the field, not from a captured variable.
- There is no dependency array to maintain. There is no lint rule to check. The correctness is implicit in the object-oriented model.

This is not to say Blazor is bug-free — it is not. Blazor has its own class of subtle bugs, particularly around the interaction between `StateHasChanged`, async operations, and the Blazor synchronization context. But the stale closure class of bugs does not exist, and that removes a significant source of subtle, hard-to-debug problems.

## Part 8: React's Evolution Away from `useEffect`

It is worth noting that the React team is aware that `useEffect` is confusing and bug-prone. Several recent React features are specifically designed to reduce the need for `useEffect`:

### Server Components (React 19)

React Server Components (RSC) run entirely on the server and never on the client. They can fetch data directly (no `useEffect` needed) and render it to HTML that is sent to the browser. For components whose primary job is data fetching, RSC eliminates the need for `useEffect` entirely:

```jsx
// A React Server Component (runs on server, NO useEffect needed)
// This is a Next.js App Router component
async function BlogPostPage({ params }) {
    // Direct async data access — no useEffect, no useState, no loading state
    const post = await fetchPost(params.id);
    
    return (
        <article>
            <h1>{post.title}</h1>
            <p>{post.summary}</p>
        </article>
    );
}
```

This is a genuinely elegant solution to the data fetching problem. But it requires a server — which means it is not available in a purely static, SPA-style deployment. If you are deploying to GitHub Pages or an S3 bucket, you cannot use Server Components.

### The `use()` Hook (React 19)

React 19's `use()` hook allows you to consume a Promise directly in a component's render, with Suspense handling the loading state:

```jsx
import { use, Suspense } from 'react';

// Create the promise outside the component (important!)
const postsPromise = fetchPosts();

function PostList() {
    // use() suspends the component until the promise resolves
    const posts = use(postsPromise);
    return <ul>{posts.map(p => <li key={p.id}>{p.title}</li>)}</ul>;
}

function App() {
    return (
        <Suspense fallback={<p>Loading...</p>}>
            <PostList />
        </Suspense>
    );
}
```

This eliminates the need for `useEffect` + `useState` for simple data fetching cases. But it has the constraint that the promise must be created outside the component to avoid recreating it on every render — which creates its own set of questions about where that promise lives and how to refresh it.

### React 19.2's `useEffectEvent` Hook

The `useEffectEvent` hook (added in React 19.2, released October 2025) addresses one specific `useEffect` pattern that has been a source of bugs: the need for event handlers inside effects that need to read current state without being dependencies of the effect.

```jsx
// Before useEffectEvent (React 19.0 and earlier):
// This has a problem: 'onNotification' changes every render (new function reference)
// If we add it to deps, the effect re-runs every render, which is wrong.
// If we don't add it to deps, it's stale.
useEffect(() => {
    const socket = openSocket('/notifications');
    socket.onmessage = event => onNotification(event.data); // 'onNotification' is stale!
    return () => socket.close();
}, []); // ← onNotification not listed = stale closure bug

// After useEffectEvent (React 19.2):
// useEffectEvent creates a "non-reactive" handler — it always reads the latest
// value of its dependencies, but doesn't trigger effect re-execution when they change.
const onNotificationEvent = useEffectEvent(data => {
    onNotification(data); // Always reads the latest 'onNotification'
});

useEffect(() => {
    const socket = openSocket('/notifications');
    socket.onmessage = event => onNotificationEvent(event.data);
    return () => socket.close();
}, []); // ← Correct: socket doesn't depend on onNotification
```

`useEffectEvent` is specifically designed to handle "event-like" code inside effects that needs current state but should not trigger effect re-subscription when the values change. This is a direct acknowledgment that the dependency array model is too coarse-grained for some patterns.

## Part 9: Server Components in Blazor — A Different Approach to the Same Problem

Blazor also has a server-rendering model. Blazor Server runs component logic on the server and streams DOM updates to the client via SignalR. This is conceptually similar to React Server Components in that rendering happens on the server — but the mechanism is different.

Blazor's render mode infrastructure (introduced in .NET 8 and refined in .NET 10) allows you to annotate components with a render mode:

```razor
@* Render this component on the server, with interactivity via SignalR *@
@rendermode InteractiveServer

@* Render this component in the browser via WebAssembly *@
@rendermode InteractiveWebAssembly

@* Render this component on the server to static HTML only (no interactivity) *@
@rendermode null  @* or just omit — default is static SSR *@

@* Choose at runtime based on environment *@
@rendermode InteractiveAuto  @* Uses Server first, then WASM if available *@
```

For a static site deployment (our focus in this series), all interactive components use `InteractiveWebAssembly`. For a server-hosted application, you have the full flexibility of mixing render modes within a single application. This is Blazor's answer to the React ecosystem's Server Components story.

## Conclusion: The Lifecycle Mental Model

Let us be completely honest. React's hook-based lifecycle model is powerful and enables patterns that are difficult or impossible in a class-based system. The ability to encapsulate lifecycle behavior in custom hooks — reusing complex patterns across many components — is genuinely elegant:

```jsx
// A custom hook that encapsulates all the data fetching logic
function usePost(postId) {
    const [post, setPost] = useState(null);
    const [loading, setLoading] = useState(true);
    const [error, setError] = useState(null);
    
    useEffect(() => {
        setLoading(true);
        const controller = new AbortController();
        
        fetchPost(postId, { signal: controller.signal })
            .then(setPost)
            .catch(err => {
                if (err.name !== 'AbortError') setError(err);
            })
            .finally(() => setLoading(false));
        
        return () => controller.abort();
    }, [postId]);
    
    return { post, loading, error };
}

// Any component can use this without duplicating the logic:
function BlogPostPage({ postId }) {
    const { post, loading, error } = usePost(postId);
    if (loading) return <p>Loading...</p>;
    if (error) return <p>Error: {error.message}</p>;
    return <h1>{post.title}</h1>;
}
```

This is genuinely nice. The `usePost` hook is a reusable, testable unit that any component can use without knowing the implementation details.

But it requires understanding closures, dependency arrays, and the specific timing of effect execution. For developers who come from object-oriented backgrounds and think in terms of mutable state on objects, these concepts require a real shift in mental model.

Blazor's lifecycle methods — `OnInitializedAsync`, `OnParametersSetAsync`, `OnAfterRenderAsync`, `Dispose` — are less flexible but more predictable. They map to familiar object-oriented concepts. They do not have stale closure bugs. They are easier to reason about and easier to test.

The right choice depends on your team's background and the specific patterns your application needs. In Part 4, we are going to look at the scenario where the choice is forced: when you need a React library that has no Blazor equivalent, and you have to run both in the same application.
