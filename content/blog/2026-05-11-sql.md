# Relational Databases, SQL, and Second Normal Form: A Complete Guide for the Developer Who Has Been Getting Away With It

There is a particular kind of developer who has been building web applications for years — maybe even a decade — and has never truly understood their database. They write `SELECT * FROM Orders WHERE CustomerId = @id`, they call `SaveChanges()` on an Entity Framework `DbContext`, they paste a Stack Overflow migration script when something breaks, and somehow, improbably, the thing mostly works. The application ships. The customer is (usually) happy. Nobody notices.

Until they do.

This article is for that developer. It is not a gentle introduction. It is not a blog post that ends with "and that is all you need to know about databases!" and leaves you nodding contentedly in your ignorance. It is a long, thorough, sometimes uncomfortable reckoning with the foundations of relational databases — specifically the theory of normalization and the critical concept of Second Normal Form (2NF). If you are that developer, you will recognize yourself in several of the bad examples. That is intentional. Recognition is the beginning of repair.

We will start at the very beginning: what a relational database actually is, where it came from, and why it was invented. We will cover what data means, what a table means, what a key means. We will walk through First Normal Form (1NF), explain why it is not enough, and then spend a very long time on Second Normal Form — what it means, why partial dependencies are a disaster, and how to fix them. We will write a lot of SQL. We will write C# code that talks to that SQL. We will look at bad designs, explain why they are bad in precise terms, and then fix them. We will cover the practical implications for your ASP.NET applications, your Entity Framework models, and your day-to-day development workflow.

By the end, you should be able to look at a database schema and know whether it is in 2NF, why it matters, and what to do if it is not.

Let us begin.

## Part 1: What Is a Relational Database, and Why Should You Care?

### 1.1 The Problem That Relational Databases Solve

Before there were relational databases, there were flat files. Imagine it is 1965. You are a programmer at a large company — an insurance company, say. Your job is to track policyholders and their claims. You are using a magnetic tape. The tape stores records sequentially. A record looks like this:

```
SMITH,JOHN,123 MAIN ST,DALLAS,TX,75201,POL-001,AUTO,2024-01-15,COLLISION,FORD,F150
```

This works. Barely. You can write a COBOL program that reads through the tape looking for John Smith's record. But now your manager says: "We need to also track John Smith's dependents. And each policy can have multiple claims." Suddenly your flat file explodes. You either duplicate John Smith's address on every claim record, or you invent some baroque encoding scheme that only you understand.

This is not a hypothetical. Entire industries ran on this kind of data management for decades. The problems were severe and well-documented:

- **Redundancy**: The same information stored in dozens of places
- **Inconsistency**: When John Smith moves to Fort Worth, you update his address on three records but not on the fourth, and now you have contradictory data
- **Insertion anomalies**: You cannot record that Ford makes a truck called the F-150 unless you have a policyholder who owns one
- **Deletion anomalies**: If you delete the last claim for a policy, you lose the policy itself
- **Update anomalies**: Changing one fact requires finding and changing every copy of that fact

These are not minor inconveniences. They are fundamental flaws in how information is organized. Data that is stored redundantly is data that will eventually become inconsistent. Inconsistent data causes real harm: incorrect bills, lost records, bad business decisions, compliance failures.

Edgar F. Codd — a British computer scientist working at IBM — looked at these problems and saw that they had a mathematical solution.

### 1.2 Edgar F. Codd and the Relational Model

In June 1970, Edgar Codd published a paper titled "A Relational Model of Data for Large Shared Data Banks" in the journal *Communications of the ACM*. This paper is, without exaggeration, one of the most important documents in the history of computing. It introduced the relational model of data.

Codd's key insight was that data should be stored in **relations** — which you can think of as tables, though Codd was more precise about the term. A relation has a set of **attributes** (columns), and a set of **tuples** (rows). The order of rows does not matter. The order of columns does not matter. Each row represents a fact about the world. The relation as a whole represents a set of facts of the same kind.

Here is what this looks like in practice:

```sql
-- A relation called Customer
CREATE TABLE Customer (
    CustomerId   INT          NOT NULL,
    FirstName    VARCHAR(100) NOT NULL,
    LastName     VARCHAR(100) NOT NULL,
    Email        VARCHAR(255) NOT NULL,
    City         VARCHAR(100) NOT NULL,
    StateCode    CHAR(2)      NOT NULL,
    CONSTRAINT PK_Customer PRIMARY KEY (CustomerId)
);
```

Each row in this table represents a single customer. The table as a whole represents the set of all customers. Every column contains a single, indivisible value. There are no hidden arrays, no comma-separated lists, no "etc."

Codd followed his 1970 paper with "Further Normalization of the Data Base Relational Model" in 1971, in which he defined the first, second, and third normal forms. In 1974, Codd and Raymond F. Boyce jointly defined the Boyce-Codd Normal Form (BCNF). These papers established the theoretical foundation that underlies every relational database system in use today — Oracle, SQL Server, PostgreSQL, MySQL, SQLite. All of them.

The point is not historical trivia. The point is that normalization is not a style preference or a performance optimization tip. It is the theoretical basis for what relational databases are and why they work. When you violate normalization, you are not making a pragmatic shortcut. You are actively breaking the model.

### 1.3 What SQL Is and What It Is For

SQL — Structured Query Language — was developed at IBM in the early 1970s as a way to query relational databases. The language was originally called SEQUEL (Structured English Query Language). The first commercial SQL database system was Oracle in 1979. IBM released its own SQL system, DB2, in 1983. SQL was standardized by ANSI in 1986 and by ISO in 1987. Since then, the SQL standard has gone through numerous revisions: SQL-89, SQL-92, SQL:1999, SQL:2003, SQL:2008, SQL:2011, SQL:2016, SQL:2023.

SQL is a declarative language. You tell the database *what* you want, not *how* to get it. This is enormously powerful. You say:

```sql
SELECT c.FirstName, c.LastName, o.OrderDate, o.TotalAmount
FROM Customer c
INNER JOIN [Order] o ON o.CustomerId = c.CustomerId
WHERE c.StateCode = 'TX'
ORDER BY o.OrderDate DESC;
```

And the database figures out the most efficient way to actually retrieve that data — whether to use an index, how to perform the join, in what order to execute the operations. You do not need to know. That is the database's job.

This is only possible because the relational model gives the database engine strong guarantees about the structure and meaning of the data. A well-normalized schema makes it possible for the query optimizer to reason about what it is doing. A poorly normalized schema makes that reasoning unreliable.

### 1.4 The Vocabulary You Must Know

Before we can talk about normalization, we need precise definitions. These are not optional. Fuzzy understanding of these terms is exactly what allows bad database designs to persist.

**Relation**: A table. A set of tuples (rows) that share the same attributes (columns). In SQL, you create a relation with `CREATE TABLE`.

**Attribute**: A column in a table. Each attribute has a name and a domain (the set of allowed values — e.g., integers, strings up to 255 characters, dates).

**Tuple**: A row in a table. A tuple assigns a specific value from each attribute's domain to that attribute.

**Domain**: The set of valid values for an attribute. `StateCode CHAR(2)` has a domain of all two-character strings, or more specifically, the set of valid US state codes if you add a check constraint.

**Candidate key**: A minimal set of attributes that uniquely identifies every tuple in a relation. "Minimal" means you cannot remove any attribute from the set and still have it be unique. A table can have multiple candidate keys.

**Primary key**: The candidate key chosen to be the official identifier. Every table should have exactly one primary key. In SQL, you declare it with `PRIMARY KEY`.

**Foreign key**: An attribute (or set of attributes) in one table whose values correspond to the primary key of another table. Enforces referential integrity. Declared with `FOREIGN KEY ... REFERENCES`.

**Superkey**: Any set of attributes that uniquely identifies tuples. Every candidate key is a superkey, but not every superkey is a candidate key (because some superkeys include extra, unnecessary attributes).

**Prime attribute**: An attribute that is part of at least one candidate key.

**Non-prime attribute**: An attribute that is not part of any candidate key. Also called a non-key attribute.

**Functional dependency**: Attribute A functionally determines attribute B (written A → B) if every tuple with the same value of A has the same value of B. In other words, knowing A tells you B.

These definitions are precise. Learn them. We will use all of them.

## Part 2: The Disaster Zone — Unnormalized Data

### 2.1 A Case Study: The Order Management Database That Ate Itself

Let me tell you about Marcus. Marcus is a mid-level ASP.NET developer at a small e-commerce company. He has been building web applications for about four years. He knows C#. He can write LINQ queries. He uses Entity Framework. When a manager asks him to build a new feature for the order management system, he opens up his `DbContext`, creates a new entity class, adds a migration, and ships it.

The original order management database was designed in 2019 by a contractor who was paid by the hour and had strong incentives to finish quickly. The main table looks like this:

```sql
CREATE TABLE Orders (
    OrderId          INT          NOT NULL PRIMARY KEY,
    CustomerId       INT          NOT NULL,
    CustomerName     VARCHAR(200) NOT NULL,
    CustomerEmail    VARCHAR(255) NOT NULL,
    CustomerPhone    VARCHAR(20)  NULL,
    CustomerAddress  VARCHAR(500) NOT NULL,
    CustomerCity     VARCHAR(100) NOT NULL,
    CustomerState    CHAR(2)      NOT NULL,
    CustomerZip      VARCHAR(10)  NOT NULL,
    ProductId        INT          NOT NULL,
    ProductName      VARCHAR(200) NOT NULL,
    ProductCategory  VARCHAR(100) NOT NULL,
    ProductSKU       VARCHAR(50)  NOT NULL,
    SupplierName     VARCHAR(200) NOT NULL,
    SupplierCountry  VARCHAR(100) NOT NULL,
    Quantity         INT          NOT NULL,
    UnitPrice        DECIMAL(10,2) NOT NULL,
    OrderDate        DATETIME     NOT NULL,
    ShippingMethod   VARCHAR(50)  NOT NULL,
    TrackingNumber   VARCHAR(100) NULL
);
```

And here is some sample data:

```
OrderId | CustomerId | CustomerName | CustomerEmail      | ProductId | ProductName | ProductCategory | SupplierName   | SupplierCountry | Quantity | UnitPrice | ...
--------|------------|--------------|--------------------|-----------|-----------  |-----------------|----------------|-----------------|----------|-----------|
1       | 42         | Jane Doe     | jane@example.com   | 101       | Widget Pro  | Tools           | Acme Corp      | USA             | 2        | 19.99     |
2       | 42         | Jane Doe     | jane@example.com   | 102       | Gadget X    | Electronics     | TechCo Ltd     | China           | 1        | 89.99     |
3       | 43         | Bob Smith    | bob@example.com    | 101       | Widget Pro  | Tools           | Acme Corp      | USA             | 5        | 19.99     |
4       | 42         | Jane Doe     | jane.doe@new.com   | 103       | Doohickey   | Hardware        | Parts Plus     | Germany         | 1        | 34.99     |
```

Do you see the problems? Take a moment and look.

Jane Doe's email address changed between row 2 and row 4 — or did it? Is `jane@example.com` her old address and `jane.doe@new.com` her new one? Or is this a data entry error, and one of them is wrong? We cannot tell. The data is **inconsistent**.

Widget Pro (ProductId 101) appears in rows 1 and 3. Both rows store `ProductCategory = 'Tools'` and `SupplierName = 'Acme Corp'` and `SupplierCountry = 'USA'`. What happens if Acme Corp is acquired by a German company? You have to update every row that mentions Widget Pro — how many are there? Dozens? Thousands? And if you miss one, your data is inconsistent. This is an **update anomaly**.

What if you want to add a new product to the catalog — one nobody has ordered yet? You cannot. There is no product table. You can only record a product by creating an order for it. This is an **insertion anomaly**.

What if you delete order 3 — Bob Smith's only order? You have just destroyed the information that Widget Pro is made by Acme Corp in the USA. That information existed nowhere else. This is a **deletion anomaly**.

Marcus inherited this table. He has been adding to it for two years. He now has 847,000 rows. He is afraid to touch the schema because he does not fully understand it. When the manager asks "what supplier makes Widget Pro?", he writes a query that does `SELECT DISTINCT SupplierName FROM Orders WHERE ProductName = 'Widget Pro'` and hopes the results only have one row.

They usually do. Until they do not.

### 2.2 Why This Happens

Marcus did not set out to make a mess. He was working quickly, under pressure, in a codebase that already looked like this. The original contractor took the path of least resistance: put everything in one table, worry about it later.

This pattern — the massive, flat, do-everything table — is one of the most common forms of database dysfunction in the wild. It has a name: the **God Table** or **Universal Base Table**. You will recognize it by these symptoms:

- Column names with prefixes: `CustomerName`, `CustomerEmail`, `CustomerCity` — all in the same table as order data
- Repeated information across many rows
- NULL columns that only apply to certain types of rows
- Columns whose meaning changes depending on the value of another column
- Columns named things like `ExtraField1`, `ExtraField2`, `Notes`, `Misc`

The God Table is the result of someone modeling a database the way they would model a spreadsheet. In a spreadsheet, you put everything on one sheet. In a relational database, you put related things in related tables. These are fundamentally different approaches to information organization.

### 2.3 Counting the Redundancy

Let us count how much redundant data the Orders table above contains. For Jane Doe (CustomerId 42), we store her name, email, phone, address, city, state, and zip on every single order she places. If she places 50 orders in a year, we store her address 50 times. If there are 10,000 customers with an average of 20 orders each, we have 200,000 rows of customer contact information.

How much storage does that waste? Less than you think, actually — disk is cheap. But that is not the point. The point is **consistency**. Every copy of Jane's address is an opportunity for those copies to disagree. And they will.

More importantly, the duplication makes the meaning of the data ambiguous. When you see two different email addresses for the same CustomerId, you have no idea which one is correct. The database has no mechanism to tell you. If you had a separate `Customer` table with one row per customer, the email would appear exactly once. Contradiction would be impossible.

## Part 3: First Normal Form — Getting the Basics Right

### 3.1 What 1NF Requires

First Normal Form (1NF) is the most basic level of normalization. A table is in 1NF if and only if:

1. Every column contains only **atomic** (indivisible) values — no arrays, no comma-separated lists, no repeating groups
2. Every row is uniquely identified — there is a primary key

That is it. Two requirements.

The word "atomic" comes from the Greek *atomos*, meaning "indivisible." In the context of databases, an atomic value is a single, scalar value that cannot be meaningfully broken down further in the context of this database.

### 3.2 Violations of 1NF

Here are the most common violations of 1NF, along with their symptoms and fixes.

**Violation Type 1: Comma-separated lists in a single column**

```sql
-- BAD: This violates 1NF
CREATE TABLE BlogPost (
    PostId  INT          PRIMARY KEY,
    Title   VARCHAR(500) NOT NULL,
    Tags    VARCHAR(500) NOT NULL  -- stores "blazor,csharp,dotnet"
);
```

This looks harmless. Developers do this all the time. Here is why it is catastrophic:

- How do you find all posts tagged "blazor"? You have to use `LIKE '%blazor%'`, which cannot use an index and is fragile — what if a tag is "blazorwasm"?
- How do you count how many posts have each tag? You cannot, without parsing every Tags column in application code
- How do you enforce that "blazor" is a valid tag? You cannot
- How do you rename a tag from "csharp" to "csharp-14"? You have to parse and rewrite every row

The fix is always a separate table:

```sql
-- GOOD: Properly normalized tag structure
CREATE TABLE BlogPost (
    PostId  INT          NOT NULL,
    Title   VARCHAR(500) NOT NULL,
    CONSTRAINT PK_BlogPost PRIMARY KEY (PostId)
);

CREATE TABLE Tag (
    TagId   INT          NOT NULL,
    Name    VARCHAR(100) NOT NULL,
    CONSTRAINT PK_Tag PRIMARY KEY (TagId),
    CONSTRAINT UQ_Tag_Name UNIQUE (Name)
);

CREATE TABLE BlogPostTag (
    PostId  INT NOT NULL,
    TagId   INT NOT NULL,
    CONSTRAINT PK_BlogPostTag PRIMARY KEY (PostId, TagId),
    CONSTRAINT FK_BlogPostTag_Post FOREIGN KEY (PostId) REFERENCES BlogPost (PostId),
    CONSTRAINT FK_BlogPostTag_Tag  FOREIGN KEY (TagId)  REFERENCES Tag (TagId)
);
```

Now finding all posts tagged "blazor" is a straightforward join. Counting posts per tag is `GROUP BY`. Renaming a tag is a single `UPDATE` on one row. Every operation that was difficult or impossible is now simple.

**Violation Type 2: Repeating column groups**

```sql
-- BAD: This violates 1NF
CREATE TABLE Customer (
    CustomerId   INT          PRIMARY KEY,
    Name         VARCHAR(200) NOT NULL,
    Phone1       VARCHAR(20)  NULL,
    Phone2       VARCHAR(20)  NULL,
    Phone3       VARCHAR(20)  NULL  -- "up to three phones"
);
```

You have seen this. We all have. The developer could not put a list in one column, so they made three columns. Now:

- What if a customer has four phones?
- How do you find all customers who have a specific phone number? `WHERE Phone1 = @num OR Phone2 = @num OR Phone3 = @num` — this is a mess
- How do you count how many phone numbers each customer has? Painful
- What does it mean for a customer to have Phone3 filled in but Phone1 and Phone2 NULL? Nothing, but it happens

Fix:

```sql
-- GOOD
CREATE TABLE Customer (
    CustomerId INT          NOT NULL,
    Name       VARCHAR(200) NOT NULL,
    CONSTRAINT PK_Customer PRIMARY KEY (CustomerId)
);

CREATE TABLE CustomerPhone (
    PhoneId    INT         NOT NULL,
    CustomerId INT         NOT NULL,
    PhoneType  VARCHAR(20) NOT NULL,  -- 'Mobile', 'Home', 'Work'
    Number     VARCHAR(20) NOT NULL,
    CONSTRAINT PK_CustomerPhone PRIMARY KEY (PhoneId),
    CONSTRAINT FK_CustomerPhone_Customer FOREIGN KEY (CustomerId) REFERENCES Customer (CustomerId)
);
```

**Violation Type 3: No primary key**

```sql
-- BAD: No primary key, rows are not uniquely identifiable
CREATE TABLE EventLog (
    EventTime   DATETIME     NOT NULL,
    EventType   VARCHAR(100) NOT NULL,
    Description VARCHAR(MAX) NOT NULL
);
```

If two events occur at the same time with the same type and description, you have two indistinguishable rows. You cannot UPDATE or DELETE a specific row. You cannot create a foreign key pointing to this table. Add a surrogate key:

```sql
-- GOOD
CREATE TABLE EventLog (
    EventLogId  BIGINT       NOT NULL IDENTITY(1,1),
    EventTime   DATETIME2    NOT NULL DEFAULT SYSUTCDATETIME(),
    EventType   VARCHAR(100) NOT NULL,
    Description VARCHAR(MAX) NOT NULL,
    CONSTRAINT PK_EventLog PRIMARY KEY (EventLogId)
);
```

### 3.3 The C# Entity Framework Equivalent

In ASP.NET with Entity Framework Core, 1NF violations often look like this:

```csharp
// BAD: EF model that maps to a 1NF violation
public class Order
{
    public int OrderId { get; set; }
    public int CustomerId { get; set; }
    
    // Customer data duplicated here instead of in a Customer entity
    public string CustomerName { get; set; } = "";
    public string CustomerEmail { get; set; } = "";
    public string CustomerCity { get; set; } = "";
    
    // Comma-separated product IDs — a 1NF violation in EF clothing
    public string ProductIds { get; set; } = "";
    
    // Repeating groups
    public string? ProductName1 { get; set; }
    public string? ProductName2 { get; set; }
    public string? ProductName3 { get; set; }
}
```

This entity will map to a table that violates 1NF. EF is just a code interface to the database — if your entities are structured badly, your database will be structured badly. EF does not save you from poor design. It faithfully implements whatever design you give it.

The correct EF model has proper navigation properties:

```csharp
// GOOD: Properly structured EF entities
public class Customer
{
    public int CustomerId { get; set; }
    public string Name { get; set; } = "";
    public string Email { get; set; } = "";
    
    // Navigation property — EF will handle the join
    public ICollection<Order> Orders { get; set; } = [];
}

public class Order
{
    public int OrderId { get; set; }
    public int CustomerId { get; set; }
    public DateTime OrderDate { get; set; }
    
    public Customer Customer { get; set; } = null!;
    public ICollection<OrderLine> OrderLines { get; set; } = [];
}

public class OrderLine
{
    public int OrderLineId { get; set; }
    public int OrderId { get; set; }
    public int ProductId { get; set; }
    public int Quantity { get; set; }
    public decimal UnitPrice { get; set; }
    
    public Order Order { get; set; } = null!;
    public Product Product { get; set; } = null!;
}

public class Product
{
    public int ProductId { get; set; }
    public string Name { get; set; } = "";
    public string SKU { get; set; } = "";
    public int CategoryId { get; set; }
    
    public ProductCategory Category { get; set; } = null!;
    public ICollection<OrderLine> OrderLines { get; set; } = [];
}
```

The navigation properties (`Customer.Orders`, `Order.OrderLines`, `OrderLine.Product`) tell EF how the tables are related. EF will use these relationships to generate the appropriate `JOIN` queries when you use `.Include()`.

### 3.4 Checking 1NF: A Practical Checklist

Before you declare a table to be in 1NF, run through this checklist:

- [ ] Does every column contain exactly one value per row? (No lists, no arrays, no JSON blobs that are actually relational data in disguise)
- [ ] Are there any repeating column groups (e.g., Address1, Address2, Address3)?
- [ ] Does the table have a primary key?
- [ ] Is every row distinguishable from every other row by its primary key alone?
- [ ] Does every column have a consistent meaning across all rows? (No "this column means X if StatusCode is 1, or Y if StatusCode is 2")

If you can check all of these boxes, your table is in 1NF.

But 1NF is not enough. Not even close.

## Part 4: Functional Dependencies — The Heart of Normalization

### 4.1 What Functional Dependency Means

Before we can talk about 2NF, we need to deeply understand functional dependencies. This is the concept that everything else is built on.

**Formal definition**: Attribute B is functionally dependent on attribute A (written A → B) if, for every valid state of the relation, knowing the value of A uniquely determines the value of B. In other words, there are never two tuples with the same A value but different B values.

**Informal definition**: Knowing A tells you B.

Let us use concrete examples.

In a `Customer` table:
- `CustomerId → CustomerName` (knowing a customer's ID tells you their name — each customer has exactly one name)
- `CustomerId → Email` (knowing a customer's ID tells you their email)
- `Email → CustomerId` (if emails are unique, knowing someone's email tells you their ID)
- `CustomerName → CustomerId`? **No.** Multiple customers can have the same name. Knowing "Jane Doe" does not tell you which customer.

In a `Product` table:
- `ProductId → ProductName` (each product has one name)
- `ProductId → CategoryId` (each product belongs to one category)
- `SKU → ProductId` (if SKUs are unique, knowing the SKU tells you the product)
- `ProductName → ProductId`? **No.** You could have products with the same name.

In an `OrderLine` table with a composite primary key `{OrderId, ProductId}`:
- `{OrderId, ProductId} → Quantity` (the quantity is specific to this order-product combination)
- `{OrderId, ProductId} → UnitPrice` (the price at time of ordering is specific to this combination)
- `OrderId → OrderDate` (knowing the order ID tells you the order date — but this depends only on OrderId, not the combination!)
- `ProductId → ProductName` (knowing the product tells you its name — depends only on ProductId!)

Those last two are the key insight for 2NF.

### 4.2 Full vs. Partial Functional Dependencies

This is where 2NF comes in.

**Full functional dependency**: B is fully functionally dependent on A if B is functionally dependent on A and is *not* functionally dependent on any proper subset of A. In other words, you need all of A to determine B.

**Partial functional dependency**: B is partially functionally dependent on A if B is functionally dependent on A but is *also* functionally dependent on some proper subset of A. In other words, you do not need all of A — part of A is enough.

Partial dependencies only occur when the key is **composite** (made up of multiple attributes). If the primary key is a single column, all non-prime attributes are automatically fully dependent on it (because there is no proper subset of a single-attribute key).

Let us look at an `OrderLine` table to see this in action:

```sql
-- A table that is in 1NF but not 2NF
CREATE TABLE OrderLine (
    OrderId         INT          NOT NULL,
    ProductId       INT          NOT NULL,
    ProductName     VARCHAR(200) NOT NULL,  -- depends only on ProductId!
    ProductCategory VARCHAR(100) NOT NULL,  -- depends only on ProductId!
    CustomerName    VARCHAR(200) NOT NULL,  -- depends only on OrderId (via Order.CustomerId)!
    Quantity        INT          NOT NULL,  -- depends on {OrderId, ProductId} — correct
    UnitPrice       DECIMAL(10,2) NOT NULL, -- depends on {OrderId, ProductId} — correct
    CONSTRAINT PK_OrderLine PRIMARY KEY (OrderId, ProductId)
);
```

The primary key is `{OrderId, ProductId}`. Now let us analyze each non-prime attribute:

| Attribute | Depends on | Full or Partial? |
|---|---|---|
| ProductName | ProductId only | **PARTIAL** — violation |
| ProductCategory | ProductId only | **PARTIAL** — violation |
| CustomerName | OrderId only | **PARTIAL** — violation |
| Quantity | {OrderId, ProductId} | Full — correct |
| UnitPrice | {OrderId, ProductId} | Full — correct |

Three of the five non-prime attributes have partial dependencies. This table is not in 2NF.

### 4.3 Why Partial Dependencies Are Dangerous

Let us see exactly what goes wrong when you have partial dependencies.

Suppose the `OrderLine` table has this data:

```
OrderId | ProductId | ProductName  | ProductCategory | CustomerName | Quantity | UnitPrice
--------|-----------|--------------|-----------------|--------------|----------|----------
1001    | 5         | Widget Pro   | Tools           | Jane Doe     | 2        | 19.99
1001    | 8         | Gadget X     | Electronics     | Jane Doe     | 1        | 89.99
1002    | 5         | Widget Pro   | Tools           | Bob Smith    | 3        | 19.99
1003    | 5         | Widget Pro   | Tools           | Alice Brown  | 1        | 19.99
```

**Update anomaly**: Acme Corp renames "Widget Pro" to "Widget Professional Edition." You run:

```sql
UPDATE OrderLine SET ProductName = 'Widget Professional Edition' WHERE ProductId = 5;
```

This updates three rows. But what if your `UPDATE` statement had a `WHERE` clause with an additional condition and you only updated two rows? Now you have inconsistent data — some rows say "Widget Pro" and others say "Widget Professional Edition," and both refer to the same product.

**Insertion anomaly**: You want to add a new product — "Doohickey" in category "Hardware" — but nobody has ordered it yet. You cannot. There is no place to record a product without an order. The product information lives inside the `OrderLine` table, which requires an order to exist.

**Deletion anomaly**: Bob Smith (order 1002) cancels his order. You delete the row. You have just lost the information that ProductId 5 belongs to the "Tools" category and is named "Widget Pro" — unless other orders for the same product still exist. If all orders for a product are deleted, the product itself vanishes from your database.

These are not edge cases. They are the normal operation of any system that has partial dependencies. They guarantee that your data will become inconsistent over time.

### 4.4 Identifying Functional Dependencies in Practice

How do you actually identify functional dependencies when you are looking at a real schema? There is a mechanical approach.

For each pair of attributes A and B in a table, ask: "Could two rows ever have the same value of A but different values of B?" If the answer is "no," then A → B.

**Tip**: Think about the real-world entities your data represents. A product has one name. A product has one category. A customer has one email (assuming email is unique). An order has one date. The functional dependencies follow from the nature of the entities.

**Another tip**: Primary key → everything else in the table. That is the whole point of a primary key. The question is whether there are also *other* functional dependencies — specifically, whether any non-prime attribute is determined by only *part* of the primary key.

Here is a worked example. You are looking at this table:

```sql
CREATE TABLE StudentCourse (
    StudentId    INT          NOT NULL,
    CourseId     INT          NOT NULL,
    StudentName  VARCHAR(200) NOT NULL,
    StudentEmail VARCHAR(255) NOT NULL,
    CourseName   VARCHAR(200) NOT NULL,
    Credits      INT          NOT NULL,
    InstructorId INT          NOT NULL,
    Grade        CHAR(2)      NULL,
    CONSTRAINT PK_StudentCourse PRIMARY KEY (StudentId, CourseId)
);
```

Analyze each non-prime attribute:

- `StudentName`: Determined by `StudentId` alone — same student, same name, regardless of which course. **Partial dependency.**
- `StudentEmail`: Determined by `StudentId` alone. **Partial dependency.**
- `CourseName`: Determined by `CourseId` alone. **Partial dependency.**
- `Credits`: Determined by `CourseId` alone (a course has a fixed number of credits). **Partial dependency.**
- `InstructorId`: Determined by `CourseId` alone (a course has one instructor). **Partial dependency.**
- `Grade`: Determined by `{StudentId, CourseId}` — the grade is specific to a student's enrollment in a specific course. **Full dependency. Correct.**

Everything except `Grade` is a partial dependency. This table has severe 2NF violations.

## Part 5: Second Normal Form — The Definition and the Fix

### 5.1 The Formal Definition of 2NF

A relation is in **Second Normal Form (2NF)** if and only if:

1. It is in First Normal Form (1NF), **and**
2. Every non-prime attribute is **fully functionally dependent** on every candidate key

Equivalently: a relation is in 2NF if it is in 1NF and has **no partial dependencies**.

Note the precise language: it is every non-prime attribute and every candidate key. Not just the primary key you chose — all candidate keys. This matters in tables with multiple candidate keys, though it is less common in practice.

Also note: **2NF is only violated by composite keys**. If your primary key is a single column, and you have no other composite candidate keys, your table is automatically in 2NF (assuming it is in 1NF). This is why surrogate integer primary keys (identity columns) help prevent certain kinds of normalization violations — they are always single-column keys.

### 5.2 Achieving 2NF: The Decomposition Process

The fix for a 2NF violation is always the same: **decompose the relation**. Move each partial dependency into its own relation where the partially-determining attribute(s) become the primary key.

Let us fix the `OrderLine` table from section 4.2.

**Step 1: Identify all partial dependencies.**

From our earlier analysis:
- `ProductId → {ProductName, ProductCategory}` (partial dependency on ProductId)
- `OrderId → {CustomerName}` (partial dependency on OrderId)
- `{OrderId, ProductId} → {Quantity, UnitPrice}` (full dependency — keep this here)

**Step 2: Create a new relation for each partial dependency.**

For `ProductId → {ProductName, ProductCategory}`:

```sql
CREATE TABLE Product (
    ProductId       INT          NOT NULL,
    ProductName     VARCHAR(200) NOT NULL,
    ProductCategory VARCHAR(100) NOT NULL,
    CONSTRAINT PK_Product PRIMARY KEY (ProductId)
);
```

For `OrderId → {CustomerName}`:

Wait. This is actually revealing a problem. `CustomerName` should not be in `OrderLine` at all. It is information about the customer (or the order), not about the order-line combination. We need a proper `Order` table:

```sql
CREATE TABLE [Order] (
    OrderId      INT          NOT NULL,
    CustomerId   INT          NOT NULL,
    CustomerName VARCHAR(200) NOT NULL,  -- denormalized for display convenience
    OrderDate    DATETIME2    NOT NULL,
    CONSTRAINT PK_Order PRIMARY KEY (OrderId)
);
```

Actually, `CustomerName` should not be in `Order` either — we are still duplicating it. Let us do this properly:

```sql
CREATE TABLE Customer (
    CustomerId INT          NOT NULL,
    Name       VARCHAR(200) NOT NULL,
    Email      VARCHAR(255) NOT NULL,
    CONSTRAINT PK_Customer PRIMARY KEY (CustomerId)
);

CREATE TABLE [Order] (
    OrderId    INT       NOT NULL,
    CustomerId INT       NOT NULL,
    OrderDate  DATETIME2 NOT NULL,
    CONSTRAINT PK_Order PRIMARY KEY (OrderId),
    CONSTRAINT FK_Order_Customer FOREIGN KEY (CustomerId) REFERENCES Customer (CustomerId)
);
```

**Step 3: Rebuild the original relation with only its genuine attributes.**

After removing all partial dependencies, `OrderLine` retains only what truly depends on the composite key `{OrderId, ProductId}`:

```sql
CREATE TABLE OrderLine (
    OrderId   INT           NOT NULL,
    ProductId INT           NOT NULL,
    Quantity  INT           NOT NULL,
    UnitPrice DECIMAL(10,2) NOT NULL,
    CONSTRAINT PK_OrderLine PRIMARY KEY (OrderId, ProductId),
    CONSTRAINT FK_OrderLine_Order   FOREIGN KEY (OrderId)   REFERENCES [Order] (OrderId),
    CONSTRAINT FK_OrderLine_Product FOREIGN KEY (ProductId) REFERENCES Product (ProductId)
);
```

Now let us verify: every non-prime attribute in `OrderLine` is `Quantity` and `UnitPrice`. Do they depend on `{OrderId, ProductId}`? Yes — the quantity ordered and the price at the time of the order are specific to a particular product on a particular order. No partial dependencies.

**The complete, 2NF-compliant schema:**

```sql
CREATE TABLE Customer (
    CustomerId INT          NOT NULL,
    Name       VARCHAR(200) NOT NULL,
    Email      VARCHAR(255) NOT NULL,
    CONSTRAINT PK_Customer PRIMARY KEY (CustomerId)
);

CREATE TABLE ProductCategory (
    CategoryId   INT          NOT NULL,
    CategoryName VARCHAR(100) NOT NULL,
    CONSTRAINT PK_ProductCategory PRIMARY KEY (CategoryId)
);

CREATE TABLE Product (
    ProductId  INT          NOT NULL,
    Name       VARCHAR(200) NOT NULL,
    SKU        VARCHAR(50)  NOT NULL,
    CategoryId INT          NOT NULL,
    CONSTRAINT PK_Product PRIMARY KEY (ProductId),
    CONSTRAINT UQ_Product_SKU UNIQUE (SKU),
    CONSTRAINT FK_Product_Category FOREIGN KEY (CategoryId) REFERENCES ProductCategory (CategoryId)
);

CREATE TABLE [Order] (
    OrderId    INT       NOT NULL,
    CustomerId INT       NOT NULL,
    OrderDate  DATETIME2 NOT NULL,
    CONSTRAINT PK_Order PRIMARY KEY (OrderId),
    CONSTRAINT FK_Order_Customer FOREIGN KEY (CustomerId) REFERENCES Customer (CustomerId)
);

CREATE TABLE OrderLine (
    OrderId   INT           NOT NULL,
    ProductId INT           NOT NULL,
    Quantity  INT           NOT NULL,
    UnitPrice DECIMAL(10,2) NOT NULL,
    CONSTRAINT PK_OrderLine PRIMARY KEY (OrderId, ProductId),
    CONSTRAINT FK_OrderLine_Order   FOREIGN KEY (OrderId)   REFERENCES [Order] (OrderId),
    CONSTRAINT FK_OrderLine_Product FOREIGN KEY (ProductId) REFERENCES Product (ProductId)
);
```

This schema has five tables instead of one. Each table has a clear, single purpose. Each fact is stored exactly once. The update anomaly is gone: to rename a product, you update one row in `Product`. The insertion anomaly is gone: you can add a product before anyone orders it. The deletion anomaly is gone: deleting an order does not delete the product.

### 5.3 The StudentCourse Example, Fully Fixed

Let us also fix the `StudentCourse` table from section 4.4.

Original violations:
- `StudentId → {StudentName, StudentEmail}` (partial)
- `CourseId → {CourseName, Credits, InstructorId}` (partial)
- `{StudentId, CourseId} → {Grade}` (full — keep)

Decomposed:

```sql
CREATE TABLE Student (
    StudentId INT          NOT NULL,
    Name      VARCHAR(200) NOT NULL,
    Email     VARCHAR(255) NOT NULL,
    CONSTRAINT PK_Student PRIMARY KEY (StudentId)
);

CREATE TABLE Instructor (
    InstructorId INT          NOT NULL,
    Name         VARCHAR(200) NOT NULL,
    Email        VARCHAR(255) NOT NULL,
    CONSTRAINT PK_Instructor PRIMARY KEY (InstructorId)
);

CREATE TABLE Course (
    CourseId     INT          NOT NULL,
    Name         VARCHAR(200) NOT NULL,
    Credits      INT          NOT NULL,
    InstructorId INT          NOT NULL,
    CONSTRAINT PK_Course PRIMARY KEY (CourseId),
    CONSTRAINT FK_Course_Instructor FOREIGN KEY (InstructorId) REFERENCES Instructor (InstructorId)
);

CREATE TABLE Enrollment (
    StudentId INT    NOT NULL,
    CourseId  INT    NOT NULL,
    Grade     CHAR(2) NULL,
    CONSTRAINT PK_Enrollment PRIMARY KEY (StudentId, CourseId),
    CONSTRAINT FK_Enrollment_Student FOREIGN KEY (StudentId) REFERENCES Student (StudentId),
    CONSTRAINT FK_Enrollment_Course  FOREIGN KEY (CourseId)  REFERENCES Course (CourseId)
);
```

Now:
- Update a student's email: one `UPDATE` on `Student`. Done.
- Add a new course before any students enroll: `INSERT INTO Course (...)`. Done.
- Drop a course (delete all enrollments): course information is preserved in `Course`. Done.
- Find all students in a given course: join `Enrollment` and `Student`. Straightforward.

### 5.4 The Toothbrush Example from Wikipedia

The 2NF Wikipedia article (which references Codd's original work) uses an elegant toothbrush manufacturing example that is worth covering because it is compact and clear.

Consider this relation in 1NF:

```
Toothbrush(Manufacturer, Model, ManufacturerCountry)
Primary key: {Manufacturer, Model}
```

Sample data:

```
Manufacturer  | Model      | ManufacturerCountry
-----------   | ---------- | -------------------
Aquabrush Co  | Slim 200   | USA
Aquabrush Co  | Pro 500    | USA
BrushMaster   | Classic    | Germany
BrushMaster   | Sport Plus | Germany
CleanTech     | Basic      | Japan
```

The functional dependency `Manufacturer → ManufacturerCountry` is a **partial dependency**, because `ManufacturerCountry` depends only on `Manufacturer`, which is just part of the composite key `{Manufacturer, Model}`.

The fix:

```sql
-- BEFORE (violates 2NF)
CREATE TABLE Toothbrush (
    Manufacturer        VARCHAR(200) NOT NULL,
    Model               VARCHAR(200) NOT NULL,
    ManufacturerCountry VARCHAR(100) NOT NULL,
    CONSTRAINT PK_Toothbrush PRIMARY KEY (Manufacturer, Model)
);

-- AFTER (in 2NF)
CREATE TABLE Manufacturer (
    ManufacturerName    VARCHAR(200) NOT NULL,
    ManufacturerCountry VARCHAR(100) NOT NULL,
    CONSTRAINT PK_Manufacturer PRIMARY KEY (ManufacturerName)
);

CREATE TABLE Toothbrush (
    ManufacturerName VARCHAR(200) NOT NULL,
    Model            VARCHAR(200) NOT NULL,
    CONSTRAINT PK_Toothbrush PRIMARY KEY (ManufacturerName, Model),
    CONSTRAINT FK_Toothbrush_Manufacturer 
        FOREIGN KEY (ManufacturerName) REFERENCES Manufacturer (ManufacturerName)
);
```

Now `ManufacturerCountry` is in `Manufacturer`, where it belongs. If Aquabrush Co moves its headquarters from USA to Canada, you update one row in `Manufacturer`. Done. No need to find all the toothbrush models and update them individually.

## Part 6: Anomalies in Depth — What 2NF Prevents

### 6.1 Insertion Anomalies

An **insertion anomaly** occurs when you cannot record information about one entity without first recording information about another entity.

In the unnormalized `OrderLine` table (before our fix), you cannot record that a new product exists without creating an order for it. This is because product information (name, category) is stored inside order lines. There is nowhere for a product to live independently.

After normalizing to 2NF with a separate `Product` table, you can freely add products:

```sql
INSERT INTO Product (ProductId, Name, SKU, CategoryId)
VALUES (999, 'Future Gadget 3000', 'FG-3000', 5);
```

No order required. The product exists in the database independently of whether anyone has ordered it.

In the `StudentCourse` table, you could not add a new course without enrolling a student in it (because course data lived inside the enrollment record). After normalization, you insert into `Course` independently of `Enrollment`.

### 6.2 Update Anomalies

An **update anomaly** occurs when changing one real-world fact requires updating multiple rows in the database, creating the risk that some copies are updated and others are not.

**Classic example**: A company changes its name. If the company name is stored in 50 different rows in an `Order` or `Supplier` table, you must update all 50. Miss one, and your database contains contradictory information about which company supplied which products.

After 2NF normalization, the company name appears in one row of one table. Update that one row. Done. Consistency is guaranteed.

In SQL Server, you can even use `ON UPDATE CASCADE` on foreign keys to automatically propagate primary key changes:

```sql
CREATE TABLE Supplier (
    SupplierId   INT          NOT NULL,
    SupplierName VARCHAR(200) NOT NULL,
    Country      VARCHAR(100) NOT NULL,
    CONSTRAINT PK_Supplier PRIMARY KEY (SupplierId)
);

CREATE TABLE Product (
    ProductId  INT          NOT NULL,
    Name       VARCHAR(200) NOT NULL,
    SupplierId INT          NOT NULL,
    CONSTRAINT PK_Product PRIMARY KEY (ProductId),
    CONSTRAINT FK_Product_Supplier 
        FOREIGN KEY (SupplierId) REFERENCES Supplier (SupplierId)
        ON UPDATE CASCADE
        ON DELETE RESTRICT
);
```

### 6.3 Deletion Anomalies

A **deletion anomaly** occurs when deleting a tuple causes the unintentional loss of information about another entity.

**Classic example**: In the God Table from section 2.1, if you delete the last order that references a particular product, you lose all information about that product — its name, its category, its supplier. The product is gone from your system.

This is particularly nasty because it happens silently. No error is thrown. The data just vanishes. You discover it weeks later when a manager asks "why is product P-101 missing from the catalog?" and nobody knows.

After 2NF normalization with a separate `Product` table, deleting an order line only affects the `OrderLine` row. The `Product` row remains intact. You could add a foreign key constraint with `ON DELETE RESTRICT` or `ON DELETE NO ACTION` to prevent deletion of a product that still has order lines:

```sql
CONSTRAINT FK_OrderLine_Product 
    FOREIGN KEY (ProductId) REFERENCES Product (ProductId)
    ON DELETE RESTRICT
```

This makes deletion anomalies impossible by construction. The database will refuse to delete a product that has associated order lines.

### 6.4 The Anomaly Detective Game

Here is a useful mental exercise when reviewing a database schema. For each table, ask these three questions:

1. **Insertion**: Can I insert a row about X without having to also record Y? If not, why not?
2. **Update**: If I change a fact about X, how many rows do I need to update? If more than one, that is a red flag.
3. **Deletion**: If I delete a row about X, do I lose information about Y? If so, that is a red flag.

If the answers reveal problems, you have found a normalization violation. Trace it back to the functional dependencies.

## Part 7: Walking Through the Normalization Process — A Complete Case Study

### 7.1 The Scenario: A Library Management System

Let us design a library management system database from scratch, starting with a completely unnormalized approach (as a real-world developer might initially approach it), identifying all the problems, and then normalizing step by step to 2NF.

The system needs to track:
- Books and their authors
- Library members
- Book checkouts (borrowing records)
- Fines for overdue books

A hasty first draft might produce this single table:

```sql
CREATE TABLE LibraryRecord (
    RecordId        INT           NOT NULL PRIMARY KEY,
    MemberId        INT           NOT NULL,
    MemberName      VARCHAR(200)  NOT NULL,
    MemberEmail     VARCHAR(255)  NOT NULL,
    MemberPhone     VARCHAR(20)   NULL,
    BookId          INT           NOT NULL,
    BookTitle       VARCHAR(500)  NOT NULL,
    BookISBN        VARCHAR(20)   NOT NULL,
    AuthorName      VARCHAR(200)  NOT NULL,  -- assumes one author per book
    PublisherName   VARCHAR(200)  NOT NULL,
    PublisherCity   VARCHAR(100)  NOT NULL,
    GenreName       VARCHAR(100)  NOT NULL,
    CheckoutDate    DATE          NOT NULL,
    DueDate         DATE          NOT NULL,
    ReturnDate      DATE          NULL,
    FinePerDay      DECIMAL(5,2)  NOT NULL,
    TotalFine       DECIMAL(8,2)  NULL,
    FineStatus      VARCHAR(20)   NULL  -- 'PAID', 'UNPAID', 'WAIVED'
);
```

Every problem we have discussed is present here:
- `MemberName`, `MemberEmail`, `MemberPhone` depend only on `MemberId` — partial conceptual dependency (though the PK is not composite here, the God Table structure is still a problem)
- `BookTitle`, `BookISBN`, `AuthorName`, `PublisherName`, `PublisherCity`, `GenreName` depend only on `BookId`
- `FinePerDay` depends only on — what? Is it per-book, per-library-policy, per-checkout? This is ambiguous.
- Assumes one author per book (violates 1NF conceptually for multi-author books)

Let us fix it properly.

### 7.2 Step 1: Normalize to 1NF

First, let us give `RecordId` a real identity and address the single-author assumption. Books can have multiple authors; we need a junction table.

```sql
-- Separate entities into their own tables
CREATE TABLE Member (
    MemberId    INT          NOT NULL,
    Name        VARCHAR(200) NOT NULL,
    Email       VARCHAR(255) NOT NULL,
    Phone       VARCHAR(20)  NULL,
    JoinDate    DATE         NOT NULL,
    CONSTRAINT PK_Member PRIMARY KEY (MemberId),
    CONSTRAINT UQ_Member_Email UNIQUE (Email)
);

CREATE TABLE Publisher (
    PublisherId   INT          NOT NULL,
    Name          VARCHAR(200) NOT NULL,
    City          VARCHAR(100) NOT NULL,
    CONSTRAINT PK_Publisher PRIMARY KEY (PublisherId)
);

CREATE TABLE Genre (
    GenreId   INT          NOT NULL,
    Name      VARCHAR(100) NOT NULL,
    CONSTRAINT PK_Genre PRIMARY KEY (GenreId),
    CONSTRAINT UQ_Genre_Name UNIQUE (Name)
);

CREATE TABLE Book (
    BookId      INT          NOT NULL,
    Title       VARCHAR(500) NOT NULL,
    ISBN        VARCHAR(20)  NOT NULL,
    PublisherId INT          NOT NULL,
    GenreId     INT          NOT NULL,
    CONSTRAINT PK_Book PRIMARY KEY (BookId),
    CONSTRAINT UQ_Book_ISBN UNIQUE (ISBN),
    CONSTRAINT FK_Book_Publisher FOREIGN KEY (PublisherId) REFERENCES Publisher (PublisherId),
    CONSTRAINT FK_Book_Genre     FOREIGN KEY (GenreId)     REFERENCES Genre (GenreId)
);

CREATE TABLE Author (
    AuthorId INT          NOT NULL,
    Name     VARCHAR(200) NOT NULL,
    CONSTRAINT PK_Author PRIMARY KEY (AuthorId)
);

-- Junction table: a book can have multiple authors, an author can write multiple books
CREATE TABLE BookAuthor (
    BookId   INT NOT NULL,
    AuthorId INT NOT NULL,
    CONSTRAINT PK_BookAuthor PRIMARY KEY (BookId, AuthorId),
    CONSTRAINT FK_BookAuthor_Book   FOREIGN KEY (BookId)   REFERENCES Book (BookId),
    CONSTRAINT FK_BookAuthor_Author FOREIGN KEY (AuthorId) REFERENCES Author (AuthorId)
);

CREATE TABLE Checkout (
    CheckoutId   INT           NOT NULL,
    MemberId     INT           NOT NULL,
    BookId       INT           NOT NULL,
    CheckoutDate DATE          NOT NULL,
    DueDate      DATE          NOT NULL,
    ReturnDate   DATE          NULL,
    CONSTRAINT PK_Checkout PRIMARY KEY (CheckoutId),
    CONSTRAINT FK_Checkout_Member FOREIGN KEY (MemberId) REFERENCES Member (MemberId),
    CONSTRAINT FK_Checkout_Book   FOREIGN KEY (BookId)   REFERENCES Book (BookId)
);

CREATE TABLE Fine (
    FineId     INT           NOT NULL,
    CheckoutId INT           NOT NULL,
    AmountDue  DECIMAL(8,2)  NOT NULL,
    Status     VARCHAR(20)   NOT NULL DEFAULT 'UNPAID',  -- 'PAID', 'UNPAID', 'WAIVED'
    PaidDate   DATE          NULL,
    CONSTRAINT PK_Fine PRIMARY KEY (FineId),
    CONSTRAINT FK_Fine_Checkout FOREIGN KEY (CheckoutId) REFERENCES Checkout (CheckoutId),
    CONSTRAINT CK_Fine_Status CHECK (Status IN ('PAID', 'UNPAID', 'WAIVED'))
);
```

Now let us check 2NF. The only composite key here is `BookAuthor.{BookId, AuthorId}`. Are there any non-prime attributes in `BookAuthor`? No — it has exactly its two key columns. A junction table with only key columns is automatically in 2NF.

All other tables have single-column primary keys. They are therefore automatically in 2NF as well (since there are no composite keys to partially depend on).

This schema is in 2NF.

### 7.3 Queries on the Normalized Schema

Let us verify that common queries are still easy to write with the normalized schema.

**Find all books checked out by a specific member:**

```sql
SELECT b.Title, b.ISBN, c.CheckoutDate, c.DueDate, c.ReturnDate
FROM Checkout c
INNER JOIN Book b ON b.BookId = c.BookId
WHERE c.MemberId = 42
ORDER BY c.CheckoutDate DESC;
```

**Find all authors of a specific book:**

```sql
SELECT a.Name
FROM BookAuthor ba
INNER JOIN Author a ON a.AuthorId = ba.AuthorId
WHERE ba.BookId = 101
ORDER BY a.Name;
```

**Find books that are currently overdue:**

```sql
SELECT 
    m.Name AS MemberName,
    m.Email AS MemberEmail,
    b.Title AS BookTitle,
    c.DueDate,
    DATEDIFF(DAY, c.DueDate, CAST(GETUTCDATE() AS DATE)) AS DaysOverdue
FROM Checkout c
INNER JOIN Member m ON m.MemberId = c.MemberId
INNER JOIN Book b ON b.BookId = c.BookId
WHERE c.ReturnDate IS NULL
  AND c.DueDate < CAST(GETUTCDATE() AS DATE)
ORDER BY c.DueDate;
```

**Find unpaid fines with member and book information:**

```sql
SELECT 
    m.Name AS MemberName,
    m.Email AS MemberEmail,
    b.Title AS BookTitle,
    f.AmountDue,
    f.Status
FROM Fine f
INNER JOIN Checkout c ON c.CheckoutId = f.CheckoutId
INNER JOIN Member m ON m.MemberId = c.MemberId
INNER JOIN Book b ON b.BookId = c.BookId
WHERE f.Status = 'UNPAID'
ORDER BY f.AmountDue DESC;
```

These queries are more complex than `SELECT * FROM LibraryRecord` — but they are precise, efficient, and reliable. Each query retrieves exactly what it needs without redundant information.

```csharp
using Microsoft.EntityFrameworkCore;
using Library.Data.Models;

namespace Library.Data;

public class LibraryDbContext : DbContext
{
    public LibraryDbContext(DbContextOptions<LibraryDbContext> options) : base(options) { }

    public DbSet<Member> Members => Set<Member>();
    public DbSet<Publisher> Publishers => Set<Publisher>();
    public DbSet<Genre> Genres => Set<Genre>();
    public DbSet<Book> Books => Set<Book>();
    public DbSet<Author> Authors => Set<Author>();
    public DbSet<BookAuthor> BookAuthors => Set<BookAuthor>();
    public DbSet<Checkout> Checkouts => Set<Checkout>();
    public DbSet<Fine> Fines => Set<Fine>();

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        // Composite key for junction table
        modelBuilder.Entity<BookAuthor>()
            .HasKey(ba => new { ba.BookId, ba.AuthorId });

        // Unique constraints
        modelBuilder.Entity<Member>()
            .HasIndex(m => m.Email)
            .IsUnique();

        modelBuilder.Entity<Book>()
            .HasIndex(b => b.ISBN)
            .IsUnique();

        modelBuilder.Entity<Genre>()
            .HasIndex(g => g.Name)
            .IsUnique();

        // Fine status check constraint
        modelBuilder.Entity<Fine>()
            .HasCheckConstraint("CK_Fine_Status", "Status IN ('PAID', 'UNPAID', 'WAIVED')");

        // Decimal precision for Fine.AmountDue
        modelBuilder.Entity<Fine>()
            .Property(f => f.AmountDue)
            .HasPrecision(8, 2);

        // Many-to-many: Book <-> Author
        modelBuilder.Entity<BookAuthor>()
            .HasOne(ba => ba.Book)
            .WithMany(b => b.BookAuthors)
            .HasForeignKey(ba => ba.BookId);

        modelBuilder.Entity<BookAuthor>()
            .HasOne(ba => ba.Author)
            .WithMany(a => a.BookAuthors)
            .HasForeignKey(ba => ba.AuthorId);

        // One-to-one: Checkout -> Fine
        modelBuilder.Entity<Fine>()
            .HasOne(f => f.Checkout)
            .WithOne(c => c.Fine)
            .HasForeignKey<Fine>(f => f.CheckoutId);
    }
}
```

```csharp
        CancellationToken ct = default)
    {
        var today = DateOnly.FromDateTime(DateTime.UtcNow);

        return await _db.Checkouts
            .Include(c => c.Member)
            .Include(c => c.Book)
            .Where(c => c.ReturnDate == null && c.DueDate < today)
            .OrderBy(c => c.DueDate)
            .ToListAsync(ct);
    }

    public async Task<IReadOnlyList<BookAuthor>> GetBookAuthorsAsync(
        int bookId,
        CancellationToken ct = default)
    {
        return await _db.BookAuthors
            .Include(ba => ba.Author)
            .Where(ba => ba.BookId == bookId)
            .ToListAsync(ct);
    }

    public async Task<Checkout> CheckOutBookAsync(
        int memberId,
        int bookId,
        CancellationToken ct = default)
    {
        // Verify the book exists and is not currently checked out
        var isCheckedOut = await _db.Checkouts
            .AnyAsync(c => c.BookId == bookId && c.ReturnDate == null, ct);

        if (isCheckedOut)
            throw new InvalidOperationException($"Book {bookId} is already checked out.");

        var today = DateOnly.FromDateTime(DateTime.UtcNow);
        var checkout = new Checkout
        {
            MemberId = memberId,
            BookId = bookId,
            CheckoutDate = today,
            DueDate = today.AddDays(14)
        };

        _db.Checkouts.Add(checkout);
        await _db.SaveChangesAsync(ct);
        return checkout;
    }
}
```

## Part 8: Common Mistakes and Misconceptions

### 8.1 "Surrogate Keys Fix Everything"

A surrogate key (an auto-increment integer like `IDENTITY(1,1)` in SQL Server or `SERIAL` in PostgreSQL, or a GUID) is an artificial primary key that has no business meaning. It is added purely for database management purposes.

Some developers believe that using surrogate keys everywhere automatically keeps you in 2NF. This is almost true but not quite.

It is true that if every table has a single-column surrogate primary key, there are no composite candidate keys. And without composite candidate keys, there can be no partial dependencies (since partial dependencies require a composite key to partially depend on).

But consider this scenario:

```sql
CREATE TABLE Employee (
    EmployeeId   INT          NOT NULL IDENTITY(1,1),
    SSN          CHAR(9)      NOT NULL,  -- Social Security Number
    Name         VARCHAR(200) NOT NULL,
    DepartmentId INT          NOT NULL,
    DeptName     VARCHAR(200) NOT NULL,  -- stored directly on employee row
    CONSTRAINT PK_Employee PRIMARY KEY (EmployeeId),
    CONSTRAINT UQ_Employee_SSN UNIQUE (SSN)
);
```

The primary key is `EmployeeId` — single column. But `SSN` is also a candidate key (it is unique and identifies an employee). Does `DeptName` partially depend on the primary key? No, because the primary key is a single column. But wait — `DeptName` is determined by `DepartmentId`, which is a non-prime attribute. This is a **transitive dependency**, which is a 3NF violation (not 2NF), but it demonstrates that surrogate keys do not save you from all normalization problems.

More to the point: surrogate keys do not save you from redundancy. If `DeptName` is stored on every employee row, and you have 500 employees in the Sales department, you have 500 copies of "Sales." That is redundant, and it causes update anomalies — changing the department name requires updating 500 rows.

Surrogate keys are excellent. Use them. But they are not a substitute for normalization analysis.

### 8.2 "We Denormalized for Performance"

"We denormalized for performance" is the most common excuse for a poorly normalized database. It is sometimes valid. It is usually not.

The valid case: you have a read-heavy reporting workload where you execute a complex multi-table join millions of times per second, and profiling has proven this join is the bottleneck, and a denormalized summary table solves the problem measurably. In this case, denormalization (or a read replica, or a materialized view, or a separate OLAP database) can be appropriate.

The invalid case: a developer who does not want to write join queries, or who is afraid of the schema complexity, or who is pre-optimizing without measurements.

Here is the thing about joins in a properly indexed database: they are very fast. The database was designed for this. Its entire query execution engine is optimized for joins. A join between two tables where the foreign key and primary key are both indexed (which they should be, and EF Core creates these indexes automatically) is not slow.

What is slow:
- Table scans on large tables with no indexes
- Queries that return far more data than needed
- Missing indexes on join columns
- N+1 queries (querying in a loop instead of using a single set-based query)

What is not intrinsically slow:
- Joins between well-indexed tables

The diagnosis before denormalization must be:
1. Run the application under load
2. Identify the specific slow query
3. Examine the query execution plan
4. Add or fix indexes
5. Rewrite the query if it has logical problems
6. *Only if all of the above are insufficient*, consider denormalization

And even then, denormalization should usually be done in a **separate read model** — a materialized view, a reporting table, a read replica — not by corrupting your primary normalized schema. You maintain the normalized schema as the source of truth and project denormalized views for read performance.

### 8.3 "My ORM Takes Care of This"

No. Your ORM absolutely does not take care of this.

Entity Framework Core, Dapper, NHibernate — these are data access tools. They help you query and persist data. They do not know anything about the correct design of your schema. If you give EF Core a badly designed entity model, EF Core will faithfully create a badly designed database. It has no opinion about normalization.

In fact, EF Core can make normalization violations easier to create, because its conventions encourage putting properties on entities in a flat way. Consider:

```csharp
// EF Core makes this easy to write — and it maps to a 2NF violation
public class Order
{
    public int OrderId { get; set; }
    public int ProductId { get; set; }
    
    // These are product attributes, not order attributes
    // But EF will happily put them on the Orders table
    public string ProductName { get; set; } = "";
    public decimal ProductBasePrice { get; set; }
    
    public int Quantity { get; set; }
    public decimal UnitPrice { get; set; }  // this one is correct
}
```

EF does not warn you that `ProductName` and `ProductBasePrice` should be in a `Product` table. It just creates `Orders.ProductName` and `Orders.ProductBasePrice` columns. You have created a 2NF violation, and EF helped you do it quickly.

The responsibility for normalization is yours, not your ORM's.

### 8.4 "This Table Has a Single-Column PK, So It Must Be in 2NF"

This is true in the narrow sense that 2NF is automatically satisfied when there is a single-column primary key with no other composite candidate keys. But this thinking can lead to complacency.

Consider:

```sql
CREATE TABLE Employee (
    EmployeeId     INT          NOT NULL IDENTITY(1,1),
    Name           VARCHAR(200) NOT NULL,
    DepartmentId   INT          NOT NULL,
    DepartmentName VARCHAR(200) NOT NULL,  -- REDUNDANT
    CONSTRAINT PK_Employee PRIMARY KEY (EmployeeId)
);
```

This table is technically in 2NF. `DepartmentName` fully depends on `EmployeeId` (transitively, through `DepartmentId`). But it still has a problem: `DepartmentName` is duplicated for every employee in the same department. This is a 3NF violation (transitive dependency), not a 2NF violation. But calling it "in 2NF" and stopping there misses the real problem.

The lesson: 2NF is necessary but not sufficient. It is a waypoint on the road to 3NF.

### 8.5 The JSON Column Trap

Modern SQL databases support storing JSON in a column. SQL Server has JSON functions (`JSON_VALUE`, `JSON_QUERY`). PostgreSQL has the `jsonb` type. This is a legitimate feature for genuinely semi-structured data. But it is frequently misused to evade normalization.

```sql
-- This looks modern and flexible
-- It is actually a 1NF violation in JSON clothing
CREATE TABLE Order (
    OrderId   INT      NOT NULL PRIMARY KEY,
    OrderData NVARCHAR(MAX) NOT NULL  -- JSON: {"customer": {...}, "items": [...]}
);
```

The problems:
- You cannot query individual fields efficiently without either a computed column and index, or JSON_VALUE in a WHERE clause (which defeats indexes)
- You cannot join on fields inside the JSON without computed columns
- The database cannot enforce constraints on JSON fields
- You cannot reference JSON fields with foreign keys
- Schema evolution requires parsing and rewriting every row

JSON columns are appropriate for:
- Truly semi-structured data where the schema is unknown or varies per row
- Audit logs where you want to preserve a historical snapshot of an object
- Configuration data that is only ever read/written as a whole unit

JSON columns are NOT appropriate for:
- Relational data that you will need to query, join, or index
- Any data that benefits from foreign key enforcement
- Data whose schema is known and consistent

When you see a JSON column used to store structured relational data, treat it as a red flag — likely a 1NF (or worse) violation in disguise.

## Part 9: 2NF in Real ASP.NET Applications — Practical Patterns

### 9.1 Structuring EF Core Migrations

When you start a new ASP.NET project with EF Core, the migrations system creates your database schema based on your entity model. Following 2NF from the beginning means your entities should mirror the normalized schema.

Here is a concrete example: starting a multi-tenant SaaS application. A common mistake is to design like this:

```csharp
// BAD: Tenant information duplicated on every entity
public class Invoice
{
    public int InvoiceId { get; set; }
    public int TenantId { get; set; }
    public string TenantName { get; set; } = "";      // 2NF violation
    public string TenantEmail { get; set; } = "";     // 2NF violation
    public string TenantPlan { get; set; } = "";      // 2NF violation
    // ... invoice-specific fields
}
```

The correct approach:

```csharp
// GOOD: Tenant is a separate entity
public class Tenant
{
    public int TenantId { get; set; }
    public string Name { get; set; } = "";
    public string ContactEmail { get; set; } = "";
    public string Plan { get; set; } = "free";
    
    public ICollection<Invoice> Invoices { get; set; } = [];
}

public class Invoice
{
    public int InvoiceId { get; set; }
    public int TenantId { get; set; }
    public decimal Amount { get; set; }
    public DateOnly InvoiceDate { get; set; }
    public string Status { get; set; } = "pending";
    
    public Tenant Tenant { get; set; } = null!;
}
```

And the EF Core migration:

```bash
dotnet ef migrations add InitialCreate --project src/MyApp.Data --startup-project src/MyApp.Web
dotnet ef database update --project src/MyApp.Data --startup-project src/MyApp.Web
```

### 9.2 Using Projections to Get Data Without N+1

One concern developers have with normalized schemas is that they require more joins, and joins seem complex. With EF Core, you can use **projections** to retrieve exactly the data you need in a single query:

```csharp
// DTO for displaying invoice in a list
public record InvoiceListItem(
    int InvoiceId,
    string TenantName,
    decimal Amount,
    DateOnly InvoiceDate,
    string Status
);

// Service method using projection — single SQL query, no N+1
public async Task<IReadOnlyList<InvoiceListItem>> GetRecentInvoicesAsync(
    int tenantId,
    int count = 20,
    CancellationToken ct = default)
{
    return await _db.Invoices
        .Where(i => i.TenantId == tenantId)
        .OrderByDescending(i => i.InvoiceDate)
        .Take(count)
        .Select(i => new InvoiceListItem(
            i.InvoiceId,
            i.Tenant.Name,   // EF will join to Tenant automatically
            i.Amount,
            i.InvoiceDate,
            i.Status
        ))
        .ToListAsync(ct);
}
```

EF Core translates this `.Select()` projection into a single SQL query with a JOIN. You are not loading entire entity objects — you are loading only the columns you need. This is both correct (normalized) and efficient.

The generated SQL will look approximately like:

```sql
SELECT TOP(20) 
    i.InvoiceId, 
    t.Name AS TenantName, 
    i.Amount, 
    i.InvoiceDate, 
    i.Status
FROM Invoices i
INNER JOIN Tenants t ON t.TenantId = i.TenantId
WHERE i.TenantId = @tenantId
ORDER BY i.InvoiceDate DESC;
```

One round trip. One join. Exactly the columns needed.

### 9.3 The Repository Pattern with Normalized Schemas

When working with normalized schemas in ASP.NET, the Repository pattern helps organize your data access code:

```csharp
public interface IOrderRepository
{
    Task<Order?> GetByIdAsync(int orderId, CancellationToken ct = default);
    Task<IReadOnlyList<OrderSummary>> GetByCustomerAsync(int customerId, CancellationToken ct = default);
    Task<Order> CreateAsync(CreateOrderRequest request, CancellationToken ct = default);
}

public sealed class OrderRepository : IOrderRepository
{
    private readonly AppDbContext _db;

    public OrderRepository(AppDbContext db) => _db = db;

    public async Task<Order?> GetByIdAsync(int orderId, CancellationToken ct = default)
    {
        return await _db.Orders
            .Include(o => o.Customer)
            .Include(o => o.OrderLines)
                .ThenInclude(ol => ol.Product)
                    .ThenInclude(p => p.Category)
            .FirstOrDefaultAsync(o => o.OrderId == orderId, ct);
    }

    public async Task<IReadOnlyList<OrderSummary>> GetByCustomerAsync(
        int customerId,
        CancellationToken ct = default)
    {
        return await _db.Orders
            .Where(o => o.CustomerId == customerId)
            .Select(o => new OrderSummary
            {
                OrderId = o.OrderId,
                OrderDate = o.OrderDate,
                TotalItems = o.OrderLines.Sum(ol => ol.Quantity),
                TotalAmount = o.OrderLines.Sum(ol => ol.Quantity * ol.UnitPrice)
            })
            .OrderByDescending(o => o.OrderDate)
            .ToListAsync(ct);
    }

    public async Task<Order> CreateAsync(
        CreateOrderRequest request,
        CancellationToken ct = default)
    {
        var order = new Order
        {
            CustomerId = request.CustomerId,
            OrderDate = DateTime.UtcNow,
            OrderLines = request.Lines.Select(l => new OrderLine
            {
                ProductId = l.ProductId,
                Quantity = l.Quantity,
                UnitPrice = l.UnitPrice
            }).ToList()
        };

        _db.Orders.Add(order);
        await _db.SaveChangesAsync(ct);
        return order;
    }
}
```

Notice: the `CreateAsync` method accepts a `CreateOrderRequest` and creates both the `Order` and its `OrderLine` children in a single `SaveChanges` call (EF handles the insert order and foreign key values). This is clean, testable, and works correctly precisely because the schema is normalized — we do not have to duplicate product information onto the order.

## Part 10: Beyond 2NF — A Map of the Territory

### 10.1 Third Normal Form (3NF) — The Next Step

We have spent most of this article on 2NF, but it is important to know where you are going after 2NF. Third Normal Form (3NF) eliminates **transitive dependencies**.

A transitive dependency occurs when a non-prime attribute determines another non-prime attribute. In other words: A → B and B → C, therefore A → C transitively. If B and C are both non-prime, this is a 3NF violation.

**Example**:

```sql
-- In 2NF (no partial dependencies) but NOT in 3NF (transitive dependency)
CREATE TABLE Employee (
    EmployeeId     INT          NOT NULL,
    Name           VARCHAR(200) NOT NULL,
    DepartmentId   INT          NOT NULL,
    DepartmentName VARCHAR(200) NOT NULL,  -- transitive dependency: DeptId → DeptName
    CONSTRAINT PK_Employee PRIMARY KEY (EmployeeId)
);
```

Here: `EmployeeId → DepartmentId` and `DepartmentId → DepartmentName`. Therefore `EmployeeId → DepartmentName` transitively. But `DepartmentName` is really a property of `DepartmentId`, not of `EmployeeId` directly.

Fix: create a `Department` table:

```sql
CREATE TABLE Department (
    DepartmentId   INT          NOT NULL,
    DepartmentName VARCHAR(200) NOT NULL,
    CONSTRAINT PK_Department PRIMARY KEY (DepartmentId)
);

CREATE TABLE Employee (
    EmployeeId   INT          NOT NULL,
    Name         VARCHAR(200) NOT NULL,
    DepartmentId INT          NOT NULL,
    CONSTRAINT PK_Employee PRIMARY KEY (EmployeeId),
    CONSTRAINT FK_Employee_Department FOREIGN KEY (DepartmentId) REFERENCES Department (DepartmentId)
);
```

For most practical applications, designing to 3NF is the appropriate goal. You will eliminate the vast majority of redundancy and anomalies while keeping the schema comprehensible.

### 10.2 Boyce-Codd Normal Form (BCNF)

BCNF (defined by Codd and Boyce in 1974) is a slightly stronger version of 3NF. A relation is in BCNF if, for every non-trivial functional dependency A → B, A is a superkey.

BCNF only differs from 3NF in tables that have multiple overlapping candidate keys. These situations are relatively uncommon in practice, but they do occur. If you are in 3NF and you have no overlapping candidate keys, you are automatically in BCNF.

### 10.3 Fourth and Fifth Normal Forms

Fourth Normal Form (4NF, defined by Ronald Fagin in 1977) deals with **multi-valued dependencies** — situations where one attribute independently determines multiple values of two other attributes in the same table.

Fifth Normal Form (5NF, also defined by Fagin in 1979) deals with **join dependencies** — situations where a table can only be reconstructed by joining three or more tables.

These higher normal forms address increasingly subtle redundancies. In practice, most database designers target 3NF or BCNF. Violations of 4NF and 5NF are less common and less severe.

### 10.4 When Not to Normalize

Normalization is the right default for transactional (OLTP) databases. But there are contexts where it is not appropriate:

**Data warehouses and analytical databases (OLAP)**: These systems are optimized for reading aggregated data, not for transactional updates. They often use dimensional modeling (star schema or snowflake schema), which deliberately denormalizes to avoid expensive joins on very large datasets. Tools like Power BI, Tableau, and SSAS are designed to work with this kind of schema.

**Event stores**: An event store records an immutable log of events. Each event is a fact about what happened at a point in time. Events are never updated; they are only appended. Normalization is less relevant because the data is append-only and historical.

**Read models in CQRS**: In Command-Query Responsibility Segregation (CQRS), you separate the "write side" (commands) from the "read side" (queries). The write side should be normalized. The read side can be denormalized for read performance, because it is a derived projection, not the source of truth.

**Reporting tables and materialized views**: Pre-aggregated or pre-joined data for reporting is often intentionally denormalized. The key is that this data is derived from the normalized source and is updated via ETL processes or triggers.

In all of these cases, the normalized database remains the source of truth. Denormalization is a deliberate, controlled transformation for specific read purposes — not an excuse for a bad primary schema.

## Part 11: The Bad Code Gallery — Real-World Violations and Their Consequences

### 11.1 The Premature Optimization Disaster

Here is a C# class that a real developer wrote to "avoid extra queries":

```csharp
// BAD: "I'll just cache the product name to avoid the join"
public class CartItem
{
    public int CartItemId { get; set; }
    public int UserId { get; set; }
    public int ProductId { get; set; }
    public string ProductName { get; set; } = "";    // 2NF violation
    public string ProductImageUrl { get; set; } = ""; // 2NF violation
    public decimal ProductPrice { get; set; }         // 2NF violation — is this current price?
    public int Quantity { get; set; }
}
```

The developer thought: "Every time I display the cart, I need the product name and image. If I store them on the cart item, I can avoid the join to the Products table."

The consequences:
1. Product is renamed → all cart items with that product show the old name
2. Product image is updated → all cart items show the old image
3. Is `ProductPrice` the current price or the price when added to cart? The column name does not say. If it is current price and the product price changes, all cart items are stale. If it is the price at time of adding, it should be named `PriceWhenAdded` — and even then, `ProductName` and `ProductImageUrl` are still 2NF violations.

Correct approach:

```csharp
// GOOD: Normalized. Join to Product when you need product info.
public class CartItem
{
    public int CartItemId { get; set; }
    public int UserId { get; set; }
    public int ProductId { get; set; }
    public int Quantity { get; set; }
    public decimal PriceWhenAdded { get; set; }  // snapshot of price at add time
    
    public Product Product { get; set; } = null!;  // navigation property for joins
}
```

And the query to display the cart uses a projection:

```csharp
var cartItems = await _db.CartItems
    .Where(ci => ci.UserId == userId)
    .Select(ci => new CartItemDisplay
    {
        CartItemId = ci.CartItemId,
        ProductId = ci.ProductId,
        ProductName = ci.Product.Name,        // JOIN happens in SQL
        ProductImageUrl = ci.Product.ImageUrl, // same JOIN
        PriceWhenAdded = ci.PriceWhenAdded,
        CurrentPrice = ci.Product.Price,       // if you need to show both
        Quantity = ci.Quantity,
        Subtotal = ci.Quantity * ci.PriceWhenAdded
    })
    .ToListAsync();
```

This is one SQL query with a join. Fast. Correct. Normalized.

### 11.2 The Status Code Table That Ate Its Lookup Data

```sql
-- BAD: Lookup table data embedded in the main table
CREATE TABLE Task (
    TaskId         INT          NOT NULL PRIMARY KEY,
    Title          VARCHAR(500) NOT NULL,
    AssigneeId     INT          NOT NULL,
    AssigneeName   VARCHAR(200) NOT NULL,   -- 2NF-adjacent violation
    AssigneeEmail  VARCHAR(255) NOT NULL,   -- redundant
    PriorityCode   INT          NOT NULL,   -- 1 = Low, 2 = Medium, 3 = High
    PriorityLabel  VARCHAR(50)  NOT NULL,   -- "Low", "Medium", "High" — redundant
    StatusCode     INT          NOT NULL,
    StatusLabel    VARCHAR(50)  NOT NULL,   -- redundant
    StatusColor    VARCHAR(20)  NOT NULL    -- "#808080", "#0000FF" etc. — redundant
);
```

This is a mess. `PriorityLabel` is determined by `PriorityCode`. `StatusLabel` and `StatusColor` are determined by `StatusCode`. `AssigneeName` and `AssigneeEmail` are determined by `AssigneeId`.

Fix:

```sql
CREATE TABLE Priority (
    PriorityCode  INT         NOT NULL,
    Label         VARCHAR(50) NOT NULL,
    CONSTRAINT PK_Priority PRIMARY KEY (PriorityCode)
);

CREATE TABLE TaskStatus (
    StatusCode INT         NOT NULL,
    Label      VARCHAR(50) NOT NULL,
    Color      VARCHAR(20) NOT NULL,
    CONSTRAINT PK_TaskStatus PRIMARY KEY (StatusCode)
);

CREATE TABLE AppUser (
    UserId INT          NOT NULL,
    Name   VARCHAR(200) NOT NULL,
    Email  VARCHAR(255) NOT NULL,
    CONSTRAINT PK_AppUser PRIMARY KEY (UserId)
);

CREATE TABLE Task (
    TaskId       INT          NOT NULL,
    Title        VARCHAR(500) NOT NULL,
    AssigneeId   INT          NOT NULL,
    PriorityCode INT          NOT NULL,
    StatusCode   INT          NOT NULL,
    CONSTRAINT PK_Task PRIMARY KEY (TaskId),
    CONSTRAINT FK_Task_Assignee  FOREIGN KEY (AssigneeId)   REFERENCES AppUser (UserId),
    CONSTRAINT FK_Task_Priority  FOREIGN KEY (PriorityCode) REFERENCES Priority (PriorityCode),
    CONSTRAINT FK_Task_Status    FOREIGN KEY (StatusCode)   REFERENCES TaskStatus (StatusCode)
);
```

Seed the lookup tables once:

```sql
INSERT INTO Priority (PriorityCode, Label) VALUES (1, 'Low'), (2, 'Medium'), (3, 'High');
INSERT INTO TaskStatus (StatusCode, Label, Color) 
VALUES (1, 'Pending', '#808080'), (2, 'In Progress', '#0000FF'), (3, 'Done', '#00AA00');
```

Now priority labels and status colors are defined in one place, can be changed without touching task records, and the database enforces that only valid priority and status codes can be used.

```sql
    ModifiedBy       NVARCHAR(100),
    ModifiedAt       DATETIME,
    IsDeleted        BIT DEFAULT 0,   -- soft delete
    DeletedBy        NVARCHAR(100),
    DeletedAt        DATETIME,
    ExtraField1      NVARCHAR(MAX),   -- what does this contain? nobody knows.
    ExtraField2      NVARCHAR(MAX),   -- nobody knows
    LegacyId         NVARCHAR(50)     -- imported from the old system
);
```

One hundred million rows. The application has been running since 2018. Three developers have left the company without documentation. There is no `Customer` table. There is no `Product` table. The "database" is this one table plus thirty stored procedures that nobody understands.

This is a real situation. Not an exaggeration. It happens in small companies, medium companies, and large companies. The question is not "how did this happen?" (it happened because of short-term pressure, lack of knowledge, and accumulated technical debt). The question is: what do you do about it?

The answer is: carefully, incrementally, with tests at every step. You do not rewrite everything at once. You:

1. Document the actual behavior of the existing system
2. Write integration tests that verify that behavior
3. Create properly normalized tables alongside the old table
4. Migrate data in stages
5. Switch application code to use the new tables for new features
6. Eventually retire the old table once all code has migrated

This is a multi-month or multi-year project. It is worth doing. But it requires patience, discipline, and the support of management.

## Part 12: SQL Skills for the Normalized World

### 12.1 The Essential Joins

Working with a normalized schema means writing joins. If you are uncomfortable with joins, fix that now. Joins are not advanced SQL — they are the fundamental operation of a relational database.

**INNER JOIN**: Returns rows that have matching values in both tables.

```sql
-- Orders with their customer names
SELECT o.OrderId, c.Name AS CustomerName, o.OrderDate
FROM [Order] o
INNER JOIN Customer c ON c.CustomerId = o.CustomerId;
```

If an order has no corresponding customer (which should not happen with a foreign key constraint, but might in a dirty database), that order will not appear in the results.

**LEFT JOIN**: Returns all rows from the left table, and matching rows from the right table. Non-matching right-side rows are NULL.

```sql
-- All customers, with their order counts (including customers who have never ordered)
SELECT c.CustomerId, c.Name, COUNT(o.OrderId) AS OrderCount
FROM Customer c
LEFT JOIN [Order] o ON o.CustomerId = c.CustomerId
GROUP BY c.CustomerId, c.Name;
```

**Multi-table joins**:

```sql
-- Order lines with full detail: customer name, product name, category name
SELECT 
    c.Name AS CustomerName,
    o.OrderDate,
    p.Name AS ProductName,
    pc.Name AS CategoryName,
    ol.Quantity,
    ol.UnitPrice,
    ol.Quantity * ol.UnitPrice AS LineTotal
FROM OrderLine ol
INNER JOIN [Order] o   ON o.OrderId   = ol.OrderId
INNER JOIN Customer c  ON c.CustomerId = o.CustomerId
INNER JOIN Product p   ON p.ProductId  = ol.ProductId
INNER JOIN ProductCategory pc ON pc.CategoryId = p.CategoryId
ORDER BY o.OrderDate DESC, c.Name;
```

This query joins five tables. In a properly indexed database, it will execute efficiently.

### 12.2 Aggregations with Joins

Combining joins and aggregations is where SQL becomes genuinely powerful:

```sql
-- Total revenue by product category, last 30 days
SELECT 
    pc.Name AS CategoryName,
    COUNT(DISTINCT ol.OrderId) AS OrderCount,
    SUM(ol.Quantity) AS UnitsSold,
    SUM(ol.Quantity * ol.UnitPrice) AS Revenue
FROM OrderLine ol
INNER JOIN [Order] o ON o.OrderId = ol.OrderId
INNER JOIN Product p ON p.ProductId = ol.ProductId
INNER JOIN ProductCategory pc ON pc.CategoryId = p.CategoryId
WHERE o.OrderDate >= DATEADD(DAY, -30, GETUTCDATE())
GROUP BY pc.CategoryId, pc.Name
ORDER BY Revenue DESC;
```

This is impossible to write against the God Table without parsing comma-separated fields or doing multiple queries. Against a normalized schema, it is a straightforward GROUP BY.

### 12.3 Window Functions for Advanced Analytics

SQL window functions perform calculations across rows related to the current row, without collapsing the result set like GROUP BY does:

```sql
-- Running total of revenue per customer, chronologically
SELECT 
    c.Name AS CustomerName,
    o.OrderDate,
    SUM(ol.Quantity * ol.UnitPrice) AS OrderTotal,
    SUM(SUM(ol.Quantity * ol.UnitPrice)) 
        OVER (PARTITION BY c.CustomerId ORDER BY o.OrderDate) AS RunningTotal
FROM [Order] o
INNER JOIN Customer c ON c.CustomerId = o.CustomerId
INNER JOIN OrderLine ol ON ol.OrderId = o.OrderId
GROUP BY c.CustomerId, c.Name, o.OrderId, o.OrderDate
ORDER BY c.Name, o.OrderDate;
```

Window functions — `SUM() OVER`, `ROW_NUMBER() OVER`, `LAG()`, `LEAD()`, `RANK()` — are enormously useful for analytical queries. They only work sensibly on a normalized schema where each fact is stored once.

### 12.4 The Explain Plan — Understanding What the Database Is Actually Doing

When a query is slow, the first step is to look at the execution plan. In SQL Server:

```sql
SET STATISTICS IO ON;
SET STATISTICS TIME ON;

-- Your query here
SELECT ...

SET STATISTICS IO OFF;
SET STATISTICS TIME OFF;
```

Or use `EXPLAIN` (PostgreSQL) / `EXPLAIN ANALYZE` / `SET SHOWPLAN_ALL ON` (SQL Server) to see what operations the query optimizer chooses.

Signs of trouble in an execution plan:
- **Table Scan** (full scan of a table with no index): Usually bad for large tables
- **High estimated row count errors**: The optimizer has bad statistics
- **Nested Loops on large tables**: Can be catastrophically slow
- **Missing index warnings**: SQL Server will suggest indexes

The normalized schema makes index design straightforward: you index foreign key columns (EF Core does this automatically) and add indexes on columns you frequently filter or sort by.

```csharp
using Microsoft.EntityFrameworkCore;
using BookStore.Data;
using BookStore.Models;

var builder = WebApplication.CreateBuilder(args);

    options.UseSqlite("Data Source=bookstore.db"));

        .Select(b => new

            b.ISBN,
            b.Price,
            b.Stock,
            Category = new { b.Category.CategoryId, b.Category.Name },
            Publisher = new { b.Publisher.PublisherId, b.Publisher.Name, b.Publisher.City },
            Authors = b.BookAuthors.Select(ba => new { ba.Author.AuthorId, ba.Author.Name, ba.Author.Bio }).ToList()
        })
        .FirstOrDefaultAsync();

    return book is null ? Results.NotFound() : Results.Ok(book);
});

// POST /orders — create an order
app.MapPost("/orders", async (CreateOrderRequest req, BookStoreDbContext db) =>
{
    // Validate customer exists
    var customer = await db.Customers.FindAsync(req.CustomerId);
    if (customer is null)
        return Results.BadRequest($"Customer {req.CustomerId} not found.");

    // Validate all books exist and have sufficient stock
    foreach (var line in req.Lines)
    {
        var book = await db.Books.FindAsync(line.BookId);
        if (book is null)
            return Results.BadRequest($"Book {line.BookId} not found.");
        if (book.Stock < line.Quantity)
            return Results.BadRequest($"Insufficient stock for book {line.BookId}.");
    }

    // Create the order
    var order = new Order
    {
        CustomerId = req.CustomerId,
        OrderDate = DateTime.UtcNow,
        OrderLines = req.Lines.Select(l => new OrderLine
        {
            BookId = l.BookId,
            Quantity = l.Quantity,
            UnitPrice = db.Books.Find(l.BookId)!.Price  // price snapshot
        }).ToList()
    };

    // Deduct stock
    foreach (var line in req.Lines)
    {
        var book = await db.Books.FindAsync(line.BookId);
        book!.Stock -= line.Quantity;
    }

    db.Orders.Add(order);
    await db.SaveChangesAsync();

    return Results.Created($"/orders/{order.OrderId}", new { order.OrderId });
});

// GET /reports/sales-by-category — revenue breakdown
app.MapGet("/reports/sales-by-category", async (BookStoreDbContext db) =>
{
    var report = await db.OrderLines
        .GroupBy(ol => ol.Book.Category.Name)
        .Select(g => new
        {
            Category = g.Key,
            TotalUnitsSold = g.Sum(ol => ol.Quantity),
            TotalRevenue = g.Sum(ol => ol.Quantity * ol.UnitPrice)
        })
        .OrderByDescending(r => r.TotalRevenue)
        .ToListAsync();

    return Results.Ok(report);
});

app.Run();
```

## Part 14: Summary and Principles to Carry Forward

### 14.1 The Rules, Stated Plainly

Here is everything we have covered, distilled to the rules you should internalize:

**Rule 1: Every table needs a primary key.** No exceptions. If you cannot identify what makes a row unique, you do not yet understand what the table represents.

**Rule 2: Every column must contain atomic values.** No comma-separated lists. No JSON-encoded relational data. No repeating column groups (Phone1, Phone2, Phone3).

**Rule 3: Every non-prime attribute must fully depend on the entire primary key.** If your table has a composite key and some column only depends on part of that key, extract it to a new table.

**Rule 4: Learn to identify functional dependencies.** For every attribute B in a table, ask: "What is the minimum information I need to know to determine B?" That minimum information is what B depends on. If it is less than the full primary key, you have a partial dependency.

**Rule 5: Redundancy is not a performance feature.** Storing the same information in multiple places does not make queries faster in any meaningful way. It makes your data inconsistent.

**Rule 6: Joins are not expensive by default.** A properly indexed join between two tables is fast. If it is not, the problem is an index, a query, or a missing index — not the fact that you normalized the schema.

**Rule 7: Normalize by default, denormalize deliberately.** Start with a normalized schema. If profiling reveals that a specific query is slow and normalization is the cause, add a materialized view, a read model, or a summary table. Do not corrupt your primary schema.

**Rule 8: Your ORM does not save you from bad design.** EF Core, Dapper, NHibernate — they implement whatever design you give them. The design is your responsibility.

**Rule 9: Surrogate keys help, but they do not replace normalization analysis.** Using identity columns as primary keys prevents some partial dependency scenarios, but you must still identify and eliminate transitive dependencies (3NF), multi-valued dependencies (4NF), and conceptual redundancy.

**Rule 10: Insertion, update, and deletion anomalies are the symptoms; partial and transitive dependencies are the cause.** When you encounter data inconsistency, work backward to the dependencies that caused it.

### 14.2 The Habit of Normalization

Normalization is not something you do once and forget. It is a habit of thought that shapes how you model data from the beginning.

When you design a new table, ask: "What is the one thing this table is about?" A table should represent exactly one type of entity or relationship. If you find yourself saying "this table is about orders and customers and products," that is a sign you have not decomposed it properly.

When you see a column in a table, ask: "Does this attribute belong to this entity, or does it really belong to a different entity?" Customer information belongs in a Customer table. Product information belongs in a Product table. They do not belong in an Order table.

When you see data repeated across multiple rows, ask: "Why is this repeated? What dependency structure is causing the repetition?" The answer will lead you to a normalization violation.

When you see a NULL column that "only applies to certain rows," ask: "Is this a sign that the table is trying to represent two different types of entities at once?" Often it is, and the right fix is to split the table.

### 14.3 For Marcus — A Specific Action Plan

Let us return to Marcus from section 2.1. He has inherited a God Table with 847,000 rows, no separate Customer or Product tables, and inconsistent data everywhere. What should he actually do?

**Phase 1: Understand and document (2 weeks)**
- Write down every column and what it actually contains (not what the name implies)
- Document which columns are always populated, which are sometimes NULL, which are never populated
- Write queries to find all inconsistencies: `SELECT CustomerId, COUNT(DISTINCT CustomerEmail) AS EmailCount FROM Orders GROUP BY CustomerId HAVING COUNT(DISTINCT CustomerEmail) > 1`
- Document the results of these inconsistency checks

**Phase 2: Create tests for existing behavior (1 week)**
- Write integration tests that verify the current behavior of all application features that use this table
- These tests are your safety net for the migration

**Phase 3: Create normalized tables alongside the existing table (2 weeks)**
- Add `Customer`, `Product`, `ProductCategory` tables
- Do NOT delete the old table yet
- Migrate data: `INSERT INTO Customer (CustomerId, Name, Email) SELECT DISTINCT CustomerId, MAX(CustomerName), MAX(CustomerEmail) FROM Orders GROUP BY CustomerId`
- This "MAX" approach picks one value for each customer — document the inconsistencies you are resolving this way

**Phase 4: Update application code incrementally (4-8 weeks)**
- For each new feature, use the normalized tables
- For each existing feature you touch, migrate it to use the normalized tables
- Run your integration tests after every change

**Phase 5: Deprecate the old table (ongoing)**
- Once all code uses the normalized tables, the old table becomes a read-only archive
- Eventually rename it to `Legacy_Orders` and keep it for reference
- Eventually delete it when you are confident nothing depends on it

This is not fast. It is not glamorous. But it is the correct path.
